Stream Mode Overview
7 种 stream_mode，粒度从粗到细
checkpoints → values → updates → tasks → messages → custom
模式何时推送推送内容适用场景
values每个超级步结束完整 State状态面板、节点高亮
updates每个超级步结束节点返回的增量带宽敏感、增量更新
messagesLLM 生成每个 token(AIMessageChunk, metadata)打字机效果、实时对话

tasks任务 start/finishTaskPayload / TaskResultPayload执行监控、性能分析

LangGraph 渐进式教程 LG-07 / 11
LangGraph 1.x 提供 7 种 stream_mode，按粒度从粗到细排列。checkpoints 最粗，推送 Checkpoint 快照；values 推送完整 State；updates 只推送节点返回值；tasks 自动追踪节点生命周期；messages 是 LLM 每生成一个 token 就推送；custom 最灵活，通过 writer() 主动推送任意数据；debug 是 checkpoints + tasks 的组合，数据量最大，只建议在开发阶段使用。选型原则：人读的东西流式，机器读的东西非流式。

State Observation
values 看全局，updates 看变化

stream_mode="values" — 完整状态快照

{ "type": "values", "ns": (), # 命名空间 "data": { # 完整的 State "messages": [...], "outline": ["引言", "背景"], "current_node": "outline_planner" } }

每次推送都是完整 State，直接替换即可。State 大时带宽消耗高。

stream_mode="updates" — 增量更新

{ "type": "updates", "ns": (), "data": { "outline_planner": { "outline": ["引言", "背景"] } } }

只推送节点返回值，数据量小。前端需要 merge 到现有 State。

vs 对比

values 适合 State 小、需要完整视图的场景；updates 适合 State 大、带宽敏感的场景（如移动端）。并行节点执行时，updates 会分开发送每个节点的结果。

LangGraph 渐进式教程 LG-07 / 11
values 每次超级步结束后推送完整的 State。优点是前端处理简单——直接替换整个 State。缺点是如果 State 很大，带宽消耗高。updates 只推送每个节点返回的增量数据，数据结构是 {node_name: node_output}。优点是数据量小，适合带宽敏感场景。缺点是前端需要把增量 merge 到现有 State。并行节点同时执行时，updates 会分开发送每个节点的结果。
Token Streaming
messages 模式让 LLM 的每个 token 实时可见
async for chunk in graph.astream(input, stream_mode="messages"): msg, metadata = chunk["data"] print(f"来自节点: {metadata['langgraph_node']}") print(f"Token: {msg.content}")

数据结构

{ "type": "messages", "ns": (), "data": ( AIMessageChunk(content="我"), { "langgraph_step": 3, "langgraph_node": "writer", "langgraph_triggers": ["researcher"] } ) }

前端拼接（关键！）

full_text = "" async for chunk in graph.astream(...): msg, metadata = chunk["data"] if hasattr(msg, "content"): full_text += msg.content # ← 追加，不是替换！ update_ui(full_text)
LangGraph 渐进式教程 LG-07 / 11
messages 模式是实现打字机效果的核心。数据结构是 (AIMessageChunk, metadata) 元组。metadata 包含 langgraph_step、langgraph_node、langgraph_triggers 等信息——告诉你这个 token 来自哪个节点、第几步、被谁触发。前端拼接有一个最容易踩的坑：token 是增量，必须用 += 追加，不能赋值替换。messages 模式只捕获通过 LangChain ChatModel 调用产生的 token。如果节点内直接 return 字符串，不走 LLM，不会触发 messages 事件。
Custom Events & Lifecycle
custom 看进度，tasks 看生命周期

stream_mode="custom" — 自定义事件

async def research_node(state, writer): sources = ["财经数据", "新闻", "竞品"] for i, source in enumerate(sources): writer({ "type": "progress", "source": source, "percent": (i + 1) / len(sources) * 100, "status": "searching" }) await search(source)

writer 是自动注入的——只要在 Node 函数签名中声明 writer 参数，LangGraph 就会自动传入。

stream_mode="tasks" — 任务生命周期

# Task Start {"type": "tasks", "data": { "id": "outline_planner-abc", "name": "outline_planner", "input": {...}, "triggers": ["__start__"] }} # Task Finish {"type": "tasks", "data": { "id": "outline_planner-abc", "name": "outline_planner", "error": None, "result": {"outline": [...]} }}

自动追踪 start/finish，无需在 Node 内手动 yield。失败时 error 字段包含完整错误信息。

LangGraph 渐进式教程 LG-07 / 11
custom 的核心是 writer 函数。只要在 Node 函数签名中声明 writer 参数，LangGraph 就会自动注入。你在 Node 内部调用 writer({...}) 推送任意自定义数据。tasks 是自动的——不需要你在 Node 里写任何代码，LangGraph 自动发送每个节点的 start 和 finish 事件。通过 start 和 finish 的时间差，可以计算每个节点的执行时长。custom 是手动的、业务级的、可以多次推送的，适合前端进度条；tasks 是自动的、系统级的、节点级粒度的，适合性能监控和错误追踪。
Combined Modes & Nested Agents
组合多种 mode，同时实现打字机 + 进度条 + 监控
# 同时获取 messages + custom + tasks async for mode, chunk in graph.astream( input, stream_mode=["messages", "custom", "tasks"], subgraphs=True # ← 多 Agent 嵌套时必须开启 ): if mode == "messages": msg, metadata = chunk["data"] append_token(msg.content) # 打字机效果 elif mode == "custom": update_progress(chunk["data"]["percent"]) # 进度条 elif mode == "tasks": highlight_node(chunk["data"]["name"]) # 节点高亮

ns 命名空间含义

  • () = 顶层图
  • ("subgraph",) = 名为 subgraph 的子图
  • ("supervisor", "weather_expert") = supervisor 图中的 weather_expert 子图

前端处理策略

  • 顶层图事件 → 主面板显示
  • 子图事件 → 可折叠详情面板
  • 多层嵌套 → 按层级分组展示
LangGraph 渐进式教程 LG-07 / 11
实际项目中很少只用一种 stream_mode。最常见的组合是 messages + custom + tasks。组合 mode 时，每次迭代返回 (mode, chunk) 元组——mode 是字符串，告诉你这是什么类型的事件；chunk 是事件数据。多 Agent 嵌套时必须设置 subgraphs=True，子图的事件才会流出来。每个 StreamPart 都包含 ns 字段，用来标识事件来源的图层级。前端处理策略：顶层图的事件直接展示在主面板；子图的事件默认折叠在详情面板里，点击才展开。
Frontend Protocol & Business Decision
三段式流式协议：Start → Processing → End

start

占位，创建消息容器，插入进度条

processing

内容增量，追加到容器，更新打字机

end

收尾，移除光标，启用复制，触发后续

场景是否流式理由
LLM 生成自然语言必须流式消除等待焦虑、提升 perceived speed
思考过程/推理链建议流式让用户看到"AI 在思考"，建立信任
工具调用参数非流式结构化数据，一次性展示更高效
工具执行结果非流式结果完整性要求高，片段无意义
错误信息非流式错误需要立即、完整呈现
LangGraph 渐进式教程 LG-07 / 11
对于必须流式的消息，推荐三段式协议：start 阶段告知前端"这个流开始了"，创建消息容器；processing 阶段推送实际的内容增量；end 阶段告知前端"这个流结束了"，可以做后续操作。没有 end 的问题很严重：前端不知道流什么时候结束，光标永远闪烁，无法触发后续操作，网络抖动时无法区分"还在生成"还是"已断连"。核心原则：人读的东西流式，机器读的东西非流式。LLM 生成的自然语言流式，工具调用和错误信息非流式。

Summary
核心要点回顾

5 种核心 Stream Mode

  • values = 完整状态快照——适合状态面板
  • updates = 增量更新——适合带宽敏感场景
  • messages = token 级流式——适合打字机效果
  • custom = 自定义事件——适合进度条
  • tasks = 任务生命周期——适合监控

关键技巧与注意事项

  • 组合 mode 时返回 (mode, chunk) 元组，需要解包
  • 多 Agent 嵌套必须设置 subgraphs=True
  • 通过 ns 命名空间区分事件来源
  • messages 的 token 用 += 追加，不是替换
  • 三段式协议：start → processing → end

"values 看全局，updates 看变化，messages 看逐字，custom 看进度，tasks 看生命周期，subgraphs=True 看嵌套。"
