## LG-07：流式输出与实时监控

**目标学员**：已掌握 StateGraph 和预构建 Agent，想构建实时交互式 AI 应用的开发者。

**学完你能做什么**：
- 用 5 种 stream_mode 实现不同粒度的实时反馈
- 为 Node 添加 `StreamWriter` 推送自定义进度事件
- 组合多种 stream_mode 构建复杂的前端实时 UI
- 处理多 Agent 嵌套时的流式事件层级

---

## 1. 开场引入

**开场白**：
> "大家用过 ChatGPT 吧？它的打字机效果——一个字一个字往外蹦——不只是为了好看，更重要的是消除等待焦虑。你知道它在工作，你看到了进度。
>
> 但我们前面写的图都是『黑盒执行』：调用 `invoke()`，等几秒或几十秒，然后蹦出一个最终结果。用户在这几十秒里完全不知道发生了什么。
>
> 今天的主题是：**把黑盒变成透明盒**——让每一个中间状态、每一条生成的 token、每一次节点执行，都实时推送到用户眼前。"

**生活化类比：外卖订单追踪**

> "点外卖时，你最焦虑的不是等 30 分钟，而是『外卖到哪了』的未知。所以美团给你显示了完整的流程：商家已接单→骑手已到店→骑手已取餐→骑手距您 500 米。
>
> 流式输出就是在给我们的 LangGraph 应用做『外卖追踪』。"

---

## 2. Stream Mode 全景概览

LangGraph 1.x 提供 7 种 `stream_mode`：

```python
# langgraph/types.py:120-134
StreamMode = Literal[
"values", # 完整 State 快照
"updates", # 增量更新
"messages", # Token 级流式
"custom", # 自定义事件
"tasks", # 任务生命周期
"checkpoints", # Checkpoint 事件
"debug", # 调试事件
]
```

**一张图看懂选型**：

```
粒度从粗到细：

checkpoints ──→ values ──→ updates ──→ tasks ──→ messages ──→ custom
│ │ │ │ │ │
│ │ │ │ │ └─ 你定义什么就推送什么
│ │ │ │ └─ LLM 每个 token
│ │ │ └─ 节点 start/finish
│ │ └─ 节点返回的数据
│ └─ 完整 State
└─ Checkpoint 快照

debug = checkpoints + tasks
```

| 模式 | 何时推送 | 推送内容 | 适用场景 |
|------|---------|---------|---------|
| `values` | 每个超级步结束 | 完整 State | 状态面板、节点高亮 |
| `updates` | 每个超级步结束 | 节点返回的增量 | 带宽敏感、增量更新 |
| `messages` | LLM 生成每个 token | `(AIMessageChunk, metadata)` | 打字机效果、实时对话 |
| `custom` | Node 内调用 `writer()` | 你自定义的任何数据 | 进度条、业务通知 |
| `tasks` | 任务 start/finish | `TaskPayload` / `TaskResultPayload` | 执行监控、性能分析 |
| `checkpoints` | Checkpoint 创建 | `CheckpointPayload` | 状态恢复调试 |
| `debug` | 以上两种组合 | `DebugPayload` | 开发调试 |

## 3. 每种 Stream Mode 深度讲解

### 3.1 `stream_mode="values"` —— 完整状态快照

**概念**：每次超级步（step）结束后，推送当前完整的 State。

**源码类型**：

```python
# langgraph/types.py:252-262
class ValuesStreamPart(TypedDict, Generic[OutputT]):
type: Literal["values"]
ns: tuple[str, ...] # namespace，标识事件来源的图层级
data: OutputT # 完整的 State
interrupts: tuple[Interrupt, ...] # 中断信息
```

**代码示例**：

```python
async for chunk in graph.astream(
{"messages": [HumanMessage("写一篇文章")]},
stream_mode="values",
):
print(f"类型: {chunk['type']}")
print(f"数据: {chunk['data']}") # 完整的 State
print(f"命名空间: {chunk['ns']}") # () 表示顶层图
```

**输出示例**：

```python
# Step 1: outline_planner 执行完
{
"type": "values",
"ns": (),
"data": {
"messages": [
HumanMessage("写一篇文章"),
AIMessage("我来为你规划文章大纲..."),
],
"outline": ["引言", "背景", "分析", "结论"],
"current_node": "outline_planner",
},
"interrupts": (),
}

# Step 2: researcher 执行完
{
"type": "values",
"ns": (),
"data": {
"messages": [...],
"outline": ["引言", "背景", "分析", "结论"],
"research_data": {"news": [...], "stats": [...]},
"current_node": "researcher",
},
"interrupts": (),
}
```

**适用场景**：
- 实时状态面板：显示当前执行到哪个节点
- 节点高亮：前端根据 State 变化高亮当前活跃的节点
- 完整的 State 可视化

**注意**：每次推送都是**完整 State**，不是增量。如果 State 很大（messages 很长），带宽消耗较大。

---

### 3.2 `stream_mode="updates"` —— 增量更新

**概念**：只推送每个节点返回的增量数据（即节点函数的返回值）。

**源码类型**：

```python
# langgraph/types.py:264-273
class UpdatesStreamPart(TypedDict):
type: Literal["updates"]
ns: tuple[str, ...]
data: dict[str, Any] # {node_name: node_output}
```

**代码示例**：

```python
async for chunk in graph.astream(
{"messages": [HumanMessage("写一篇文章")]},
stream_mode="updates",
):
print(chunk)
```

**输出示例**：

```python
# outline_planner 节点返回的数据
{
"type": "updates",
"ns": (),
"data": {
"outline_planner": {
"outline": ["引言", "背景", "分析", "结论"],
}
},
}

# researcher 节点返回的数据
{
"type": "updates",
"ns": (),
"data": {
"researcher": {
"research_data": {"news": [...], "stats": [...]},
}
},
}

# 并行节点同时执行时，分开发送
{
"type": "updates",
"ns": (),
"data": {
"search_news": {"news": [...]},
},
}
{
"type": "updates",
"ns": (),
"data": {
"search_stats": {"stats": [...]},
},
}
```

**vs. values**：

| | values | updates |
|--|--------|---------|
| 推送内容 | 完整 State | 仅节点返回值 |
| 数据量 | 大（包含所有字段） | 小（仅变化的部分） |
| 前端处理 | 直接替换整个 State | 需要 merge 到现有 State |
| 适用 | State 小、需要完整视图 | State 大、带宽敏感 |

**适用场景**：
- 带宽敏感的前端（移动端）
- 只需要增量更新的场景
- 并行节点执行时，分开发送每个节点的结果

---

### 3.3 `stream_mode="messages"` —— Token 级流式（重点）

**概念**：LLM 每生成一个 token，立即推送到前端。这是实现"打字机效果"的核心。

**源码类型**：

```python
# langgraph/types.py:276-286
class MessagesStreamPart(TypedDict):
type: Literal["messages"]
ns: tuple[str, ...]
data: tuple[AnyMessage, dict[str, Any]]
    # data[0] = BaseMessage（如 AIMessageChunk）
    # data[1] = metadata（包含 langgraph_step, langgraph_node 等）
```

**代码示例**：

```python
async for chunk in graph.astream(
{"messages": [HumanMessage("写一篇文章")]},
stream_mode="messages",
):
msg, metadata = chunk["data"]
print(f"来自节点: {metadata['langgraph_node']}")
print(f"Token: {msg.content}")
```

**输出示例**：

```python
# LLM 生成 "我"
{
"type": "messages",
"ns": (),
"data": (
AIMessageChunk(content="我"),
{
"langgraph_step": 3,
"langgraph_node": "writer",
"langgraph_triggers": ["researcher"],
},
),
}

# LLM 生成 "来"
{
"type": "messages",
"ns": (),
"data": (
AIMessageChunk(content="来"),
{
"langgraph_step": 3,
"langgraph_node": "writer",
"langgraph_triggers": ["researcher"],
},
),
}

# ... 逐字推送，直到生成完毕
```

**前端拼接代码**：

```python
# 前端（如 React）如何拼接 token
full_text = ""
async for chunk in graph.astream(input, stream_mode="messages"):
msg, metadata = chunk["data"]
if hasattr(msg, "content"):
full_text += msg.content # ← 追加，不是替换！
update_ui(full_text) # 实时更新显示
```

**关键点**：
1. `messages` 模式只捕获**通过 LangChain ChatModel 调用**产生的 token
2. 每个 chunk 都带有 `metadata`，告诉你这个 token 来自哪个节点、第几步
3. 如果节点内直接 `return` 字符串（不走 LLM），不会触发 `messages` 事件

**适用场景**：
- 打字机效果（ChatGPT 风格）
- 实时对话机器人
- 长文本生成时的用户等待体验

---

### 3.4 `stream_mode="custom"` —— 自定义事件

**概念**：在 Node 函数内部，通过 `StreamWriter` 主动推送任意自定义数据。

**源码类型**：

```python
# langgraph/types.py:136-139
StreamWriter = Callable[[Any], None]
# 接受任意参数，写入输出流

# langgraph/types.py:289-297
class CustomStreamPart(TypedDict):
type: Literal["custom"]
ns: tuple[str, ...]
data: Any # 你自定义的任何数据
```

**代码示例**：

```python
async def research_node(state, writer):
"""Node 函数接收 writer 参数，用于推送自定义事件。"""
sources = ["财经数据", "新闻", "竞品分析", "专家观点"]

for i, source in enumerate(sources):
        # 推送进度事件
writer({
"type": "progress",
"source": source,
"percent": (i + 1) / len(sources) * 100,
"status": "searching",
})

        # 执行实际的搜索...
await search(source)

        # 推送完成事件
writer({
"type": "progress",
"source": source,
"percent": (i + 1) / len(sources) * 100,
"status": "completed",
})

return {"research_data": results}
```

**消费端**：

```python
async for chunk in graph.astream(input, stream_mode="custom"):
data = chunk["data"]
if data["type"] == "progress":
update_progress_bar(
source=data["source"],
percent=data["percent"],
)
```

**输出示例**：

```python
{
"type": "custom",
"ns": (),
"data": {
"type": "progress",
"source": "财经数据",
"percent": 25.0,
"status": "searching",
},
}
{
"type": "custom",
"ns": (),
"data": {
"type": "progress",
"source": "财经数据",
"percent": 25.0,
"status": "completed",
},
}
```

**适用场景**：
- 进度条（百分比）
- 业务通知（"资料检索中..."）
- 中间结果预览（"已找到 3 条相关新闻"）
- 任何需要"推送通知"的场景

**注意**：
- `writer` 是自动注入的——只要在 Node 函数签名中声明 `writer` 参数，LangGraph 就会自动传入
- 只有在 `stream_mode="custom"` 或包含 `custom` 的组合中，`writer` 的调用才会生效

### 3.5 `stream_mode="tasks"` —— 任务生命周期

**概念**：自动追踪每个任务（节点）的生命周期：start → finish，包含执行时长和错误信息。

**源码类型**：

```python
# langgraph/types.py:142-167
class TaskPayload(TypedDict):
id: str # 任务唯一 ID
name: str # 节点名称
input: Any # 输入数据
triggers: list[str] # 触发该任务的条件

class TaskResultPayload(TypedDict):
id: str
name: str
error: str | None # 错误信息（成功为 None）
interrupts: list[dict] # 中断列表
result: dict[str, Any] # 节点返回值

# langgraph/types.py:308-320
class TasksStreamPart(TypedDict):
type: Literal["tasks"]
ns: tuple[str, ...]
data: TaskPayload | TaskResultPayload
```

**代码示例**：

```python
async for chunk in graph.astream(input, stream_mode="tasks"):
data = chunk["data"]

if "result" not in data:
        # Task Start 事件
print(f"▶️ 节点开始: {data['name']} (id={data['id']})")
else:
        # Task Finish 事件
if data["error"]:
print(f"❌ 节点失败: {data['name']}, 错误: {data['error']}")
else:
print(f"✅ 节点完成: {data['name']}")
```

**输出示例**：

```python
# Task Start
{
"type": "tasks",
"ns": (),
"data": {
"id": "outline_planner-abc123",
"name": "outline_planner",
"input": {"messages": [...]},
"triggers": ["__start__"],
},
}

# Task Finish
{
"type": "tasks",
"ns": (),
"data": {
"id": "outline_planner-abc123",
"name": "outline_planner",
"error": None,
"interrupts": [],
"result": {"outline": ["引言", "背景", "分析", "结论"]},
},
}
```

**独特价值**：

1. **自动生命周期追踪**：无需在 Node 内手动 yield，自动发送 start/finish
2. **内置错误捕获**：失败时 `error` 字段包含完整错误信息
3. **性能监控**：通过 start 和 finish 的时间差，计算每个节点的执行时长

**适用场景**：
- 系统性能监控（记录每个节点的执行时长）
- 错误追踪与报警
- 简单的进度追踪（已完成 X / 总共 Y 个节点）

**vs. custom**：

| | tasks | custom |
|--|-------|--------|
| 推送时机 | 自动（start/finish） | 手动（Node 内调用 writer） |
| 数据内容 | 系统级（节点名、ID、错误） | 业务级（进度百分比、中间结果） |
| 粒度 | 节点级 | 任意（可多次推送） |
| 适用 | 监控、调试 | 用户界面反馈 |

---

### 3.6 `stream_mode="debug"` —— 调试事件

**概念**：组合了 `checkpoints` + `tasks`，提供最详细的调试信息。

**源码类型**：

```python
# langgraph/types.py:323-328
class DebugStreamPart(TypedDict, Generic[StateT]):
type: Literal["debug"]
ns: tuple[str, ...]
data: DebugPayload[StateT]
    # DebugPayload = CheckpointPayload | TaskPayload | TaskResultPayload
```

**适用场景**：
- 开发阶段调试图的执行流程
- 排查 State 在哪个步骤发生了变化
- 配合 LangGraph Studio 的可视化调试

**注意**：生产环境不建议开启，数据量太大。

## 4. 多 Mode 组合与事件过滤

### 4.1 组合多种 Stream Mode

```python
# 同时获取 messages（打字机）和 custom（进度条）
async for mode, chunk in graph.astream(
input,
stream_mode=["messages", "custom"],
):
if mode == "messages":
msg, metadata = chunk["data"]
append_token(msg.content) # 打字机效果

elif mode == "custom":
data = chunk["data"]
if data["type"] == "progress":
update_progress(data["percent"]) # 进度条
```

**输出格式**：当使用组合 mode 时，每次迭代返回 `(mode_name, chunk)` 元组。

### 4.2 事件过滤：Namespace

所有 StreamPart 都包含 `ns`（namespace）字段：

```python
{
"type": "values",
"ns": ("supervisor", "weather_expert"), # ← 来自 supervisor 图下的 weather_expert 子图
"data": {...},
}
```

**ns 的含义**：
- `()`：顶层图（根图）
- `("subgraph",)`：名为 "subgraph" 的子图
- `("supervisor", "weather_expert")`：supervisor 图中的 weather_expert 子图

**过滤示例**：

```python
async for chunk in graph.astream(input, stream_mode="values", subgraphs=True):
if chunk["ns"] == ():
print("来自顶层图")
elif chunk["ns"] == ("writer",):
print("来自 writer 子图")
elif len(chunk["ns"]) > 1:
print(f"来自嵌套子图: {' -> '.join(chunk['ns'])}")
```

**注意**：要接收子图的事件，必须设置 `subgraphs=True`。

## 5. 多 Agent 嵌套时的 Stream 事件

### 5.1 问题场景

当你使用 `create_supervisor` 或 `create_swarm` 时，图结构是嵌套的：

```
顶层图 (supervisor)
├── supervisor 节点（LLM 调度）
├── weather_expert 子图
│ ├── agent 节点（LLM）
│ └── tools 节点（ToolNode）
├── order_expert 子图
│ ├── agent 节点（LLM）
│ └── tools 节点（ToolNode）
└── ...
```

**问题**：默认情况下，`stream()` 只返回顶层图的事件。子图内部发生了什么，你是看不到的。

### 5.2 解决方案：`subgraphs=True`

```python
# 开启 subgraphs=True，子图的事件也会流出来
async for chunk in app.astream(
input,
stream_mode=["messages", "tasks"],
subgraphs=True, # ← 关键！
):
mode, data = chunk # 当 stream_mode 是列表时
    # 或 chunk = data  # 当 stream_mode 是单个字符串时
```

**嵌套事件示例**：

```python
# 1. Supervisor 自己调用 LLM（顶层图）
{
"type": "messages",
"ns": (),
"data": (AIMessageChunk(content="我"), {...}),
}

# 2. Supervisor 决定调用 weather_expert
#    handoff 后，weather_expert 子图开始执行
{
"type": "tasks",
"ns": ("weather_expert",), # ← 来自子图！
"data": {
"id": "agent-xyz",
"name": "agent",
"input": {...},
"triggers": [...],
},
}

# 3. weather_expert 内部的 LLM 生成 token
{
"type": "messages",
"ns": ("weather_expert",), # ← 子图的 LLM token
"data": (AIMessageChunk(content="北"), {...}),
}

# 4. weather_expert 内部的 ToolNode 执行
{
"type": "tasks",
"ns": ("weather_expert",),
"data": {
"id": "tools-abc",
"name": "tools",
"error": None,
"result": {...},
},
}

# 5. weather_expert 完成，回到 Supervisor
{
"type": "tasks",
"ns": ("weather_expert",),
"data": {
"id": "agent-xyz",
"name": "agent",
"error": None,
"result": {...},
},
}

# 6. Supervisor 再次调用 LLM，给用户最终回复
{
"type": "messages",
"ns": (),
"data": (AIMessageChunk(content="北"), {...}),
}
```

### 5.3 前端如何处理嵌套事件

```python
# 前端根据 ns 区分事件来源
async for mode, chunk in app.astream(
input,
stream_mode=["messages", "tasks", "custom"],
subgraphs=True,
):
ns = chunk["ns"]
data = chunk["data"]

if not ns:
        # 顶层图事件
handle_root_event(mode, data)
elif len(ns) == 1:
        # 直接子图
subgraph_name = ns[0]
handle_subgraph_event(subgraph_name, mode, data)
else:
        # 多层嵌套
path = " -> ".join(ns)
handle_nested_event(path, mode, data)
```

**可视化建议**：
- 顶层图的事件：主面板显示
- 子图的事件：可折叠的详情面板，点击展开看子图内部执行

## 6. `astream_events`（v3，实验性）

LangGraph 1.x 引入了 `astream_events(version="v3")`，提供比传统 stream_mode 更灵活的事件流：

```python
# v3 API（实验性）
run = await app.astream_events(input, version="v3")

# 并发读取不同类型的投影
async for chunk in run.output:
    # 输出流（类似 values）
print(f"State: {chunk}")

async for event in run.interrupts:
    # 中断事件
print(f"中断: {event}")
```

**v3 的特点**：
1. 返回 `AsyncGraphRunStream`，支持并发投影
2. 自动包含所有子图事件（不需要 `subgraphs=True`）
3. 内置 `StreamMux` 自动分类事件

**注意**：v3 是实验性 API，可能变化。生产环境建议使用稳定的 `stream_mode`。

## 7. 业务视角：流式策略设计与前端协议

> 前面的技术讲解回答了「怎么实现流式」。但在真实业务中，更重要的是回答「什么时候该流式、什么时候不该流式、流出来的东西前端怎么渲染」。这一节从业务视角拆解流式策略的设计原则。

### 7.1 流式 vs 非流式：业务决策矩阵

不是所有输出都需要流式。错误的流式策略会带来两个问题：
1. **该流的不流** → 用户盯着白屏等 10 秒，体验差
2. **不该流的也流** → 前端频繁重绘、带宽浪费、代码复杂

**决策矩阵**：

| 场景 | 是否流式 | 理由 | 典型消息类型 |
|------|---------|------|------------|
| LLM 生成自然语言内容 | **必须流式** | 消除等待焦虑、提升 perceived speed | `content` |
| 思考过程/推理链 | **建议流式** | 让用户看到「AI 在思考」，建立信任 | `thinking` |
| 工具调用参数 | **非流式** | 结构化数据，一次性展示更高效 | `tool_call` |
| 工具执行结果 | **非流式** | 结果完整性要求高，片段无意义 | `tool_result` |
| 错误信息 | **非流式** | 错误需要立即、完整呈现，不能逐字蹦 | `error` |
| 中断/审批请求 | **非流式** | 需要用户立即做决策，必须完整展示 | `interrupt` |
| 计划/步骤列表 | **视情况** | 短计划非流式，长计划可流式 | `plan` |
| 进度通知 | **非流式** | 百分比/状态变更，一次性推送即可 | `processing` |

**核心原则**：
> **「人读的东西流式，机器读的东西非流式」**。LLM 生成的自然语言是给人看的，流式能提升体验；工具调用、错误码是给程序处理的，非流式更简洁。

### 7.2 消息类型与显示样式映射

前端需要根据消息类型决定渲染方式。以下是生产环境的推荐映射：

```
┌─────────────┬────────────────────────────────────────────┐
│ 消息类型 │ 前端渲染样式 │
├─────────────┼────────────────────────────────────────────┤
│ content │ Markdown 渲染 + 打字机效果 │
│ thinking │ 折叠面板/灰色小字（类似 ChatGPT 的 reasoning）│
│ tool_call │ 卡片式展示：工具名 + 参数 JSON（可展开） │
│ tool_result │ 结果卡片：成功/失败状态 + 返回值（可折叠） │
│ plan │ 步骤列表：序号 + 状态图标（pending/done） │
│ search │ 引用卡片：来源链接 + 摘要高亮 │
│ summary │ 结构化面板：标题 + 要点列表 │
│ processing │ 进度条/Toast 通知（3 秒后自动消失） │
│ error │ 红色 Alert 横幅 + 重试按钮 │
│ interrupt │ 模态框/底部操作栏：表单 + 确认/取消按钮 │
│ files │ 文件列表：图标 + 文件名 + 下载链接 │
└─────────────┴────────────────────────────────────────────┘
```

**关键设计**：每种消息类型对应唯一的渲染组件。前端不需要「猜测」怎么展示——`message_type` 字段直接决定渲染器。

### 7.3 三段式流式协议（Start → Processing → End）

对于必须流式的消息，推荐采用**三段式协议**：

```
第一次推送 中间 N 次推送 一次推送
┌─────────┐ ┌─────────────┐ ┌─────────┐
│ start │ →→→ │ processing │ →→→ │ end │
│ 空内容 │ │ 内容增量 │ │ 空内容 │
│ 状态:开始│ │ 状态:处理中 │ │ 状态:结束│
└─────────┘ └─────────────┘ └─────────┘
```

**为什么需要三段式？**

| 阶段 | 作用 | 前端行为 |
|------|------|---------|
| `start` | 占位。告知前端「这个流开始了」，创建消息容器 | 插入空消息气泡/进度条 |
| `processing` | 内容。实际的数据增量 | 追加到容器内，更新打字机效果 |
| `end` | 收尾。告知前端「这个流结束了」，可以做后续操作 | 移除光标、启用复制按钮、触发摘要 |

**没有 `end` 的问题**：
- 前端不知道流什么时候结束，光标永远闪烁
- 无法触发后续操作（如生成追问、保存历史）
- 网络抖动时前端无法区分「还在生成」vs「已断连」

### 7.4 消息信封设计（MessageEnvelope）

统一的消息结构，让前端处理逻辑极简：

```python
{
"message_type": "content", # 消息类型 → 决定渲染组件
"content": "实际文本内容", # 文本内容（流式时可能是片段）
"status": "processing", # start / processing / end
"message_id": "msg_abc123", # 唯一 ID，同一条流的消息 ID 相同
"node_name": "writer", # 产生这个消息的节点
"agent_name": "deepresearch", # 所属 Agent（多 Agent 时用于分组）
"agent_hierarchy": ["supervisor", "researcher"], # Agent 嵌套路径
"process_step": "3/5", # 步骤进度（如 "大纲规划"）
"timestamp": 1716643200.0, # 时间戳
"format": { # 渲染格式提示（可选）
"type": "markdown",
"code_language": "python"
},
"data": { # 业务数据（可选）
"delta": "增量文本",
"mode": "append" # append（追加）vs replace（替换）
},
"metadata": { # 元数据（可选）
"finish_reason": "stop", # LLM 结束标记
"source_type": "llm" # 来源：llm / tool / human
}
}
```

**前端解析逻辑（统一入口）**：

```javascript
function handleMessage(envelope) {
// 1. 根据 message_type 选择渲染器
const renderer = getRenderer(envelope.message_type);

// 2. 根据 status 决定操作
switch (envelope.status) {
case "start":
renderer.createPlaceholder(envelope.message_id);
break;
case "processing":
renderer.appendContent(envelope.message_id, envelope.content);
break;
case "end":
renderer.finalize(envelope.message_id);
break;
}

// 3. 根据 agent_hierarchy 决定展示位置
const container = getContainer(envelope.agent_hierarchy);
container.render(renderer);
}
```

### 7.5 多 Agent 嵌套时的显示策略

当 Supervisor → Researcher → Writer 多级嵌套时，前端面临一个问题：**消息从哪来、该往哪放？**

**方案：Agent 层级路径（agent_hierarchy）**

```python
# Supervisor 自己生成的内容
{
"message_type": "content",
"agent_hierarchy": [], # 空 = 顶层
"content": "我来为您深入研究这个问题..."
}

# Researcher 子图生成的内容
{
"message_type": "thinking",
"agent_hierarchy": ["researcher"],
"content": "正在分析市场数据..."
}

# Writer 子图（嵌套在 Researcher 内）生成的内容
{
"message_type": "content",
"agent_hierarchy": ["researcher", "writer"],
"content": "根据调研结果，报告显示..."
}
```

**前端展示策略**：

```
主对话面板（agent_hierarchy=[]）
├── 用户输入
├── Supervisor: "我来为您深入研究..."
└── Supervisor: "研究完成，结果如下..."

可折叠详情面板（agent_hierarchy=["researcher"]）
├── ▶ researcher 思考过程
│ ├── thinking: "正在分析市场数据..."
│ └── search: "找到 3 篇相关论文"
│
└── ▶ researcher > writer 报告生成
└── content: "根据调研结果..."
```

**设计原则**：
- 顶层消息直接展示在主对话区
- 子图消息默认折叠在「详情面板」，点击展开
- 用户可通过配置决定哪些子图的消息直接展示（如 writer 的 content 直接展示）

### 7.6 性能与体验权衡

**带宽优化**：

| 策略 | 做法 | 效果 |
|------|------|------|
| 批量推送 | 后端缓存 50ms 的 token，批量发送 | 减少 80% 网络包数量 |
| 压缩内容 | 空 content 的 end 消息只发 header | 节省 30% 带宽 |
| 选择性订阅 | 前端只订阅需要展示的消息类型 | 减少无效数据处理 |

**前端渲染优化**：

```javascript
// 反例：每次 token 都强制重排
function badAppend(text) {
element.innerHTML = text; // ❌ 强制重排
}

// 正例：使用 requestAnimationFrame 批量更新
let pendingText = "";
function goodAppend(chunk) {
pendingText += chunk;
}
function flush() {
element.textContent = pendingText; // ✅ 每帧只更新一次
requestAnimationFrame(flush);
}
```

**关键指标**：
- **端到端延迟**：用户输入到第一个 token 出现 < 500ms
- **渲染帧率**：token 推送频率应与屏幕刷新率匹配（60Hz ≈ 16ms/帧）
- **内存占用**：长对话中消息历史应虚拟滚动，只渲染可视区域

## 8. 案例驱动：LiveWriter

### 8.1 案例设计

实时写作助手：用户输入主题 → AI 并行检索资料 → 逐段生成文章 → 实时显示进度。

### 8.2 各 Stream Mode 的体现

| 概念 | LiveWriter 中的体现 |
|------|-------------------|
| values | 实时状态面板显示当前执行到哪个节点 |
| updates | 段落进度条只接收新增的内容 |
| messages | 打字机效果，逐字生成文章 |
| custom | 自定义进度事件：「大纲完成 100%」「资料检索中...」 |
| tasks | 并行任务的执行状态（pending→running→completed） |

### 8.3 代码演示

```python
async def live_writer(user_input: str):
"""LiveWriter 的流式处理。"""

async for mode, chunk in graph.astream(
{"messages": [HumanMessage(user_input)]},
stream_mode=["messages", "custom", "tasks"],
subgraphs=True,
):
if mode == "messages":
msg, metadata = chunk["data"]
yield {
"type": "token",
"content": msg.content,
"node": metadata.get("langgraph_node"),
}

elif mode == "custom":
data = chunk["data"]
if data.get("type") == "progress":
yield {
"type": "progress",
"source": data["source"],
"percent": data["percent"],
}

elif mode == "tasks":
data = chunk["data"]
if "result" not in data:
yield {
"type": "task_start",
"node": data["name"],
}
else:
yield {
"type": "task_end",
"node": data["name"],
"success": data["error"] is None,
}
```

### 8.4 前端消费示例

```javascript
// WebSocket 接收后端流式事件
ws.onmessage = (event) => {
const msg = JSON.parse(event.data);

switch (msg.type) {
case "token":
// 打字机效果：追加 token
document.getElementById("article").textContent += msg.content;
break;

case "progress":
// 更新进度条
updateProgressBar(msg.source, msg.percent);
break;

case "task_start":
// 高亮当前执行的节点
highlightNode(msg.node);
break;

case "task_end":
// 节点执行完毕，更新状态
markNodeComplete(msg.node, msg.success);
break;
}
};
```

## 9. 常见误区与注意事项

| 坑 | 现象 | 预警话术 |
|----|------|---------|
| **用 values 做打字机效果** | 每次推送完整 State，前端需要 diff 才能知道新增了什么 | "打字机效果必须用 messages mode，values 不适合 token 级流式。" |
| **忘记前端拼接 token** | 每个 token 覆盖了上一个，只显示最后一个字 | "messages mode 的 token 是增量，前端需要 `text += new_token`，不是 `text = new_token`。" |
| **custom writer 不在 Node 中** | 自定义事件没有推送 | "custom mode 的 writer 必须在 Node 函数内部调用，且图要用 astream。" |
| **子图事件没出来** | 只看到顶层图的事件，子图内部黑盒 | "嵌套多 Agent 时，必须设置 `subgraphs=True` 才能看到子图内部事件。" |
| **混淆 sync/async** | `stream` 和 `astream` 混用导致不流式 | "异步图必须用 `astream`，同步图用 `stream`。LangGraph 的流式依赖 async generator。" |
| **组合 mode 时没解包** | 只返回了一个值，不知道是哪种 mode | "组合 stream_mode 时，每次迭代返回 `(mode, chunk)` 元组，需要解包。" |
| **v3 API 不稳定** | 代码在新版本 LangGraph 中失效 | "`astream_events(version='v3')` 是实验性 API，生产环境建议用稳定的 `stream_mode`。" |

### 调试技巧

1. **打印每个 event**：先不用前端，直接在终端 `print(event)` 观察事件结构
2. **区分 event 类型**：`event["type"]` 字段标识事件类型（values/updates/messages/custom/tasks）
3. **检查 namespace**：`event["ns"]` 告诉你事件来自哪个图层级
4. **用 Gradio 快速验证**：Gradio 的 `chatbot` 组件原生支持流式消息

## 10. 阶段小结

### 核心要点回顾

1. **values = 完整状态快照**：每次超级步后推送完整 State，适合状态面板
2. **updates = 增量更新**：只推送节点返回值，适合带宽敏感场景
3. **messages = token 级流式**：LLM 每生成一个 token 就推送，适合打字机效果
4. **custom = 自定义事件**：Node 内通过 `writer()` 主动推送，适合进度条
5. **tasks = 任务生命周期**：自动追踪节点 start/finish，适合监控
6. **多 Agent 嵌套**：设置 `subgraphs=True` 接收子图事件，通过 `ns` 区分来源
7. **组合使用**：`stream_mode=["messages", "custom"]` 可同时实现打字机+进度条

### Stream Mode 选型决策树

```
你需要什么样的实时反馈？
│
├─ 需要看 LLM 逐字生成？
│ └─ messages ✅
│
├─ 需要显示进度百分比？
│ └─ custom ✅（Node 内 writer 推送）
│
├─ 需要知道节点什么时候开始/结束？
│ └─ tasks ✅
│
├─ 需要显示当前 State 全貌？
│ └─ values ✅
│
├─ 只需要增量数据，省带宽？
│ └─ updates ✅
│
└─ 多种需求都要？
└─ stream_mode=["messages", "custom", "tasks"] ✅
```

### 记忆口诀

> **"values 看全局，updates 看变化，messages 看逐字，custom 看进度，tasks 看生命周期，subgraphs=True 看嵌套。"**
