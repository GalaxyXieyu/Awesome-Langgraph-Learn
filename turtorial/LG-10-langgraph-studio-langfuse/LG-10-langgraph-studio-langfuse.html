// part 01 · langgraph studio

Studio 让你实时看到图的每一次执行

安装与启动

pip install langgraph-cli
# langgraph.json 配置图路径
langgraph dev

浏览器自动打开 localhost:2024

常见坑：langgraph.json 里的 Python 路径要写对，指向 graph = builder.compile() 那个变量

界面四区

  • ① 图结构视图 — Mermaid 动态渲染，节点逐个高亮
  • ② 运行配置区 — 输入 State、选择 Stream/Invoke 模式
  • ③ 状态面板 — 每个节点执行前后的完整 State（可展开）
  • ④ 时间线/历史 — 每次运行记录，可回放到任意节点
LG-10 · LangGraph Studio + Langfuse 3 / 10

LangGraph Studio 是什么？一句话说——它是图的"解剖台"。你写的 StateGraph 跑起来的时候，Studio 让你实时看到每一个节点怎么被调用、State 怎么变化。

安装很简单，pip install langgraph-cli，然后需要一个 langgraph.json 配置文件告诉 Studio 你的图在哪。启动命令是 langgraph dev，浏览器自动打开 localhost:2024。

界面分成四个区。左边是图结构视图，用 Mermaid 动态渲染，节点逐个变绿。右边上面是运行配置区，填输入 State、选 Stream 还是 Invoke 模式。右边中间是状态面板——这是最重要的区域。右边下面是时间线，每次运行的记录都在这里，可以点进去回放到任意节点。

// part 01 · core operations

Studio 的时间旅行调试让排查 bug 速度提升 10 倍

操作一：运行并观察节点高亮

填入输入 → 点击 Run → 看节点逐个变绿

START → parse_intent → extract_city → query_weather → format_weather_reply → END

操作二：查看节点 State（最重要）

点击任意已执行节点，看三种 State：

  • Input State — 执行前收到的完整 State
  • Output Update — 返回的增量 dict
  • Merged State — 框架合并后的新 State

操作三：修改 State 重新运行

时间线里点击节点 → 手动改 State → 点 "Replay from here"

不用改代码、不用重启，直接看新走向

操作四：打断点

节点右键 → "Set breakpoint"

运行到该节点自动暂停，查看 State、修改变量，然后 Continue 继续

LG-10 · LangGraph Studio + Langfuse 4 / 10

Studio 有四个核心操作。

操作一：运行并观察节点高亮。你在输入区填一个消息，点击 Run，图结构视图里的节点会逐个变绿。你能清楚看到执行路径。每个节点的耗时也会实时显示。

操作二：查看节点 State。这是 Studio 最有价值的功能。点击任意已执行的节点，状态面板会展开显示三种 State。你可以验证 parse_intent 的 output 里 intent 是不是 "weather"，extract_city 的 output 里 city 是不是 "北京"。

操作三：修改 State 重新运行。这是 Studio 最强大的能力——"时间旅行调试"。运行完一次后，你在时间线里点击 parse_intent 节点，把 intent 改成 "time"，然后点 "Replay from here"。图会重新从 parse_intent 开始跑，这次走向 get_time 分支。不用改代码、不用重启，排查 bug 时极快。

操作四：打断点。在节点上右键 Set breakpoint，下次运行到该节点会自动暂停，你可以查看 State、修改变量，然后 Continue 继续。这跟调试普通程序一模一样。

LG-10 · LangGraph Studio + Langfuse 5 / 10

好，进入第二部分——Langfuse。先回答一个常见问题：Studio 不够吗？为什么还要 Langfuse？

我直接给一张对比表。Studio 在实时可视化执行上很强，节点高亮、改 State 重跑、打断点——这些都是开发阶段的刚需。但 Studio 的历史记录只在本地内存，关了浏览器就没了；一次只能看一个请求，没法批量对比；不显示 Token 用量和成本——你不知道哪个节点在烧 API 费用。

Langfuse 正好补上了这些缺口。它的历史记录持久化存储在云端或本地数据库；支持批量筛选和对比多个请求；精确统计每次 LLM 调用的 Token 用量；还能按模型、按时间段做成本分析；支持多用户团队协作；是生产级的可观测性平台。

所以记住这句话：Studio 是"开发时的显微镜"，Langfuse 是"上线后的仪表盘"。两者互补，不是替代。

// part 02 · core concepts

Trace / Span / Event 三层结构完整记录每一次执行

Trace（一次用户请求） ├── Span: Node - parse_intent (23ms) │ └── Event: LLM Call (312ms, tokens: 120→45) ├── Span: Node - extract_city (15ms) ├── Span: Node - query_weather (1203ms) │ └── Event: HTTP Request (1198ms) └── Span: Node - format_weather_reply (45ms) └── Event: LLM Call (2300ms, tokens: 800→120)

Trace

一次完整的图执行，对应一个 graph.invoke() 调用

Span

一个节点的执行，包含开始时间、耗时、输入输出

Event

节点内部的具体操作：LLM 调用、HTTP 请求、数据库查询

LG-10 · LangGraph Studio + Langfuse 6 / 10

Langfuse 的核心概念有三层：Trace、Span、Event。理解这三层，你就理解了 Langfuse 怎么组织数据。

Trace 是一次完整的图执行，对应你代码里的一次 graph.invoke() 调用。用户发一条消息，就是一个 Trace。

Span 是一个节点的执行。比如 parse_intent 执行了 23 毫秒，这就是一个 Span。每个 Span 包含开始时间、耗时、输入输出。

Event 是节点内部的具体操作。比如 parse_intent 里调了一次 LLM，耗时 312 毫秒，输入 120 tokens、输出 45 tokens——这是一个 Event。再比如 query_weather 里发了一个 HTTP 请求，耗时 1198 毫秒——这也是一个 Event。

看左边这棵树，你能一眼定位问题：query_weather 节点显示 1203 毫秒，但点进去发现 HTTP Request 占了 1198 毫秒。所以问题不是图的问题，是天气 API 响应慢。优化方向是加缓存或换 API。

这就是 Langfuse 的价值——瀑布图帮你精确找到瓶颈在哪一层。

// part 02 · integration

接入 Langfuse 只需要一行代码

Step 1 & 2：装包 + 配环境变量

pip install langfuse

# 去 cloud.langfuse.com 注册，拿到 key
export LANGFUSE_PUBLIC_KEY="pk-xxx"
export LANGFUSE_SECRET_KEY="sk-xxx"

Step 3：一行代码接入

from langfuse.callback import CallbackHandler

langfuse_handler = CallbackHandler(
  session_id="weather-bot-demo",
  user_id="student-01",
)

result = graph.invoke(
  {"messages": [HumanMessage("...")]},
  config={"callbacks": [langfuse_handler]}
)

完了。不需要改图的定义，不需要改节点代码。LangGraph 会自动把每个节点、每次 LLM 调用都上报到 Langfuse。

LG-10 · LangGraph Studio + Langfuse 7 / 10

接入 Langfuse 比你想象的简单得多，5 分钟搞定。

第一步，pip install langfuse。第二步，去 cloud.langfuse.com 注册一个账号，创建项目，拿到三个 key：PUBLIC_KEY、SECRET_KEY、HOST。Cloud 用户 HOST 用默认值就行。把前两个配成环境变量。

第三步，也是最关键的一步——一行代码接入。从 langfuse.callback 导入 CallbackHandler，创建一个实例，可以选填 session_id 和 user_id 来打标签。然后在 graph.invoke() 的时候，通过 config 参数把 callbacks 传进去。

完了。不需要改图的定义，不需要改任何节点代码。LangGraph 的回调机制会自动把每个节点的执行、每次 LLM 调用、每次工具调用都上报到 Langfuse。

LG-10 · LangGraph Studio + Langfuse 8 / 10

接入之后，在 Langfuse 界面看两个页面。

第一个是 Trace 列表页。每次 invoke 一条记录，耗时、Token 数、成本一眼就能看到。你可以按标签、用户、时间段筛选。比如只看今天生产环境的请求，或者只看某个用户的请求。

第二个是 Trace 详情页，也就是瀑布图。这个页面是排查性能问题的利器。看右边这个示例：parse_intent 23 毫秒，extract_city 15 毫秒，query_weather 1203 毫秒——这里最慢。但点进去发现 HTTP Request 占了 1198 毫秒，所以问题不是图逻辑的问题，是外部 API 慢。

再看 format_weather_reply，总共 2345 毫秒，但 LLM Call 占了 2300 毫秒。这说明节点本身的逻辑很快，慢的是 LLM 推理。优化方向是换更快的模型或者精简 prompt。

下面三个小卡片是进阶功能：Tags 按环境打标签，避免开发数据混到生产；Score 给 Trace 打质量分数，做趋势监控；点击任意节点可以看完整的 prompt、输出、Token 用量、模型参数。

// part 03 · hands-on

WeatherBot 加上 Langfuse 后，每次执行都有迹可循

# === 接入 Langfuse（在原有代码最后加这几行）===
from langfuse.callback import CallbackHandler

langfuse_handler = CallbackHandler(
  tags=["env:development", "tutorial:lg-10"]
)

for query in ["北京今天天气怎么样？", "现在几点了？", "讲个笑话"]:
  result = graph.invoke(
    {"messages": [HumanMessage(query)]},
    config={"callbacks": [langfuse_handler]}
  )
  print(f"用户: {query}")
  print(f"Bot: {result['response']}\n")
