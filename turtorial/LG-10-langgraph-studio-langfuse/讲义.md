# LangGraph Studio 可视化调试与 Langfuse 可观测性入门

**目标学员**：刚学完 LG-01，写出了第一个 StateGraph，想知道"图到底怎么跑的"和"出问题了怎么看"的新手。

**定位**：工具课。不写新 Agent，而是把 LG-01 的 WeatherBot 用可视化工具"打开来看"，建立调试和监控的直觉。

**学完你能做什么**：
- 用 LangGraph Studio 启动可视化界面，看到图的实际执行过程
- 在 Studio 里打断点、改 State、重跑节点，像调试普通程序一样调试图
- 接入 Langfuse，在 Web 界面看到每次请求的完整执行链路（Trace）
- 通过瀑布图定位慢节点，通过 Token 用量监控成本

## 1. 开场引入

**开场白**：
> "上一节课你写出了 WeatherBot，代码能跑、结果也对。但有个问题——图跑起来的时候，你其实是'盲'的。你只知道输入和输出，不知道中间每个节点收到了什么 State、返回了什么、哪个节点最慢、哪个节点在烧 API 费用。
>
> 今天这节课，我们要把图从'黑盒'变成'玻璃盒'。"

**生活化类比**：
> "普通写代码像开一辆没有仪表盘的车——你知道油门踩了、方向打了，但看不到转速、油量、水温。Studio 是 HUD 抬头显示，Langfuse 是行车记录仪 + 油耗分析。"

## 2. LangGraph Studio —— 图的"解剖台"

### 2.1 安装与启动

```bash
pip install langgraph-cli
```

Studio 需要一个 `langgraph.json` 配置文件来描述你的图在哪：

```json
{
"dependencies": ["."],
"graphs": {
"weather_bot": "./weather_bot.py:graph"
}
}
```

启动：

```bash
langgraph dev
```

浏览器自动打开 `http://localhost:2024`，看到 Studio 界面。

**常见坑**：`langgraph.json` 里的 Python 路径要写对，指向你的 `graph = builder.compile()` 那个变量。

### 2.2 界面四区

```
┌─────────────────────────────────────────────┐
│ ① 图结构视图（Mermaid 动态渲染） │
│ 节点逐个高亮，显示当前走到哪一步 │
├─────────────────────────────────────────────┤
│ ② 运行配置区 │
│ 输入 State、选择 Stream/Invoke 模式 │
├─────────────────────────────────────────────┤
│ ③ 状态面板 │
│ 每个节点执行前后的完整 State（可展开查看） │
├─────────────────────────────────────────────┤
│ ④ 时间线/历史 │
│ 每次运行的记录，可点击回放到任意节点 │
└─────────────────────────────────────────────┘
```

### 2.3 核心操作演示

**操作一：运行图并观察节点高亮**

1. 在输入区填入 `{"messages": [{"type": "human", "content": "北京今天天气怎么样？"}]}`
2. 点击 Run
3. 看图结构视图：节点逐个变绿，从 START → parse_intent → extract_city → query_weather → format_weather_reply → END
4. 观察时间线：每个节点的耗时实时显示

**操作二：查看节点 State（最重要！）**

点击任意已执行的节点，状态面板展开显示：
- **Input State**：节点执行前收到的完整 State
- **Output Update**：节点返回的增量 dict
- **Merged State**：框架合并后的新 State

> 学员现场验证：点击 `extract_city` 节点，看 Input State 里的 `intent` 是不是 `"weather"`，Output Update 里的 `city` 是不是 `"北京"`。

**操作三：修改 State 重新运行（时间旅行调试）**

1. 运行完一次后，在时间线里点击 `parse_intent` 节点
2. 在状态面板里手动把 `intent` 改成 `"time"`
3. 点击 "Replay from here"
4. 观察图走向：`parse_intent` → `get_time` → END（而不是天气分支）

> **这就是 Studio 最强大的能力**：不用改代码、不用重启，直接在中间节点修改状态，看图会怎么走。排查 bug 时极快。

**操作四：打断点**

在节点上右键 → "Set breakpoint"，下次运行到该节点会自动暂停，你可以查看 State、修改变量，然后点击 Continue 继续执行。

## 3. Langfuse —— 全链路"行车记录仪"

### 3.1 为什么需要 Langfuse？Studio 不够吗？

| 能力 | Studio | Langfuse |
|------|--------|----------|
| 实时可视化执行 | ✅ 强 | ❌ 弱（事后看） |
| 历史记录持久化 | ❌ 只在本地内存 | ✅ 云端/本地数据库存储 |
| 多请求对比 | ❌ 一次只能看一个 | ✅ 批量筛选、对比 |
| Token 用量统计 | ❌ 不显示 | ✅ 精确到每个 LLM 调用 |
| 成本分析 | ❌ 不显示 | ✅ 按模型、按时间段统计 |
| 团队协作 | ❌ 单机 | ✅ 多用户共享项目 |
| 生产环境监控 | ❌ 开发工具 | ✅ 生产级可观测性 |

> **关系**：Studio 是"开发时的显微镜"，Langfuse 是"上线后的仪表盘"。两者互补，不是替代。

### 3.2 核心概念：Trace / Span / Event

```
Trace（一次用户请求）
└── Span: Node - parse_intent (23ms)
│ └── Event: LLM Call (312ms, 输入tokens: 120, 输出tokens: 45)
└── Span: Node - extract_city (15ms)
└── Span: Node - query_weather (1203ms)
│ └── Event: HTTP Request (1198ms)
└── Span: Node - format_weather_reply (45ms)
└── Event: LLM Call (2300ms, 输入tokens: 800, 输出tokens: 120)
```

- **Trace**：一次完整的图执行（对应一个 `graph.invoke()` 调用）
- **Span**：一个节点的执行
- **Event**：节点内部的具体操作（LLM 调用、HTTP 请求、数据库查询）

### 3.3 接入 Langfuse（5 分钟搞定）

**Step 1：注册账号**

去 [cloud.langfuse.com](https://cloud.langfuse.com) 注册，创建项目，拿到三个 key：
- `LANGFUSE_PUBLIC_KEY`
- `LANGFUSE_SECRET_KEY`
- `LANGFUSE_HOST`（Cloud 用户用默认值即可）

**Step 2：装包 + 配环境变量**

```bash
pip install langfuse
```

```bash
export LANGFUSE_PUBLIC_KEY="pk-xxx"
export LANGFUSE_SECRET_KEY="sk-xxx"
```

**Step 3：一行代码接入**

```python
from langfuse.callback import CallbackHandler

# 创建回调处理器
langfuse_handler = CallbackHandler(
session_id="weather-bot-demo", # 可选：给这批请求打标签
user_id="student-01", # 可选：标识用户
)

# 运行时传入 config
result = graph.invoke(
{"messages": [HumanMessage("北京今天天气怎么样？")]},
config={"callbacks": [langfuse_handler]}
)
```

**完了。** 不需要改图的定义，不需要改节点代码。LangGraph 会自动把每个节点、每次 LLM 调用都上报到 Langfuse。

### 3.4 在 Langfuse 界面看什么？

**Trace 列表页**：
- 每次 `invoke()` 一条记录
- 耗时、Token 数、成本一眼看到
- 支持按标签、用户、时间段筛选

**Trace 详情页（瀑布图）**：

```
01:23:45.123 parse_intent ████ (23ms)
01:23:45.147 extract_city ██ (15ms)
01:23:45.163 query_weather ████████████████ (1203ms) ← 这里最慢！
01:23:46.367 format_weather ████████ (2345ms)
└── LLM Call ████████████████ (2300ms) ← 实际慢的是 LLM
```

点击任意节点，展开看：
- 完整的输入 prompt
- 完整的输出结果
- Token 用量（输入/输出分别多少）
- 模型名称和参数

**现场验证**：
> 问学员："query_weather 节点显示 1203ms，点进去发现 HTTP Request 占了 1198ms。优化方向是什么？"
> 答：不是图的问题，是天气 API 响应慢。考虑加缓存或换 API。

### 3.5 Tags 和 Score（进阶但重要）

给不同环境打标签，避免开发数据混到生产：

```python
langfuse_handler = CallbackHandler(
tags=["env:development", "project:weather-bot"]
)
```

手动上报自定义指标：

```python
from langfuse import Langfuse

langfuse = Langfuse()

# 给某次 Trace 打分数（0-1）
langfuse.score(
trace_id=trace_id,
name="response_quality",
value=0.9,
comment="回答准确，格式清晰"
)
```

## 4. 实战：WeatherBot 的"体检报告"

### 目标

给 LG-01 的 WeatherBot 加上 Langfuse，在 Studio 里跑通，然后在 Langfuse 界面分析一次完整的执行。

### 完整代码

```python
from typing import TypedDict, Annotated
from operator import add
from datetime import datetime
from langchain_core.messages import HumanMessage
from langgraph.graph import StateGraph, START, END
from langfuse.callback import CallbackHandler

# === State ===
class WeatherState(TypedDict):
messages: Annotated[list, add]
intent: str
city: str
weather_data: dict
response: str

# === Nodes ===
def parse_intent(state: WeatherState):
text = state["messages"][-1].content
if "天气" in text or "温度" in text:
return {"intent": "weather"}
elif "时间" in text or "几点" in text:
return {"intent": "time"}
return {"intent": "other"}

def extract_city(state: WeatherState):
text = state["messages"][-1].content
city = "北京" if "北京" in text else "上海" if "上海" in text else ""
return {"city": city}

def query_weather(state: WeatherState):
return {"weather_data": {"temp": 25, "condition": "晴"}}

def format_weather_reply(state: WeatherState):
data = state["weather_data"]
reply = f"{state['city']}今天{data['condition']}，{data['temp']}度"
return {"response": reply}

def get_time(state: WeatherState):
return {"response": f"现在是 {datetime.now().strftime('%H:%M')}"}

def fallback_reply(state: WeatherState):
return {"response": "抱歉，我暂时只能回答天气和时间相关的问题。"}

# === Routers ===
def route_by_intent(state: WeatherState) -> str:
intent = state["intent"]
if intent == "weather":
return "extract_city"
elif intent == "time":
return "get_time"
return "fallback_reply"

def route_after_city(state: WeatherState) -> str:
if state.get("city"):
return "query_weather"
return "fallback_reply"

# === Build Graph ===
builder = StateGraph(WeatherState)
builder.add_node("parse_intent", parse_intent)
builder.add_node("extract_city", extract_city)
builder.add_node("query_weather", query_weather)
builder.add_node("format_weather_reply", format_weather_reply)
builder.add_node("get_time", get_time)
builder.add_node("fallback_reply", fallback_reply)

builder.add_edge(START, "parse_intent")
builder.add_edge("query_weather", "format_weather_reply")
builder.add_edge("format_weather_reply", END)
builder.add_edge("get_time", END)
builder.add_edge("fallback_reply", END)

builder.add_conditional_edges("parse_intent", route_by_intent)
builder.add_conditional_edges("extract_city", route_after_city)

graph = builder.compile()

# === 接入 Langfuse ===
langfuse_handler = CallbackHandler(
tags=["env:development", "tutorial:lg-09"]
)

# === Run with Langfuse ===
for query in ["北京今天天气怎么样？", "现在几点了？", "讲个笑话"]:
result = graph.invoke(
{"messages": [HumanMessage(query)]},
config={"callbacks": [langfuse_handler]}
)
print(f"用户: {query}")
print(f"Bot: {result['response']}\n")
```

## 5. Studio vs Langfuse 选型速查

| 场景 | 推荐工具 | 为什么 |
|------|---------|--------|
| 开发阶段调试图逻辑 | Studio | 实时高亮、改状态重跑、打断点 |
| 排查线上问题 | Langfuse | 历史记录、完整链路、持久化存储 |
| 分析 API 成本和 Token 用量 | Langfuse | 精确统计到每次 LLM 调用 |
| 团队协作 review Agent 行为 | Langfuse | 多用户共享、按标签筛选 |
| 教学演示（给学生看） | Studio | 界面直观、节点高亮、互动性强 |
| 长期监控 Agent 质量趋势 | Langfuse | Score 时序图、Dataset 回归测试 |

## 附录：常见问题

**Q：Studio 支持改代码后热重载吗？**
A：支持。保存 Python 文件后 Studio 自动重新加载图定义，不用手动重启。

**Q：Langfuse 数据存在哪里？安全吗？**
A：Cloud 版数据存在 Langfuse 官方服务器（SOC2 合规）。也可以自建（开源版），数据完全在自己的服务器上。

**Q：Studio 只能本地用吗？**
A：目前主要作为本地开发工具。LangGraph Cloud 提供云端部署能力，那属于生产部署范畴（LG-09 讲）。

**Q：CallbackHandler 会影响性能吗？**
A： overhead 极小（主要是异步上报网络开销），对单次请求延迟的影响通常在 10ms 以内。
