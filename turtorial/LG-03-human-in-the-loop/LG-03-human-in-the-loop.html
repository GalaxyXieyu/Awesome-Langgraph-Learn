THREE-LAYER ARCHITECTURE
HITL 三层架构：组件关系全景图
粒度递增，风险递增——选择正确的层级解决问题

Layer 1：编译时配置

  • interrupt_before=["node_a"]
  • interrupt_after=["node_c"]
  • 无需改节点代码

适用：固定审批点

Layer 2：预构建 Agent

  • create_react_agent(..., interrupt_before=["tools"])
  • 节点名固定："agent" / "tools"
  • 透传给 compile()

适用：ReAct Agent 工具审批

Layer 3：节点内动态中断

  • interrupt({"question": "确认吗？"})
  • 最灵活但风险最大
  • resume 时节点重执行

⚠ 必须配合拆分模式

层级控制粒度改节点代码动态条件重执行风险
Layer 1节点级否否无
Layer 2Agent 内节点级否否无
Layer 3节点内任意位置是是有
LangGraph 渐进式教程 LG-03 / 10
先讲整体架构，再看细节。三层不是独立的，而是有明确的调用关系和适用范围。Layer 1 和 2 是声明式的，Layer 3 是动态的。决策树：是否需要运行时数据决定中断？是 → Layer 3（但注意拆分）；否 → 用预构建 Agent？是 → Layer 2；否 → Layer 1。
COMPILE-TIME INTERRUPT
interrupt_before vs interrupt_after
声明式中断：一个执行前拦截，一个执行后审查

interrupt_before：执行前拦截

  • 节点代码完全不执行
  • resume 后节点第一次执行
  • state 没有节点输出
graph = builder.compile( checkpointer=InMemorySaver(), interrupt_before=["publish"], # 执行前暂停 )

适用：审批输入、高风险操作前置确认

interrupt_after：执行后审查

  • 节点代码完整执行
  • writes 已保存到 state
  • resume 后直接流转到下一节点
graph = builder.compile( checkpointer=InMemorySaver(), interrupt_after=["generate"], # 执行完暂停 )

适用：AI 生成后确认、工具执行后审查

易混淆点：interrupt_before=["publish"] + resume 后，publish 第一次执行。interrupt_after=["generate"] + resume 后，generate 不会再次执行。

LangGraph 渐进式教程 LG-03 / 10
这是最容易混淆的两个概念。interrupt_before 是执行前暂停，节点还没跑；interrupt_after 是执行完暂停，节点已经跑完了。resume 后的行为完全不同——before 是第一次执行，after 是继续往下走。
THE RE-EXECUTION PITFALL
interrupt() 的最大坑：重执行陷阱
节点恢复时从头重新执行，interrupt() 前的代码会重跑

错误写法（计算 + 中断混在一起）

def bad_node(state): result = expensive_compute(state) # ← resume 时重跑！ user_input = interrupt({"data": result}) return {"output": process(user_input)}

用户点通过，又卡 3 秒——生产环境不可接受

正确写法（拆分节点）

def compute_node(state): result = expensive_compute(state) return {"analysis_result": result} # 写入 state def confirm_node(state): result = state["analysis_result"] # 从 state 读取 decision = interrupt({"content": result}) return {"confirmed": decision == "approve"}

resume 时秒过，无重计算

拆分模式设计原则：计算节点不抛 interrupt；中断节点只读 state、不做耗时操作；通过 state 传递数据。

LangGraph 渐进式教程 LG-03 / 10
这是本节课最重要的一个坑。interrupt() 恢复时，整个节点从头重新执行。如果节点里有 API 调用、数据库查询、LLM 调用——全部重来。解决方案只有一个：拆成「计算节点 → 中断节点」。计算结果写入 state，中断节点只读 state。
PREBUILT AGENT HITL
create_react_agent 的 HITL 配置
透传 interrupt_before/after 到 compile，节点名固定

场景 1：工具调用前审批

agent = create_react_agent( "openai:gpt-4", tools=[check_weather, send_email], interrupt_before=["tools"], checkpointer=InMemorySaver(), )

LLM 决定调用工具后 → 暂停 → 用户确认 → 执行工具

场景 2：LLM 输出后审查

agent = create_react_agent( "openai:gpt-4", tools=[check_weather, send_email], interrupt_after=["agent"], checkpointer=InMemorySaver(), )

LLM 每次输出后暂停 → 用户审查/修改 → 继续

组合使用

interrupt_before=["tools"], # 工具执行前确认 interrupt_after=["agent"] # LLM 输出后审查

post_model_hook：自动审查

def guardrail_hook(state): if "免费送" in last_message.content: return {"messages": [{"role": "assistant", "content": "[系统拦截]"}]} return {}
LangGraph 渐进式教程 LG-03 / 10
create_react_agent 的 interrupt_before/after 直接透传给 workflow.compile()，本身不做额外处理，只是限制可用节点名为 "agent" 和 "tools"。post_model_hook 是自动化审查，不需要人工介入，和 interrupt_after 的区别要搞清楚。
RESUME MECHANISM
Command(resume=...) 恢复机制
三种 resume 值类型，匹配 interrupt() 的调用顺序

单值

Command(resume="approve")

最简单场景：通过 / 拒绝

列表

Command(resume=["Alice", 30, "Beijing"])

节点内多个 interrupt() 按顺序匹配

字典（ID 映射）

Command(resume={ "hash_a": "approve", "hash_b": "reject", })

并行子图批量恢复

中断类型resume 值给谁节点是否重新执行
interrupt_before被中断的节点本身是（第一次执行）
interrupt_after下一个接收流转的节点否（原节点已完成）
interrupt() in node节点内部的 interrupt() 调用是（整个节点重执行）
LangGraph 渐进式教程 LG-03 / 10
多 interrupt 意味着节点会被重新执行多次。结合重执行陷阱，如果节点内有昂贵操作 + 多 interrupt，性能会极差。这就是为什么拆分模式对多中断场景更加重要。
PRODUCTION ENCAPSULATION
生产级 HITL 封装模式
从「代码里的魔法值」到「结构化的协议」

ToolInteractionEnvelope

工具返回信封而非直接中断，让上层决定要不要中断

{ "status": "needs_confirmation", "summary": "确认预约 A阅览室？", "action_request": {"action": "library_booking", "risk": "write"}, "render_schema": {...} # 前端渲染契约 }

RenderSchema：中断即 UI

标准字段类型驱动前端渲染

"fields": [ {"id": "date", "kind": "date_input", "label": "日期"}, {"id": "room", "kind": "single_select_card", ...}, {"id": "file", "kind": "file_upload", ...} ]

Resume 协议

normalize + build 统一接口，支持 submit / continue / revise / regenerate / retry

工具包装器

Copilot 模式（自动执行）vs Interactive 模式（需要确认），state.auto 动态切换

LangGraph 渐进式教程 LG-03 / 10
生产环境直接写 interrupt() 太原始。封装的目标：把中断从「代码里的魔法值」变成「结构化的协议」。工具声明需要什么，前端知道怎么渲染，用户操作后能统一恢复。这是从「能跑」到「能上线」的关键一步。
PITFALLS
这些坑，踩一个就炸
从真实项目提炼的避坑指南
#坑现象正确做法
01 忘记 checkpointer interrupt 后无法恢复，State 丢失 HITL 必须配 checkpointer
02 在工具里用 interrupt() 工具异常，图不暂停 用 interrupt_before=["tools"] 代替
03 interrupt 和异常混用 raise Exception 终止执行 HITL 必须用 interrupt() 原语

五个最常见的坑。第一个最致命：没有 checkpointer，HITL 完全无法工作。第二个很多人踩：在 tool 函数里用 interrupt()，结果异常被 ToolNode 吞掉。第三个：用 raise 代替 interrupt，整个图终止而不是暂停。第四个：重执行陷阱。第五个：hooks 的副作用。
SUMMARY
核心要点回顾

三层架构

  • Layer 1 compile：声明式，不动代码就生效
  • Layer 2 prebuilt：透传，限制节点名
  • Layer 3 in-node：最灵活，但必须拆分防重跑

中断机制

  • interrupt_before：节点不执行，resume 后第一次执行
  • interrupt_after：节点已执行完，resume 后流转下一节点
  • interrupt()：节点恢复时从头重新执行

生产级封装

  • ToolInteractionEnvelope：工具返回信封，上层决策
  • RenderSchema：中断即 UI，声明式驱动前端
  • Resume 协议：统一恢复接口，支持 edit/revise
  • Copilot/Interactive 双模式：state.auto 动态切换
