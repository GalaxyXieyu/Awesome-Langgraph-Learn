BUSINESS FIRST

为什么需要中断？因为“识别风险”不等于“拦截风险”

没有中断的自动化

生成文案→ 自动审核→ 直接发布
系统可能已经算出高风险，但流程仍然继续执行写操作。

HITL 自动化

生成文案→ 自动审核→ 暂停审批→ 发布 / 拒绝

HITL 的核心价值：把不可逆动作变成可审查动作，把人的判断写回流程状态。

SCENARIO

本节案例：内容发布审批

外部影响

公众号、短信、App Push 一旦发出去，就会触达用户，不能只靠模型自动决定。

自动判断有限

规则能命中“全网最低”“百分百治愈”，但是否允许发布仍需要业务责任人确认。

责任需要记录

审批人、审批意见、是否修改、最终发布内容，都应该进入审计链路。

案例背景3 / 15
CORE MODEL

LangGraph 中断恢复的三个角色

interrupt(payload)

在节点内部暂停，把人工处理需要的上下文交出去。

checkpointer

保存暂停点、state、thread 信息，让之后可以恢复。

Command(resume=value)

把人工回复交回图，成为 `interrupt(...)` 的返回值。

第一次执行到 interrupt(payload)：暂停，返回 __interrupt__
恢复时传入 Command(resume=value)：interrupt(...) 得到 value，节点继续完成
讲师备注：这里的“保存现场”要多解释一句。生产系统通常不会让一个请求、一个进程一直挂着等人审批。审批可能几分钟后回来，也可能第二天才回来；如果每个等待中的任务都占着线程、连接、内存和运行上下文，堆多了机器就会被拖垮。所以遇到人工审批时，系统会先把这次运行断开：生成待办，同时把 thread_id、暂停点和当时的 state 写进 checkpoint。等人处理完，再用 Command(resume=...) 从这个现场接着跑。

HUMAN ACTIONS

中断后用户可以做什么？

用户动作回复内容业务含义后续结果
通过decision=approve原文可以继续发布发布原始草稿
修改后通过decision=edit_and_approve
revised_draft=...		人类修正文案风险发布人工修改稿
拒绝decision=reject不允许发布进入拒绝分支，结束
中断 payload 不能只写“是否继续”。它应该明确给出上下文、风险证据和可选操作。

中断上下文6 / 15
RESUME PROTOCOL

用户怎么回复？用 `Command(resume=...)`

回复会变成业务事实

人工回复不应该只停留在前端按钮里。恢复后，它会写入 state，成为后续 router 的判断依据。

`resume` 的值会成为暂停位置里 `interrupt(...)` 的返回值。
Command(resume={
  "decision": "edit_and_approve",
  "reviewer_id": "reviewer_demo_001",
  "comment": "删除绝对化和收益承诺表达后发布。",
  "revised_draft": "618 智能空调活动开启..."
})
恢复协议7 / 15
ROUTING

回复后如何影响 router 走向？

review_decision = approve
  → publish_content：发布原文

review_decision = edit_and_approve
  → publish_content：发布 revised_draft

review_decision = reject
  → reject_content：不发布，流程结束
human_review→ route_after_review→ publish_content reject_content

人不是在流程外“点了个按钮”，而是把一个新的业务事实写回 state。

路由状态8 / 15
CHECKPOINT

为什么中断恢复必须配置 checkpointer？

恢复时必须知道

  • 哪个 thread 停住了
  • 停在哪个节点
  • 当时的 state 是什么
  • resume 值交还给哪次 interrupt
  • 恢复后从哪里继续

为什么不能只靠调用栈

真实审批可能几分钟后才发生，甚至第二天才发生。系统不能让请求、线程或进程一直挂在那里等人，否则大量待审批任务会占住线程、连接和内存，机器会被拖垮。

所以中断时要先断开这次运行，只留下待办和 checkpoint。人处理完，再从保存的现场恢复。

类型适合限制
MemorySaverNotebook、demo、单元测试进程重启后状态丢失
SQLite本地服务、小型应用并发和运维能力有限
Postgres / Redis生产服务、多实例、长期等待审批需要额外基础设施
恢复基础设施9 / 15
INTERRUPT TYPES

执行前、执行后、节点内中断分别适合什么？

类型暂停时机节点是否已执行恢复后发生什么常见场景
interrupt_before节点执行前否节点第一次执行发布、扣款、发短信、删除数据前拦截
interrupt_after节点执行后是进入后续节点生成文案后审查、检索后选证据、计划后确认
interrupt()节点内部取决于写法同一节点拿到 resume 后继续根据风险评分、金额、权限等运行态动态审批
选择逻辑：运行时数据决定是否暂停 → 节点内 `interrupt()`；固定写操作前拦截 → `interrupt_before`；固定产物后审查 → `interrupt_after`。

PITFALLS

这节课最容易讲漏的点

误区正确讲法
只讲 `interrupt()` API先讲业务为什么必须暂停，再讲 API 如何承载这个暂停。
中断后不知道用户能干什么明确 approve / edit_and_approve / reject 及其业务后果。
没讲回复如何影响路由人工回复写回 state，router 根据 `review_decision` 选择路径。
没讲 checkpointer没有 checkpoint，就不知道暂停点、state 和 thread，无法可靠恢复。
混淆 before / afterbefore 拦截副作用；after 审查已生成产物。
教学重点14 / 15
SUMMARY

一句话总结

HITL 的核心不是暂停，而是：在关键业务动作前保留状态，把上下文交给人，拿到人工回复后写回 state，再让 router 选择新的执行路径。

interrupt→ checkpointer→ Command(resume)→ router
