### 1. 开场引入

#### 引入策略：从「AI 失控」引入

**开场白建议**：
> "假设你是一家电商公司的运营。你用 LangGraph 做了一个自动发布营销文案的系统。AI 生成了文案，直接自动发到官方微博、微信公众号、抖音...
>
> 然后 AI 把『五一促销』写成了『五一大促，全场免费送』——因为训练数据里的某个梗。等你看手机时，已经 10 万转发，公司准备赔钱了。
>
> 这就是『完全自动化』的风险。在很多场景下，AI 可以生成，但人类必须审批。今天的主题是：**让图学会『停下来问人』**。
>
> 但这里有个巨大的坑——如果你把中断直接写在业务节点里，恢复时整个节点会重新执行一遍。你的 AI 生成逻辑、数据库查询、外部 API 调用——全部重来。如果生成一次要 30 秒，用户每次审批后都要再等 30 秒。这节课，我们要把这个坑挖出来，再填上。"

#### 生活化类比：电梯的急停按钮

> "自动扶梯是自动运行的，但每部扶梯都有一个红色的急停按钮。紧急情况下，任何人都可以按下它，让扶梯停下来。
>
> HITL（Human-in-the-Loop）就是这个急停按钮——图在正常执行，但在关键节点，它可以停下来等人类输入，然后再决定是继续、修改还是终止。
>
> 但注意：急停按钮不会『倒回』已经走过的台阶。LangGraph 的 `interrupt()` 却会让节点重新走一遍——除非你学会『拆分中断节点』这个设计模式。"

#### 与 LG-02 的衔接

> "上节课我们让图学会了调用工具。今天我们要解决一个更本质的问题：**当图要执行高风险操作时，谁来把关？**
>
> 我们会从三个层面讲清楚：图的编译配置（`interrupt_before/after`）、预构建 Agent 的配置（`create_react_agent` 的参数）、以及节点内部的动态中断（`interrupt()` 原语）。这三个层面不是独立的，而是有明确的调用关系和适用范围。
>
> 但『会写 interrupt』只是起点。生产环境里，你需要一套**封装模式**：工具怎么和人类结构化对话？前端怎么知道该渲染什么 UI？用户选了「修改」后怎么恢复？这些都是从 '能跑' 到 '能上线' 必须解决的问题。这节课后半段，我会从真实项目提炼一套完整的生产级封装方案——`ToolInteractionEnvelope` 协议、`RenderSchema` 前端契约、统一的 Resume 协议、以及 Copilot/Interactive 双模式工具包装器。"

### 2. 核心内容

#### 2.1 HITL 的三层架构：组件关系全景图

**讲解顺序**：
1. **三层架构图**—— 先看整体，再拆细节
2. **每层能做什么、不能做什么**
3. **选择决策树**—— 遇到场景该用哪一层

**核心关系图**：

```
┌─────────────────────────────────────────────────────────────────────────────┐
│ HITL 三层架构 │
├─────────────────────────────────────────────────────────────────────────────┤
│ Layer 1: 图编译时配置 (Graph Compile-Time) │
│ ├─ interrupt_before=["node_a", "node_b"] ← 节点执行前暂停 │
│ ├─ interrupt_after=["node_c"] ← 节点执行后暂停 │
│ └─ 作用域: 整图所有线程统一生效，无需改节点代码 │
│ 实现: StateGraph.compile(interrupt_before=..., interrupt_after=...) │
├─────────────────────────────────────────────────────────────────────────────┤
│ Layer 2: 预构建 Agent 配置 (Prebuilt Agent) │
│ ├─ create_react_agent(..., interrupt_before=["tools"]) │
│ ├─ create_react_agent(..., interrupt_after=["agent"]) │
│ └─ 作用域: 只对 Agent 内部的 "agent"/"tools" 节点生效 │
│ 实现: 透传给 workflow.compile()，节点名固定为 "agent" 和 "tools" │
├─────────────────────────────────────────────────────────────────────────────┤
│ Layer 3: 节点内动态中断 (Dynamic In-Node) │
│ ├─ def my_node(state): │
│ │ result = expensive_compute(state) ← 恢复时这里会重跑！ │
│ │ user_input = interrupt({"question": "确认吗？"}) │
│ │ return {"data": process(user_input)} │
│ └─ 作用域: 单个节点内部，最灵活但风险最大 │
│ 实现: interrupt() 抛出 GraphInterrupt，恢复时从节点开头重新执行 │
└─────────────────────────────────────────────────────────────────────────────┘
```

**每层的能力边界**：

| 层级 | 控制粒度 | 是否需要改节点代码 | 支持动态条件 | 重执行风险 | 适用场景 |
|------|---------|-------------------|-------------|-----------|---------|
| Layer 1 (compile) | 节点级 | 否 | 否（静态列表） | 无 | 固定审批点，如所有发布前必审 |
| Layer 2 (prebuilt) | Agent 内节点级 | 否 | 否 | 无 | ReAct Agent 的工具调用审批 |
| Layer 3 (in-node) | 节点内任意位置 | 是 | 是 | **有** | 根据运行态数据决定要不要中断 |

**决策树**：
```
是否需要根据运行时数据决定中断？
├─ 是 → Layer 3 (interrupt())
│ └─ 但节点内有昂贵操作？
│ ├─ 是 → 拆成「计算节点 → 中断节点」
│ └─ 否 → 直接写 interrupt()
└─ 否 → 使用预构建 Agent？
├─ 是 → Layer 2 (create_react_agent 参数)
└─ 否 → Layer 1 (compile 参数)
```

#### 2.2 `interrupt_before` / `interrupt_after` 编译时配置

**讲解顺序**：
1. **源码级实现原理**—— 理解为什么它是"声明式"的
2. **interrupt_before：执行前中断**
3. **interrupt_after：执行后中断**
4. **两者的本质区别与版本检查机制**

**2.2.1 源码实现原理**（`langgraph/pregel/_loop.py:650-703`）

```python
# === tick() 方法：每次 superstep 执行前调用 ===
def tick(self) -> bool:
    # ... 准备 tasks ...

    # BEFORE 检查：在节点执行前判断是否要中断
if self.interrupt_before and should_interrupt(
self.checkpoint, self.interrupt_before, self.tasks.values()
):
self.status = "interrupt_before"
raise GraphInterrupt() # ← 这里抛出，节点不会执行

    # ... 执行 tasks ...

def after_tick(self) -> None:
    # ... writes 已应用，checkpoint 已保存 ...

    # AFTER 检查：在节点执行完、writes 应用后判断
if self.interrupt_after and should_interrupt(
self.checkpoint, self.interrupt_after, self.tasks.values()
):
self.status = "interrupt_after"
raise GraphInterrupt() # ← 这里抛出，节点已执行完毕
```

**关键函数 `should_interrupt`**（`langgraph/pregel/_algo.py:155-185`）：

```python
def should_interrupt(checkpoint, interrupt_nodes, tasks):
    # 1. 检查自上次中断以来，是否有任何 channel 被更新
any_updates_since_prev_interrupt = any(
version > seen.get(chan, null_version)
for chan, version in checkpoint["channel_versions"].items()
)
    # 2. 如果有更新，且当前 step 要执行的节点在 interrupt 列表中 → 中断
return [
task for task in tasks
if task.name in interrupt_nodes # 或 == "*"
] if any_updates_since_prev_interrupt else []
```

> **讲师强调**：`should_interrupt` 的第一个条件是 "自上次中断以来是否有 channel 更新"。这意味着：如果一个节点被标记了 `interrupt_after`，但它的执行没有产生任何 state 变更，是不会触发中断的。这防止了无意义的中断循环。

**2.2.2 `interrupt_before` —— 执行前拦截**

**语义**：在节点的 `proc.invoke()` 被调用之前，抛出 `GraphInterrupt`。节点代码**完全不执行**。

**适用场景**：
- 审批员先看输入，决定是否让 AI 处理
- 高风险操作（发布、转账、删除）执行前的确认
- 需要人工补充信息后才能执行的节点

**代码示例**：
```python
from langgraph.graph import StateGraph, START
from langgraph.checkpoint.memory import InMemorySaver
from typing import TypedDict

class State(TypedDict):
content: str
approved: bool

def generate_node(state: State):
    # 这个节点在 resume 后才会真正执行
return {"content": f"生成内容: {state['content']}"}

def publish_node(state: State):
print(f"发布: {state['content']}")
return {"approved": True}

builder = StateGraph(State)
builder.add_node("generate", generate_node)
builder.add_node("publish", publish_node)
builder.add_edge(START, "generate")
builder.add_edge("generate", "publish")

graph = builder.compile(
checkpointer=InMemorySaver(),
interrupt_before=["publish"], # ← publish 执行前暂停
)

# 第一次调用：走到 publish 前中断
config = {"configurable": {"thread_id": "demo-1"}}
for chunk in graph.stream({"content": "五一促销文案"}, config):
print(chunk)
# 输出: {'generate': {'content': '生成内容: 五一促销文案'}}
# 然后中断，status = "interrupt_before"

# 用户审批后恢复
from langgraph.types import Command
for chunk in graph.stream(Command(resume="approve"), config):
print(chunk)
# 输出: {'publish': {'approved': True}}
```

**2.2.3 `interrupt_after` —— 执行后拦截**

**语义**：节点代码**完整执行**，writes 应用到 state，checkpoint 保存后，再抛出 `GraphInterrupt`。

**适用场景**：
- AI 生成内容后让人类确认
- 工具执行后审查结果
- 需要看到输出才能决定是否继续

**代码示例**：
```python
graph = builder.compile(
checkpointer=InMemorySaver(),
interrupt_after=["generate"], # ← generate 执行完后暂停
)

# 第一次调用：generate 执行完，writes 已保存，然后中断
for chunk in graph.stream({"content": "五一促销文案"}, config):
print(chunk)
# 输出: {'generate': {'content': '生成内容: 五一促销文案'}}
# 然后中断，status = "interrupt_after"
# 关键：此时 state 里已经有 content 了

# 查看当前 state
snapshot = graph.get_state(config)
print(snapshot.values["content"]) # "生成内容: 五一促销文案"

# 用户确认后继续
for chunk in graph.stream(Command(resume="confirmed"), config):
print(chunk)
# 输出: {'publish': ...}
```

**2.2.4 核心区别总结**

| 维度 | `interrupt_before` | `interrupt_after` |
|------|-------------------|-------------------|
| 节点执行了吗 | **否** | **是** |
| resume 后节点执行吗 | **是**（第一次执行） | **否**（已执行过） |
| state 有节点的输出吗 | 否 | 是 |
| 适用场景 | 审批输入/前置确认 | 审查输出/后置确认 |
| status 值 | `"interrupt_before"` | `"interrupt_after"` |
| 源码位置 | `_loop.py:654` | `_loop.py:702` |

**易混淆点澄清**：

> `interrupt_before=["publish"]` + resume 后：publish 节点**第一次执行**。
>
> `interrupt_after=["generate"]` + resume 后：generate 节点**不会再次执行**，因为它已经执行过了，resume 只是让图继续流转到下一个节点。

#### 2.3 `interrupt()` 原语：动态中断与「重执行陷阱」

**讲解顺序**：
1. **`interrupt()` 的工作机制**
2. **⚠️ 重执行陷阱：源码级证明**—— 这是最大的坑
3. **中断节点拆分模式：解决方案**
4. **多 `interrupt()` 与顺序匹配**

**2.3.1 `interrupt()` 工作机制**

`interrupt()` 不是普通的函数返回，它内部实现（`langgraph/types.py:801-924`）是这样的：

```python
def interrupt(value: Any) -> Any:
conf = get_config()["configurable"]
scratchpad = conf[CONFIG_KEY_SCRATCHPAD]
idx = scratchpad.interrupt_counter() # 当前是第几个 interrupt

    # 检查是否有 resume 值可用
if scratchpad.resume:
if idx < len(scratchpad.resume):
            # ← 恢复时走这里：直接返回 resume 值，不抛异常
return scratchpad.resume[idx]

    # 第一次执行：抛出 GraphInterrupt
raise GraphInterrupt((Interrupt.from_ns(value, ns=conf[CONFIG_KEY_CHECKPOINT_NS]),))
```

**执行流程**：
```
第 1 次调用 interrupt("question"):
→ 没有 resume 值
→ raise GraphInterrupt
→ 图暂停，value 传给客户端

客户端发送 Command(resume="answer")

第 2 次调用（恢复后，节点从头重新执行）:
→ scratchpad.resume = ["answer"]
→ scratchpad.interrupt_counter() 返回 0
→ idx (0) < len(resume) (1)，返回 resume[0] = "answer"
→ 不抛异常，继续执行后续代码
```

**2.3.2 ⚠️ 重执行陷阱（The Re-execution Pitfall）**

**源码原文**（`langgraph/types.py:814`）：

> "The graph resumes from the start of the node, **re-executing** all logic."

**问题复现**：
```python
from langgraph.types import interrupt
import time

class State(TypedDict):
query: str
result: str
confirmed: bool

def expensive_analysis_node(state: State):
    # ===== 这部分在 resume 后会重新执行！=====
print("[重执行] 开始耗时分析...")
time.sleep(3) # 模拟 3 秒分析
analysis = f"分析结果: {state['query']} -> 推荐方案A"
print(f"[重执行] 分析完成: {analysis}")
    # ========================================

    # interrupt 在这里
decision = interrupt({
"type": "approval",
"title": "请确认分析结果",
"content": analysis
})

    # 只有 interrupt 之后的代码是"新的"
return {"result": analysis, "confirmed": decision == "approve"}
```

**问题演示**：
```
用户: "分析市场需求"
→ 打印 "[重执行] 开始耗时分析..."
→ sleep 3 秒
→ 打印 "[重执行] 分析完成..."
→ 中断，弹出审批面板

用户点击"通过"（发送 Command(resume="approve")）
→ 节点从头重新执行！
→ 打印 "[重执行] 开始耗时分析..." ← 又来了！
→ sleep 3 秒 ← 又等 3 秒！
→ 打印 "[重执行] 分析完成..."
→ 这次 interrupt() 返回 "approve"
→ 执行 return
```

> **学员现场感受**："用户点一下通过，结果又卡了 3 秒？这在生产环境是不可接受的。"

**2.3.3 中断节点拆分模式（Split Interrupt Node Pattern）**

**解决方案**：把「计算」和「中断」拆成两个节点。

**错误写法**（计算 + 中断混在一起）：
```python
def bad_node(state: State):
result = expensive_compute(state) # ← resume 时重跑
user_input = interrupt({"data": result}) # ← 中断
return {"output": process(user_input)}
```

**正确写法**（拆分节点）：
```python
def compute_node(state: State):
"""纯计算，无中断。只执行一次。"""
result = expensive_compute(state)
return {"analysis_result": result} # ← 写入 state，持久化

def human_confirm_node(state: State):
"""纯中断，无重计算。resume 时秒过。"""
result = state["analysis_result"] # ← 从 state 读取，不计算
decision = interrupt({
"type": "approval",
"title": "请确认分析结果",
"content": result
})
return {"confirmed": decision == "approve"}

def process_node(state: State):
"""根据确认结果处理"""
if state["confirmed"]:
return {"output": finalize(state["analysis_result"])}
return {"output": "已取消"}

builder = StateGraph(State)
builder.add_node("compute", compute_node)
builder.add_node("confirm", human_confirm_node) # ← 单独的中断节点
builder.add_node("process", process_node)
builder.add_edge(START, "compute")
builder.add_edge("compute", "confirm")
builder.add_edge("confirm", "process")

graph = builder.compile(checkpointer=InMemorySaver())
```

**拆分后的执行流程**：
```
第 1 次执行:
compute_node: expensive_compute() → 写 state → 完成
confirm_node: interrupt() → 暂停

用户发送 Command(resume="approve"):
compute_node: 已经执行过，不再执行（writes 已有，task.writes 非空）
confirm_node: 重新执行，但 interrupt() 发现 resume 值，立即返回
→ 写 confirmed=True → 完成
process_node: 执行 finalize()
```

**拆分模式设计原则**：

| 原则 | 说明 |
|------|------|
| 计算节点不抛 interrupt | 所有 `interrupt()` 调用放在独立的专用节点中 |
| 中断节点只读 state | 不执行任何有副作用或耗时的操作 |
| 中断节点只做一件事 | 调用 `interrupt()`，然后根据返回值写 state |
| 通过 state 传递数据 | 计算结果通过 state 字段传给中断节点 |

**2.3.4 多 `interrupt()` 与顺序匹配**

如果一个节点内有多个 `interrupt()` 调用，LangGraph 按**调用顺序**匹配 resume 值：

```python
def multi_interrupt_node(state: State):
    # interrupt #0
name = interrupt({"question": "你的名字？"})
print(f"名字: {name}")

    # interrupt #1
age = interrupt({"question": "你的年龄？"})
print(f"年龄: {age}")

    # interrupt #2
city = interrupt({"question": "你的城市？"})
print(f"城市: {city}")

return {"profile": {"name": name, "age": age, "city": city}}
```

```python
# 第一次中断：question 0 (name)
graph.stream(input, config)

# 恢复：提供第一个值
graph.stream(Command(resume="Alice"), config)
# → 节点重新执行，interrupt #0 返回 "Alice"
# → 走到 interrupt #1，再次中断

# 恢复：提供第二个值
graph.stream(Command(resume=30), config)
# → 节点重新执行，interrupt #0 返回 "Alice"（从 scratchpad.resume[0]）
# → interrupt #1 返回 30（从 scratchpad.resume[1]）
# → 走到 interrupt #2，再次中断

# 恢复：提供第三个值
graph.stream(Command(resume="Beijing"), config)
# → 全部通过，节点完成
```

> **警告**：多 interrupt 意味着节点会被重新执行多次。结合前面的陷阱，如果节点内有昂贵操作 + 多 interrupt，性能会极差。这就是为什么**拆分模式**对多中断场景更加重要。

**2.4.1 预构建 Agent 的内部节点**

**2.4.2 Agent 级别的中断配置**

`create_react_agent` 的 `interrupt_before` / `interrupt_after` 参数（源码第 302-304 行）：

```python
def create_react_agent(
model, tools, *,
interrupt_before: list[str] | None = None, # 只能是 "agent" 或 "tools"
interrupt_after: list[str] | None = None, # 只能是 "agent" 或 "tools"
...
):
```

文档说明（第 447-454 行）：
> "Should be one of the following: `'agent'`, `'tools'`. This is useful if you want to add a user confirmation or other interrupt before taking an action."

**场景 1：工具调用前审批**（`interrupt_before=["tools"]`）
```python
from langgraph.prebuilt import create_react_agent

def check_weather(location: str) -> str:
return f"{location} 天气晴朗"

def send_email(to: str, content: str) -> str:
return f"邮件已发送给 {to}"

# 创建 Agent，工具执行前需要人工确认
agent = create_react_agent(
"openai:gpt-4",
tools=[check_weather, send_email],
interrupt_before=["tools"], # ← LLM 决定调用工具后，先暂停等用户确认
checkpointer=InMemorySaver(),
)

# 用户问："给 boss@company.com 发邮件说项目延期"
# 执行流程：
# 1. agent 节点：LLM 生成 tool_call (send_email)
# 2. 要进入 tools 节点前 → interrupt_before 触发 → 暂停
# 3. 用户看到："AI 要执行 send_email(to='boss@company.com', content='...')，确认吗？"
# 4. 用户发送 Command(resume="yes") → 进入 tools 节点执行
```

**场景 2：LLM 输出后审查**（`interrupt_after=["agent"]`）
```python
agent = create_react_agent(
"openai:gpt-4",
tools=[check_weather, send_email],
interrupt_after=["agent"], # ← LLM 每次输出后暂停，用户可审查/修改
checkpointer=InMemorySaver(),
)

# 用户问："查北京天气"
# 执行流程：
# 1. agent 节点：LLM 输出 "我来查北京天气" + tool_call (check_weather)
# 2. agent 节点执行完 → interrupt_after 触发 → 暂停
# 3. 用户看到 LLM 的完整输出，可以确认或要求修改
# 4. 如果 LLM 没有 tool_call（直接回答），用户也可以在这里审查答案
```

**场景 3：组合使用**
```python
agent = create_react_agent(
"openai:gpt-4",
tools=[check_weather, send_email],
interrupt_before=["tools"], # 工具执行前确认（防止误操作）
interrupt_after=["agent"], # LLM 输出后审查（质量控制）
checkpointer=InMemorySaver(),
)
```

> 注意：`create_react_agent` 的 `interrupt_before/after` **直接透传**给 `workflow.compile()`（第 995-1002 行）。它本身不做额外处理，只是限制了可用节点名。

**2.4.3 `post_model_hook`：轻量级审查点**

如果你需要在每次 LLM 输出后做一些**自动化的**审查（不需要人工审批），用 `post_model_hook` 而不是 `interrupt_after`：

```python
def guardrail_hook(state):
"""自动审查 LLM 输出，无需人工介入"""
messages = state["messages"]
last_message = messages[-1]

    # 自动检查敏感词
if hasattr(last_message, "content") and "免费送" in last_message.content:
        # 触发自动拦截，用特殊消息替换
return {
"messages": [
{
"role": "assistant",
"content": "[系统拦截：检测到违规用语]"
}
]
}

    # 通过审查，不做修改
return {}

agent = create_react_agent(
"openai:gpt-4",
tools=[...],
post_model_hook=guardrail_hook, # ← 自动化审查
)
```

`post_model_hook` vs `interrupt_after` 的区别：

| 维度 | `post_model_hook` | `interrupt_after=["agent"]` |
|------|-------------------|---------------------------|
| 时机 | agent 节点后 | agent 节点后 |
| 是否需要人工 | 否（自动） | 是（人工） |
| 实现方式 | 额外节点 | compile 配置 |
| 可以修改输出 | 是 | 是（通过 resume 值） |
| 延迟 | 无 | 取决于用户响应时间 |
| 适用 | 自动 guardrails | 人工审批 |

#### 2.5 工具层面的 HITL 思考

**讲解顺序**：
1. **工具不能直接用 `interrupt()`**
2. **替代方案：工具节点级中断 + 条件路由**

**2.5.1 为什么工具不能直接用 `interrupt()`**

工具函数在 `ToolNode` 内部被调用，它的执行环境**没有** graph 的 `get_config()` 上下文，`interrupt()` 无法正常工作。

```python
# ❌ 错误：在 tool 函数里用 interrupt
def risky_tool(input: str) -> str:
decision = interrupt({"question": "确认执行？"}) # ← 抛异常，但不会被正确捕获
if decision == "yes":
return do_risky_thing(input)
return "取消"

# ToolNode 内部执行工具时，interrupt() 抛出的 GraphInterrupt
# 会被 ToolNode 当作普通异常处理，导致 tool 返回错误而不是中断整个图
```

**2.5.2 替代方案：Agent 图层面的工具审批**

方案 A：`interrupt_before=["tools"]` —— 所有工具执行前都审批（最简单）

方案 B：自定义条件路由 —— 只对高风险工具审批
```python
from langgraph.types import Command
from langchain_core.messages import AIMessage, ToolMessage

def should_interrupt_tools(state) -> str:
"""条件路由：只有高风险工具才中断"""
messages = state["messages"]
last_message = messages[-1]

if isinstance(last_message, AIMessage) and last_message.tool_calls:
high_risk_tools = {"send_email", "transfer_money", "delete_user"}
tool_names = {call["name"] for call in last_message.tool_calls}

if tool_names & high_risk_tools: # 有交集
return "need_approval" # 走审批节点

return "direct_execute" # 直接执行工具

# 在自定义图中使用
builder.add_conditional_edges("agent", should_interrupt_tools, {
"need_approval": "tool_approval",
"direct_execute": "tools",
})
builder.add_node("tool_approval", approval_node) # 这个节点里用 interrupt()
builder.add_edge("tool_approval", "tools")
```

#### 2.6 `Command(resume=...)` 恢复机制深度解析

**讲解顺序**：
1. **Resume 值的三种传递方式**
2. **Resume 与 `interrupt_before/after` 的关系**
3. **批量恢复多个中断**

**2.6.1 Resume 值的传递路径**

```
客户端发送: Command(resume=value)
↓
_loop.py: 解析 resume → scratchpad.resume = [value]
↓
节点重新执行: interrupt() 检查 scratchpad
↓
scratchpad.resume 非空 → 返回 value（不抛异常）
```

**三种 resume 值类型**：

```python
# 1. 单值：最简单的场景
Command(resume="approve")

# 2. 列表：节点内有多个 interrupt()
Command(resume=["Alice", 30, "Beijing"])
# interrupt #0 得到 "Alice"
# interrupt #1 得到 30
# interrupt #2 得到 "Beijing"

# 3. 字典（ID 映射）：恢复特定中断（多任务并行时）
Command(resume={
"abc123hash": "approve", # task A 的中断
"def456hash": "reject", # task B 的中断
})
```

**2.6.2 Resume 与 `interrupt_before/after` 的关系**

| 中断类型 | resume 值给谁 | 节点是否重新执行 |
|---------|-------------|---------------|
| `interrupt_before` | 被中断的节点本身 | **是**（第一次执行） |
| `interrupt_after` | 下一个接收流转的节点 | 否（原节点已完成） |
| `interrupt()` in node | 节点内部的 `interrupt()` 调用 | **是**（整个节点重执行） |

```python
# interrupt_before=["publish"] 场景
def publish_node(state: State):
    # resume 值不会直接传给 publish_node 的参数
    # 但如果 publish_node 内部有 interrupt()，resume 值会被它消费
decision = interrupt({"question": "确认发布？"}) # ← 收到 resume
if decision == "yes":
do_publish(state["content"])
return {"published": decision == "yes"}
```

**2.6.3 批量恢复**

当并行子图或多个节点同时中断时：

```python
# 假设有两个并行的中断任务
# task_id_a 的中断 ID = "hash_a"
# task_id_b 的中断 ID = "hash_b"

Command(resume={
"hash_a": "approve",
"hash_b": "reject",
})
```

#### 2.7 Pre/Post Model Hooks

**讲解顺序**：
1. **Pre-model Hook：LLM 调用前的预处理**
2. **Post-model Hook：LLM 调用后的处理**
3. **`llm_input_messages` 的临时注入机制**

**2.7.1 Pre-model Hook**

```python
def trim_messages_hook(state):
"""在 LLM 调用前截断过长的消息历史"""
messages = state["messages"]
if len(messages) > 10:
        # 保留 system message + 最近 9 条
trimmed = [messages[0]] + messages[-9:]
return {
"llm_input_messages": trimmed # ← 临时注入，不修改 state["messages"]
}
return {}

agent = create_react_agent(
model,
tools=[...],
pre_model_hook=trim_messages_hook,
)
```

**关键区别**：
- `return {"messages": [...]}` → **修改 state**，后续节点可见
- `return {"llm_input_messages": [...]}` → **只影响本次 LLM 输入**，不修改 state

**2.7.2 Post-model Hook**

```python
def review_hook(state):

if isinstance(last_message, AIMessage):
        # 检查是否包含 tool_calls

```

#### 2.8 生产级 HITL 封装模式（从真实项目提炼）

**讲解顺序**：
1. **为什么需要封装**—— 每次都写 `interrupt()` 太原始
2. **ToolInteractionEnvelope**—— 工具与人类的结构化通信协议
3. **InterruptWriter + RenderSchema**—— 中断即 UI，前端直接渲染
4. **Resume 协议**—— 统一的恢复接口，不只是 "approve/reject"
5. **工具包装器**—— Copilot 模式 vs Interactive 模式
6. **完整实战：图书馆订位系统**

**2.8.1 为什么需要封装？**

前面讲的 `interrupt()` 是原子能力，但直接用在生产环境会有几个问题：

| 问题 | 现象 | 后果 |
|------|------|------|
| 每次手写 interrupt 格式不统一 | 有的传 `{"question": "确认吗？"}`，有的传 `{"title": "审批", "content": "..."}` | 前端解析困难，UI 混乱 |
| resume 值格式散乱 | 有的返回字符串 `"approve"`，有的返回 `{"decision": "confirm", "values": {...}}` | 后端处理逻辑碎片化 |
| 工具与前端耦合 | 工具函数里直接写死中断文案 | 改文案要改代码，无法配置化 |
| 复杂交互无法表达 | 只有 "接受/拒绝"，没有 "修改参数"、"选日期"、"上传文件" | 用户体验差 |

**封装的目标**：把中断从 "代码里的魔法值" 变成 "结构化的协议"——工具声明需要什么，前端知道怎么渲染，用户操作后能统一恢复。

---

**2.8.2 ToolInteractionEnvelope：工具与人类的结构化通信协议**

在生产环境中，工具不应该直接抛出 `interrupt()`。相反，工具返回一个**信封（Envelope）**，告诉上层：
- 我执行完了，状态是什么？（ok / needs_input / needs_confirmation / error）
- 如果需要人类介入，需要什么形式的输入？
- 这个动作的风险等级是什么？

**信封结构**：

```python
from typing import TypedDict, Literal, Dict, Any, Optional

ToolInteractionStatus = Literal["ok", "needs_input", "needs_confirmation", "error"]
ToolActionRisk = Literal["read", "write"]

class ToolActionRequest(TypedDict, total=False):
    action: str          # 动作标识，如 "library_booking"
    args: Dict[str, Any] # 动作参数
    risk: ToolActionRisk # read = 只读，write = 会修改数据
    requires_confirmation: bool  # 是否必须人工确认

class ToolInteractionEnvelope(TypedDict, total=False):
    version: str         # 协议版本，如 "tool-interaction/v1"
    status: ToolInteractionStatus
    domain: str          # 业务域，如 "library", "finance"
    kind: str            # 具体类型，如 "booking_confirmation"
    summary: str         # 给 LLM 看的摘要
    data: Dict[str, Any] # 结构化数据
    render_schema: Dict[str, Any]  # 前端渲染契约
    action_request: ToolActionRequest  # 动作请求
    state_patch: Dict[str, Any]      # 要 patches 到 state 的字段
    error: Dict[str, Any]            # 错误信息
```

**工具返回信封而非直接中断**：

```python
from src.infrastructure.ai.agent.interaction_protocol import (
    build_tool_interaction_envelope,
    build_tool_interaction_response,
)

def library_booking_tool(date: str, room_id: str) -> str:
    """图书馆订位工具"""
    # 1. 执行业务逻辑
    result = query_library_api(date=date, room_id=room_id)

    # 2. 构建信封：需要用户确认
    envelope = build_tool_interaction_envelope(
        status="needs_confirmation",
        domain="library",
        kind="booking_confirmation",
        summary=f"确认预约 {room_id} 于 {date}？",
        data={"date": date, "room_id": room_id, "available": result["available"]},
        action_request={
            "action": "library_booking",
            "args": {"date": date, "room_id": room_id},
            "risk": "write",
            "requires_confirmation": True,
        },
        render_schema={
            "kind": "library_booking_confirmation",
            "title": "图书馆订位确认",
            "description": "请确认或修改预约信息。",
            "fields": [
                {
                    "id": "booking_date",
                    "kind": "date_input",
                    "label": "日期",
                    "required": True,
                    "value": date,
                },
                {
                    "id": "room_select",
                    "kind": "single_select_card",
                    "label": "阅览室",
                    "required": True,
                    "options": [
                        {"id": "room-a", "title": "A阅览室", "description": "安静区，24座位"},
                        {"id": "room-b", "title": "B阅览室", "description": "讨论区，12座位"},
                    ],
                },
            ],
        },
    )

    # 3. 返回 (content, artifact) 元组——content 给 LLM 看，artifact 存信封
    return build_tool_interaction_response(
        envelope,
        content=f"图书馆订位查询完成：{room_id} 在 {date} {'可预约' if result['available'] else '已满'}。"
    )
```

**关键设计决策**：
- 工具函数**不调用 `interrupt()`**——它只返回信封，让上层决定要不要中断
- `content` 是 LLM 可见的文本，`artifact`（信封）是给程序解析的结构化数据
- `render_schema` 告诉前端："这个中断应该渲染成什么 UI"

**2.8.3 InterruptWriter + RenderSchema：中断即 UI**

信封里的 `render_schema` 是前端渲染的契约。它不是随意写的 dict，而是一个有**标准字段类型**的规范：

**标准字段类型**：

```python
RenderFieldKind = Literal[
    "static_text",        # 静态文本展示
    "single_select_card", # 单选卡片（带标题、描述、徽章）
    "checkbox",           # 复选框
    "text_input",         # 单行文本输入
    "date_input",         # 日期选择
    "time_input",         # 时间选择
    "datetime_input",     # 日期时间选择
    "file_upload",        # 文件上传
]
```

**RenderSchema 结构**：

```python
class InterruptRenderSchema(TypedDict, total=False):
    kind: str                    # 业务类型，如 "library_booking_confirmation"
    title: str                   # 弹窗标题
    description: str             # 说明文字
    continuation: Dict[str, str] # {mode: "action|resume", graph_id: "..."}
    fields: List[RenderSchemaField]  # 表单字段

class RenderSchemaField(TypedDict, total=False):
    id: str           # 字段标识
    kind: RenderFieldKind  # 字段类型
    label: str        # 标签
    description: str  # 辅助说明
    value: Any        # 默认值
    required: bool    # 是否必填
    options: List[RenderSchemaOption]  # 选项（单选卡片用）
    config: Dict[str, Any]  # 额外配置（如 file_upload 的 accept）
```

**验证函数**：


**2.8.4 Resume 协议：统一恢复接口**

用户在前端操作后，发送 resume 请求。但不同场景的 resume 格式不同：
- 简单确认：`"approve"` / `"reject"`
- 带修改的确认：`{"decision": "edit", "edit_args": {"date": "2024-06-01"}}`
- 多中断批量处理：`{"int-a": {...}, "int-b": {...}}`

**Resume 信封：统一所有格式**

```python
# 旧格式（Legacy）→ 自动归一化
envelope = normalize_resume_envelope(
    {"decision": "edit", "edit_args": {"option_id": "opt-1"}}
)
# 结果：
# {
#   "operation": "submit",
#   "target": {"type": "interrupt"},
#   "submission": {"decision": "confirm", "values": {"option_id": "opt-1"}}
# }

# 新格式（直接提交）
envelope = normalize_resume_envelope({
    "operation": "submit",
    "target": {"type": "interrupt", "id": "int-1"},
    "submission": {"decision": "cancel", "values": {"reason": "no"}}
})
# submission.decision = "cancel" → 映射为 reject

# 多中断批量提交
envelope = normalize_resume_envelope({
    "operation": "submit",
    "target": {"type": "interrupt", "ids": ["a", "b"]},
    "submissions": {
        "a": {"decision": "confirm", "values": {"approved": True}},
        "b": {"decision": "cancel", "values": {}},
    },
})
```

**支持的 Operation**：

| Operation | 含义 | 场景 |
|-----------|------|------|
| `submit` | 提交决策 | 用户完成选择，提交结果 |
| `continue` | 继续执行 | 不需要额外输入，直接继续 |
| `revise` | 修改后重试 | 用户修改了参数，要求重新执行 |
| `regenerate` | 重新生成 | 对 AI 输出不满意，要求重来 |
| `retry` | 重试 | 上次执行失败，再试一次 |

**决策映射到 Command(resume=...)**：

```python
from src.application.use_cases.tasks.resume_protocol import build_command_resume_payload

# 单中断
payload = build_command_resume_payload({
    "operation": "submit",
    "target": {"type": "interrupt", "id": "int-1"},
    "submission": {"decision": "confirm", "values": {}},
})
# → ["int-1"]: {"type": "accept", "args": {}}

# 取消 → reject
payload = build_command_resume_payload({
    "operation": "submit",
    "target": {"type": "interrupt", "id": "int-1"},
    "submission": {"decision": "cancel", "values": {"reason": "时间冲突"}},
})
# → ["int-1"]: {"type": "reject", "args": {"reason": "时间冲突"}}

# 修改 → edit
payload = build_command_resume_payload({
    "operation": "submit",
    "target": {"type": "interrupt", "id": "int-1"},
    "submission": {"decision": "confirm", "values": {"date": "2024-06-02"}},
})
# → ["int-1"]: {"type": "edit", "args": {"date": "2024-06-02"}}

# 最终使用
from langgraph.types import Command
graph.stream(Command(resume=payload), config)
```

**关键映射规则**：

| Resume Decision | Interrupt Response | 含义 |
|-----------------|-------------------|------|
| `confirm` + 无 values | `{"type": "accept"}` | 直接接受 |
| `confirm` + 有 values | `{"type": "edit", "args": values}` | 接受但修改了参数 |
| `cancel` | `{"type": "reject", "args": values}` | 拒绝 |
| `revise` / `regenerate` / `retry` | `{"type": "edit", "args": {"_decision": "...", ...}}` | 修改并重试 |

**2.8.5 工具包装器：Copilot 模式 vs Interactive 模式**

生产环境工具通常有两种执行模式：
- **Copilot 模式**：自动执行，无需确认（适合只读查询）
- **Interactive 模式**：需要人工确认（适合写操作）

**包装器实现**：

```python
from src.infrastructure.ai.agent.tools.wrapper import (
    wrap_tools_by_state,
    get_tool_mode_from_state,
)

# 根据 state.auto 自动选择模式
# auto=True → copilot（自动执行）
# auto=False → interactive（需要确认）

async def build_agent_tools(state: dict):
    # 从 state 读取模式
    mode = get_tool_mode_from_state(state)  # "copilot" 或 "interactive"

    tools = [
        web_search_tool,      # 搜索——copilot 下直接执行
        library_booking_tool, # 订位——interactive 下需要确认
        send_email_tool,      # 发邮件——interactive 下需要确认
    ]

    # 自动包装：根据 mode 决定每个工具是否需要 interrupt
    return await wrap_tools_by_state(
        tools,
        state,
        writer=stream_writer,  # StreamWriter 用于输出流信息
        state=state,
    )
```

**Interactive 模式内部逻辑**（简化版）：

```python
async def interactive_tool(config, **tool_input):
    # 1. 权限检查
    denial = _authorize(config, tool, tool_input)
    if denial:
        return denial  # 权限不足，直接返回错误

    # 2. 构建中断请求
    request = {
        "action_request": {"action": tool.name, "args": tool_input},
        "config": {"allow_accept": True, "allow_reject": True, "allow_edit": True},
        "description": f"准备调用 {tool.name}，参数：{tool_input}",
    }

    # 3. 调用 interrupt，等待用户决策
    response = types.interrupt(request)

    # 4. 归一化响应（支持字符串和 dict 两种格式）
    response = _normalize_interrupt_response(response)
    # "accept" → {"type": "accept"}
    # "reject" → {"type": "reject"}
    # "edit" → {"type": "edit", "args": tool_input}

    # 5. 根据决策执行
    if response["type"] == "accept":
        return await tool.ainvoke(input=tool_input)
    elif response["type"] == "edit":
        # 用用户修改后的参数重新执行
        new_input = {**tool_input, **response.get("args", {})}
        return await tool.ainvoke(input=new_input)
    else:  # reject
        return "操作已取消。"
```

```python
    messages = state["messages"]

    # 从 ToolMessage.artifact 中提取信封

                "description": envelope["summary"],
                "action_request": action_req,
                "render_schema": render_schema,
            })

            # interrupt 返回后，decision 是用户的 resume 值
            return {"human_decision": decision}

    return {}

# ===== 第 5 步：恢复图执行 =====

def process_resume_node(state: State):

    decision = state["human_decision"]
```

### 3. ContentGuard：完整 HITL 设计

#### 案例与概念的一一对应

| 概念 | ContentGuard 中的体现 | 层级 |
|------|----------------------|------|
| `interrupt_before` | 高风险发布操作前强制审批 | Layer 1 |
| `interrupt_after` | AI 生成文案后人工确认 | Layer 1 |
| `interrupt()` + 拆分节点 | 根据内容风险分数动态决定要不要审批 | Layer 3 |
| `create_react_agent` + `interrupt_before=["tools"]` | Agent 调用发布工具前确认 | Layer 2 |
| `post_model_hook` | LLM 输出后自动敏感词扫描 | Hook |
| `pre_model_hook` | LLM 调用前注入品牌规范 | Hook |
| `Command(resume)` | 审批员选择「通过/拒绝/修改」后恢复 | 恢复机制 |

#### 完整代码架构

```python
from typing import TypedDict, Optional
from langgraph.graph import StateGraph, START
from langgraph.checkpoint.memory import InMemorySaver
from langgraph.types import interrupt, Command
from langgraph.prebuilt import create_react_agent
from langchain_core.messages import AIMessage

class State(TypedDict):
topic: str
draft: str
risk_score: float
approved: bool
published: bool

# ===== Layer 3: 动态中断（拆分节点模式）=====
def analyze_risk_node(state: State):
"""计算内容风险分数。纯计算，无中断。"""
    # 模拟风险分析
risk_keywords = ["免费送", "暴跌", "100%"] # 高风险词
risk_score = sum(1 for kw in risk_keywords if kw in state["draft"]) * 0.3
return {"risk_score": min(risk_score, 1.0)}

def risk_approval_node(state: State):
"""高风险内容需要人工确认。纯中断，无计算。"""
if state["risk_score"] > 0.5:
decision = interrupt({
"type": "approval",
"title": "⚠️ 高风险内容确认",
"content": f"风险分数: {state['risk_score']}\n内容: {state['draft']}",
"options": ["approve", "reject", "revise"]
})
return {"approved": decision == "approve"}
return {"approved": True}

# ===== Layer 1: 编译时中断（固定审批点）=====
def generate_node(state: State):
draft = f"【{state['topic']}】五一特惠，全场满100减50！"
return {"draft": draft}

def publish_node(state: State):
print(f"✅ 正式发布: {state['draft']}")
return {"published": True}

# 构建内容审核图
builder = StateGraph(State)
builder.add_node("generate", generate_node)
builder.add_node("analyze_risk", analyze_risk_node)
builder.add_node("risk_approval", risk_approval_node)
builder.add_node("publish", publish_node)

builder.add_edge(START, "generate")
builder.add_edge("generate", "analyze_risk")
builder.add_edge("analyze_risk", "risk_approval")
builder.add_edge("risk_approval", "publish")

# 编译：固定审批点 + 动态中断
graph = builder.compile(
checkpointer=InMemorySaver(),
interrupt_before=["publish"], # Layer 1: 发布前必审
interrupt_after=["generate"], # Layer 1: 生成后确认
)

# ===== Layer 2: 预构建 Agent（工具审批）=====
def post_to_weibo(content: str) -> str:
return f"微博发布成功: {content[:50]}..."

content_agent = create_react_agent(
"openai:gpt-4",
tools=[post_to_weibo],
interrupt_before=["tools"], # Layer 2: 调用发布工具前确认
checkpointer=InMemorySaver(),
)
```

### 4. 常见误区与注意事项

#### 4.1 重执行陷阱检查清单

| 检查项 | 危险信号 | 修复方案 |
|--------|---------|---------|
| 节点内有 API 调用 | `requests.post()` / `client.query()` | 拆分到独立节点 |
| 节点内有数据库写操作 | `db.insert()` | 拆分到独立节点，或用 state 传递 |
| 节点内有 LLM 调用 | `model.invoke()` | 拆分到独立节点 |
| 节点内有文件读写 | `open(...).write()` | 拆分到独立节点 |
| 节点内 interrupt() 前有逻辑 | 任何在 `interrupt()` 之前的代码 | 确保这些代码幂等，或拆分 |

#### 4.2 完整误区表

| 坑 | 现象 | 预警话术 | 层级 |
|----|------|---------|------|
| **忘记 checkpointer** | interrupt 后无法恢复，State 丢失 | "HITL **必须**配 checkpointer！没有持久化，中断后就丢失了。`interrupt()` 源码第一行检查就是 checkpointer。" | 所有 |
| **interrupt 和异常混用** | 用 raise Exception 代替 interrupt | "Exception 会终止执行，interrupt 是暂停。HITL 必须用 `interrupt()` 原语。" | Layer 3 |
| **resume 值格式不对** | 被中断节点收不到 resume 值 | "`interrupt()` 返回的是 `Any` 类型，但要和你的节点逻辑匹配。`Command(resume=...)` 的值就是 `interrupt()` 的返回值。" | 恢复 |
| **在工具里用 interrupt()** | 工具异常，图不暂停 | "工具函数里不能用 `interrupt()`，它不在 graph 的 config 上下文里。用 `interrupt_before=['tools']` 代替。" | Layer 2 |
| **interrupt_before 后期待 resume 给节点** | 节点参数收不到值 | "`interrupt_before` 的 resume 值是给节点内部的 `interrupt()` 消费的，不是传给节点函数的参数。" | Layer 1 |
| **Hooks 里修改 State** | Pre-hook 改了 State，影响后续节点 | "Pre-hook 应该用 `llm_input_messages` 临时注入，不要直接改 `messages`（除非你真的想持久化修改）。" | Hooks |
| **审批流设计太复杂** | 一个流程里中断 5 次，用户体验差 | "HITL 是『安全阀』不是『每一步都问』。只在高风险/高成本操作前中断。" | 设计 |
| **多层中断冲突** | `interrupt_before` 和 `interrupt()` 同时触发 | "一个节点可能同时命中 `interrupt_before` 和内部的 `interrupt()`。设计时确保逻辑不重叠。" | 设计 |

#### 4.3 调试技巧

1. **查看中断状态**：`graph.get_state(config)` 查看 `.next` 字段知道要执行哪个节点
2. **查看中断类型**：`snapshot = graph.get_state(config); print(snapshot.tasks)` 看 `interrupts` 字段
3. **模拟审批**：开发时用 `Command(resume=...)` 模拟不同审批结果
4. **检测重执行**：在节点内加 `print(f"[EXEC] node={name} step={step}")`，如果 resume 后看到重复打印，说明有重执行
5. **审计日志**：所有 HITL 操作记录到 Store，包含 `thread_id`, `node_name`, `interrupt_value`, `resume_value`, `timestamp`

### 5. 阶段小结

#### 核心要点回顾

1. **三层架构**：编译时配置（Layer 1）→ 预构建 Agent 配置（Layer 2）→ 节点内动态中断（Layer 3），粒度递增，风险递增
2. **`interrupt_before`**：节点**不执行**，resume 后**第一次执行**
3. **`interrupt_after`**：节点**已执行完**，resume 后直接流转到下一节点
4. **`interrupt()` 陷阱**：节点恢复时**从头重新执行**，`interrupt()` 前的代码会重跑
5. **拆分模式**：计算节点 + 中断节点分离，消除重执行风险
6. **`create_react_agent`**：透传 `interrupt_before/after` 到 compile，节点名固定为 `"agent"` / `"tools"`
7. **工具不能 `interrupt()`**：用 `interrupt_before=["tools"]` 或自定义条件路由
8. **`Command(resume)`**：单值/列表/字典三种形式，匹配 `interrupt()` 的调用顺序
9. **ToolInteractionEnvelope**：工具不直接中断，返回结构化信封让上层决策
10. **RenderSchema**：中断即 UI，标准字段类型驱动前端渲染
11. **Resume 协议**：`normalize` + `build` 统一恢复接口，支持 edit/revise/regenerate
12. **工具包装器**：Copilot/Interactive 双模式，state.auto 动态切换

#### 记忆口诀

> **"compile 配 before/after，不动代码就生效；**
> **节点内写 interrupt，千万要拆分防重跑；**
> **Agent 工具要审批，tools 节点暂停好；**
> **resume 是 playback，state 已在 checkpoint 存好。"**
