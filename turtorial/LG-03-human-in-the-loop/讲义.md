# LG-03: Human-in-the-loop 人工介入

## 1. 这一节到底要解决什么问题

Human-in-the-loop（HITL）不是“在代码里加一个暂停按钮”。它解决的是：图已经自动推进到某个关键业务节点，但这个节点的后果不能完全交给模型或规则引擎决定。

以内容发布为例：

```text
运营请求 → 生成文案 → 自动审核 → 是否需要人工判断 → 发布 / 拒绝 / 修改后发布
```

如果没有中断，流程会变成：

```text
模型生成了内容 → 系统算出风险 → 仍然继续发布
```

这就是风险点。系统“知道有风险”和系统“能在风险动作前停住”是两回事。HITL 要补上的正是后者。

## 2. 为什么业务上需要中断

中断通常出现在三个地方：

1. **结果有外部影响**：发布内容、发短信、扣款、下单、调用真实工具。
2. **自动判断不够可靠**：模型可能误判，规则也可能只覆盖一部分情况。
3. **责任需要落到人身上**：审批、合规、客服回复、合同修改，都需要记录是谁做的决定。

所以中断不是为了让流程慢下来，而是为了把“不可逆动作”变成“可审查动作”。

业务上可以这样理解：

```text
普通自动化：系统自己决定下一步
HITL 自动化：系统先准备好上下文，再把关键选择权交给人
```

## 3. 中断后用户可以做什么

中断后，图不会继续向下执行。它会返回一个 `__interrupt__`，里面应该包含人工决策需要看的上下文。

在本节案例里，中断 payload 会包含：

```json
{
  "action": "content_review_required",
  "request_id": "campaign_high_001",
  "channel": "公众号",
  "draft": "618 智能空调全网最低...",
  "risk_score": 0.9,
  "risk_level": "high",
  "matched_rules": [...],
  "options": ["approve", "edit_and_approve", "reject"]
}
```

用户拿到这些信息后，常见动作有三类：

| 用户动作 | 回复内容 | 业务含义 |
|---|---|---|
| 通过 | `decision=approve` | 原文可以继续发布 |
| 修改后通过 | `decision=edit_and_approve` + `revised_draft` | 用人工修改后的内容发布 |
| 拒绝 | `decision=reject` | 流程结束，不发布 |

也就是说，中断不是只问一句“继续吗”。中断应该把“人能做什么”和“每个选择会造成什么结果”说清楚。

## 4. 用户怎么回复

在 LangGraph 里，用户不是直接改图，也不是重新从头调用一次普通输入。恢复中断需要传入：

```python
Command(resume={
    "decision": "edit_and_approve",
    "reviewer_id": "reviewer_demo_001",
    "comment": "删除绝对化和收益承诺表达后发布。",
    "revised_draft": "618 智能空调活动开启，限时下单享专属优惠，具体权益以活动页为准。",
})
```

关键点：`resume` 里的值会成为暂停位置里 `interrupt(...)` 的返回值。

所以节点里的业务含义是：

```text
第一次执行到 interrupt(payload)：暂停，把 payload 交出去
恢复时收到 Command(resume=value)：interrupt(...) 这行得到 value，然后节点继续跑
```

## 5. 回复后如何影响 router 走向

用户的回复最终会进入 state。router 不应该直接看“用户原始输入”，而应该看整理后的业务状态。

本节会把人工回复整理成：

```json
{
  "review_decision": "edit_and_approve",
  "reviewer_id": "reviewer_demo_001",
  "reviewer_comment": "删除绝对化和收益承诺表达后发布。",
  "revised_draft": "..."
}
```

然后 router 根据 `review_decision` 决定下一步：

```text
review_decision = approve
  → publish_content

review_decision = edit_and_approve
  → publish_content，发布 revised_draft

review_decision = reject
  → reject_content，结束，不发布
```

这也是 HITL 的重点：人不是在流程外“点了个按钮”，而是把一个新的业务事实写回 state，后续路径根据这个事实变化。

## 6. 本节图结构

最终审批图会这样组织：

```text
START
  → load_request
  → auto_review
  → human_review
  → route_after_review
      ├─ publish_content
      └─ reject_content
  → END
```

其中：

- `load_request`：读取发布请求。
- `auto_review`：计算风险评分和命中规则。
- `human_review`：低风险自动通过，高风险触发 `interrupt()`。
- `route_after_review`：根据人工决策决定发布还是拒绝。
- `publish_content`：发布原文或人工修改后的内容。
- `reject_content`：记录拒绝结果。

## 7. 为什么中断必须配 checkpointer

中断不是普通函数暂停。真实业务里，中断之后可能发生这些事：

- 用户几分钟后才审批。
- 审批页面刷新了。
- 服务进程重启了。
- 同时有很多个审批流程在等待。
- 同一个用户可能有多个不同 thread 在暂停。

这里的“保存现场”可以先按生产系统来理解：任务跑到一半要等人时，系统通常不会让请求、线程或进程一直挂在那里。审批可能很快回来，也可能第二天才回来；如果每个待审批任务都占着线程、连接、内存和运行上下文，任务一多机器就会被拖垮。更合理的做法是先把这次运行挂起，只留下待办和 checkpoint。人处理完以后，再从 checkpoint 里恢复。

LangGraph 必须知道：

```text
哪个 thread 停住了？
停在哪个节点？
当时的 state 是什么？
恢复时应该把 resume 值交还给哪一次 interrupt？
恢复后应该从哪里继续？
```

这些信息不能只放在 Python 调用栈里，因为调用栈在请求结束后就没了。LangGraph 通过 `checkpointer` 保存 checkpoint，才能让中断变成可恢复的长期状态。

所以，只要你要使用中断恢复，就需要在 `compile(...)` 时配置 checkpointer。

本节先用内存版：

```python
review_graph = builder.compile(checkpointer=MemorySaver())
```

内存版适合教学和单进程演示；生产环境通常会换成 SQLite、Postgres、Redis 等持久化 checkpointer。区别不是 API 语义，而是保存位置和可靠性。

| checkpointer | 适合场景 | 风险 |
|---|---|---|
| `MemorySaver` | Notebook、单进程 demo、单元测试 | 进程重启后暂停状态丢失 |
| SQLite | 本地服务、小型应用、可恢复 demo | 并发和运维能力有限 |
| Postgres / Redis | 生产服务、多实例、长期等待审批 | 需要额外基础设施 |

## 8. 执行前中断和执行后中断的区别

LangGraph 还有声明式中断：`interrupt_before` 和 `interrupt_after`。它们不是在节点里根据业务条件动态判断，而是在编译图时指定固定暂停点。

### 8.1 执行前中断：`interrupt_before`

含义：节点还没执行，先暂停。

```text
auto_review 完成
  → 准备进入 publish_content
  → interrupt_before 命中
  → publish_content 还没有执行
```

适合场景：

- 发布前确认。
- 扣款前确认。
- 发短信前确认。
- 调用真实外部 API 前确认。
- 删除数据前确认。

它的价值是“挡住副作用”。只要没有恢复，写操作就不会发生。

### 8.2 执行后中断：`interrupt_after`

含义：节点已经执行完，输出已经写入 state，然后暂停。

```text
generate_draft 执行完成
  → 生成内容已经写入 state
  → interrupt_after 命中
  → 等人审查这个生成结果
```

适合场景：

- 文案生成后，让人看一眼。
- 工具查询后，让人确认结果是否可信。
- 检索完成后，让人选择要采用哪条证据。
- 计划生成后，让人确认是否进入执行阶段。

它的价值是“审查产物”。节点已经跑完了，人看的就是节点结果。

### 8.3 二者对比

| 类型 | 暂停时机 | 节点是否已执行 | 恢复后发生什么 | 更适合 |
|---|---|---|---|---|
| `interrupt_before` | 节点执行前 | 否 | 节点第一次执行 | 拦截副作用 |
| `interrupt_after` | 节点执行后 | 是 | 进入后续节点 | 审查结果 |
| 节点内 `interrupt()` | 节点内部任意位置 | 取决于写法 | 从同一节点继续 | 动态业务审批 |

## 9. 三种中断方式怎么选

选择逻辑可以很简单：

```text
需要根据运行时数据决定是否暂停？
  是 → 用节点内 interrupt()
  否 → 只是固定节点前后停一下？
      节点前有副作用风险 → interrupt_before
      节点后要审查结果 → interrupt_after
```

本节主案例使用节点内 `interrupt()`，因为是否暂停取决于风险评分。声明式中断用来补充演示“发布节点执行前如何被拦住”。

## 10. 朴素实现会暴露什么问题

朴素实现会读取请求、计算风险并直接发布。运行后你会看到：

```text
publish_status: published
risk_score: 0.9
risk_level: high
```

这说明它虽然算出了高风险，但没有把风险变成流程控制。

这句话很关键：

```text
风险识别不等于风险拦截。
```

HITL 的作用就是把识别结果接到流程暂停、人工输入和路由选择上。

## 11. Notebook 要观察的证据

跑通 Notebook 时，至少要看到这些证据：

1. 高风险第一次执行返回 `__interrupt__`。
2. `get_state(config).next` 显示图停在 `human_review`。
3. `Command(resume=edit_and_approve)` 后，发布的是 `revised_draft`。
4. `Command(resume=reject)` 后，流程进入拒绝分支，不发布。
5. 低风险内容不会中断，会自动发布。
6. `interrupt_before` 能在发布节点执行前拦住，恢复前没有 `publish_status`。
7. `interrupt_after` 会在生成节点执行后暂停，恢复后继续向后走。
