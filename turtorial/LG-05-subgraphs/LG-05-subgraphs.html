Fan-out（并行分发）

一个节点同时触发多个下游节点

类比：项目经理同时给 4 个人发邮件分配任务

builder.add_edge("parse", "finance")
builder.add_edge("parse", "news")
builder.add_edge("parse", "competitor")
builder.add_edge("parse", "expert")

Fan-in（自动汇聚）

多个节点都完成后，才触发下一个节点

类比：Excel 条件格式——等 4 个人都回复了，这行才变绿

builder.add_edge("finance", "compile")
builder.add_edge("news", "compile")
builder.add_edge("competitor", "compile")
builder.add_edge("expert", "compile")

✓ LangGraph 自动处理 Fan-in，无需额外配置

LangGraph 渐进式教程LG-05
Fan-out 是并行分发，Fan-in 是汇聚等待。LangGraph 的自动 Fan-in：多个上游边指向同一个节点即自动汇聚。这是并行执行的基础模式。
REDUCER

Reducer：并行的「和平协议」

问题：没有 Reducer 会怎样？

4 个并行节点都往 findings 里写数据，没有 Reducer → 写覆盖

最后一个完成的节点把前面节点的数据全盖掉了

内置 Reducer：operator.add

from operator import add

class AgentState(TypedDict):
    findings: Annotated[list, add]
    # 多个并行节点的结果自动合并为一个列表

自定义 Reducer

# 保留最近 10 条
logs: Annotated[list, lambda x, y: (x + y)[-10:]]

# 去重合并
unique: Annotated[list, lambda x, y: list(dict.fromkeys(x + y))]

# 数值累加
count: Annotated[int, lambda x, y: x + y]

关键话术

"Reducer 是并行的和平协议——规定大家怎么合并结果。没有它，并行节点就是竞态条件。"

LangGraph 渐进式教程LG-05
Reducer 是并行执行的核心机制。多个节点同时修改同一个 State 字段时，Reducer 定义合并策略。operator.add 是最常用的列表累加 Reducer。
DYNAMIC DISPATCH

Send：动态 Map-Reduce

Map-Reduce 流程

输入 companies=[腾讯,阿里,字节]
↓ dispatch 节点
子图:腾讯
子图:阿里
子图:字节
↓ Fan-in 汇聚
collect 节点 → reports=[...]

关键点

  • Send 的第二个参数是子图的初始 State
  • 子图 State 独立于父图
  • 所有子图完成后自动触发 Fan-in

Send API 代码

from langgraph.types import Send

def dispatch(state: ParentState):
    return [
        Send("analyze_company", {"company": c})
        for c in state["companies"]
    ]

# 条件边：dispatch 返回 Send 列表
parent_builder.add_conditional_edges(
    START, dispatch, ["analyze_company"]
)

⚠️ Send 分发太多子任务会耗尽内存/连接池，需做上限控制

LangGraph 渐进式教程LG-05
Send 实现动态 Map-Reduce：运行时决定分发多少子任务。每个 Send 触发一次子图执行，子图独立运行，结果通过 Reducer 合并。注意子图初始 State 是 Send 的参数，不是父图 State。
SUBGRAPH

Subgraph：模块化复用

为什么需要子图？

  • 模块化：复杂系统拆分为独立模块
  • 复用：同一子图嵌入多个父图
  • 隔离复杂度：内部细节对外隐藏
  • 可测试：子图可独立运行调试

子图定义与嵌入

# 1. 定义子图
child_builder = StateGraph(ChildState)
child_builder.add_node("fetch", fetch_data)
child_builder.add_node("summarize", summarize)
child_builder.add_edge(START, "fetch")
child_builder.add_edge("fetch", "summarize")
child_graph = child_builder.compile()

# 2. 嵌入父图
parent_builder.add_node("research", child_graph)

子图核心要点

  • 子图是独立的 StateGraph，有完整的 State/Node/Edge
  • 必须先 .compile()，再传给父图 add_node
  • 子图可单独测试：child_graph.invoke({...})
  • 子图默认继承父图的 checkpointer

子图最佳实践 Checklist

  • 先独立测试子图，确保自身逻辑正确
  • 再嵌入父图，先不加映射，确认能跑通
  • 配置 input/output，处理字段名不一致
  • 优先用公共字段，减少映射函数
  • 子图 State 要最小化，不复制整个父图
LangGraph 渐进式教程LG-05
子图是独立的 StateGraph，编译后作为节点嵌入父图。核心优势是模块化、复用和状态隔离。最佳实践：先独立测试，再嵌入，逐步加映射。
STATE MAPPING

父子图状态隔离与映射

默认行为：状态隔离

父图 State
{topic, findings}
↔
子图 State
{query, raw_data, summary}

✓ 封装性、可复用性、可测试性

input / output 映射

parent_builder.add_node(
    "research",
    child_graph,
    input=lambda s: {"query": s["topic"]},
    output=lambda s: {"findings": s["results"]},
)

input = 翻译器（父图→子图），output = 翻译器（子图→父图）

公共字段模式

class SharedState(TypedDict):
    messages: list
    user_id: str

# 同名字段自动透传，无需映射

⚠️ 风险：子图可能意外修改父图字段 → 建议只添加/追加，不删除

跨子图通信三种方式

方式场景
公共字段父子图有相同字段名
input/output 映射字段名不同，需格式转换
Store（长期记忆）多个子图共享数据
LangGraph 渐进式教程LG-05
子图默认状态隔离。父子图字段名不一致时，通过 input/output 映射配置传递规则。公共字段模式简单但有修改风险。Store 是最解耦的跨子图通信方式。
CASE STUDY

ResearchForge：并行研究系统

概念对应

概念ResearchForge 体现
Fan-out主题输入后并行分发给 4 个数据源
Fan-in4 个数据源完成后进入汇总
Reducerfindings 用 add 合并
Send动态分发子查询到搜索节点
Subgraph每个数据源是独立子图
input/outputtopic → query, results → findings

关键转折点

  • 不用 Reducer → 4 个并行节点互相覆盖
  • 4 个并行节点中 1 个失败 → Fan-in 默认等待所有节点
  • 子图 vs 普通节点 → 模块化、复用、状态隔离

常见误区

  • 并行节点修改同一非 Reducer 字段 → 竞态条件
  • Fan-in 前置有循环 → 可能永远等不到
  • 子图 State 与父图不兼容 → 字段映射错误
  • Send 分发太多 → 内存/连接池耗尽
  • 递归限制不当 → 并行+循环触发上限

调试技巧

  • 先串行跑通每个节点，再改并行
  • 打印合并前后的 State，确认无数据丢失
  • 每个子图作为独立图先跑通，再嵌入
LangGraph 渐进式教程LG-05
ResearchForge 是并行研究系统的完整实战。4 个数据源并行查询，Reducer 合并结果，Fan-in 等待汇总。每个数据源可以是独立子图，通过 input/output 映射与父图通信。
SUMMARY

核心要点回顾

并行执行模式

  • Fan-out：一个节点触发多个下游同时执行
  • Fan-in：多个节点完成后才触发下一个
  • Reducer：定义并行修改同一字段的合并策略
  • Send：运行时动态分发子任务

子图模块化

  • 子图是独立的 StateGraph，编译后嵌入父图
  • 状态默认隔离，通过 input/output 映射通信
  • 公共字段自动透传，Store 跨子图共享
  • 子图继承父图 checkpointer

记忆口诀

"Fan-out 是群发，Fan-in 是等齐人；Reducer 是合并器，Send 是动态派活，Subgraph 是模块包。"
