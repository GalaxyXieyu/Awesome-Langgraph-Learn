## 并行执行与子图

---

### 1. 开场引入

#### 引入策略：从「等待焦虑」引入

**开场白建议**：
> "假设你要写一份行业研究报告。需要做四件事：查财报数据、查新闻舆情、查竞品动态、查专家观点。如果你一个人做，要一件一件来——做完第一件再做第二件，总时间是 4 个任务之和。
>
> 但如果你有一个团队：你负责统筹，4 个研究员同时分头查，你在汇总点等他们都完成后整合报告。总时间就是最慢的那个研究员的时间。
>
> 这就是并行执行的意义。今天的主题是：**让图学会『分工合作』**。"

#### 生活化类比：餐厅后厨的并行出餐

> "你去 restaurant 点了一份套餐：主菜、配菜、汤、饮料。后厨不会等主菜做完了才做配菜——四个厨师同时开工，各自负责一道，然后在『出餐台』汇聚成一份完整的套餐端给你。
>
> 并行节点就是这四个厨师，Fan-in 汇聚点就是出餐台。"

#### 与 LG-04 的衔接

> "上节课我们学会了在关键节点让人类审批。今天我们要加速图的执行——让独立的任务并行跑，让复杂的系统模块化拆分。"

### 2. 核心内容

#### 2.1 Fan-out / Fan-in

**讲解顺序**：
1. **Fan-out**：一个节点同时触发多个下游节点
2. **Fan-in**：多个节点都完成后，汇聚到一个节点
3. **LangGraph 的自动 Fan-in**：多个上游边指向同一个节点即自动汇聚

**必须画图的点**：
```
┌─→ [查财报] ──┐
[主题输入] ─┼─→ [查新闻] ──┼─→ [汇总报告] → END
├─→ [查竞品] ──┤
└─→ [查专家] ──┘
↑ Fan-out ↑ Fan-in
```

**类比**："Fan-out 像是一个项目经理同时给 4 个人发邮件分配任务；Fan-in 像是项目经理在 Excel 里设置了条件格式——等 4 个人都回复了，这行才变绿。"

---

#### 2.2 Reducer 机制

**讲解顺序**：
1. **为什么需要 Reducer**：多个节点同时修改同一个 State 字段
2. **`operator.add`**：列表/数值累加
3. **自定义 Reducer**：去重、保留最近 N 个、排序合并

**代码演示**：
```python
from operator import add

class AgentState(TypedDict):
    # 多个并行节点都往 findings 里加内容，自动合并
findings: Annotated[list, add]
    # 自定义 reducer：保留最近的 10 条
logs: Annotated[list, lambda x, y: (x + y)[-10:]]
```

**关键话术**：
> "并行节点都往同一个字段写数据时，如果没有 Reducer，就会『写覆盖』——最后一个完成的节点把前面节点的数据全盖掉了。Reducer 是并行的『和平协议』——规定大家怎么合并结果。"

---

#### 2.3 `Send` 动态 Map-Reduce

**讲解顺序**：
1. **什么是 Map-Reduce**：动态分发子任务 → 并行执行 → 汇聚结果
2. **`Send` API**：`Send("target_node", {"key": value})`
3. **动态分发**：运行时决定分发几个子任务

**类比**："Map-Reduce 像是一个老师批改作文——有 50 篇作文，分给 5 个助教，每人 10 篇，汇总成绩。`Send` 就是『分发』这个动作。"

**完整代码示例——Send + Subgraph 组合**：

```python
from typing import TypedDict, Annotated
from operator import add
from langgraph.graph import StateGraph, START, END
from langgraph.types import Send

# ========== 1. 定义子图（每个公司的分析子图）==========
class CompanyState(TypedDict):
company: str
analysis: str

def analyze_company(state: CompanyState):
"""分析单个公司。"""
return {"analysis": f"{state['company']} 的分析报告..."}

child_builder = StateGraph(CompanyState)
child_builder.add_node("analyze", analyze_company)
child_builder.add_edge(START, "analyze")
child_builder.add_edge("analyze", END)
company_graph = child_builder.compile()

# ========== 2. 定义父图 ==========
class ParentState(TypedDict):
companies: list[str]
reports: Annotated[list[str], add] # Reducer 合并所有子图结果

def dispatch(state: ParentState):
"""
Map 阶段：动态分发每个公司到子图。
返回 Send 列表，每个 Send 触发一次子图执行。
"""
return [
Send("analyze_company", {"company": c})
for c in state["companies"]
]

def collect(state: ParentState):
"""Reduce 阶段：所有子图完成后汇聚。"""
return {"reports": state["reports"]}

parent_builder = StateGraph(ParentState)
parent_builder.add_node("dispatch", dispatch)
# 关键：子图作为节点，可以被 Send 动态触发
parent_builder.add_node("analyze_company", company_graph)
parent_builder.add_node("collect", collect)

# 条件边：dispatch 返回 Send 列表，动态分发
parent_builder.add_conditional_edges(START, dispatch, ["analyze_company"])
# Fan-in：所有 analyze_company 完成后汇聚到 collect
parent_builder.add_edge("analyze_company", "collect")
parent_builder.add_edge("collect", END)

parent_graph = parent_builder.compile()

# ========== 3. 执行 ==========
result = parent_graph.invoke({"companies": ["腾讯", "阿里", "字节"]})
print(result["reports"])
# ['腾讯 的分析报告...', '阿里 的分析报告...', '字节 的分析报告...']
```

**Send + Subgraph 的执行流程**：

```
[输入: companies=[腾讯, 阿里, 字节]]
↓
[dispatch 节点]
↓
Send("analyze_company", {company: "腾讯"})
Send("analyze_company", {company: "阿里"}) ← 并行分发
Send("analyze_company", {company: "字节"})
↓
┌─────────────────────────────────────┐
↓ ↓ ↓
[子图:腾讯] [子图:阿里] [子图:字节]
↓ ↓ ↓
└─────────────────────────────────────┘
↓
Fan-in 汇聚
↓
[collect 节点]
↓
[输出: reports=[...]]
```

**关键点**：
- `Send` 的第二个参数是**子图的初始 State**，不是父图的 State
- 子图执行时，State 从 Send 的参数开始，独立于父图
- 子图的结果通过 `output` 映射（或公共字段）写回父图
- 所有子图实例完成后，LangGraph 自动触发 Fan-in

#### 2.4 Subgraph 构建

**讲解顺序**：
1. **为什么需要子图**：模块化、复用、隔离复杂度
2. **子图的定义**：独立的 `StateGraph`，有自己的 State Schema
3. **子图作为节点**：`builder.add_node("subgraph", subgraph.compile())`
4. **子图的 Checkpointer 继承**：子图自动复用父图的持久化配置

**完整代码示例——从定义到嵌入**：

```python
from typing import TypedDict, Annotated
from operator import add
from langgraph.graph import StateGraph, START, END

# ========== 1. 定义子图 ==========
class ChildState(TypedDict):
"""子图自己的 State——完全独立定义。"""
query: str
raw_data: list
summary: str

def fetch_data(state: ChildState):
"""模拟数据获取。"""
return {"raw_data": [f"data for {state['query']}"]}

def summarize(state: ChildState):
return {"summary": f"Summary of {len(state['raw_data'])} items"}

child_builder = StateGraph(ChildState)
child_builder.add_node("fetch", fetch_data)
child_builder.add_node("summarize", summarize)
child_builder.add_edge(START, "fetch")
child_builder.add_edge("fetch", "summarize")
child_builder.add_edge("summarize", END)

# 编译子图
child_graph = child_builder.compile()

# ========== 2. 定义父图 ==========
class ParentState(TypedDict):
topic: str
findings: Annotated[list, add] # Reducer 自动合并

def parse_topic(state: ParentState):
return {"topic": state["topic"].strip()}

def compile_report(state: ParentState):
return {"findings": [f"Final report on {state['topic']}"]}

parent_builder = StateGraph(ParentState)
parent_builder.add_node("parse", parse_topic)
# 关键：把编译后的子图作为节点嵌入父图
parent_builder.add_node("research", child_graph)
parent_builder.add_node("compile", compile_report)

parent_builder.add_edge(START, "parse")
parent_builder.add_edge("parse", "research")
parent_builder.add_edge("research", "compile")
parent_builder.add_edge("compile", END)

parent_graph = parent_builder.compile()

# ========== 3. 执行 ==========
result = parent_graph.invoke({"topic": "AI 芯片"})
print(result)
# {'topic': 'AI 芯片', 'findings': ['Final report on AI 芯片']}
```

**核心要点**：
- 子图是一个**独立的 `StateGraph`**，有自己完整的 State、Node、Edge 定义
- 子图必须先 `.compile()`，然后作为 action 传给父图的 `add_node`
- 子图可以单独测试：`child_graph.invoke({"query": "test"})`
- 子图默认**继承父图的 checkpointer**（如果父图配置了的话）

#### 2.5 子图状态隔离与映射

##### 2.5.1 状态隔离：子图是「黑盒」

**默认行为**：子图的 State 与父图完全隔离。

```
父图 State: {topic, findings} 子图 State: {query, raw_data, summary}
↑ ↑
父图可见 子图内部可见
子图不可见 父图不可见
```

**隔离的好处**：
1. **封装性**：子图内部的中间状态（`raw_data`）不会污染父图
2. **可复用性**：同一个子图可以嵌入多个不同的父图
3. **可测试性**：子图可以独立运行和调试

**隔离的代价**：
- 父子图字段名不一致时，数据不会自动传递
- 必须显式配置映射关系

---

##### 2.5.2 `input` / `output` 映射配置

当父子图的 State 字段名不一致时，需要通过 `input` 和 `output` 参数配置映射。

**API**：
```python
parent_builder.add_node(
"research", # 节点名
child_graph, # 编译后的子图
input=..., # 进入子图前的字段映射
output=..., # 子图返回后的字段映射
)
```

**完整示例——字段映射**：

```python
from typing import TypedDict
from langgraph.graph import StateGraph, START, END

# 父图 State
class ParentState(TypedDict):
topic: str # 父图叫 "topic"
findings: list # 父图叫 "findings"

# 子图 State（字段名完全不同）
class ChildState(TypedDict):
query: str # 子图叫 "query"
results: list # 子图叫 "results"

def child_fetch(state: ChildState):
return {"results": [f"Result for {state['query']}"]}

child_builder = StateGraph(ChildState)
child_builder.add_node("fetch", child_fetch)
child_builder.add_edge(START, "fetch")
child_builder.add_edge("fetch", END)
child_graph = child_builder.compile()

# 父图
parent_builder = StateGraph(ParentState)
parent_builder.add_node("research", child_graph)

# 配置 input/output 映射
# input:  父图 "topic" → 子图 "query"
# output: 子图 "results" → 父图 "findings"
parent_builder.add_node(
"research",
child_graph,
input=lambda state: {"query": state["topic"]},
output=lambda state: {"findings": state["results"]},
)

parent_builder.add_edge(START, "research")
parent_builder.add_edge("research", END)
```

**映射函数签名**：
```python
# input: 接收父图完整 state，返回子图需要的字段
input_mapper = lambda parent_state: {
"子图字段1": parent_state["父图字段A"],
"子图字段2": parent_state["父图字段B"],
}

# output: 接收子图完整 state，返回父图需要的字段
output_mapper = lambda child_state: {
"父图字段X": child_state["子图字段1"],
"父图字段Y": child_state["子图字段2"],
}
```

**映射的本质**：
- `input` 函数 = **「翻译器」**：把父图的数据格式翻译成子图能理解的格式
- `output` 函数 = **「翻译器」**：把子图的结果格式翻译回父图的数据格式

---

##### 2.5.3 公共字段模式：父子共享 State

当父子图有**相同字段名**时，不需要配置映射，字段自动透传。

```python
class SharedState(TypedDict):
messages: list # 父子图都有 "messages"
user_id: str # 父子图都有 "user_id"

def child_node(state: SharedState):
    # 子图可以直接读写父图的 "messages"
return {"messages": state["messages"] + [{"role": "child", "content": "..."}]}

# 无需 input/output 映射——"messages" 和 "user_id" 自动共享
parent_builder.add_node("child", child_graph)
```

**共享字段的注意事项**：
- 优点：简单，不需要写映射函数
- 风险：子图可能意外修改父图的关键字段 → 建议子图只添加/追加，不要删除

---

##### 2.5.4 跨子图通信的三种方式

| 方式 | 适用场景 | 代码示例 |
|------|---------|---------|
| **公共字段** | 父子图有相同字段名 | `messages` 在父子图中同名，自动透传 |
| **input/output 映射** | 字段名不同，需要格式转换 | `input=lambda s: {"query": s["topic"]}` |
| **Store（长期记忆）** | 多个子图需要共享数据 | `store.aput(user_id, "key", data)` |

**Store 跨子图通信示例**：

```python
from langgraph.store.base import BaseStore

def child_a(state, store: BaseStore):
    # 子图 A 写入共享数据
store.aput("shared", "intermediate_result", {"data": "..."})
return {"status": "done"}

def child_b(state, store: BaseStore):
    # 子图 B 读取共享数据
result = store.aget("shared", "intermediate_result")
return {"output": result}
```

> Store 是 LG-04 的内容。如果子图间需要复杂的数据交换，Store 是最解耦的方式。

#### 2.6 复杂 state 组合策略

##### 2.6.1 嵌套子图的状态扁平化

当子图嵌套多层时（父 → 子 → 孙），状态可能变得非常深。推荐做法：

```python
# 不推荐：深层嵌套状态
class DeepState(TypedDict):
level1: {"level2": {"level3": {"data": ...}}}

# 推荐：扁平化
class FlatState(TypedDict):
data: ...
meta: ...
```

**扁平化原则**：
- 子图的 State 字段尽量与父图保持同一层级
- 不要用嵌套 dict 表达层级关系，用字段前缀区分

```python
# 推荐
class State(TypedDict):
news_results: list # 新闻子图的结果
finance_results: list # 财经子图的结果
final_report: str # 汇总结果
```

---

##### 2.6.2 状态聚合模式

多个子图并行执行后，父图需要聚合它们的结果。常用模式：

```python
from operator import add

class ParentState(TypedDict):
    # Reducer 自动聚合多个子图的结果
findings: Annotated[list, add]

def aggregate(state: ParentState):
    # 所有子图完成后，对 findings 做最终处理
return {"final_report": f"基于 {len(state['findings'])} 条发现生成报告"}
```

---

##### 2.6.3 子图最佳实践 checklist

1. **先独立测试子图**：`subgraph.invoke({...})` 确保子图自身逻辑正确
2. **再嵌入父图**：逐步集成，先不加映射，确认能跑通
3. **加映射**：配置 input/output，处理字段名不一致的情况
4. **优先用公共字段**：如果父子图可以用相同字段名，就不要写映射函数
5. **子图 State 要最小化**：只放子图真正需要的字段，不要复制整个父图 State

### 3. ResearchForge：并行研究系统实战

#### 案例与概念的一一对应

| 概念 | ResearchForge 中的体现 |
|------|----------------------|
| Fan-out | 主题输入后并行分发给 4 个数据源 |
| Fan-in | 4 个数据源都完成后才进入汇总节点 |
| Reducer | 各分支的 findings 用 `add` 合并 |
| Send | 动态分发多个子查询到搜索节点 |
| Subgraph | 每个数据源是一个独立子图（财经子图、新闻子图等） |
| input/output 映射 | 父图的 "topic" → 子图的 "query" |
| 状态隔离 | 各子图有自己的中间状态，互不干扰 |

#### 代码演示的讲解话术

**Step 1 - Fan-out 并行检索**：
> "看这张图——输入主题『新能源汽车』后，4 个箭头同时亮起来。这就是 Fan-out。`parse_intent` 节点有 4 条出边，分别指向 4 个数据源节点。LangGraph 会自动并行执行这 4 个节点。"

**Step 2 - Reducer 合并结果**：
> "4 个节点并行执行，每个节点都会往 `findings` 里加数据。注意 State 定义——`findings: Annotated[list, add]`。`add` Reducer 保证了 4 个节点的结果不会互相覆盖，而是自动合并成一个列表。"

**Step 3 - Fan-in 汇聚**：
> "4 个节点都完成后，`compile_report` 节点才会执行。这就是 Fan-in——LangGraph 自动等待所有前置节点完成。注意看这张图，`compile_report` 的入度是 4，4 条边都『变绿』后它才开始。"

**Step 4 - 子图结构**：
> "点击这个『财经数据』节点，展开它的内部结构。大家看——它不是一个简单的函数，而是一个完整的子图！有自己的 State（`FinanceState`）、自己的节点（`fetch_stock` → `analyze_trend`）、自己的边。子图执行完后，通过 output 映射把 `results` 传回父图的 `findings`。"

**Step 5 - 动态 Map-Reduce**：
> "再看这个动态分发场景。系统分析了主题后，发现需要查 7 个公司的数据。用 `Send` 动态分发 7 个子任务：`[Send("fetch_company", {"company": c}) for c in companies]`。这 7 个任务并行执行，结果通过 Reducer 合并。"

#### 关键转折点（需要停下来确认学员理解）

1. **Reducer 必要性**：问 "如果 4 个并行节点都返回 `{"result": "..."}`，不用 Reducer 会怎样？"（期望回答：互相覆盖，只有一个结果保留）
2. **Fan-in 触发条件**：问 "如果 4 个并行节点中有一个失败了，Fan-in 会触发吗？"（期望回答：LangGraph 默认等待所有节点，失败可以用 try/except 或特殊 Reducer 处理）
3. **子图 vs 普通节点**：问 "子图和普通函数节点相比，优势是什么？"（期望回答：模块化、复用、状态隔离、内部复杂度隐藏）

### 4. 常见误区与注意事项

| 坑 | 现象 | 预警话术 |
|----|------|---------|
| **并行节点修改同一个非 Reducer 字段** | 数据互相覆盖 | "并行节点共享的字段必须用 Annotated + Reducer，否则是竞态条件。" |
| **Fan-in 节点前置有循环** | Fan-in 触发条件不明确 | "如果并行分支里有循环，确保循环有明确终止条件，否则 Fan-in 永远等不到。" |
| **子图 State 与父图不兼容** | 字段映射错误，数据传不过去 | "子图的 input/output 映射必须显式配置，字段名不一致时不会自动匹配。" |
| **Send 分发太多子任务** | 内存/连接池耗尽 | "Send 是动态分发，要做上限控制。`max_concurrency` 限制同时执行的子任务数。" |
| **递归限制不当** | 并行 + 循环导致超级步过多 | "并行图更容易触达 recursion_limit，生产环境要合理设置。" |

#### 调试技巧

1. **逐个节点验证**：先不并行，串行跑通每个节点，再改成并行
2. **Reducer 验证**：打印合并前后的 State，确认没有数据丢失
3. **子图单独测试**：每个子图作为独立图先跑通，再嵌入父图

### 5. 阶段小结

#### 核心要点回顾

1. **Fan-out = 并行分发**：一个节点触发多个下游节点同时执行
2. **Fan-in = 汇聚等待**：多个节点都完成后才触发下一个节点
3. **Reducer = 并行和平协议**：定义多个节点同时修改同一字段时的合并策略
4. **Send = 动态 Map-Reduce**：运行时决定分发多少子任务
5. **Subgraph = 模块化**：独立 State、独立逻辑、可复用、可嵌套
6. **input/output 映射 = 父子通信**：显式配置字段传递规则

#### 记忆口诀

> **"Fan-out 是群发，Fan-in 是等齐人；Reducer 是合并器，Send 是动态派活，Subgraph 是模块包。"**
