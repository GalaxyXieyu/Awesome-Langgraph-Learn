# LG-05：复杂任务的分层解构

> 核心冲突句：**人类会自然把复杂任务拆成层次，但图不会；图只知道哪些职责被拆成节点、哪些重复工作被批量执行、哪些工具能一起调用、哪些内部流程要封装成子图。**

这节课不要从 `Fan-out`、`Send`、`Subgraph` 开始，而是从一个真实 Agent 产品场景开始：**ResearchForge 行业研究助手**。

> 用户只给一句话：“请快速判断新能源汽车供应链公司 Q2 是否值得继续跟踪。”
>
> 如果只有一个大 Agent 串行做，它会遇到四个压力：速度慢、重复等待多、工具能力选择写不清、结构越来越难扩展。
>
> 人类研究团队会自然拆开：财经研究员查数据，新闻研究员看舆情，竞品研究员做对比，专家研究员整理观点。继续往下看，每个研究员内部也会拆：财经研究员可能同时处理多段材料；Agent 助理可能一次调用搜索、数据库、计算器；财经研究这件事本身还可以封装成一个小组流程。
>
> 所以今天这节课不是只讲“怎么并行更快”，而是讲：**复杂任务应该在哪一层拆开，拆开后如何并行、合并、复用和扩展**。拆得好的结果，是把能力原子化：工具可以被不同 Agent 复用，子图可以被不同父图复用。

本节 notebook 必须用真实 LLM 跑通：串行研究、并行研究、节点内部摘要、工具调用规划、动态文档摘要、财经子图总结都由模型生成结构化 JSON。工具函数可以是课堂里的可控数据源，但“研究判断”和“工具调用计划”不能写死。

---

## 0. 开篇：为什么会需要并行、tools、subgraph / subagent

### 要讲清楚的问题

学生容易把“并行”理解成一个词：慢了就并行，复杂了就子图，任务多了就 Send。这样会把不同层级的机制混在一起。

ResearchForge 同时遇到四类产品压力：

| 压力 | 如果不拆会怎样 | 需要的机制 | 课堂证据 |
|---|---|---|---|
| 速度 | 财务、新闻、竞品、专家依次等，用户等很久 | node 并行 | 4 个 LLM 研究节点同时跑 |
| 效率 | 一个财经节点内部要处理多段材料，逐条等很慢 | 节点内部并发 | 一个节点内 `asyncio.gather` 多个 LLM 摘要 |
| 能力 | Agent 临场发现要搜索、查库、计算，固定边写不完 | tools 并行 | LLM 生成工具调用计划，再并发执行工具 |
| 结构与拓展 | 财经研究内部越来越复杂，父图越来越乱，能力也难复用 | subgraph；更进一步是 subagent | 财经研究封装成 3 节点 LLM 子图，可被其他父图复用 |

### 配图

使用 `images/decomposition-layers-cn.svg`。

### 推荐话术

> 今天不是把所有并行 API 讲完，而是训练一个判断：这个复杂任务应该拆成节点、留在节点内部、交给工具调用，还是升级成子图？判断对了，API 只是最后一步。

### 真实 LLM 要求

不要用硬编码“财经发现”“新闻发现”假装研究员。课堂中的研究发现、工具调用计划、文档摘要和子图总结都应该由真实 LLM 生成。可控工具可以返回课堂数据，但工具要由模型规划调用，报告要由模型汇总。

## 1. 模型配置：本节用真实 LLM，不用假研究员

### 要讲清楚的问题

这节课讲的是 Agent / Graph 如何拆解真实 LLM 工作流。fake 研究员会让学生只看到机制玩具，看不到真实产品里的延迟、结构和工具调用问题。

### 推荐演示

显式读取：

```text
OPENAI_MODEL
OPENAI_BASE_URL
OPENAI_API_KEY
OPENAI_TEMPERATURE
```

缺配置时直接报错，不做 fake fallback。

### 教学要点

Prompt 要在 notebook 中展示：研究员 prompt、汇总 prompt、工具规划 prompt、文档摘要 prompt、财经子图 prompt 都是课程材料。

## 2. Baseline：一个 Agent 串行做完所有研究

### 人类动作

人类研究团队会自然分工。但如果我们只写一个大 Agent，它会依次做：财经 → 新闻 → 竞品 → 专家。

### Graph 的限制

Graph 不会自动知道这四件事互相独立。你不画出分支，它就只能按你写的顺序执行。

### 推荐演示

用真实 LLM 串行调用 4 次：

```text
财经研究员 LLM
新闻研究员 LLM
竞品研究员 LLM
专家观点研究员 LLM
```

每个研究员都返回 JSON：`role / finding / evidence / risk / next_question`。

### 证据输出

```text
串行 LLM 研究次数: 4
串行执行耗时: x.xx 秒
是否能看到 Graph 职责边界: 否 — 职责只藏在 for-loop 里
串行研究样例: {...}
```

### 教学要点

慢只是表层问题。更深的问题是：职责边界没有进入图结构，后面要加缓存、审批、替换模块都困难。

---

## 3. Node 并行：把独立职责拆成 Graph 节点

### 人类动作

项目经理把四个独立职责同时派出去：财经、新闻、竞品、专家。

### Graph 的限制

Graph 不会自动知道“这些节点可以同时跑”。你需要用多条边把它们显式画出来。

### API 映射

- 多条边从 `START` 指向多个 LLM 节点：Fan-out。
- 多个上游节点指向同一个汇总节点：Fan-in。
- `findings: Annotated[list[dict], add]` 合并所有研究发现。

### 配图

使用 `images/node-parallel-reducer-cn.svg`。

### 证据输出

```text
并行 LLM 研究节点数: 4
并行执行耗时: x.xx 秒
是否等齐四类发现后生成报告: 是
Reducer 是否保留全部发现: 是
汇总报告 JSON: {...}
```

### 教学要点

Node 并行不是只为了快。它让四个职责进入图结构：以后可以单独替换财经节点、给新闻节点加缓存、给专家节点加人工审批。

---

## 4. Reducer：并行 LLM 结果必须有合并协议

### 人类动作

四个研究员都会往同一个报告素材库里交材料。人类会自然把材料放进同一个文件夹，不会互相覆盖。

### Graph 的限制

多个并行节点如果更新同一个 state 字段，Graph 必须知道怎么合并这些更新。否则就会报错或丢结果。

### 推荐演示

1. 先构造两个并行节点同时写 `findings`，但不给 reducer。
2. 观察 `InvalidUpdateError` 一类错误。
3. 回到正确图，证明 `Annotated[list[dict], add]` 保留 4 个 LLM findings。

### 证据输出

```text
是否触发并行写入冲突: 是
错误类型: InvalidUpdateError
正确做法: findings: Annotated[list[dict], add]
是否有结果丢失: 否
```

### 教学要点

并行提速只是第一步；并行结果正确合并，图才能稳定工作。

---

## 5. 节点内部并发：同一职责里的多个 LLM 子任务不要打碎父图

### 人类动作

财经研究员负责“查财务”。他会同时处理收入、利润、现金流、费用多段材料。对项目经理来说，这仍然只是“财经研究”这一个职责。

### Graph 的限制

如果把每段材料都拆成父图节点，父图会被 I/O 和摘要细节淹没，职责边界反而不清楚。

### 推荐演示

在一个财经节点内部，用 `asyncio.gather` 并发调用真实 LLM，总结 4 段财经材料。

### 配图

使用 `images/inside-node-vs-graph-node-cn.svg`。

### 证据输出

```text
父图看到的财经节点数量: 1
财经节点内部 LLM 子任务数: 4
是否节点内部并发: 是 — asyncio.gather
返回摘要数: 4
财经内部摘要样例: {...}
```

### 教学要点

同一职责里的批量等待，留在节点内部并发；需要独立路由、审批、恢复，才升级成 Graph node。

## 6. Tools 并行：Agent 临场决定要调用多个外部能力

### 人类动作

主管问助理：“帮我查一下这家公司。”助理可能同时发起三件事：搜索新闻、查财务库、算增长率。等三条结果都回来，再统一写结论。

### Agent 的限制

工具并行不是开发者手写多条 graph edge，而是模型临场规划要调用哪些外部能力。工具的另一个价值是能力原子化：`search_news`、`query_finance`、`calculate_growth` 一旦做成工具，就可以被不同 Agent、不同图、不同课程案例复用。不同模型提供商对原生 `bind_tools` / `tool_calls` 的支持不完全一样，所以课堂里先让真实 LLM 输出明确的工具调用计划 JSON，再把它转换成工具执行。

### 和 Node 并行的区别

| 对比项 | 图节点并行 | 工具并行 |
|---|---|---|
| 谁决定并行任务 | 开发者通过边设计 | 模型规划要调用哪些工具 |
| 并行单位 | Graph node | Tool call / 工具调用计划 |
| 合并位置 | State + Reducer / Fan-in 节点 | ToolMessage / 工具结果 + 后续 Agent 节点 |
| 适合场景 | 固定职责、多源流程、可视化工作流 | Agent 临场决定需要哪些外部能力 |
| 风险边界 | 节点逻辑和 state 合并 | 工具副作用、权限、审批、安全边界 |

### 配图

使用 `images/tool-parallel-cn.svg`。

### 证据输出

```text
模型本轮规划工具调用数: 3
工具名称: search_news, query_finance, calculate_growth
工具结果返回数: 3
```

### 安全提醒

如果工具有副作用（发邮件、下单、写数据库、发布内容），不能因为它们可以并行就直接并行执行。工具并行要和 LG-03 的人工审批边界联动：危险工具需要 wrapper 级审批。

## 7. Send：运行时才知道要拆多少份

### 人类动作

今天要处理 3 篇文档，明天要处理 50 篇文档。人类会根据任务数量临时分派，而不是提前画死 50 条路线。

### Graph 的限制

静态 Fan-out 适合“固定几个职责”。如果任务数量运行时才知道，就需要动态分发。

### 推荐演示

读取 `map_reduce_documents.json`，取前 4 篇材料。`dispatch_documents` 根据运行时文档数返回 N 个 `Send`。每个 `summarize_document` 都调用真实 LLM 返回 JSON 摘要。

### 证据输出

```text
输入文档数: 4
Send 分发任务数: 4
LLM 摘要结果数: 4
最终摘要是否包含所有文档: 是
动态文档汇总结论: {...}
```

### 教学要点

`Send` 解决“数量运行时才知道”的问题，不是 Subgraph 的专属机制。目标可以是普通 node，也可以是 subgraph node。

## 8. Subgraph：一个职责内部也有稳定流程

### 人类动作

财经研究不是一个动作，它内部也有流程：整理材料、评估指标、写财经摘要。这个流程稳定、可复用，就像一个小研究组。

### Graph 的限制

如果把财经研究内部的所有节点都摊在父图里，父图会越来越大，中间状态也容易污染父图。

### 白话模型

```text
父图只看见：finance_analysis

finance_analysis 内部：
prepare_context → evaluate_metrics(LLM) → write_summary(LLM)
```

### 配图

使用 `images/subgraph-boundary-cn.svg`。

### 证据输出

```text
财经子图内部节点数: 3
子图是否可独立运行: 是
子图是否使用真实 LLM: 是 — evaluate_metrics / write_summary
子图输出摘要: {...}
```

### 教学要点

Subgraph 不是为了“看起来高级”，而是当一个职责内部已经有稳定多步流程时，用来封装复杂度、复用流程、隔离 state。它也是能力原子化：`finance_subgraph` 可以被研报父图、风控父图、投后跟踪父图重复使用。

## 9. 父子图状态边界：mapper 是翻译器

### 要讲清楚的问题

子图隔离不是障碍，而是好事。它让父图不用知道子图内部的 `snippets`、`metric_review` 等临时字段。

### 白话模型

```text
父图 topic → input mapper → 子图 query
子图 summary → output mapper → 父图 findings
```

### 证据输出

```text
父图字段: topic / findings / final_report
子图字段: query / snippets / metric_review / summary
input mapper 是否完成翻译: 是
output mapper 是否只返回父图需要的结果: 是
父图是否包含 metric_review: 否
```

### 教学要点

Mapper 的本质不是“复杂配置”，而是父图和子图边界处的数据翻译器。

## 10. Subgraph 和 Subagent 的边界

用户会自然追问：“既然都在拆复杂任务，那 subgraph 和 subagent 有什么区别？”

| 机制 | 白话理解 | 适用边界 |
|---|---|---|
| Subgraph | 把稳定流程封装成一张小图 | 内部步骤明确，状态和边由开发者设计 |
| Subagent | 把复杂职责交给一个专家 Agent | 需要独立模型、工具、记忆或自主循环 |

这节课讲 subgraph，因为我们还在学习单张图内部如何分层。下一节再进入更强自治程度的预构建 Agent / 多 Agent 模式。

## 11. ResearchForge 全景：不是为了并行而并行

| 压力 | 本节机制 | 这层解决什么 |
|---|---|---|
| 用户不想等 | node 并行 | 四个独立研究职责同时推进 |
| 节点内部还有很多等待 | 节点内部并发 | 同一职责里的多段材料并发处理 |
| Agent 临场需要外部能力 | tools 并行 | 模型一次规划多个工具调用 |
| 某个职责越来越复杂 | subgraph | 把稳定流程封装成可复用能力，并隔离 state |
| 输入数量运行时变化 | Send | 动态分发 N 个任务 |
| 多分支都写结果 | Reducer | 保证合并不丢、不冲突 |

一句话：**并行是表象，分层解构才是主线。**

## 12. 常见误区

| 误区 | 现象 | 正确说法 |
|---|---|---|
| 只把并行理解成提速 | 代码变快了，但结构仍然混乱 | 先判断复杂度来自哪一层 |
| 用硬编码函数替代 LLM 研究 | 学生只看到机制玩具，看不到 Agent 产品压力 | 本课研究、摘要、工具计划必须用真实 LLM |
| 把所有慢任务都拆成 graph node | 父图巨大，职责边界模糊 | 不同职责才拆 node |
| 把同一职责里的 URL 请求拆成很多节点 | I/O 细节污染父图 | 批量 I/O 留在节点内部并发 |
| 忘记 Reducer | 多个 LLM 节点写 `findings` 报错 | 并行写同一字段必须有合并协议 |
| 把 tools 并行写成固定 edge | 模型临场能力选择被写死 | 工具调用由模型规划 |
| 子图复制整个父图 state | 子图不可复用，字段互相污染 | 子图 state 最小化，边界处用 mapper |
| 把 subgraph 和 subagent 混为一谈 | 不知道什么时候需要 Agent 自治 | subgraph 是流程封装；subagent 是角色自治 |

## 13. 本节小结：先判断在哪一层拆

```text
不同职责 → node 并行
同一职责里的批量请求 → 节点内部 async 并发
模型临场选择多个外部能力 → tools 并行
一个职责内部有稳定多步流程 → subgraph
复杂职责需要自治模型和工具循环 → subagent（下一阶段）
多个并行结果写同一字段 → reducer
运行时才知道 N 个任务 → Send
```

学生应该能不用 API 名解释机制：

> ResearchForge 不是把一个大 Agent 写得更长，而是把复杂任务按层拆开：职责交给图节点，重复等待留在节点内部，外部能力原子化成 tools，稳定流程原子化成 subgraph。拆开以后，再显式设计等齐、合并、边界翻译和复用方式。
