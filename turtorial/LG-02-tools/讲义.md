## 2. Tools 深度掌握

### 2.1 Tool 的五种定义方式（含 LangGraph 运行时注入）

LangChain / LangGraph 提供了五种创建 Tool 的方式，从简单原型到生产级运行时注入：

**方式 1：`@tool` 装饰器（最简单，快速原型）**

```python
from langchain_core.tools import tool

@tool
def query_weather(city: str) -> str:
"""查询指定城市的实时天气。

Args:
city: 城市名称，如"北京"、"Shanghai"

Returns:
天气描述字符串，如"北京今天晴，25度"
"""
return f"{city}今天晴，25度"
```

框架自动从 docstring + 类型注解生成 tool schema。**docstring = LLM 看到的工具说明书**，写不清楚，LLM 就不会用。

**方式 2：`@tool` + `args_schema`（复杂参数，生产推荐）**

当参数多、需要校验、需要详细描述时，用 Pydantic BaseModel：

```python
from pydantic import BaseModel, Field
from langchain_core.tools import tool

class WeatherInput(BaseModel):
city: str = Field(description="城市名称，如'北京'、'Shanghai'")
date: str | None = Field(
default=None,
description="查询日期，格式 YYYY-MM-DD。不填则查今天。"
)
include_aqi: bool = Field(
default=False,
description="是否同时返回空气质量指数"
)

@tool(args_schema=WeatherInput)
def query_weather(city: str, date: str | None = None, include_aqi: bool = False) -> str:
"""查询指定城市的实时天气，可选返回空气质量。"""
...
```

> **为什么用 `args_schema`？**
> - 参数有默认值时，LLM 知道哪些是可选的
> - `Field(description=...)` 会被直接写入 tool schema，LLM 看到的就是这个描述
> - Pydantic 自动做类型校验，LLM 传错类型会提前报错

**方式 3：`StructuredTool.from_function()`（程序化创建）**

适合动态生成 tool、从配置文件中加载 tool 的场景：

```python
from langchain_core.tools import StructuredTool
from pydantic import BaseModel, Field

class CalculatorInput(BaseModel):
expression: str = Field(description="数学表达式，如 '23 * 45'、'sqrt(16)'")

def calculate_impl(expression: str) -> str:
try:
return str(eval(expression))
except Exception as e:
return f"计算错误：{e}"

calculator_tool = StructuredTool.from_function(
func=calculate_impl,
name="calculator",
description="计算数学表达式。",
args_schema=CalculatorInput,
return_direct=False, # 默认：结果回到 LLM 继续推理
handle_tool_error=True, # 自动捕获异常并返回错误信息
)
```

**方式 4：继承 `BaseTool`（完全自定义，最灵活）**

适合需要自定义元数据、权限控制、状态管理的场景：

```python
from langchain_core.tools import BaseTool
from pydantic import BaseModel, Field
from typing import Type

class QueryWeatherInput(BaseModel):
city: str = Field(description="城市名称")

class QueryWeatherTool(BaseTool):
name: str = "query_weather"
description: str = "查询指定城市的实时天气。"
args_schema: Type[BaseModel] = QueryWeatherInput

    # 同步执行
def _run(self, city: str) -> str:
return query_weather_api(city)

    # 异步执行（可选但推荐）
async def _arun(self, city: str) -> str:
return await query_weather_api_async(city)

weather_tool = QueryWeatherTool()
```

**方式 5：`InjectedState` / `InjectedStore`（LangGraph 运行时注入，推荐用于预构建 Agent）**

LangGraph 1.x 支持通过类型注解将 graph state 或持久化 store 直接注入 tool，**无需 Factory 闭包**即可访问运行时状态：

```python
from typing import Annotated
from langchain_core.tools import tool, InjectedState, InjectedStore
from langgraph.store.base import BaseStore

@tool
async def query_user_history(
query: str,
state: Annotated[dict, InjectedState()], # 运行时自动注入当前 graph state
store: Annotated[BaseStore, InjectedStore()], # 运行时自动注入持久化 store
) -> str:
"""查询用户历史记录。"""
user_id = state.get("user_id")
memories = await store.asearch(("memories", user_id), query=query)
return f"找到 {len(memories)} 条相关记录。"
```

> **为什么用注入机制替代 Factory 闭包？**
> - 不需要 `build_xxx_tool()` 工厂函数，tool 本身是无状态的
> - state / store 由 LangGraph 运行时自动注入，代码更干净
> - 兼容 `create_react_agent`、`ToolNode` 等预构建节点，注入透明无感知
> - 避免闭包导致 tool 实例被重复创建或序列化问题

**五种方式选型**：

| 方式 | 适用场景 | 复杂度 |
|------|---------|--------|
| `@tool` | 参数少、快速验证 | 低 |
| `@tool` + `args_schema` | 参数多、需要校验、生产环境 | 中 |
| `StructuredTool.from_function()` | 动态生成、配置化 | 中 |
| 继承 `BaseTool` | 需要自定义元数据、权限、状态 | 高 |
| `InjectedState` / `InjectedStore` | 预构建 Agent，需访问 state/store | 中 |

### 2.2 `return_direct`：控制结果流向

`return_direct` 决定 tool 的输出是回到 LLM 继续推理，还是直接返回给用户：

```python
# return_direct=False（默认）：结果回到 LLM
# LLM 看到工具结果后，可以继续思考、调用其他工具
@tool
def query_weather(city: str) -> str:
"""查询天气。结果需要 LLM 进一步整理回复。"""
return query_weather_api(city)

# return_direct=True：结果直出，不再经过 LLM
# 适合工具本身已经给出最终答案的场景
@tool(return_direct=True)
def get_time() -> str:
"""获取当前时间。结果直接展示给用户，不需要 LLM 再加工。"""
return f"现在是 {datetime.now().strftime('%H:%M')}"
```

**使用场景对比**：

| 场景 | return_direct | 原因 |
|------|--------------|------|
| 查询天气后需要 LLM 整理回复 | False | 原始数据需要 LLM 加工成自然语言 |
| 计算器返回数值 | True | 数值就是最终答案 |
| 查询订单状态后需要判断下一步 | False | 需要根据状态决定后续操作 |
| 返回固定格式的卡片/图片 | True | 前端直接渲染，不需要 LLM 加工 |

### 2.3 生产级 Tool 设计模式

参考实际项目中的 Tool 设计，生产级 Tool 需要处理五个层面的问题：

```
┌─────────────────────────────────────────────────────────────┐
│ 生产级 Tool 五层设计 │
├─────────────────────────────────────────────────────────────┤
│ ① 输入层：Pydantic 校验 + 缺失参数检查 + 归一化 │
│ ② 执行层：业务逻辑 + 外部 API 调用 │
│ ③ 容错层：异常捕获 + 候选匹配 + 重试机制 │
│ ④ 输出层：LLM 摘要 + 原始数据写入 State │
│ ⑤ 透传层：State 写入 + 格式化文本返回 + 日志记录 │
└─────────────────────────────────────────────────────────────┘
```

**完整示例：书房预约查询 Tool**

```python
from pydantic import BaseModel, Field
from langchain_core.tools import tool

# ========== ① 输入层：Pydantic args_schema ==========
class LibraryAvailabilityInput(BaseModel):
library_id: str | None = Field(
default=None, description="书房真实 ID。没有就留空，不要编造。"
)
library_name: str | None = Field(
default=None, description="书房名称，例如 平原轩。"
)
booking_date: str | None = Field(
default=None, description="预约日期，格式 YYYY-MM-DD。没有明确日期时不要猜。"
)
start_time: str | None = Field(
default=None, description="开始时间，格式 HH:MM。"
)


# ========== ⑤ 透传层：Factory 模式 + 依赖注入 ==========
def build_availability_tool(
*,
state: dict, # 共享状态
client: Any, # 外部服务客户端
) -> Any:
"""Factory：根据当前上下文构建 tool 实例。"""

@tool(
args_schema=LibraryAvailabilityInput,
return_direct=False,
)
async def availability_tool(
library_id: str | None = None,
library_name: str | None = None,
booking_date: str | None = None,
start_time: str | None = None,
) -> str:
"""查询书房可预约时段。"""

        # ========== ① 输入层：缺失参数检查 ==========
missing = []
if not (library_id or library_name):
missing.append("书房名称或 ID")
if not booking_date:
missing.append("预约日期")
if missing:
return f"缺少必要参数：{', '.join(missing)}。请先提供这些信息。"

        # ========== ② 执行层：业务逻辑 ==========
try:
result = await client.query_availability(
library_id=library_id,
library_name=library_name,
date=booking_date,
start_time=start_time,
)
except Exception as e:
            # ========== ③ 容错层：异常处理 ==========
return f"查询失败：{str(e)}。请稍后再试。"

        # ========== ③ 容错层：结果校验 ==========
if not result or not result.items:
return f"抱歉，{library_name or library_id} 在 {booking_date} 没有可预约时段。"

        # ========== ④ 输出层：格式化 ==========
        # 生成给 LLM 看的自然语言摘要
summary = _build_llm_summary(result)

        # ========== ⑤ 透传层：State 写入 + 文本返回 ==========
        # 把查询结果写入 State，下游节点可以直接用
state["availability_result"] = result.to_dict()
state["booking_candidates"] = [item.to_dict() for item in result.items[:5]]

return summary

return availability_tool
```

**设计要点解析**：

1. **`args_schema` 做输入校验**：Pydantic 自动校验类型，LLM 传 `date="明天"` 会提前报错
2. **缺失参数检查**：不依赖 LLM 一次传齐所有参数，tool 自己检查缺什么，返回明确的补充提示
3. **异常捕获**：`try/except` 包裹外部调用，返回错误信息而非抛异常。抛异常会打断整个图，返回错误信息让 LLM 决定下一步
4. **输出分层**：
- `summary`（return 值）：给 LLM 看的自然语言文本（进入 message history）
- `state["availability_result"]`：给下游节点用的原始数据（通过 state 旁路传递，不占用 LLM 上下文）
5. **Factory 模式**：通过 `build_availability_tool()` 注入 `state`、`client` 等依赖，而不是用全局变量

**深度兜底：参数归一化、候选匹配与可恢复重试**

上述五层框架是骨架，真实项目还需要以下工程细节：

```python
def _normalize_text(value) -> str:
"""① 参数归一化：处理 None、空字符串、多余空格。"""
if value is None:
return ""
return str(value).strip()


def _business_token_from_state(state: dict) -> str | None:
"""② 多层嵌套 state 查找：token 可能藏在多层 dict 里。"""
for key in ("nanshantong_token", "business_token"):
token = _normalize_text(state.get(key))
if token:
return token
for container_key in ("params", "metadata"):
container = state.get(container_key)
if isinstance(container, dict):
for key in ("nanshantong_token", "business_token"):
token = _normalize_text(container.get(key))
if token:
return token
return None


def _value_matches(value: str, *candidates) -> bool:
"""③ 候选模糊匹配：处理 LLM 传"8号桌"但候选里是"8"的情况。"""
normalized = _normalize_text(value).lower()
if not normalized:
return True
for candidate in candidates:
candidate_text = _normalize_text(candidate).lower()
if not candidate_text:
continue
if normalized == candidate_text or normalized in candidate_text or candidate_text in normalized:
return True
return False


def _error_result_text(*, summary: str, data: dict) -> str:
"""④ 结构化错误返回：让 LLM 知道错误原因和可恢复路径。"""
import json
return json.dumps(
{"summary": summary, "status": "error", "data": data},
ensure_ascii=False,
)


def _missing_booking_summary(missing_fields: list[str]) -> str:
"""⑤ 缺失参数友好提示：告诉 LLM 缺什么，而不是抛异常。"""
field_labels = {
"library": "书房",
"area": "区域",
"seat": "座位",
"start_datetime": "开始时间",
"end_datetime": "结束时间",
}
return (
"缺少确认预约所需参数："
+ "、".join(field_labels.get(key, key) for key in missing_fields)
+ "。请先查询可预约候选，再选择一个候选继续提交。"
)
```

**兜底机制清单**：

| 问题 | 机制 | 代码体现 |
|------|------|---------|
| LLM 传空字符串 `""` | 参数归一化 | `_normalize_text()` |
| token 在多层嵌套 dict 中 | 多层查找 | `_business_token_from_state()` |
| LLM 传"8号桌"匹配候选"8" | 模糊匹配 | `_value_matches()` |
| 缺少必要参数 | 缺失检查 + 友好提示 | `_missing_booking_summary()` |
| 执行失败 | 结构化错误返回 | `_error_result_text()` |
| 结果需要下游使用 | State 写入 | `state["result"] = raw_data` |
| 下游需要原始数据 | state 透传 | `state["booking_request"] = result` |

### 2.4 Tool 的缓存机制

LangChain 官方提供了 **LLM 调用缓存**，不是 Tool 结果缓存。但 Tool 结果缓存在生产中同样重要。

**官方机制 1：LLM 调用缓存（省钱）**

```python
from langchain_core.globals import set_llm_cache
from langchain_core.caches import InMemoryCache, SQLiteCache

# 内存缓存（开发用）
set_llm_cache(InMemoryCache())

# SQLite 持久化缓存（本地用）
set_llm_cache(SQLiteCache(database_path=".langchain_cache.db"))

# Redis 分布式缓存（生产用）
from langchain_redis import RedisCache
set_llm_cache(RedisCache(redis_url="redis://localhost:6379/0"))
```

> **作用**：相同的 prompt 第二次调用时，直接从缓存返回，不花钱调 API。

**官方机制 2：`BaseStore` + `InjectedStore`（状态级持久化）**

LangGraph 没有专门的"Tool 结果缓存"API，但可以通过持久化 store 间接实现跨会话的状态复用：

```python
from typing import Annotated
from langchain_core.tools import tool, InjectedStore
from langgraph.store.base import BaseStore

@tool
async def query_weather_cached(
city: str,
store: Annotated[BaseStore, InjectedStore()],
) -> str:
"""查询天气，优先从 store 读取缓存。"""
    # 尝试从持久化 store 读取缓存
cached = await store.aget(("weather_cache", city))
if cached:
return f"[缓存] {cached['value']}"

result = await query_weather_api(city)

    # 写入缓存（TTL 600 秒由 store 后端控制）
await store.aput(("weather_cache", city), {"value": result})
return result
```

> **注意**：`BaseStore` 是 LangGraph 1.x 引入的持久化抽象，需要配置 `InMemoryStore`（内存）或外部存储后端（如 Postgres、Redis）。它不是传统意义上的"缓存"，而是跨图执行的持久状态。

**机制 3：Tool 结果缓存（常用做法）**

进程内 Tool 缓存没有官方 API，推荐用 `functools.lru_cache` 或自定义装饰器：

```python
from functools import lru_cache

@tool
@lru_cache(maxsize=128)
def query_weather(city: str, date: str | None = None) -> str:
"""查询天气（单进程内缓存 128 条）。"""
return query_weather_api(city, date)
```

> ⚠️ 注意：`lru_cache` 要求参数可哈希。`dict` 和 `list` 不能作为参数直接缓存。

**缓存策略选择**：

| 机制 | 缓存对象 | 适用场景 | 持久化 |
|------|---------|---------|--------|
| `set_llm_cache(InMemoryCache())` | LLM 调用 | 相同 prompt 重复查询 | ❌ |
| `set_llm_cache(RedisCache())` | LLM 调用 | 生产环境多实例共享 | ✅ |
| `@lru_cache` | Tool 结果 | 单进程内重复 tool 调用 | ❌ |
| `BaseStore` + `InjectedStore()` | 状态/记忆 | 跨会话持久化、历史检索 | ✅ |
| 自定义 Redis wrapper | Tool 结果 | 跨会话共享 tool 结果 | ✅ |

### 2.5 Tool 直接作为 Node 使用：定义一次，到处复用

**核心洞察**：LangGraph 中，Tool 本身就是一等公民——它实现了 LangChain 的 `Runnable` 接口，可以直接被 `builder.add_node()` 当作普通节点使用。

**为什么需要这样做？**

在真实业务中，同一个能力往往需要在多个图、多个流程中复用：

```
订单查询 Tool ──┬──→ 客服对话图（查询用户订单）
├──→ 售后处理图（查询订单状态）
├──→ 数据报表图（批量查询订单）
└──→ 管理员后台图（订单管理）
```

你不需要在每个图里重新写一遍查询逻辑。**定义一次 Tool，任何图都可以直接把它当节点用**。

**两种方式把 Tool 变成节点**：

**方式 A：Tool 直接 add_node（精准业务流程）**

当你需要**精确控制** tool 何时执行、如何与上下游衔接时，直接把 tool 当节点：

```python
from langgraph.graph import StateGraph

@tool
def query_order(order_id: str) -> str:
"""查询订单详情。"""
return order_service.get(order_id)

builder = StateGraph(State)

# ① 把 tool 直接当节点用
builder.add_node("query_order", query_order)

# ② 精确编排：先验证身份，再查订单，再生成回复
builder.add_edge("verify_user", "query_order")
builder.add_edge("query_order", "generate_reply")
```

**方式 B：ToolNode（LLM 自主决策调用）**

当你想让 LLM **自己决定**要不要调、调哪个时，用 `ToolNode` 包装一组 tools：

```python
from langgraph.prebuilt import ToolNode, tools_condition

tools = [query_order, cancel_order, refund_order]
tool_node = ToolNode(tools)

builder.add_node("tools", tool_node)

# LLM 输出 tool_calls → 走 tools 节点
# LLM 输出普通回复 → 直接结束
builder.add_conditional_edges("llm", tools_condition)
```

**两种方式对比**：

| 场景 | 方式 | 控制权 |
|------|------|-------|
| 必须按固定顺序执行（先查订单再退款） | Tool 直接 add_node | 开发者精确控制 |
| LLM 自主判断调哪个工具 | ToolNode + tools_condition | LLM 自主决策 |
| 同一个工具在多个图中复用 | Tool 直接 add_node | 定义一次，到处复用 |

> **关键理解**：`ToolNode` 本质上就是 LangGraph 帮你写好的"自动路由节点"——它检查 LLM 的 `tool_calls`，自动分发到对应的 tool。而 Tool 直接 `add_node` 是你在手动编排流程。

### 2.6 ToolNode 与 tools_condition

**ToolNode**：把一组 tools 包装成图节点。

```python
from langgraph.prebuilt import ToolNode

tools = [query_weather, query_stock, calculator]
tool_node = ToolNode(tools)

builder.add_node("tools", tool_node)
```

**tools_condition**：自动判断 LLM 是否想调用工具。

```python
from langgraph.prebuilt import tools_condition

# LLM 节点后面接条件边
# 如果 LLM 输出包含 tool_calls → 走 tools 节点
# 如果 LLM 输出是普通回复 → 直接结束
builder.add_conditional_edges("llm", tools_condition)
```

**ToolNode 的工作原理**：
1. 检查 LLM 输出的 `tool_calls` 字段
2. 根据 `name` 找到对应的 tool 函数
3. 传入 `arguments` 执行 tool
4. 把结果包装成 `ToolMessage` 返回

**常见坑**：ToolNode 必须跟在 LLM 节点后面，且 LLM 输出的是 `AIMessage` 带 `tool_calls`。

## 3. 上下文管理：防止 Tool 结果撑爆窗口

### 3.1 问题：Tool 结果太长怎么办？

```python
# 用户让查"Python 文档"
# Tool 返回了 5000 字的网页内容
# 塞进 message history → Token 暴涨 → 费用暴涨 → 可能超出模型上下文上限
```

### 3.2 策略 1：Tool 结果 Truncate（截断）

```python
@tool
def fetch_url(url: str, max_length: int = 2000) -> str:
"""获取网页内容，自动截断过长内容。"""
content = requests.get(url).text
if len(content) > max_length:
content = content[:max_length] + "\n...[内容已截断]"
return content
```

### 3.3 策略 2：选择性传递 Tool 结果

不是把 tool 的原始输出全部塞给 LLM，而是提取关键信息：

```python
@tool
def query_weather(city: str) -> str:
"""查询天气，只返回 LLM 需要的关键信息。"""
raw_data = query_weather_api(city) # 原始数据很大

    # 只提取 LLM 关心的字段
simplified = {
"city": raw_data["city"],
"temperature": raw_data["temp"],
"condition": raw_data["weather"][0]["description"],
"humidity": raw_data["humidity"]
}
return json.dumps(simplified, ensure_ascii=False)
```

### 3.4 策略 3：Message Trimming（自动裁剪历史）

```python
from langchain_core.messages import trim_messages

# 只保留最近的 N 条消息，避免历史太长
trimmed_messages = trim_messages(
messages,
max_tokens=4000,
strategy="last", # 保留最近的消息
token_counter=llm # 用模型自己的 tokenizer 计数
)
```

### 3.4 策略 4：Summarization（历史消息总结）

当对话历史太长时，把早期消息总结成一条，释放 Token：

```python
def summarize_history(messages: list) -> list:
"""当消息超过阈值时，总结早期消息。"""
if count_tokens(messages) < 6000:
return messages

    # 早期的消息交给 LLM 总结
early_messages = messages[:5]
recent_messages = messages[5:]

summary = llm.invoke([
SystemMessage("请总结以下对话的关键信息，保留所有事实性内容。"),
*early_messages
])

return [summary] + recent_messages
```

### 3.5 策略 5：格式化文本返回 + State 写入（标准机制）

Tool 的 return 值**永远是纯文本字符串**，会被 `ToolNode` 自动包装成 `ToolMessage` 塞进 `messages`，成为 LLM 的上下文。**不要把原始 JSON 直接 return 给 LLM**——它会看不懂、浪费 token。

LangGraph 官方标准做法是：

1. **`return` = 精心格式化的自然语言文本**（给 LLM 看，必须进 message）
2. **`state` 写入 = 原始结构化数据**（给下游节点用，不占用 LLM 上下文）

```python
from typing import TypedDict

class AgentState(TypedDict):
messages: Annotated[list, add_messages]
weather_result: dict | None # 原始数据放这里

@tool
async def query_weather(city: str) -> str:
"""查询天气。"""
raw_data = await weather_api(city) # 原始 JSON 很大

    # ① 把原始数据写入 state，下游节点可以直接用
    # 注意：这个写入需要在调用 tool 的节点里做，或 tool 返回 dict

    # ② return 的只有 LLM 可读的自然语言摘要
return (
f"{city}今天{city}，气温{raw_data['temp']}°C，"
f"湿度{raw_data['humidity']}%，{raw_data['wind']}。"
)

# 在图节点中使用：
def weather_node(state: AgentState):
result = query_weather.invoke({"city": state["city"]})
return {
"messages": [result], # ToolMessage 进 LLM 上下文
"weather_result": raw_data, # 原始数据进 state，不进 message
}
```

**更直接的方式：在 tool 里 return dict 同时更新 state**

```python
@tool
async def query_weather(state: Annotated[dict, InjectedState()], city: str) -> str:
"""查询天气，原始数据写入 state，return 文本给 LLM。"""
raw_data = await weather_api(city)

    # 原始数据写入 state（下游节点可用，不占用 LLM 上下文）
state["weather_raw"] = raw_data

    # return 纯文本摘要（进入 message history 给 LLM）
return f"{city}今天{city}，气温{raw_data['temp']}°C。"
```

**核心原则**：

| 数据去向 | 方式 | 接收方 | 是否进 LLM 上下文 |
|---------|------|-------|-----------------|
| 给 LLM 看 | `return` 字符串 | LLM | ✅ 必须进 |
| 给下游节点 | `state["key"] = value` | 下游节点 | ❌ 通过 state 传递 |

> **不要做的事**：把原始 JSON 直接 `return` 给 LLM。
>
> **要做的事**：`return` 永远放自然语言摘要；原始数据通过 `state` 旁路传递。

**五种策略对比**：

| 策略 | 实现难度 | 信息损失 | 适用场景 |
|------|---------|---------|---------|
| Truncate（截断） | 低 | 高（后半截丢了） | 网页抓取、文档阅读 |
| 选择性传递 | 中 | 低（按需提取） | API 返回结构化数据 |
| Message Trimming | 低 | 中（早期消息丢了） | 长对话场景 |
| Summarization | 高 | 可控（LLM 智能压缩） | 需要保留上下文的复杂对话 |
| 格式化文本 + State 写入 | 中 | 无（原始数据放 state 不发 LLM） | 大结果量 + 下游需原始数据 |

**现场验证**：
> 问学员："一个查询数据库的 tool 返回了 10000 行结果，直接塞给 LLM 会怎样？"
> 答：Token 爆炸，费用暴涨，可能超出上下文限制，LLM 也看不过来。
> 再问："正确的做法是什么？"
> 答：在 tool 内部做分页/聚合，只返回摘要（如前 10 行 + 统计信息）。
>
> 再问："那下游节点需要这 10000 行的原始数据做排序/导出，怎么办？"
> 答：return 值放自然语言摘要给 LLM，原始数据通过 `state["raw_data"] = result` 写入 state，下游节点直接读取。

## 4. ToolHub：工具集成实战

### 案例与概念的一一对应

| 概念 | ToolHub 中的体现 |
|------|-----------------|
| Tool 定义 | 天气/订单/计算器 tool 都有完整 docstring 和类型注解 |
| Tool 直接当 Node | `query_order` 在客服图、售后图、报表图三个图中复用 |
| ToolNode | 智能助手图用 `ToolNode` 让 LLM 自主决定调哪个 tool |
| Tool 缓存 | 天气查询加了 10 分钟缓存（`lru_cache`） |
| 上下文管理 | fetch_url tool 自动截断超过 2000 字的内容 |

### 代码演示讲解话术

**Step 1 - Tool 定义与复用**：
> "看 `query_order` 这个 tool。它定义了一次——查订单接口、校验权限、格式化输出。然后在三个图里被复用：客服对话图直接把它当节点 `add_node('query_order', query_order)`，售后处理图也用它，数据报表图还是用它。定义一次，到处复用。"

**Step 2 - docstring 的重要性**：
> "看 `query_weather` 这个 tool。它的 docstring 写了什么，LLM 就看到什么。如果我把 docstring 写成『查天气』，LLM 可能不知道参数是城市名还是城市代码。docstring 就是工具的说明书，写不清楚，LLM 就不会用。"

**Step 3 - ToolNode 与直接 add_node 对比**：
> "在智能助手图里，我们用 `ToolNode` 包装了天气、订单、计算器三个 tool。LLM 自己判断用户要查什么，自动选工具。而在客服流程图里，我们直接 `add_node('query_order', query_order)`——因为客服流程固定是'先验证身份，再查订单'，不需要 LLM 做决策。"

**Step 4 - 缓存效果验证**：
> "现在连续问两次『北京天气』。看 Langfuse Trace——第一次调用了天气 API（耗时 1.2s），第二次直接从缓存读取（耗时 2ms）。成本从 $0.01 降到了 $0。"

### 关键转折点（需要停下来确认学员理解）

1. **Tool docstring**：给出一个写得不好的 docstring，让学员指出 LLM 会怎么误解这个工具。
2. **复用场景**：问"如果公司有 5 个业务线都要查订单，是每个线自己写查询逻辑，还是复用同一个 order_tool？"（期望：复用同一个，统一权限校验、统一错误处理、统一缓存策略）
3. **缓存失效**：问"天气数据缓存了 10 分钟，但 5 分钟后突然暴风雨了，用户收到的是过期数据，怎么办？"（期望：业务上权衡新鲜度和成本，可以按数据类型设不同 TTL）
4. **上下文爆炸**：问"如果 fetch_url 返回 10MB 的网页内容，直接塞进 message history 会怎样？怎么解决？"（期望：在 tool 内部 truncate 或提取关键内容）

## 5. 常见误区与注意事项

| 坑 | 现象 | 预警话术 |
|----|------|---------|
| **Tool docstring 写得太简略** | LLM 不调用或乱传参数 | "docstring 就是 LLM 看到的工具说明书，参数、返回值、示例都要写清楚。" |
| **Tool 抛异常而不是返回错误** | 整个图中断 | "Tool 应该返回错误信息（字符串），让 LLM 决定下一步。抛异常会打断整个流程。" |
| **缓存不设 TTL** | 返回过期数据 | "所有缓存必须有 TTL，根据业务新鲜度需求设置。天气 10 分钟，汇率 1 分钟。" |
| **Tool 返回原始 JSON 给 LLM** | LLM 理解困难，token 浪费 | "Tool 必须 return 纯文本摘要；原始 JSON 写入 state，不塞进 message。" |
| **Tool 结果太长直接塞给 LLM** | Token 暴涨，费用爆炸 | "Tool 返回前要做截断或摘要。原始数据给 LLM 看，它会懵的。" |

### 调试技巧

1. **Tool 调用追踪**：Langfuse 里看 ToolMessage 的输入输出，确认 LLM 传了什么参数
2. **State 透传检查**：在节点间打印 `state.keys()`，确认下游节点能拿到上游写入的数据
3. **缓存命中率**：用 Langfuse Score 上报缓存命中/未命中，监控效果
4. **Tool 复用验证**：同一个 tool 在多个图中使用时，确认每个图的调用参数和返回格式一致

## 6. 阶段小结

### 核心要点回顾

1. **Tool 的 docstring = LLM 的说明书**：写不清楚，LLM 就不会用
2. **Tool 定义五种方式**：从 `@tool` 到 `InjectedState`/`InjectedStore`，按场景选型
3. **Tool 直接当 Node**：`add_node('query_order', query_order)`，定义一次、到处复用
4. **ToolNode**：LLM 自主决策调哪个 tool，适合开放域场景
5. **生产级 Tool = 五层兜底**：输入校验 → 执行 → 容错 → 输出格式化 → 状态透传
6. **格式化文本 + State 写入**：return 纯文本给 LLM，原始数据写入 state 给下游，两者职责分离
7. **Tool 缓存 = 省钱利器**：`lru_cache` 进程内、`BaseStore` 跨会话、TTL 必须设
8. **上下文管理 = 成本控制**：Truncate、选择性传递、Trimming、Summarization、格式化文本 + State 写入

### 记忆口诀

> **"docstring 是说明书，注入替闭包。Tool 当节点能复用，五层兜底要记牢。return 文本 state 存数据。"**
