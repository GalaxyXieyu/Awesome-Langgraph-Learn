# LG-02：Tools 深度掌握

**适用基础**：你已经掌握 StateGraph 基础，想让图具备调用外部系统的能力。

**完成后你能做什么**：
- 写出带 description、类型校验、异常处理的 production-ready Tool
- 区分 Tool 直接作为节点和交给 `ToolNode` 的使用边界
- 理解 `return_direct`、运行时注入、缓存、上下文管理各自解决什么问题
- 让 Tool 不只是“能调”，而是“能复用、能兜底、能上线”

---

## 1. 案例背景：书房预约助手

这一节从一个真实业务开始。你要做一个 `StudyRoomBookingBot`，用户会连续提出这些请求：

> - “帮我查一下明天下午平原轩还有没有空位”
> - “把 15:00 之后能约的时段列出来”
> - “那就帮我约 16:00-18:00”
> - “看看我最近的预约记录”

如果只写普通节点，通常会拆成三件事：

- 识别用户意图
- 生成回复
- 做确认

但业务一真实起来，就会碰到四个问题：

1. 查空位要连接预约系统。
2. 查预约记录要连接数据库或业务后端。
3. 提交预约失败时，错误信息要回到 LLM，而不是直接把图打断。
4. 同一个“查可预约时段”能力，要能在客服图、预约图、后台图里复用。

Tool 是图连接外部世界的标准接口。本节用一个统一主案例来讲。

## 主案例：书房预约助手 `StudyRoomBookingBot`

它支持四类请求：

- 查询可预约时段
- 查询预约规则
- 提交预约
- 查询历史预约

这四类请求都不能只靠 LLM 生成文本，必须调用外部系统。本节回答四个问题：

1. Tool 怎么定义，LLM 才知道什么时候调、参数怎么传？
2. Tool 结果怎么回到图里，而不是只变成一句散落的字符串？
3. Tool 什么时候应该直接返回给用户，什么时候应该回到 LLM 继续推理？
4. 同一个 Tool 怎么在多个图里复用？

---

## 2. 完整业务流：Tool 出现的位置

先看整体流程。

```text
用户输入
   │
   ▼
parse_request
   │
   ├── query_availability → availability_tool → summarize_slots
   ├── query_rules       → rules_tool → reply_rules
   ├── create_booking    → booking_tool → confirm_booking
   └── query_history     → history_tool → format_history
```

上一节的 State 解决“数据怎么在图里流动”。这一节的 Tool 解决“图怎么把外部系统接进来”。

换到胰腺癌辅助问诊，Tool 的边界会更明显：模型可以组织语言和追问，但不能凭空编造检验结果、影像报告或指南规则。它必须通过 Tool 去读外部系统，再把可读摘要交给 LLM，把原始结构化数据留在 State 里供下游节点继续判断。

![胰腺癌辅助问诊中的 Tool 边界](images/pancreatic-cancer-tools-boundary.svg)

这张图强调三个判断：

- Tool 不是装饰性函数，而是 Graph / Agent 接入外部事实的边界。
- 给 LLM 的结果应该是短摘要，避免把完整病历、检查 JSON 或影像元数据全部塞进消息流。
- 查不到、缺参数、系统异常时，要返回可恢复错误，让图继续追问或转人工，而不是直接中断整条链路。

书房预约场景有两个特点：

- **Tool 的必要性强**：查空位、提交预约、查规则都必须调用真实系统。
- **Tool 的返回结果类型丰富**：有的返回结构化列表，有的返回简单文本，有的还要写回 State 给下游节点继续用。

## 3. Tool 的五种定义方式

五种定义方式可以看成同一个业务能力的不同实现形态。下面统一围绕“查书房可预约时段”这个能力展开。

### 3.1 方式 1：`@tool` 装饰器

用于快速原型。

```python
from langchain_core.tools import tool


@tool
def query_room_rules(room_name: str) -> str:
    """查询书房预约规则。

    Args:
        room_name: 书房名称，例如“平原轩”“兰台”

    Returns:
        对应书房的预约规则说明文本
    """
    rules = {
        "平原轩": "平原轩每天可预约 2 小时，最晚需提前 30 分钟预约。",
        "兰台": "兰台支持 1-4 小时预约，超过 2 小时需要管理员审批。",
    }
    return rules.get(room_name, f"未找到 {room_name} 的预约规则。")
```

`docstring` 就是 LLM 看到的工具说明书。说明书写不清楚，LLM 就不会稳定调用。

### 3.2 方式 2：`@tool + args_schema`

参数开始复杂时，就该使用显式 schema。

```python
from pydantic import BaseModel, Field
from langchain_core.tools import tool


class AvailabilityInput(BaseModel):
    room_name: str = Field(description="书房名称，例如 平原轩")
    booking_date: str = Field(description="预约日期，格式 YYYY-MM-DD")
    start_after: str | None = Field(
        default=None,
        description="只返回这个时间之后的可用时段，格式 HH:MM",
    )


@tool(args_schema=AvailabilityInput)
def query_availability(
    room_name: str,
    booking_date: str,
    start_after: str | None = None,
) -> str:
    """查询书房可预约时段。"""
    slots = ["14:00-15:00", "16:00-18:00", "19:00-20:00"]
    if start_after:
        slots = [slot for slot in slots if slot[:5] >= start_after]
    return f"{room_name} 在 {booking_date} 可预约时段：{', '.join(slots)}"
```

这一段重点看三件事：

- 参数一多，LLM 更容易传错。
- `Field(description=...)` 会直接进入 schema。
- schema 越清楚，LLM 调用越稳。

### 3.3 方式 3：`StructuredTool.from_function()`

用于动态生成 Tool，比如按租户、按配置开关、按功能模块组装。

```python
from langchain_core.tools import StructuredTool


def calculate_booking_duration(start_time: str, end_time: str) -> str:
    """计算预约时长。"""
    start_hour = int(start_time.split(":")[0])
    end_hour = int(end_time.split(":")[0])
    return f"预约时长 {end_hour - start_hour} 小时"


duration_tool = StructuredTool.from_function(
    func=calculate_booking_duration,
    name="calculate_booking_duration",
    description="计算预约时长",
    return_direct=True,
    handle_tool_error=True,
)
```

有些 Tool 不是固定写死的函数，而是在系统运行时按配置拼出来的。

### 3.4 方式 4：继承 `BaseTool`

当 Tool 需要权限控制、埋点、统一日志、复杂元数据时，可以继承 `BaseTool`。

```python
from typing import Type
from langchain_core.tools import BaseTool


class BookingHistoryInput(BaseModel):
    user_id: str = Field(description="用户 ID")


class QueryBookingHistoryTool(BaseTool):
    name: str = "query_booking_history"
    description: str = "查询指定用户最近的书房预约记录"
    args_schema: Type[BaseModel] = BookingHistoryInput

    def _run(self, user_id: str) -> str:
        return f"用户 {user_id} 最近预约记录：平原轩 2025-05-28 16:00-18:00"
```

当 Tool 本身已经像一个“小组件”或“小服务”，`BaseTool` 比简单函数更自然。

### 3.5 方式 5：`InjectedState / InjectedStore`

有些 Tool 需要读取运行时 State 或持久化 Store，但这些参数不应该让 LLM 自己传。

```python
from typing import Annotated
from langchain_core.tools import tool
from langgraph.prebuilt import InjectedState, InjectedStore
from langgraph.store.base import BaseStore


@tool
async def query_recent_preferences(
    room_name: str,
    state: Annotated[dict, InjectedState()],
    store: Annotated[BaseStore, InjectedStore()],
) -> str:
    """查询用户过去的预约偏好。"""
    user_id = state.get("user_id", "unknown")
    memories = await store.asearch(("booking_preferences", user_id), query=room_name)
    return f"用户 {user_id} 找到 {len(memories)} 条和 {room_name} 相关的预约偏好。"
```

这类注入方式带来四个好处：

- 不用闭包工厂函数
- Tool 本身保持无状态
- State / Store 由运行时自动注入
- 和 `ToolNode`、预构建 Agent 更兼容

### 五种方式怎么选

| 方式 | 使用场景 | 复杂度 |
|------|---------|--------|
| `@tool` | 快速原型、参数少 | 低 |
| `@tool + args_schema` | 参数多、需要校验 | 中 |
| `StructuredTool.from_function()` | 动态组装、配置化 | 中 |
| 继承 `BaseTool` | 权限/日志/元数据复杂 | 高 |
| `InjectedState / InjectedStore` | 需要读运行时状态或持久化记忆 | 中 |

## 4. `return_direct`：结果回到图还是直接给用户

`return_direct` 决定 Tool 执行后的结果去向。预约场景里可以直接对比两类工具。

### 场景 1：回到 LLM 继续推理

比如“查询可预约时段”。Tool 查出来的只是原始结果，LLM 还要进一步做两件事：

- 理解用户意图
- 组织成自然语言答复，甚至推荐一个候选时段

```python
@tool
def query_availability_for_llm(room_name: str, booking_date: str) -> str:
    """查询书房可预约时段，结果回到 LLM 继续整理。"""
    return f"{room_name} 在 {booking_date} 可预约：14:00-15:00，16:00-18:00"
```

### 场景 2：直接返回用户

比如“计算预约时长”或者“获取当前系统时间”这类工具，结果本身就是最终答案。

```python
@tool(return_direct=True)
def calculate_booking_duration(start_time: str, end_time: str) -> str:
    """计算预约时长，结果直接返回。"""
    start_hour = int(start_time.split(":")[0])
    end_hour = int(end_time.split(":")[0])
    return f"预约时长 {end_hour - start_hour} 小时"
```

### 判断标准

| 场景 | return_direct | 原因 |
|------|--------------|------|
| 查询可预约时段 | False | 结果还需要 LLM 解释和引导下一步 |
| 查询预约规则 | False | 可能还要结合用户上下文解释限制 |
| 计算预约时长 | True | 结果本身就是最终答案 |
| 返回二维码 / 固定卡片 | True | 前端直接渲染即可 |

核心区别是：原始结果回到 LLM，最终答案可以直接返回。

## 5. 生产级 Tool 五层设计

只会写 `@tool` 还不够。一个能上线的 Tool，至少要处理输入、执行、容错、输出、透传五件事。

### 五层结构

```text
① 输入层：参数校验、缺失检查、归一化
② 执行层：业务逻辑、外部 API / 数据库调用
③ 容错层：异常捕获、候选匹配、可恢复错误返回
④ 输出层：给 LLM 的摘要文本
⑤ 透传层：原始数据写入 State / Store，供下游继续使用
```

### 完整例子：查询书房可预约时段

```python
from pydantic import BaseModel, Field
from langchain_core.tools import tool


class StudyRoomAvailabilityInput(BaseModel):
    room_name: str | None = Field(default=None, description="书房名称，例如 平原轩")
    booking_date: str | None = Field(default=None, description="日期，格式 YYYY-MM-DD")
    start_after: str | None = Field(default=None, description="开始时间下限，格式 HH:MM")


def build_availability_tool(*, state: dict, client: object):
    """根据当前图上下文构建 tool。"""

    @tool(args_schema=StudyRoomAvailabilityInput, return_direct=False)
    async def availability_tool(
        room_name: str | None = None,
        booking_date: str | None = None,
        start_after: str | None = None,
    ) -> str:
        """查询书房可预约时段。"""

        # ① 输入层
        missing = []
        if not room_name:
            missing.append("书房名称")
        if not booking_date:
            missing.append("预约日期")
        if missing:
            return f"缺少必要参数：{', '.join(missing)}。请先补充这些信息。"

        # ② 执行层
        try:
            result = await client.query_availability(
                room_name=room_name,
                booking_date=booking_date,
                start_after=start_after,
            )
        except Exception as e:
            # ③ 容错层
            return f"查询失败：{str(e)}。请稍后再试。"

        if not result or not result.get("slots"):
            return f"{room_name} 在 {booking_date} 没有可预约时段。"

        slots = result["slots"]

        # ④ 输出层：给 LLM 的文本摘要
        summary = f"{room_name} 在 {booking_date} 可预约时段：{', '.join(slots[:5])}"

        # ⑤ 透传层：把原始数据写进 state，给下游节点继续用
        state["availability_result"] = result
        state["booking_candidates"] = slots

        return summary

    return availability_tool
```

这段代码重点看三件事：

- Tool 不能只处理成功路径。
- 失败、缺参数、查无结果，都要变成**可恢复错误**。
- 原始数据和 LLM 文本摘要要分层。

也就是：

> **return 给 LLM 的，是它看得懂、需要继续推理的文本；  
> 给下游节点继续用的结构化原始数据，放到 State 里。**

## 6. Tool 缓存：缓存什么、不能缓存什么

书房预约助手里，至少有两类查询可以考虑缓存：

- 预约规则：变更不频繁
- 某天的可预约时段：短时间内频繁重复查询

### 6.1 最简单的进程内缓存

```python
from functools import lru_cache


@tool
@lru_cache(maxsize=128)
def query_room_rules_cached(room_name: str) -> str:
    """查询书房规则（单进程缓存）。"""
    rules = {
        "平原轩": "平原轩每天可预约 2 小时。",
        "兰台": "兰台超过 2 小时需审批。",
    }
    return rules.get(room_name, f"未找到 {room_name} 的规则。")
```

### 6.2 用 `InjectedStore` 做跨会话复用

```python
from typing import Annotated
from langchain_core.tools import tool
from langgraph.prebuilt import InjectedStore
from langgraph.store.base import BaseStore


@tool
async def query_room_rules_persistent(
    room_name: str,
    store: Annotated[BaseStore, InjectedStore()],
) -> str:
    """查询书房规则，优先从持久化 store 读取。"""
    cached = await store.aget(("room_rules", room_name))
    if cached:
        return f"[缓存] {cached['value']}"

    result = f"{room_name} 每天可预约 2 小时。"
    await store.aput(("room_rules", room_name), {"value": result})
    return result
```

缓存判断逻辑：

- 高频重复查、变化不频繁：可以缓存。
- 强实时性、后果严重：谨慎缓存。

在这个案例里：

- 书房规则：可以缓存。
- 当天实时可预约时段：可以短 TTL 缓存。
- 提交预约结果：不要缓存。

## 7. Tool 直接作为 Node vs ToolNode

这是 Tools 设计里最重要的分界线：流程由开发者控制，还是让 LLM 自己选择工具。

### 7.1 Tool 直接作为 Node

当流程是确定的，就直接把 Tool 当节点。

比如用户说：

> “帮我查一下平原轩明天下午有哪些空位”

这时流程很固定：

```text
parse_request → availability_tool → summarize_slots
```

对应代码：

```python
builder.add_node("availability_tool", availability_tool)
builder.add_edge("parse_request", "availability_tool")
builder.add_edge("availability_tool", "summarize_slots")
```

这里不需要 LLM 决定调哪个工具，因为业务流程本身已经确定。

### 7.2 ToolNode

开放式助手会遇到更多输入：

> 用户可能问规则、问空位、问历史记录，也可能让系统帮他提交预约

这时可以让 LLM 自己判断调哪个工具。

```python
from langgraph.prebuilt import ToolNode, tools_condition

tools = [
    query_availability,
    query_room_rules,
    query_booking_history,
    create_booking,
]

tool_node = ToolNode(tools)

builder.add_node("tools", tool_node)
builder.add_conditional_edges("llm", tools_condition)
```

### 两种方式怎么选

| 场景 | 方式 | 原因 |
|------|------|------|
| 固定业务流程 | Tool 直接 `add_node` | 开发者精确控制步骤 |
| 开放式助手 | `ToolNode + tools_condition` | 让 LLM 自主决策 |
| 同一工具多图复用 | Tool 直接定义后多处挂载 | 定义一次，到处复用 |

`ToolNode` 本质上是 LangGraph 提供的自动工具路由节点。

## 8. 上下文管理：防止 Tool 结果撑爆窗口

预约场景里，`query_booking_history` 可能一次返回用户最近 100 条预约记录。如果全塞进 LLM 上下文，会产生三个问题：

- token 暴涨
- 成本增加
- LLM 不一定能有效利用全部记录

### 8.1 策略 1：截断

```python
@tool
def query_booking_history_brief(user_id: str, max_items: int = 5) -> str:
    """查询最近预约记录，只返回最近几条。"""
    records = [
        "平原轩 2025-05-28 16:00-18:00",
        "兰台 2025-05-26 14:00-15:00",
        "平原轩 2025-05-20 19:00-20:00",
    ]
    return "最近预约记录：" + "；".join(records[:max_items])
```

### 8.2 策略 2：选择性传递

```python
@tool
def query_availability_selective(room_name: str, booking_date: str) -> str:
    """查询可预约时段，只返回关键字段。"""
    raw_data = {
        "room_name": room_name,
        "date": booking_date,
        "slots": ["14:00-15:00", "16:00-18:00", "19:00-20:00"],
        "metadata": {"source": "booking_api", "version": "v2"},
    }
    return f"{room_name} 在 {booking_date} 可预约时段：{', '.join(raw_data['slots'])}"
```

### 8.3 策略 3：文本给 LLM，原始数据进 State

```python
class BookingAgentState(TypedDict):
    messages: list
    availability_raw: dict | None


def availability_node(state: BookingAgentState) -> dict:
    raw_data = {
        "room_name": "平原轩",
        "slots": ["14:00-15:00", "16:00-18:00"],
    }
    return {
        "messages": ["平原轩可预约时段：14:00-15:00，16:00-18:00"],
        "availability_raw": raw_data,
    }
```

数据去向要分清楚：

| 数据去向 | 方式 | 是否进 LLM 上下文 |
|---------|------|-----------------|
| 给 LLM 看 | `return` / `messages` 文本摘要 | 是 |
| 给下游节点用 | `state["raw_data"]` 结构化数据 | 否 |

## 9. 本节小结

回到书房预约助手这个案例，普通节点不能独立完成所有事情，因为预约系统、规则系统、历史记录都在图外部。

本节需要掌握四点：

- 图要连接外部系统，所以需要 Tool。
- Tool 结果有时要继续推理，有时要直接返回，所以要理解 `return_direct`。
- 流程有时固定、有时开放，所以要区分 Tool 直接当节点和 `ToolNode`。
- 真实系统会出错、会重复查、会结果过长，所以 Tool 必须有兜底、缓存和上下文管理。

## 10. 常见误区

| 坑 | 现象 | 怎么避免 |
|----|------|---------|
| Tool 示例和主案例无关 | 只知道语法，看不出设计目标 | 用同一条业务线贯穿所有示例 |
| docstring 写得过短 | LLM 不会稳定调用工具 | 把 docstring 当成说明书来写 |
| Tool 抛异常 | 整张图中断 | 返回可恢复错误信息，让 LLM 决定下一步 |
| 原始 JSON 直接 return 给 LLM | token 浪费、模型难理解 | return 文本摘要，原始数据写入 State |
| 什么都交给 ToolNode | 业务流程失控 | 固定流程直接 `add_node(tool)` |
| 什么都手工编排 | 失去开放式 Tool 决策能力 | 开放域助手用 `ToolNode + tools_condition` |
