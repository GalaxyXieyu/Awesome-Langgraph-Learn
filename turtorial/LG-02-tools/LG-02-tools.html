开场白：上节课我们画了一张主要在图里流转状态的图。但真实世界里，用户要查可预约时段、查规则、提交预约、查历史记录，图必须能调用外部能力。今天聚焦核心问题：图怎么调用外部能力（Tool）。更重要的是，Tool 本身就是 LangGraph 的一等公民，可以直接当节点用。
Scenario First
先看场景：一个更像生产环境的书房预约助手

目标场景

用户输入
  → LLM 判断要不要调工具、调几个工具
    ├─ query_availability(room_name, booking_date)
    │  └─ 透传 user_id / tenant_id / role 做权限校验
    ├─ query_room_rules(room_name)
    │  └─ 透传上下文命中偏好或缓存
    ├─ create_booking(slot_id)
    └─ query_booking_history(user_id)
  → 某些结果直出，某些结果回 LLM 继续组织答复

这不是简单 demo，而是一个带权限、上下文透传、结果流向控制、Tool 复用的真实预约业务问题。

问题复杂度

这个场景同时包含工具选择、权限校验、上下文透传、缓存命中和结果流向控制，不是单一工具调用。

核心设计点

Tool 负责封装外部能力；运行时注入负责透传状态与存储；`return_direct` 控制结果回路；ToolNode 负责自动分发工具。

系统目标

在一个统一的预约入口中，让多个 Tool 可组合调用、可复用、可控权限，并支持后续由 LLM 继续组织最终答复。

LangGraph 渐进式教程 LG-02
同一个预约入口里，查空位、查规则、提交预约、查历史四个能力可能同时出现；其中预约工具依赖 user_id、tenant_id、role 做权限校验，规则工具可以结合历史偏好和缓存，部分结果需要直出，部分结果需要回到 LLM 继续组织答复。

RESULT FLOW CONTROL
return_direct：控制结果流向
决定 Tool 结果是回到 LLM 继续推理，还是直接返回给用户

return_direct=False（默认）

查可预约时段后，结果回到 LLM，继续组织回复或推荐候选时段

@tool def query_availability(room_name: str, booking_date: str) -> str: """查询可预约时段。结果需要 LLM 进一步整理回复。""" return booking_api_query(room_name, booking_date)

适用：查询空位后需 LLM 整理、预约状态需判断下一步

return_direct=True

结果直出，不再经过 LLM 加工

@tool(return_direct=True) def calculate_booking_duration(start_time: str, end_time: str) -> str: """计算预约时长。结果直接展示给用户。""" return "预约时长 2 小时"

适用：预约时长、固定格式卡片/图片

LangGraph 渐进式教程 LG-02 / 10
return_direct 控制 Tool 执行后的结果去向。默认 False 用于查空位、查规则这种还需要 LLM 继续解释的场景；True 用于算预约时长、返回固定结果这类工具本身已给出最终答案的场景。

生产级 Tool 五层兜底设计
从输入到透传，每一层都有明确的职责边界

① 输入层

  • Pydantic 校验
  • 缺失参数检查
  • 参数归一化

② 执行层

  • 业务逻辑
  • 外部 API 调用

③ 容错层

  • 异常捕获
  • 候选模糊匹配
  • 可恢复重试

④ 输出层

  • LLM 摘要
  • 结构化错误返回

⑤ 透传层

  • State 写入
  • 格式化文本返回
  • 日志记录

原始数据写入 state

CACHING STRATEGIES
Tool 缓存机制：省钱利器
三种缓存对象、五种机制，按场景选型
机制 缓存对象 适用场景 持久化
InMemoryCache() LLM 调用 相同 prompt 重复查询（开发） 否
RedisCache() LLM 调用 生产环境多实例共享 是
@lru_cache Tool 结果 单进程内重复 tool 调用 否
BaseStore + InjectedStore() 状态/记忆 跨会话持久化、历史检索 是
自定义 Redis wrapper Tool 结果 跨会话共享 tool 结果 是
# 进程内 Tool 缓存（最简单） from functools import lru_cache @tool @lru_cache(maxsize=128) def query_room_rules(room_name: str) -> str: """查询书房规则（单进程内缓存 128 条）。""" return room_rules_api(room_name)
LangGraph 渐进式教程 LG-02 / 10
所有缓存必须有 TTL，根据业务新鲜度设置。书房规则可以更长，实时空位要更短。LangChain 官方提供的是 LLM 调用缓存，不是 Tool 结果缓存，后者需要自行实现。
TOOL AS FIRST-CLASS NODE
Tool 直接作为 Node：定义一次，到处复用
Tool 实现 Runnable 接口，可直接 add_node，也可包装成 ToolNode

方式 A：Tool 直接 add_node

精确控制 Tool 何时执行，用于固定预约流程

builder.add_node("query_order", query_order) builder.add_edge("verify_user", "query_order") builder.add_edge("query_order", "generate_reply")

控制权：开发者精确控制

方式 B：ToolNode + tools_condition

LLM 自主决定调哪个 Tool，用于开放式预约助手

tools = [query_order, cancel_order] tool_node = ToolNode(tools) builder.add_node("tools", tool_node) builder.add_conditional_edges("llm", tools_condition)

控制权：LLM 自主决策

核心洞察：同一个 query_availability Tool 可在预约图、客服图、后台管理图中复用——统一参数校验、统一错误处理、统一缓存策略。

LangGraph 渐进式教程 LG-02 / 10
ToolNode 本质上就是 LangGraph 帮你写好的“自动路由节点”——检查 LLM 的 tool_calls，自动分发到对应 Tool。Tool 直接 add_node 是手动编排流程。常见坑：ToolNode 必须跟在 LLM 节点后面。
CONTEXT MANAGEMENT
防止 Tool 结果撑爆上下文窗口
五种策略，从简单粗暴到智能优雅
策略 实现难度 信息损失 适用场景
Truncate（截断） 低 高（后半截丢了） 网页抓取、文档阅读
选择性传递 中 低（按需提取） API 返回结构化数据
Message Trimming 低 中（早期消息丢了） 长对话场景
Summarization 高 可控（LLM 智能压缩） 需保留上下文的复杂对话
格式化文本 + State 写入 中 无 大结果量 + 下游需原始数据
# 标准机制：return 纯文本摘要，原始数据写入 state @tool async def query_availability(room_name: str, booking_date: str) -> str: raw_data = await weather_api(city) state["weather_raw"] = raw_data # 原始数据 → state（下游可用，不占 LLM 上下文） return f"{city}今天晴，气温{raw_data['temp']}°C。" # 纯文本 → LLM
LangGraph 渐进式教程 LG-02 / 10
核心原则：不要把原始 JSON 直接 return 给 LLM。return 永远放自然语言摘要；原始数据通过 state 旁路传递。这是 LangGraph 官方推荐的标准做法。
MODEL CONTEXT PROTOCOL
MCP：标准化工具协议
Anthropic 推出的开放协议，统一大模型与外部工具的集成方式

传统方式：每个工具写适配器

预约系统 API → booking_adapter 预约记录 API → history_adapter 数据库 → db_adapter 每个工具都要写不同代码...

MCP 方式：统一协议

所有工具统一暴露为 MCP Server ↓ LangGraph 通过 MCP Client 连接 ↓ 自动获得 LangChain-compatible tools

三种 Transport

  • stdio：本地脚本、开发调试
  • SSE：远程服务、微服务架构
  • streamable_http：生产环境、高并发

核心代码

from langchain_mcp_adapters import MultiServerMCPClient async with MultiServerMCPClient({...}) as client: tools = await client.get_tools() agent = create_react_agent(model, tools)
LangGraph 渐进式教程 LG-02 / 10
MCP 的核心思想：工具的实现细节留在 MCP Server 里，LangGraph 只关心有哪些工具、参数是什么。支持多语言写 Server，Python/Node/Go 都可以。项目早期可以用传统 @tool，多团队协作时再引入 MCP。

PROMPT & SKILLS MANAGEMENT
Prompt 集中管理 + Skills 动态加载
从散落在代码各处到版本可控、配置化启用

LangSmith Hub：Push / Pull

from langsmith import Client client = Client() # 推送 url = client.push_prompt("my-prompts/cs", object=prompt) # 拉取 prompt = client.pull_prompt("my-prompts/cs")
  • 版本控制：语义化版本号
  • A/B 测试：多版本并行
  • 回退：远程失败用本地 fallback

Skills 注册表：按需组合

SKILL_REGISTRY = { "availability": lambda: build_availability_tool(...), "rules": lambda: build_rules_tool(...), "history": lambda: build_history_tool(...), "booking": lambda: build_booking_tool(...), } tools = [SKILL_REGISTRY[name]() for name in enabled_skills] agent = create_react_agent(model, tools)
  • 普通用户：availability + rules + history
  • 管理员：availability + rules + history + booking
  • 从 YAML 配置动态加载

ConfigurableField / ConfigurableAlternatives：运行时切换模型、参数、prompt，不用改代码

LangGraph 渐进式教程 LG-02 / 10
Prompt 集中到 Hub 或配置文件管理。Skills 本质是 Tools 的集合，用注册表模式按需组合。运行时动态切换用 ConfigurableField/Alternatives。
