TOOL DEFINITION
Tool 的五种定义方式
从快速原型到生产级运行时注入，按场景选型
方式 适用场景 复杂度 推荐度
@tool 参数少、快速验证 低 原型阶段
@tool + args_schema 参数多、需要校验、生产环境 中 生产推荐
StructuredTool.from_function() 动态生成、配置化 中 动态场景
继承 BaseTool 自定义元数据、权限、状态 高 深度定制

# 核心原则：docstring = LLM 看到的工具说明书 @tool(args_schema=WeatherInput) def query_weather(city: str, date: str | None = None) -> str: """查询指定城市的实时天气，可选返回空气质量。""" ...
LangGraph 渐进式教程 LG-02 / 10
强调 docstring 的重要性：LLM 看到的就是 docstring + args_schema 的描述。写不清楚，LLM 就不会用。InjectedState/InjectedStore 是 LangGraph 1.x 的新能力，替代 Factory 闭包，代码更干净。
RESULT FLOW CONTROL
return_direct：控制结果流向
决定 tool 输出是回到 LLM 继续推理，还是直接返回给用户

return_direct=False（默认）

结果回到 LLM，可继续思考、调用其他工具

@tool def query_weather(city: str) -> str: """查询天气。结果需要 LLM 进一步整理回复。""" return query_weather_api(city)

适用：查询天气后需 LLM 整理、订单状态需判断下一步

return_direct=True

结果直出，不再经过 LLM 加工

@tool(return_direct=True) def get_time() -> str: """获取当前时间。结果直接展示给用户。""" return f"现在是 {datetime.now().strftime('%H:%M')}"

适用：计算器数值、固定格式卡片/图片

LangGraph 渐进式教程 LG-02 / 10
return_direct 是一个容易被忽略但很重要的参数。默认 False 适合需要 LLM 二次加工的场景；True 适合工具本身已给出最终答案的场景，减少一次 LLM 调用，省钱。
PRODUCTION-READY PATTERN
生产级 Tool 五层兜底设计
从输入到透传，每一层都有明确的职责边界

① 输入层

  • Pydantic 校验
  • 缺失参数检查
  • 参数归一化

② 执行层

  • 业务逻辑
  • 外部 API 调用

③ 容错层

  • 异常捕获
  • 候选模糊匹配
  • 可恢复重试

④ 输出层

  • LLM 摘要
  • 结构化错误返回

⑤ 透传层

  • State 写入
  • 格式化文本返回
  • 日志记录

核心原则
return 纯文本给 LLM
原始数据写入 state
下游节点直接读取

LangGraph 渐进式教程 LG-02 / 10
五层设计是骨架，真实项目还需要参数归一化、多层嵌套 state 查找、模糊匹配、结构化错误返回等兜底机制。关键是：Tool 应该返回错误信息（字符串）而非抛异常，抛异常会打断整个图。
CACHING STRATEGIES
Tool 缓存机制：省钱利器
三种缓存对象、五种机制，按场景选型
机制 缓存对象 适用场景 持久化
InMemoryCache() LLM 调用 相同 prompt 重复查询（开发） 否
RedisCache() LLM 调用 生产环境多实例共享 是
@lru_cache Tool 结果 单进程内重复 tool 调用 否
BaseStore + InjectedStore() 状态/记忆 跨会话持久化、历史检索 是
自定义 Redis wrapper Tool 结果 跨会话共享 tool 结果 是
# 进程内 Tool 缓存（最简单） from functools import lru_cache @tool @lru_cache(maxsize=128) def query_weather(city: str, date: str | None = None) -> str: """查询天气（单进程内缓存 128 条）。""" return query_weather_api(city, date)
LangGraph 渐进式教程 LG-02 / 10
所有缓存必须有 TTL，根据业务新鲜度设置。天气 10 分钟，汇率 1 分钟。LangChain 官方提供的是 LLM 调用缓存，不是 Tool 结果缓存，后者需要自行实现。
TOOL AS FIRST-CLASS NODE
Tool 直接作为 Node：定义一次，到处复用
Tool 实现 Runnable 接口，可直接 add_node，也可包装成 ToolNode

方式 A：Tool 直接 add_node

精确控制 tool 何时执行，适合固定业务流程

builder.add_node("query_order", query_order) builder.add_edge("verify_user", "query_order") builder.add_edge("query_order", "generate_reply")

控制权：开发者精确控制

方式 B：ToolNode + tools_condition

LLM 自主决定调哪个 tool，适合开放域

tools = [query_order, cancel_order] tool_node = ToolNode(tools) builder.add_node("tools", tool_node) builder.add_conditional_edges("llm", tools_condition)

控制权：LLM 自主决策

核心洞察：同一个 query_order Tool 可在客服图、售后图、报表图、管理后台图中复用——统一权限校验、统一错误处理、统一缓存策略。

LangGraph 渐进式教程 LG-02 / 10
ToolNode 本质上就是 LangGraph 帮你写好的"自动路由节点"——检查 LLM 的 tool_calls，自动分发到对应 tool。Tool 直接 add_node 是手动编排流程。常见坑：ToolNode 必须跟在 LLM 节点后面。
CONTEXT MANAGEMENT
防止 Tool 结果撑爆上下文窗口
五种策略，从简单粗暴到智能优雅
策略 实现难度 信息损失 适用场景
Truncate（截断） 低 高（后半截丢了） 网页抓取、文档阅读
选择性传递 中 低（按需提取） API 返回结构化数据
Message Trimming 低 中（早期消息丢了） 长对话场景
Summarization 高 可控（LLM 智能压缩） 需保留上下文的复杂对话
格式化文本 + State 写入 中 无 大结果量 + 下游需原始数据
# 标准机制：return 纯文本摘要，原始数据写入 state @tool async def query_weather(city: str) -> str: raw_data = await weather_api(city) state["weather_raw"] = raw_data # 原始数据 → state（下游可用，不占 LLM 上下文） return f"{city}今天晴，气温{raw_data['temp']}°C。" # 纯文本 → LLM
LangGraph 渐进式教程 LG-02 / 10
核心原则：不要把原始 JSON 直接 return 给 LLM。return 永远放自然语言摘要；原始数据通过 state 旁路传递。这是 LangGraph 官方推荐的标准做法。
MODEL CONTEXT PROTOCOL
MCP：标准化工具协议
Anthropic 推出的开放协议，统一大模型与外部工具的集成方式

传统方式：每个工具写适配器

天气 API → weather_adapter 股票 API → stock_adapter 数据库 → db_adapter 每个工具都要写不同代码...

MCP 方式：统一协议

所有工具统一暴露为 MCP Server ↓ LangGraph 通过 MCP Client 连接 ↓ 自动获得 LangChain-compatible tools

三种 Transport

  • stdio：本地脚本、开发调试
  • SSE：远程服务、微服务架构
  • streamable_http：生产环境、高并发

核心代码

from langchain_mcp_adapters import MultiServerMCPClient async with MultiServerMCPClient({...}) as client: tools = await client.get_tools() agent = create_react_agent(model, tools)
LangGraph 渐进式教程 LG-02 / 10
MCP 的核心思想：工具的实现细节留在 MCP Server 里，LangGraph 只关心有哪些工具、参数是什么。支持多语言写 Server，Python/Node/Go 都可以。选型建议：项目早期用传统 @tool，多团队协作用 MCP。
PROMPT & SKILLS MANAGEMENT
Prompt 集中管理 + Skills 动态加载
从散落在代码各处到版本可控、配置化启用

LangSmith Hub：Push / Pull

from langsmith import Client client = Client() # 推送 url = client.push_prompt("my-prompts/cs", object=prompt) # 拉取 prompt = client.pull_prompt("my-prompts/cs")
  • 版本控制：语义化版本号
  • A/B 测试：多版本并行
  • 回退：远程失败用本地 fallback

Skills 注册表：按需组合

SKILL_REGISTRY = { "weather": lambda: build_weather_tool(...), "stock": lambda: build_stock_tool(...), "calculator": lambda: build_calculator_tool(), } tools = [SKILL_REGISTRY[name]() for name in enabled_skills] agent = create_react_agent(model, tools)
  • 普通用户：weather + calculator
  • VIP 用户：全部 skills
  • 从 YAML 配置动态加载

ConfigurableField / ConfigurableAlternatives：运行时切换模型、参数、prompt，不用改代码

LangGraph 渐进式教程 LG-02 / 10
Prompt 不要散落在代码各处，集中到 Hub 或配置文件管理。Skills 本质是 Tools 的集合，用注册表模式按需组合。运行时动态切换用 ConfigurableField/Alternatives。
