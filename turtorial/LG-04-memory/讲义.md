## 记忆系统全景

---

### 1. 开场引入

> "前三节课的图有一个共同问题：State 存在内存里，`invoke()` 一结束就消失，进程重启清零，刷新页面断片。
>
> 这节课解决一个问题：**让图记住**——不是简单的『存到数据库』，而是一套分层的记忆系统：State（瞬时）、Checkpoint（短期自动）、Store（长期手动）、Cache（加速层）。每层有不同的存储方式、查询方式和成本结构。"

#### 与 LG-03 的衔接

> "上节课的人机协作依赖 checkpoint 保存中断状态。今天我们要把这套机制讲透——从开发环境的一行代码，到生产环境的一套架构。"

### 2. 核心内容

#### 2.1 记忆分层全景图

**讲解顺序**：
1. **四层记忆模型**：State / Checkpoint / Store / Cache
2. **各层对比表**：生命周期、管理方、查询方式、成本
3. **为什么推荐 PostgreSQL 作为统一底座**—— 预告，后面展开

**四层记忆架构图**：
```
┌─────────────────────────────────────────────────────────────────────┐
│ 记忆分层全景图 │
├─────────────────────────────────────────────────────────────────────┤
│ Layer 1: 工作记忆 (Working Memory) │
│ ├─ 载体: State（Python dict / TypedDict / Pydantic） │
│ ├─ 生命周期: 一个 super-step（节点执行期间存在） │
│ ├─ 管理方: 你直接读写 │
│ └─ 丢失: 节点 return 后，只保留写入 channel 的字段 │
├─────────────────────────────────────────────────────────────────────┤
│ Layer 2: 短期记忆 (Short-term Memory) = Checkpoint │
│ ├─ 载体: Postgres / SQLite / Redis 中的 checkpoints 表 │
│ ├─ 生命周期: 整个会话（按 thread_id 隔离） │
│ ├─ 管理方: LangGraph 自动保存（每个 super-step 结束） │
│ ├─ 用途: 对话续接、中断恢复、时间旅行 │
│ └─ 成本: 每次 super-step 一次写，线性增长 │
├─────────────────────────────────────────────────────────────────────┤
│ Layer 3: 长期记忆 (Long-term Memory) = Store │
│ ├─ 载体: Postgres store 表（JSONB + 可选向量索引） │
│ ├─ 生命周期: 你决定（可设 TTL，可永久） │
│ ├─ 管理方: 你在节点中手动 `aput` / `aget` / `asearch` │
│ ├─ 用途: 用户偏好、知识库、历史产物 │
│ └─ 成本: 按需读写，O(1) 精确查询 / O(log n) 向量检索 │
├─────────────────────────────────────────────────────────────────────┤
│ Layer 4: 加速记忆 (Fast Memory) = Cache │
│ ├─ 载体: Postgres / Redis（复用同一连接池） │
│ ├─ 生命周期: TTL 控制（30 秒 ~ 7 天） │
│ ├─ 管理方: LangGraph 自动命中/写入 │
│ ├─ 用途: 避免重复计算（API 调用、LLM 调用、DB 查询） │
│ └─ 成本: 几乎为零（命中时直接返回，未命中时多一次写） │
└─────────────────────────────────────────────────────────────────────┘
```

**各层对比表**：

| 维度 | State（工作记忆） | Checkpoint（短期） | Store（长期） | Cache（加速） |
|------|----------------|-------------------|--------------|--------------|
| **谁在管** | 你 | LangGraph 自动 | 你手动 | LangGraph 自动 |
| **存在多久** | 1 个 super-step | 整个 thread 生命周期 | 永久（或 TTL） | TTL 到期 |
| **按什么隔离** | 当前执行上下文 | `thread_id` | `namespace + key` | 输入哈希 |
| **怎么查** | 直接 dict 访问 | `get_state(config)` | `aget/aput/asearch` | 自动命中 |
| **数据内容** | 完整 state | 完整 state 快照 | 提炼后的业务数据 | 节点输出结果 |
| ** Postgres 表** | 内存 | `checkpoints` | `store` / `store_vectors` | `checkpoints`（特殊前缀） |
| **典型成本** | 零 | 每次 step 一次写 | 按需读写 | 一次哈希比较 |

#### 2.2 Checkpoint：短期记忆

**讲解顺序**：
1. **Checkpoint 的本质：每个 super-step 结束后的状态快照**
2. **自动保存触发点**：节点执行完 → writes 应用 → checkpoint 写入
3. **数据库里长什么样**：Postgres 表结构 + 现场 SQL 查询
4. **为什么选 PostgreSQL**：ACID + JSONB + pgvector + pgBouncer
5. **开发到生产的迁移路径**：MemorySaver → PostgresSaver

**2.2.1 Checkpoint 的本质**

LangGraph 的执行单元是 **super-step**（超级步）。每个 super-step 的完整流程：

```
[读取 State] → [执行节点] → [节点返回 Writes] → [应用 Writes 到 Channel]
→ [生成新 Checkpoint] → [写入 Saver]
```

**关键**：Checkpoint 是**自动**的。你不需要在节点里写任何保存逻辑，只要编译时传了 `checkpointer=...`，LangGraph 就会在每次 super-step 后自动保存。

**Checkpoint 的组成**：
- `channel_values`: 当前所有 channel 的值（即 State）
- `channel_versions`: 每个 channel 的版本号（用于判断是否需要更新）
- `versions_seen`: 每个节点看到的版本号（用于去重和一致性）
- `pending_writes`: 已产生但尚未应用的 writes（中断恢复时用到）

**2.2.2 Postgres 中的表结构（实操）**

当你使用 `PostgresSaver` 时，LangGraph 会在数据库中创建 4 张表：

```sql
-- 1. checkpoints 主表：状态快照
CREATE TABLE checkpoints (
thread_id TEXT NOT NULL,
checkpoint_ns TEXT NOT NULL DEFAULT '',
checkpoint_id TEXT NOT NULL,
parent_checkpoint_id TEXT,
type TEXT,
checkpoint JSONB NOT NULL,
metadata JSONB NOT NULL DEFAULT '{}',
PRIMARY KEY (thread_id, checkpoint_ns, checkpoint_id)
);

-- 2. checkpoint_blobs：大对象存储（消息历史等二进制数据）
CREATE TABLE checkpoint_blobs (
thread_id TEXT NOT NULL,
checkpoint_ns TEXT NOT NULL DEFAULT '',
channel TEXT NOT NULL,
version TEXT NOT NULL,
type TEXT NOT NULL,
blob BYTEA,
PRIMARY KEY (thread_id, checkpoint_ns, channel, version)
);

-- 3. checkpoint_writes：写操作日志（事务级记录）
CREATE TABLE checkpoint_writes (
thread_id TEXT NOT NULL,
checkpoint_ns TEXT NOT NULL DEFAULT '',
checkpoint_id TEXT NOT NULL,
task_id TEXT NOT NULL,
task_path TEXT NOT NULL,
idx INTEGER NOT NULL,
channel TEXT NOT NULL,
type TEXT,
blob BYTEA NOT NULL,
PRIMARY KEY (thread_id, checkpoint_ns, checkpoint_id, task_id, idx)
);

-- 4. checkpoint_migrations：版本管理
CREATE TABLE checkpoint_migrations (
v INTEGER NOT NULL PRIMARY KEY
);
```

**现场实操**：
```bash
# 连接数据库，直接查看 checkpoint 内容
psql $DATABASE_URL -c "
SELECT
thread_id,
checkpoint_id,
jsonb_pretty(checkpoint->'channel_values') as state_preview,
metadata
FROM checkpoints
WHERE thread_id = 'demo-thread-1'
ORDER BY checkpoint_id;
"
```

**2.2.3 为什么 PostgreSQL 是生产首选？**

LangGraph 官方支持 MemorySaver、PostgresSaver、RedisSaver、SQLiteSaver。但生产环境首推 PostgreSQL，原因有三：

**原因一：ACID + JSONB —— Checkpoint 需要强一致性**

Checkpoint 是状态快照，一旦丢失就意味着用户对话断片。PostgreSQL 的 ACID 事务保证写入要么成功要么回滚，不会留下半吊子状态。JSONB 类型完美兼容 LangGraph state 的嵌套字典结构，同时又可以用 SQL 做运维查询：

```sql
-- 查询某个 thread 的最新 checkpoint
SELECT checkpoint_id, metadata->>'timestamp' as ts
FROM checkpoints
WHERE thread_id = 'u123_s456'
ORDER BY checkpoint_id DESC
LIMIT 1;

-- 统计每个 thread 占用了多少空间
SELECT thread_id,
pg_size_pretty(sum(pg_column_size(checkpoint))) as total_size,
count(*) as checkpoint_count
FROM checkpoints
GROUP BY thread_id
ORDER BY sum(pg_column_size(checkpoint)) DESC
LIMIT 10;

-- 查找包含特定 tool_call 的 checkpoint（调试用）
SELECT thread_id, checkpoint_id
FROM checkpoints
WHERE checkpoint->'channel_values'->'messages' @> '[{"tool_calls": [{}]}]';
```

对比：
- **Redis**：纯 KV，无事务保证，AOF 可能丢几秒数据
- **SQLite**：单文件，多进程写入会锁竞争，不适合生产并发
- **Memory**：进程一挂全部清零

**原因二：pgvector 扩展 —— 不需要单独维护向量数据库**

LangGraph Store 支持语义检索（向量搜索），而 pgvector 是 PostgreSQL 的官方扩展。这意味着：
- 一张表 `store` 存结构化数据（JSONB）
- 一张表 `store_vectors` 存向量嵌入
- 同一个数据库、同一个连接池、同一个备份策略

不需要额外维护 Milvus、Pinecone、Weaviate 一套基础设施，减少了运维复杂度、网络延迟和成本。

```sql
-- pgvector 创建的向量表
CREATE TABLE store_vectors (
prefix TEXT NOT NULL,
key TEXT NOT NULL,
field_name TEXT NOT NULL,
embedding VECTOR(1536), -- 向量类型，维度可配
created_at TIMESTAMPTZ,
PRIMARY KEY (prefix, key, field_name)
);

-- HNSW 近似最近邻索引
CREATE INDEX store_vectors_hnsw
ON store_vectors
USING hnsw (embedding vector_cosine_ops);
```

pgvector 支持的索引和距离：

| 索引类型 | 适用场景 | 距离类型 |
|---------|---------|---------|
| `hnsw`（默认） | 高维向量，查询快，构建慢 | `cosine`、`l2`、`inner_product` |
| `ivfflat` | 中等规模，构建快，查询稍慢 | `cosine`、`l2`、`inner_product` |
| `flat` | 小规模或精确查询，无近似误差 | 任意 |

向量类型选择：
- `vector`：全精度，默认推荐
- `halfvec`：半精度，省 50% 内存，精度损失通常可忽略

**原因三：pgBouncer —— 高并发下的连接管理**

LangGraph 每个 super-step 都会触发 checkpoint 读写。100 个并发用户就意味着每秒几十上百次数据库操作。直接连 Postgres 会快速耗尽 `max_connections`（默认 100）。pgBouncer 作为中间代理层解决这个：

```yaml
# pgbouncer.ini 推荐配置
[databases]
langgraph = host=pg-primary port=5432 dbname=mydb

[pgbouncer]
listen_port = 6432
pool_mode = transaction # 事务级复用，性能最好
max_client_conn = 10000
default_pool_size = 25
reserve_pool_size = 5
server_lifetime = 3600
```

| 场景 | 直接连 PG | 经 pgBouncer |
|------|----------|-------------|
| 100 并发用户 | 100+ 连接，可能打满 | 25 个实际连接，事务级复用 |
| 短连接风暴 | 频繁创建/销毁，开销大 | 连接复用，毫秒级 |
| 连接泄漏 | bug 导致不释放，最终耗尽 | 统一管理超时回收 |
| 故障切换 | 应用感知主从切换 | pgbouncer 层配置故障转移 |

应用层 DSN：
```python
# 应用 → pgBouncer → PostgreSQL
DSN = "postgresql://user:pass@pgbouncer:6432/langgraph"
pool = psycopg_pool.AsyncConnectionPool(DSN, max_size=50)
saver = PostgresSaver(pool)
```

> **总结**：PostgreSQL = 关系型查询能力（ACID + JSONB SQL）+ 向量检索能力（pgvector）+ 连接管理能力（pgBouncer）。一张数据库解决三个问题，这是选择它作为 LangGraph 生产底座的核心逻辑。

**2.2.4 开发到生产的迁移路径**

```python
# ===== 开发环境 =====
from langgraph.checkpoint.memory import MemorySaver

graph = builder.compile(checkpointer=MemorySaver())
# 零配置，零依赖，进程重启清零

# ===== 生产环境 =====
from langgraph.checkpoint.postgres import PostgresSaver
import psycopg_pool

DSN = (
"postgresql://user:pass@pgbouncer:6432/langgraph"
"?options=-c%20search_path%3Dai_ns%2Cpublic"
)
pool = psycopg_pool.AsyncConnectionPool(DSN)
saver = PostgresSaver(pool)
await saver.setup() # 幂等建表

graph = builder.compile(checkpointer=saver)
# 持久化、可恢复、可查询、可扩展
```

**资源工厂模式（生产推荐）**：

```python
# backend/src/infrastructure/ai/graph/memory/saver.py
import psycopg_pool
from langgraph.checkpoint.postgres import PostgresSaver

_saver_instance: PostgresSaver | None = None

async def initialize_checkpointer(db_key: str = "MAIN_DB"):
global _saver_instance
dsn = build_dsn_from_env(db_key)
pool = psycopg_pool.AsyncConnectionPool(dsn)
_saver_instance = PostgresSaver(pool)
await _saver_instance.setup()

def get_global_checkpointer() -> PostgresSaver:

```

**2.2.5 `thread_id` 与会话隔离**

```python
# 每次调用必须传入 config，包含 thread_id
config = {
"configurable": {
"thread_id": "user-123-session-456",
}
}

# 第一次调用：创建新 checkpoint
result = graph.invoke({"messages": [("user", "你好")]}, config)

# 第二次调用（同一个 thread_id）：自动加载上次 checkpoint
result = graph.invoke({"messages": [("user", "我叫小明")]}, config)
```

`thread_id` 设计建议：

| 场景 | thread_id 格式 | 示例 |
|------|---------------|------|
| 单用户单会话 | `{user_id}_{session_id}` | `u123_s456` |
| 单用户多会话 | `{user_id}_{timestamp}` | `u123_20241013` |
| 匿名用户 | `anon_{uuid}` | `anon_a1b2c3d4` |

> **⚠️ 关键警告**：两个用户使用同一个 `thread_id`，他们会看到彼此的对话历史。`thread_id` **不做用户隔离**，用户隔离靠 `user_id` + Store namespace。

#### 2.3 Store：长期记忆

**讲解顺序**：
1. **Checkpoint vs Store：本质区别**
2. **命名空间设计：数据隔离的底线**
3. **Store API 实操：aput / aget / asearch**
4. **语义检索：pgvector 向量搜索**
5. **何时用 Store、何时用 Saver？**

**2.3.1 Checkpoint vs Store**

| 维度 | Checkpoint（短期记忆） | Store（长期记忆） |
|------|-------------------|-------------------|
| **谁在管** | LangGraph 自动 | 你手动读写 |
| **存在多久** | 随对话自动增长 | 永久（或 TTL） |
| **数据内容** | 完整的执行状态 | 提炼后的业务数据 |
| **查询方式** | 按 thread_id 线性遍历 | 按 namespace + key 精确/语义检索 |
| **典型用途** | 对话恢复、中断续跑 | 用户偏好、知识库、历史记录 |
| **类比** | 游戏自动存档 | 玩家手动保存的装备配置 |

**2.3.2 命名空间设计**

Store 的 namespace 是元组形式，推荐 `(bucket, user_id)` 二元组：

```python
ns_preferences = ("user_prefs", user_id) # 用户偏好
ns_history = ("chat_history", user_id) # 跨会话历史摘要
ns_knowledge = ("knowledge", user_id) # 个人知识库
ns_deepresearch = ("deepresearch:artifacts", user_id) # 深研产物
```

数据库中的存储形式：
```sql
CREATE TABLE store (
prefix TEXT NOT NULL, -- "user_prefs.u123"
key TEXT NOT NULL, -- 键
value JSONB NOT NULL, -- 值（JSON）
created_at TIMESTAMPTZ,
updated_at TIMESTAMPTZ,
expires_at TIMESTAMPTZ,
ttl_minutes INT,
PRIMARY KEY (prefix, key)
);
```

**2.3.3 Store API 实操**

```python
from langgraph.store.base import BaseStore

async def save_preference_node(state, *, config, store: BaseStore):
user_id = config.get("configurable", {}).get("user_id", "anon")
ns = ("user_prefs", user_id)
await store.aput(namespace=ns, key="theme", value={"dark": True})
return {}

async def load_preference_node(state, *, config, store: BaseStore):
user_id = config.get("configurable", {}).get("user_id", "anon")
item = await store.aget(("user_prefs", user_id), "theme")
if item:
return {"preferences": item.value}
return {"preferences": {}}
```

**封装 helpers**：
```python
# src/application/memory/helpers.py
async def put_user_kv(store, config, bucket: str, key: str, value):
```

**2.3.4 语义检索（pgvector）**

```python

class EmbeddingAdapter:
async def aembed_documents(self, texts: list[str]) -> list[list[float]]:
return await embedding_client.embed(texts)

"dims": 1536,

```

向量检索：
```python
items = await store.asearch(
("report_prefs", user_id),
query="报告的格式和风格要求",
filter={"type": {"$eq": "format"}},
limit=5,

for it in items:
```

**Store API 速查表**：

| 方法 | 用途 | 关键参数 |
|------|------|---------|
| `aput(ns, key, value)` | 写入/更新 | `index=False` 跳过向量索引 |
| `aget(ns, key)` | 精确读取 | `refresh_ttl=True` 刷新过期时间 |
| `asearch(ns, ...)` | 列表/语义检索 | `query=` 语义，`filter=` JSONB 精确过滤 |

**2.3.5 何时用 Store、何时用 Checkpoint？**

| 数据类型 | 存储选择 | 理由 |
|---------|---------|------|
| 对话消息历史 | **Checkpoint** | 自动管理，支持恢复/分支 |
| 用户偏好（语言、风格） | **Store** | 跨会话长期保存，需精确读取 |
| 知识库片段 | **Store** + 向量索引 | 语义检索需求 |
| 工具调用结果缓存 | **Store** | 可选 TTL，跨会话复用 |
| 当前执行位置 | **Checkpoint** | 中断恢复需要 |

#### 2.4 Cache：加速记忆

**讲解顺序**：
1. **Cache 是记忆的第四层**
2. **CachePolicy 配置：ttl 和 key_func**
3. **缓存后端选择：复用 PostgresSaver**
4. **缓存防雪崩与 TTL 策略**

**2.4.1 Cache 的定位**

Cache 不是独立的基础设施，而是**记忆的加速层**。它和 Checkpoint 复用同一个存储后端（PostgresSaver），只是用特殊的 `thread_id` 前缀来区分：

```
Checkpoint 记录: thread_id = "u123_s456"
Cache 记录: thread_id = "cache_<hash>"
```

```python
from langgraph.types import CachePolicy

builder.add_node(
"expensive_analysis",
analysis_node,
cache_policy=CachePolicy(ttl=300) # 5 分钟缓存
)

# 复用同一个 PostgresSaver 实例
checkpointer = saver # 存 Checkpoint
cache = saver # 存节点缓存

graph = builder.compile(checkpointer=checkpointer, cache=cache)
```

**关键区分**：
- **Checkpoint**（`checkpointer=`）：存整个图的状态，用于恢复
- **Cache**（`cache=`）：存单个节点的输出，用于避免重复计算

**2.4.2 CachePolicy 详解**

```python
from langgraph.types import CachePolicy

# 基础配置
CachePolicy(ttl=300)

# 自定义缓存键：忽略随机字段
def my_key_func(args):
state = args[0]
return f"{state['user_id']}:{state['query']}"

CachePolicy(ttl=600, key_func=my_key_func)

# 条件缓存：只读操作才缓存
def conditional_key(args):
state = args[0]
if state.get("operation") == "read":
return f"read:{state['query']}"
import uuid
return str(uuid.uuid4()) # 写操作永不命中

CachePolicy(ttl=300, key_func=conditional_key)
```

**TTL 设置建议**：

| 数据类型 | TTL | 理由 |
|---------|-----|------|
| 股票价格、实时天气 | 30 秒 | 几乎实时 |
| 数据库查询结果 | 10 分钟 | 中等动态 |
| LLM 对话结果 | 30 分钟 | 对话上下文稳定 |
| 配置信息 | 24 小时 | 几乎不变 |

**2.4.3 缓存后端选择**

| 场景 | 推荐后端 | 理由 |
|------|---------|------|
| 本地开发 | MemorySaver | 简单，重启自动清理 |
| 生产环境 | PostgresSaver（复用） | 与 Checkpoint 同一后端，统一管理 |
| 极致性能 | RedisSaver | 亚毫秒级读取 |

**2.4.4 防雪崩**

```python
import random

def random_ttl():
base = 3600
offset = random.randint(-300, 300)
return base + offset

CachePolicy(ttl=random_ttl()) # 避免大量缓存同时失效
```

#### 2.5 上下文工程：控制记忆的大小

**讲解顺序**：
1. **Token 危机：为什么需要裁剪**
2. **trim_messages：Token 窗口控制**
3. **增量摘要：超长对话压缩**
4. **pre_model_hook 统一入口**

**2.5.1 Token 危机**

- 20 轮对话 ≈ 4000-6000 tokens
- 50 轮对话 ≈ 10000-15000 tokens
- 超出上下文窗口 → 报错
- 未超出但接近 → "lost in the middle"（中间信息被忽略）

**2.5.2 trim_messages**

**2.5.3 增量摘要**

```python
async def summarize_if_needed(state, *, config, llm):
messages = state.get("messages", [])
budget = config.get("configurable", {}).get("summary_budget", 400)

if count_tokens_approx(messages) > budget * 2:
early, recent = split_history(messages, keep_last=6)
summary = await llm.ainvoke(
"请将以下对话浓缩为要点摘要，保留关键实体和数字：\n"
+ format_msgs(early)
)
new_messages = [
{"type": "system", "content": f"[历史摘要] {summary.content}，如有冲突以最新输入为准"},
*recent,
]

return {}
```

**2.5.4 硬约束窗口**

```python
def window_messages(state, keep_last: int = 12):
msgs = state.get("messages", [])
if len(msgs) <= keep_last:
return {}
head = msgs[:1] if msgs and getattr(msgs[0], "type", "") == "system" else []
tail = msgs[-keep_last:]
return {"messages": head + tail}
```

> **注意**：裁剪后确保消息序列合法——以 System/Human 开始，以 Human/Tool 结束。

### 3. MemoryPal：四层记忆的实战应用

#### 案例与概念的一一对应

| 概念 | MemoryPal 中的体现 |
|------|-------------------|
| State（工作记忆） | 当前 super-step 中的消息列表 |
| Checkpoint（短期） | 多轮对话自动保存，刷新页面续接 |
| Store（长期） | 记住用户偏好：语言、风格、兴趣标签 |
| Store（语义检索） | 向量检索用户的历史记忆 |
| Cache（加速） | 频繁查询的结果缓存（天气查询） |
| trim_messages | 超长对话自动裁剪 |
| 增量摘要 | 100 轮对话压缩为摘要 |
| pre_model_hook | 每次 LLM 调用前注入偏好 + 裁剪 |
| pgBouncer | 100 并发用户的连接管理 |

#### 完整代码架构

```python
from typing import TypedDict, Annotated
from langgraph.graph import StateGraph, START
from langgraph.checkpoint.postgres import PostgresSaver
from langgraph.prebuilt import create_react_agent
from langchain_core.messages.utils import trim_messages
import psycopg_pool

# ========== 1. 初始化（应用启动时）==========
DSN = (
"postgresql://user:pass@pgbouncer:6432/langgraph"
"?options=-c%20search_path%3Dai_ns%2Cpublic"
)
pool = psycopg_pool.AsyncConnectionPool(DSN, max_size=50)
saver = PostgresSaver(pool)
await saver.setup()

# ========== 2. State ==========
class State(TypedDict):
messages: Annotated[list, add_messages]
preferences: dict

# ========== 3. 长期记忆节点 ==========
async def load_prefs_node(state, *, config, store):
user_id = config.get("configurable", {}).get("user_id", "anon")
item = await store.aget(("user_prefs", user_id), "settings")
prefs = item.value if item else {"language": "zh", "style": "concise"}
return {"preferences": prefs}

# ========== 4. 上下文裁剪 Hook ==========
def build_context_hook(max_tokens=800):
def hook(state):
msgs = state.get("messages", [])
trimmed = trim_messages(msgs, strategy="last",
token_counter=count_tokens_approximately,
max_tokens=max_tokens, include_system=True)
return {"llm_input_messages": trimmed}
return hook

# ========== 5. 构建 Agent（带缓存）==========
agent = create_react_agent(
model, tools=[weather_tool, search_tool],
pre_model_hook=build_context_hook(max_tokens=900),
)

# ========== 6. 编译（三层记忆注入）==========
app = agent.compile(
checkpointer=saver, # Checkpoint（短期记忆）
store=store, # Store（长期记忆）
cache=saver, # Cache（加速记忆）
)

# ========== 7. 运行 ==========
config = {
"configurable": {
"thread_id": "u123_session_456",
"user_id": "u123",
}
}

# 第一次对话：建立 Checkpoint + 保存偏好到 Store
await app.ainvoke(
{"messages": [("user", "你好，请用简洁的中文回答")]}, config)

# 刷新页面后，同一个 thread_id 继续——Checkpoint 自动加载
# 偏好从 Store 读出注入——长期记忆生效
await app.ainvoke(
{"messages": [("user", "我刚才说了什么偏好？")]}, config)
```

#### 关键转折点

1. **四层记忆的分工**：问 "天气查询结果应该存哪一层？"（期望：Cache，因为会重复查且不需要持久化）
2. **thread_id 与 user_id 的区别**：问 "两个用户共享 thread_id 会怎样？"（期望：看到对方对话）
3. **Cache 与 Checkpoint 复用**：问 "cache=saver 和 checkpointer=saver 会冲突吗？"（期望：不会，底层用不同 thread_id 前缀隔离）

### 4. 常见误区与注意事项

| 坑 | 现象 | 预警话术 |
|----|------|---------|
| **忘记传 config** | 每次调用都是新对话 | "持久化必须传 `config={\"configurable\": {\"thread_id\": \"...\"}}`" |
| **Store 没传 user_id** | `("prefs",)` 没有用户隔离 | "namespace 必须是 `(bucket, user_id)`" |
| **Cache 和 Checkpointer 混淆** | 以为 `cache=` 存的是状态 | "`checkpointer=` 存 Checkpoint（恢复用），`cache=` 存节点结果（性能用），可复用同一个 PostgresSaver" |
| **向量维度不匹配** | `dims=1536` 但模型输出 1024 | "检查 embedding 模型文档，pgvector 支持任意维度" |
| **没调 setup()** | 首次使用报错 "表不存在" | "`await saver.setup()` 是幂等的，每次启动都调用" |
| **上下文裁剪太激进** | LLM 丢失关键上下文 | "保留系统消息 + 最近 N 轮 + 摘要化的早期内容" |
| **pgBouncer pool_mode 设错** | 事务异常或连接不释放 | "LangGraph 用 `pool_mode = transaction`，不要用 statement" |

### 5. 阶段小结

#### 核心要点回顾

1. **四层记忆**：State（工作）→ Checkpoint（短期，自动）→ Store（长期，手动）→ Cache（加速，自动）
2. **Checkpoint**：每个 super-step 自动保存，按 `thread_id` 隔离，存在 Postgres `checkpoints` 表
3. **Store**：手动 `aput/aget/asearch`，namespace 为 `(bucket, user_id)`，支持 pgvector 语义检索
4. **Cache**：`CachePolicy(ttl, key_func)`，与 Checkpointer 复用同一 PostgresSaver，不同前缀隔离
5. **为什么 PostgreSQL**：ACID + JSONB（关系查询）+ pgvector（向量检索）+ pgBouncer（连接池），一库解决三层记忆
6. **上下文工程**：trim_messages 裁剪 + 增量摘要压缩 + pre_model_hook 统一入口

#### 记忆口诀

> **"State 是脑子，Checkpoint 是草稿，Store 是笔记本，Cache 是小纸条；**
> **PostgreSQL 装所有，pgvector 做检索，pgBouncer 管连接，四层记忆一盘棋。"**
