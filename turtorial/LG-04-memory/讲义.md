# LG-04：让图记住该记住的

> 核心冲突句：**人自然会把信息分层记忆，但图把所有东西都当作"这一次输入"。**

这节课不从 `checkpointer`、`thread_id`、`store` 开始，而是从人类世界里的记忆分类开始：

> 你问公司的 HR 助教五个问题：
>
> 1. "我叫张伟。"（**短期事实**——当前对话才需要）
> 2. "请以后用表格回答。"（**偏好**——影响未来所有回答）
> 3. "我今年36岁。"（**长期事实**——换了话题也应该记住）
> 4. "上次会议有哪三个待办？"（**会话上下文**——同一次对话的记录）
> 5. "你还记得我的偏好吗？"（**跨会话回忆**——关了窗口再开也能想起来）
>
> 人类助教会自然把每句话放到不同的"记忆位置"。但图不会。

本节课要把"记住"拆成几种机器动作：

```text
人类自然动作：把信息分层——短期、长期、偏好、事实、习惯
        ↓
Graph 的限制：每次运行只沿显式 state 流转，不会自动区分记忆类型
        ↓
学生的反直觉点：为什么同一个程序，第二次调用还会像第一次见面？
                 为什么换个窗口偏好就不见了？
                 为什么重启后所有对话都忘了？
        ↓
机制需求：短期上下文存 Checkpoint、长期偏好存 Store、持久化到数据库
        ↓
白话模型：工作台 / 聊天窗口存档 / 用户笔记本 / 数据库持久化 / 历史摘要
        ↓
API 名：state / checkpointer / thread_id / store / PostgresSaver / summary
        ↓
可运行证据：同窗口续接、换窗口失忆、跨窗口记住偏好、进程重启后恢复
```

---

## 0. 开篇：记忆不是只有一种（核心新增）

### 要讲清楚的问题

学生一听到"记忆"，可能只想到"记住刚才说过什么"。但实际上人类在自然对话中持续做着记忆分层：

- 短期：刚才说的话题是什么
- 长期：这个人的名字、年龄
- 偏好：喜欢什么格式、什么语气
- 事实：客观知识，跟具体用户无关

图不会自动做这种分层。学生需要先理解"记忆本身就有不同种类"，然后才能理解为什么 Graph 需要不同的存储机制。

### 推荐演示

用五个问题串联一个完整的 HR 助教场景，让学生自己发现"每个问题的记忆需求不同"。

### 讲法

不要上来就画 LangGraph 架构图。先让学生体会到：

> 同样是"记住一件事"，"记住刚才说过张伟"和"记住张伟喜欢表格"是两种完全不同的需求。前者只在当前窗口有效，后者需要跨窗口持久。

### 配图

使用 `images/memory-taxonomy-cn.svg`：人类记忆四种类型（短期、长期、偏好、事实）→ Graph 四种机制的映射。

### 推荐话术

> 图不会自动判断"这是一条偏好"还是"这是一个临时的聊天事实"。你需要告诉图：这条信息存哪里、用什么 ID 找回、它属于谁、生命周期是多长。

## 1. Baseline：没有存档的图为什么会失忆

### 要讲清楚的问题

人类聊天默认连续，但 Graph 调用默认不是连续聊天。它只处理这次输入和这次传入的 state。这里丢掉的，其实是**短期记忆**——上次运行时的状态。

### 推荐演示

先写一个没有保存机制的最小图：

```python
result = 图.invoke({"用户输入": "我叫张伟", "消息": []})
result = 图.invoke({"用户输入": "我是谁？", "消息": []})
```

观察输出：

```text
是否知道用户姓名: 否
原因: 第二次调用没有拿到第一次的消息——短期上下文丢失了。
```

### 讲法

不要说 "没有 checkpointer"。先说：

> 我们没有把第一轮的现场交给第二轮。人会自然记住，图不会。运行结束，现场蒸发——这就是短期记忆丢失。

---

## 2. State：当前这一轮的工作台

### 白话模型

`state` 不是随便一个字典，而是图节点之间共享的工作台。节点只能读工作台上的内容，并把更新写回工作台。

### 学生需要理解

- 节点不会读"宇宙里的所有历史"。
- 节点只能读当前 state。
- 如果历史没有进入 state，图就不知道。

### 证据输出

```text
当前工作台里的消息数: 3
第二轮是否知道姓名: 是（因为我们手动把第一轮消息搬进了 state）
```

### 教学要点

State 教给我们第一件事：图能看到什么，完全取决于 state 里有什么。短期记忆依赖 state 显式携带上下文。这一步只解释"当前运行里能看到什么"，不要急着讲持久化。

---

## 3. Checkpoint：同一个聊天窗口的自动存档

### 人类动作

聊天软件里，同一个窗口会保留上文。你第二次发消息时，窗口里还在显示前面的消息。

### Graph 的限制

Graph 不知道"这是同一个窗口"，除非你给它一个窗口编号，并让它把每次运行后的状态存起来。

### 白话模型

Checkpoint 是聊天窗口的自动存档：

```text
第一次运行结束 → 保存现场
第二次运行开始 → 根据窗口 ID 找回现场
```

### API 映射

- `MemorySaver()`：课堂里用的内存版存档本。
- `builder.compile(checkpointer=memory)`：告诉图每轮结束后自动保存。
- `config={"configurable": {"thread_id": "窗口A"}}`：告诉图去哪一个聊天窗口找存档。

### 证据输出

```text
同一个窗口第二轮是否记得姓名: 是
  第一次运行结束 → 图把现场存进'窗口A'
  第二次运行开始 → 图先找回'窗口A'的现场，再处理新输入
```

### 到这里的定位

> 我们解决了**短期记忆的自动保存和恢复**——同一个窗口，能续接。

---

## 4. thread_id：聊天窗口 ID，不是用户 ID

### 对比实验

```text
窗口A：我叫张伟 → 我是谁？
窗口B：我是谁？
```

预期证据：

```text
窗口A 是否记得姓名: 是
窗口B 是否记得姓名: 否
```

### 讲法

`thread_id` 解决的是"同一条会话线怎么续接"。它不解决"这个用户是谁"。

推荐话术：

> `thread_id` 像聊天窗口 ID。`user_id` 像账号 ID。一个账号可以开多个窗口；一个窗口只应该属于一个账号。

---

## 5. Store：换了窗口还能记住偏好

### 人类动作

助教不只是记住当前窗口，还可能记住"这个学生喜欢表格回答"。下次开新窗口时，助教仍然知道这个偏好。

### Graph 的限制

Checkpoint 跟着 `thread_id` 走。换一个窗口，旧窗口的 checkpoint 不会自动出现。

### 白话模型

Store 是用户笔记本：

```text
Checkpoint：当前聊天窗口的现场 —— 跟着 thread_id
Store：某个用户的长期资料 —— 跟着 user_id
```

### API 映射

- `store.put(("用户偏好", user_id), key="回答风格", value={...})`
- `store.get(("用户偏好", user_id), "回答风格")`
- namespace 必须带 `user_id`，否则不同用户会串资料。

### 证据输出

```text
新窗口是否记得聊天姓名（Checkpoint）: 否 — 短期记忆不跨窗口
新窗口是否读到回答风格（Store）: 是 — 长期记忆跨窗口保存
```

### 对比表

| 问题 | 放哪里 | 白话解释 | 隔离方式 | 跨窗口 |
|---|---|---|---|---|
| 当前窗口聊到哪了 | Checkpoint | 聊天窗口自动存档 | thread_id | 否 |
| 用户长期喜欢什么 | Store | 用户笔记本 | user_id + namespace | 是 |
| 当前节点正在处理什么 | State | 当前工作台 | 单次运行 | — |

## 6. 把记忆存进数据库：PostgresSaver（核心新增）

### 为什么需要这一步

前面用的 `MemorySaver` 和 `InMemoryStore` 存在 Python 进程内存里。一旦 Jupyter 重启、服务重启，短期记忆和长期记忆全部丢失。

真实产品需要持久化到数据库。LangGraph 的设计是：图的定义不变，把保存现场和保存资料的后端换掉。

### 教学策略

不要一开始就让学生配 Postgres、读 docker-compose。先讲"为什么需要"，再讲"有哪些后端"，最后讲"为什么本课选 PG"。

核心信息：

> **持久化不是重写图，而是把“内存里的存档本”换成“数据库里的存档本”。**

### LangGraph 现在支持哪些持久化后端

课堂里不要把它讲成“只有 Postgres 能存记忆”。更准确的说法是：LangGraph 把持久化能力拆成 checkpoint 和 store 两类后端。

| 后端 | 支持类型 | 课堂里怎么理解 | 参考 |
|---|---|---|---|
| 内存 | checkpoint + store | 临时草稿本，Jupyter 关掉就没了 | [Persistence 总览](https://docs.langchain.com/oss/python/langgraph/persistence) |
| SQLite | checkpoint | 本地文件级存档，适合单机实验 | [langgraph-checkpoint-sqlite](https://pypi.org/project/langgraph-checkpoint-sqlite/) |
| Postgres | checkpoint + store | 生产常用，重启、多进程、长期记忆都能放进同一套数据库 | [langgraph-checkpoint-postgres](https://pypi.org/project/langgraph-checkpoint-postgres/) |
| MongoDB | checkpoint + store | 文档数据库风格，适合已有 MongoDB 基础设施的团队 | [checkpoint](https://pypi.org/project/langgraph-checkpoint-mongodb/) / [store](https://pypi.org/project/langgraph-store-mongodb/) |
| Redis | checkpoint + store | 更偏高速读写和缓存式基础设施，适合已有 Redis 生态的团队 | [langgraph-checkpoint-redis](https://pypi.org/project/langgraph-checkpoint-redis/) |

真正选型时先看你要保存的是 checkpoint 还是 store，再看对应包是否支持你的数据库和部署方式。

### 为什么这节课选择 Postgres

本课选 PG，不是因为其他后端没价值，而是因为它最容易把生产里的记忆系统连成一条线。

| 选择 Postgres 的原因 | 对记忆系统的意义 |
|---|---|
| 同时承载 checkpoint 和 store | 短期会话现场、长期用户资料可以用同一个数据库管理 |
| 支持 `pgvector` | 长期记忆不只能按 key 读，还能做相似度检索 |
| 生态成熟 | 连接池、备份、迁移、监控、权限控制都有现成方案 |
| 可接 `PgBouncer` | Agent 请求多、并发高时，连接数更容易管理 |
| 一个数据库多种用途 | 对教学和小团队都方便，不必一上来拆很多基础设施 |

串起来就是：

```text
同一窗口续接 → 进程重启恢复 → 多进程共享 → 长期记忆检索 → 生产连接管理
```

### 对比表

| 场景 | MemorySaver + InMemoryStore | PostgresSaver + PostgresStore |
|---|---|---|
| 进程内运行 | ✅ 正常 | ✅ 正常 |
| Jupyter 内核重启后 | ❌ 所有记忆丢失 | ✅ 同一 thread_id 仍能恢复 |
| 多进程共享 | ❌ 不互通 | ✅ 同一数据库，所有进程都能读 |
| 长期记忆检索 | ⚠️ 只适合课堂演示 | ✅ 可结合 pgvector 做语义检索 |
| 适合场景 | 课堂演示、快速实验 | 生产环境、需要持久化的场景 |

### 代码变化：只改一行

```python
# 课堂
graph = builder.compile(checkpointer=MemorySaver())

# 生产（其余代码完全不变）
graph = builder.compile(checkpointer=PostgresSaver(conn_string))
```

### 证据链（三个实验）

1. **行为一致**：PostgresSaver 和 MemorySaver 同一窗口续接行为相同。
2. **核心证据**：创建新图实例（模拟进程重启），同一 thread_id 仍能恢复现场。
3. **对比证据**：MemorySaver 新实例无法恢复，证明持久化的意义。

### 环境准备

如果学生想自己跑 Postgres 部分：

```bash
# 启动 Postgres（已有 docker-compose 配置）
docker compose -f docker-compose.minimal.yml up -d postgres

# 安装 Python 包
pip install langgraph-checkpoint-postgres
```

如果没有 Postgres，本节也展示完整的代码结构和逻辑，学生可以阅读理解。

### 推荐话术

> 今天先让你看懂"为什么要持久化、怎么换后端"。图的形状完全不变，只换一行代码。这就是 LangGraph 设计的价值——机制理解后，后端切换只改一行。

## 7. 长历史需要整理：不是越多越好

### 人类动作

人不会逐字复述一整天的对话，而是记住要点。

### Graph 的限制

如果把所有消息一直塞进 state，模型输入会越来越长，成本越来越高，也更容易忽略中间信息。

### 白话模型

当聊天太长时，把早期内容压缩成一条"历史摘要"，只保留最近几轮原文。

### 证据输出

```text
整理前消息数: 10
整理后消息数: 5
是否保留最近问题: 是
是否生成历史摘要: 是
```

### 关键心智模型

> 长期运行的图不能把所有原文无限塞进上下文。早期历史要整理成更短、更稳定的记忆，但关键事实（姓名、偏好）不能丢。

## 8. 本节小结：记忆全景图

人的记忆有不同类型，图的记忆也要分开存放。

- 短期记忆（同一窗口的上下文）→ `State + Checkpoint + thread_id`
- 长期记忆（跨窗口的用户资料）→ `Store + namespace + user_id`
- 持久化（进程重启后恢复）→ `PostgresSaver + PostgresStore`
- 长历史整理（太多时压缩）→ `Summary / Trim`

这类全景图不适合只用代码 `print()`，也不优先用生成图。它需要术语、中文标签和表格列名都稳定，所以这里用 SVG：可精确改字、可版本管理、notebook 里也能直接引用。

![LangGraph 记忆全景图](images/memory-panorama-cn.svg)

一张表格收束：

| 记忆类型 | 人类类比 | Graph 机制 | 生命周期 | 隔离方式 |
|---|---|---|---|---|
| 短期 / 会话 | 刚才聊了什么 | State + Checkpoint | 同一 thread | thread_id |
| 长期 / 偏好 | 这个用户喜欢什么 | Store | 跨会话 | user_id + namespace |
| 事实 / 知识 | 产品退货政策是什么 | Store (知识) | 跨用户 / 全局 | namespace |
| 历史整理 | 会议纪要不要全文 | Summary / Trim | 当前会话 | 按需触发 |
| 持久化记忆 | 电脑关机后还能记住 | PostgresSaver / Store | 数据库生命周期 | 同上面的隔离 |

一句话收束：

> **人自然会在不同记忆位置存放不同类型的信息；Graph 需要你显式设计"写在哪里、用什么 ID 找回、能记多久"。**

## 9. 常见误区

| 误区 | 现象 | 正确说法 |
|---|---|---|
| 把 state 当长期记忆 | 换窗口或重启后找不到资料 | state 是当前工作台 |
| 忘记传 `thread_id` | 每次都像新聊天 | checkpoint 需要窗口 ID 才能找回 |
| 把 `thread_id` 当 `user_id` | 不同用户可能串话 | `thread_id` 管会话，`user_id` 管用户 |
| 把偏好存在 checkpoint | 新窗口读不到长期偏好 | 偏好应该进 Store |
| 历史越塞越多 | 输入变贵、变慢、容易丢重点 | 早期历史要摘要化 |
| 只用内存存储 | 重启后全部丢失 | 生产环境需要 Postgres 持久化 |
