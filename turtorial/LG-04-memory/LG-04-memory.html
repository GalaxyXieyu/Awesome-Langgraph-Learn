MEMORY LAYERS

四层记忆全景图

Layer 1: State（工作记忆）

载体：Python dict / TypedDict / Pydantic  |  生命周期：1 个 super-step  |  管理：你直接读写

Layer 2: Checkpoint（短期记忆）

载体：Postgres / SQLite / Redis checkpoints 表  |  生命周期：整个 thread  |  管理：LangGraph 自动保存

Layer 3: Store（长期记忆）

载体：Postgres store 表（JSONB + 向量索引）  |  生命周期：永久或 TTL  |  管理：你手动 aput/aget/asearch

Layer 4: Cache（加速记忆）

载体：Postgres / Redis（复用连接池）  |  生命周期：TTL 控制  |  管理：LangGraph 自动命中/写入

LangGraph 渐进式教程LG-04
四层记忆模型是本章核心框架。State 是瞬时工作记忆，Checkpoint 是自动保存的短期记忆，Store 是手动管理的长期记忆，Cache 是避免重复计算的加速层。每层有不同的生命周期、管理方和成本结构。

CHECKPOINT

Checkpoint：自动状态快照

Checkpoint 的组成

  • channel_values：当前所有 channel 的值
  • channel_versions：每个 channel 版本号
  • versions_seen：节点看到的版本号
  • pending_writes：尚未应用的 writes

开发到生产迁移

# 开发环境
from langgraph.checkpoint.memory import MemorySaver
graph = builder.compile(checkpointer=MemorySaver())

# 生产环境
from langgraph.checkpoint.postgres import PostgresSaver
saver = PostgresSaver(pool)
await saver.setup()
graph = builder.compile(checkpointer=saver)

thread_id 设计建议

场景格式
单用户单会话{user_id}_{session_id}
单用户多会话{user_id}_{timestamp}
匿名用户anon_{uuid}

⚠️ 两个用户共享 thread_id 会看到彼此对话历史

为什么选 PostgreSQL？

  • ACID + JSONB：强一致性 + 嵌套字典兼容
  • pgvector：官方向量扩展，无需单独维护
  • pgBouncer：事务级连接池，高并发保障

LONG-TERM MEMORY

Store：手动管理的长期记忆

Checkpoint vs Store 本质区别

维度CheckpointStore
管理方自动手动
存在时间随对话增长永久或 TTL
查询方式thread_id 遍历namespace + key
类比游戏自动存档手动保存的装备配置

命名空间设计（推荐二元组）

ns_prefs = ("user_prefs", user_id)
ns_history = ("chat_history", user_id)
ns_knowledge = ("knowledge", user_id)
ns_research = ("deepresearch:artifacts", user_id)

Store API 速查

方法用途
aput(ns, key, value)写入/更新
aget(ns, key)精确读取
asearch(ns, query, filter, limit)列表/语义检索

何时用 Store、何时用 Checkpoint？

  • 对话消息历史 → Checkpoint（自动管理）
  • 用户偏好（语言、风格）→ Store（跨会话）
  • 知识库片段 → Store + 向量索引
  • 工具调用结果缓存 → Store（可选 TTL）
  • 当前执行位置 → Checkpoint（中断恢复）
LangGraph 渐进式教程LG-04
Store 是手动管理的长期记忆，通过 namespace 做数据隔离。推荐 (bucket, user_id) 二元组。支持精确读取和语义检索（pgvector）。
CACHE LAYER

Cache：记忆的加速层

Cache 与 Checkpoint 的关键区分

  • checkpointer=：存整个图的状态，用于恢复
  • cache=：存单个节点的输出，用于避免重复计算
  • 两者可复用同一个 PostgresSaver 实例
  • 底层用不同 thread_id 前缀隔离

CachePolicy 配置

from langgraph.types import CachePolicy

# 基础 TTL
CachePolicy(ttl=300) # 5 分钟

# 自定义缓存键
CachePolicy(ttl=600, key_func=my_key_func)

# 条件缓存（只读操作才缓存）
CachePolicy(ttl=300, key_func=conditional_key)

TTL 设置建议

数据类型TTL理由
股票价格、实时天气30 秒几乎实时
数据库查询结果10 分钟中等动态
LLM 对话结果30 分钟上下文稳定
配置信息24 小时几乎不变

缓存后端选择

场景推荐后端
本地开发MemorySaver
生产环境PostgresSaver（复用）
极致性能RedisSaver

💡 防雪崩：TTL 加随机偏移，避免大量缓存同时失效

LangGraph 渐进式教程LG-04
Cache 不是独立基础设施，而是与 Checkpoint 复用同一后端的加速层。CachePolicy 控制 TTL 和缓存键。生产推荐复用 PostgresSaver。
CONTEXT ENGINEERING

上下文工程：控制记忆的大小

Token 危机

  • 20 轮对话 ≈ 4000-6000 tokens
  • 50 轮对话 ≈ 10000-15000 tokens
  • 超出窗口 → 报错
  • 未超出但接近 → "lost in the middle"

trim_messages 裁剪

def build_pre_model_hook(max_tokens=800):
  def pre_model_hook(state):
    trimmed = trim_messages(
      messages,
      strategy="last",
      max_tokens=max_tokens,
      include_system=True,
    )
    return {"llm_input_messages": trimmed}
  return pre_model_hook

增量摘要压缩

if count_tokens_approx(messages) > budget * 2:
  early, recent = split_history(messages, keep_last=6)
  summary = await llm.ainvoke(...)
  new_messages = [
    {"type": "system", "content": f"[历史摘要] {summary}"},
    *recent,
  ]

硬约束窗口

def window_messages(state, keep_last=12):
  msgs = state.get("messages", [])
  head = msgs[:1] if msgs[0].type == "system" else []
  tail = msgs[-keep_last:]
  return {"messages": head + tail}

⚠️ 裁剪后确保消息序列合法：以 System/Human 开始，以 Human/Tool 结束

CASE STUDY

MemoryPal：四层记忆的实战应用

概念对应

概念MemoryPal 体现
State当前 super-step 的消息列表
Checkpoint多轮对话自动保存，刷新续接
Store记住用户偏好、语言、风格
Store 向量检索检索用户历史记忆
Cache天气等频繁查询结果缓存
trim_messages超长对话自动裁剪

三层记忆注入编译

app = agent.compile(
  checkpointer=saver, # 短期记忆
  store=store, # 长期记忆
  cache=saver, # 加速记忆
)

常见误区

  • 忘记传 config → 每次调用都是新对话
  • Store 没传 user_id → 没有用户隔离
  • Cache 和 Checkpointer 混淆 → 以为 cache= 存的是状态
  • 向量维度不匹配 → dims=1536 但模型输出 1024
  • 没调 setup() → 首次使用报错 "表不存在"
  • 上下文裁剪太激进 → LLM 丢失关键上下文

记忆口诀

"State 是脑子，Checkpoint 是草稿，Store 是笔记本，Cache 是小纸条；
PostgreSQL 装所有，pgvector 做检索，pgBouncer 管连接，四层记忆一盘棋。"

MemoryPal 是四层记忆的完整实战案例。关键转折点：天气查询结果存 Cache（会重复查且不需要持久化）；两个用户共享 thread_id 会看到对方对话；cache=saver 和 checkpointer=saver 不会冲突，底层用不同前缀隔离。
SUMMARY

核心要点回顾

四层记忆架构

  • State：工作记忆，1 个 super-step
  • Checkpoint：短期记忆，自动保存，thread_id 隔离
  • Store：长期记忆，手动读写，namespace 隔离
  • Cache：加速记忆，TTL 控制，避免重复计算

PostgreSQL 统一底座

  • ACID + JSONB：关系查询与强一致性
  • pgvector：向量检索，无需单独维护
  • pgBouncer：事务级连接池，高并发

上下文工程

  • trim_messages：Token 窗口控制
  • 增量摘要：超长对话压缩
  • pre_model_hook：统一入口
