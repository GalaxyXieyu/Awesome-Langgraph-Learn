# LG-06：预构建 Agent，不是魔法，是标准循环的封装

> 核心冲突句：**人会自然把“查工具 → 看结果 → 再判断”当成连续动作，但图不会；如果这是标准 ReAct 循环，预构建 Agent 可以帮你快速封装，代价是中间控制权变少。**

这节课不要从 `create_agent` 开始，而是从一个真实产品场景开始：**OpsBot 客服 / 运维助手**。

> 用户问：“查一下订单 A100 的状态，如果符合规则，帮我估算退款金额。”
>
> 这个需求看起来简单，但背后有一条标准循环：模型判断要查订单、查政策、算退款；工具返回结果；模型再整合回复。
>
> 如果每次都从零写 StateGraph：agent 节点、tools 节点、条件边、循环边，会很繁琐。预构建 Agent 的价值就在这里：把常见 ReAct 循环封装成开箱即用的图。
>
> 但天下没有免费的午餐：它快，但控制力少。生产里一旦涉及审批、权限、审计、自定义状态，就要回到自定义 StateGraph。

本节课要训练的判断不是“背 API”，而是：**什么时候用预构建快速验证，什么时候必须把业务控制权拿回来自定义图。**

---

## 0. 开篇：为什么需要预构建 Agent

### 要讲清楚的问题

学生已经学过 tools、node、subgraph。现在自然会问：既然我会自己画图，为什么还需要预构建 Agent？

答案不是“预构建更高级”，而是：标准 ReAct 循环太常见，重复手写没有必要。

| 需求 | 适合的机制 | 为什么 |
|---|---|---|
| 快速验证一个能查工具的 Agent | `create_agent` | 标准 LLM → tools → LLM 循环不用重写 |
| 把订单查询、退款计算做成可复用能力 | tools | 能力原子化，可被不同 Agent 复用 |
| 多个专家协作：订单、退款、技术 | supervisor / swarm | 角色能力原子化，可被调度或交接 |
| 涉及审批、权限、审计 | 自定义 StateGraph | 需要显式状态、路由和人工边界 |

### 配图

使用 `images/prebuilt-agent-scenario-cn.svg`。

### 推荐话术

> 预构建 Agent 不是替代 StateGraph，而是把最常见的那条 Agent 循环提前封装好。它适合快速验证和标准工具问答，不适合把业务控制权藏起来。

## 1. 模型配置：继续用真实 LLM

### 要讲清楚的问题

这一节的核心行为是“模型产生工具调用”。不能用硬编码模拟，否则学生看不到真实模型如何选择工具、如何产生 `tool_calls`、如何在工具结果后再次回答。

### 推荐演示

显式读取：

```text
OPENAI_MODEL
OPENAI_BASE_URL
OPENAI_API_KEY
OPENAI_TEMPERATURE
```

缺配置时直接报错，不做 fake fallback。

---

## 2. 先把外部能力原子化成 Tools

### 人类动作

客服会把能力拆开：查订单、查退款政策、计算退款金额。每个能力都可以被不同客服、不同流程复用。

### Agent 的限制

模型不能直接访问订单库或计算规则。你要把这些能力显式做成 tool，模型才能调用。

### 推荐演示

定义三个可控工具：

- `query_order_status(order_id)`
- `query_refund_policy(product_type)`
- `calculate_refund_amount(order_amount, shipping_cost)`

### 证据输出

```text
工具数量: 3
工具名称: [...]
能力是否已原子化: 是 — 这些工具可以被不同 Agent 复用
```

---

## 3. create_agent：标准循环三行封装

### 人类动作

客服看到用户问题，会自然做：判断需要什么信息 → 查工具 → 看工具结果 → 回答用户。

### Graph 的限制

Graph 不会自然循环。你需要 agent 节点、tools 节点、条件边、循环边。预构建 Agent 帮你把这条标准循环搭好了。

### API 映射

```python
opsbot_agent = create_agent(chat_model, ops_tools, system_prompt=opsbot_prompt)
```

### 证据输出

```text
消息总数: 4+
是否产生 tool_calls: 是
工具消息数量: 3
最终回复: ...
```

### 教学要点

真实 LLM 决定调用哪些工具，预构建 Agent 负责把 tool call、tool execution、再思考循环串起来。

## 4. 白盒化：create_agent 内部到底帮你搭了什么

### 要讲清楚的问题

预构建不是魔法。它底层仍然是一张图，只是把常见结构封装了。

### 配图

使用 `images/react-loop-cn.svg`。

### 白话模型

```text
用户输入
  ↓
agent 节点调用 LLM
  ↓
如果 AIMessage 有 tool_calls → tools 节点执行工具 → 回到 agent
如果 AIMessage 没有 tool_calls → END
```

### 证据输出

通过打印 messages 链路观察：

```text
HumanMessage
AIMessage: tool_calls=3
ToolMessage: query_order_status
ToolMessage: query_refund_policy
ToolMessage: calculate_refund_amount
AIMessage: tool_calls=0
```

### 教学要点

循环是否继续，不看模型“说结束”，而看最后一条 AIMessage 是否还有 `tool_calls`。

---

## 5. 用 stream 观察黑盒内部

### 要讲清楚的问题

`invoke()` 只给最终结果，学生会觉得预构建 Agent 是黑盒。`stream()` 能看到每一步消息如何增长。

### 推荐演示

用 `stream_mode="values"` 打印：

- 当前 step。
- 最新消息类型。
- tool_calls 数量。
- 当前 messages 数。

### 教学要点

stream 的意义不是炫技，而是帮助学生看到：预构建虽然封装了图，但内部仍然是消息流、工具调用和循环路由。

---

## 6. 预构建的代价：快，但控制力少

### 要讲清楚的问题

预构建 Agent 适合标准工具问答，但危险工具不能直接交给它跑。

### 配图

使用 `images/prebuilt-vs-custom-cn.svg`。

### 推荐演示

定义危险工具：

```python
issue_refund(order_id, amount)
```

让预构建 Agent 直接执行退款，观察工具是否被调用。

### 证据输出

```text
危险工具是否被调用: 是
为什么这是问题: 退款是副作用动作，生产环境必须先走审批边界
正确迁移方向: 自定义 StateGraph + interrupt / 审批节点 / wrapper 级审批
```

### 教学要点

不要用 prompt 代替安全边界。出现副作用工具、审批、权限、审计时，必须迁移到自定义 StateGraph。

## 7. 记忆与上下文：messages 会越来越长

### 要讲清楚的问题

预构建 Agent 的核心 state 是 `messages`。每次工具调用都会追加：

```text
HumanMessage → AIMessage(tool_calls) → ToolMessage → AIMessage(final)
```

这很方便，但长期对话会膨胀。

### 推荐演示

给 `create_agent` 加 `checkpointer`，用同一个 `thread_id` 连续对话。

### 证据输出

```text
第一轮 messages 数: N
第二轮 messages 数: M
第二轮是否接上同一 thread: 是
```

### 教学要点

这里衔接 LG-04：checkpointer 保存的是可恢复现场，不是长期偏好 Store。如果你需要长期用户资料、权限、审批状态，仍然要认真设计外层状态。

## 8. 多 Agent 预构建：把专家角色原子化

### 要讲清楚的问题

LG-05 讲过 tools 和 subgraph 的能力原子化；LG-06 继续往上一层：把一个专家 Agent 原子化。

### 配图

使用 `images/supervisor-swarm-cn.svg`。

### 两种模式

| 模式 | 白话理解 | 适合场景 |
|---|---|---|
| `create_supervisor` | 一个主管调度多个专家 | 任务分发、结果汇总 |
| `create_swarm` | 多个专家平级交接 | 客服转接、连续接力 |

### 推荐演示

创建：

- `order_expert`：只处理订单查询。
- `refund_expert`：只处理退款政策和金额。
- `supervisor`：调度两个专家并汇总。

### 证据输出

```text
Supervisor messages 数: N
最终回复: ...
```

### 教学要点

Supervisor 不是把所有工具塞给一个大 Agent，而是把专家角色原子化后交给主管调度。

## 9. 什么时候从预构建迁移到自定义

| 需求 | 推荐选择 | 原因 |
|---|---|---|
| 3 天内做一个能查工具的 demo | `create_agent` | 标准循环，速度优先 |
| 多个专家角色由主管分配 | `create_supervisor` | 角色能力原子化，中心调度 |
| 客服在多个专家间平级转接 | `create_swarm` | 用 `active_agent` 保持当前负责角色 |
| 退款、重启、发布等副作用动作 | 自定义 StateGraph | 需要审批、权限、审计 |
| 需要 `user_role`、`risk_level` 等字段 | 自定义 StateGraph | 预构建核心 state 以 messages 为主 |
| 需要复杂业务分支 | 自定义 StateGraph | 不只是 ReAct 循环 |

一句话：**预构建负责把常见循环和角色调度封装好；自定义图负责把业务控制权拿回来。**

## 10. 常见误区

| 误区 | 现象 | 正确说法 |
|---|---|---|
| 用假函数模拟 Agent | 看不到模型如何真实产生工具调用 | 本节必须用真实 LLM |
| 以为预构建更高级 | 盲目替代自定义图 | 预构建只是封装标准循环 |
| 用 prompt 管危险工具 | 模型仍可能直接调用退款/发布工具 | 副作用工具必须有审批边界 |
| 把所有工具塞给一个 Agent | Agent 选择空间过大，难调试 | 复杂角色要原子化成专家 Agent |
| 把 supervisor 和 swarm 混用 | 不知道谁负责调度 | 任务分发用 supervisor，连续转接用 swarm |
| 忽略 messages 膨胀 | 长对话成本越来越高 | 用 checkpointer、裁剪、摘要、工具结果精简 |

## 11. 本节小结

学生应该能不用 API 名解释：

> 如果需求只是“模型判断工具、工具返回结果、模型再回答”，这是一条标准 ReAct 循环，可以先用预构建 Agent 快速验证。预构建的代价是中间状态、路由和审批不够可控；一旦业务需要显式状态、人工审批或复杂路由，就要迁移到自定义 StateGraph。

判断口诀：

```text
标准 ReAct → create_agent
主管分发专家 → create_supervisor
平级连续转接 → create_swarm
副作用审批 / 自定义状态 / 复杂路由 → 自定义 StateGraph
```
