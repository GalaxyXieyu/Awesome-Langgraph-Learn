## 2. create_react_agent 详解

### 2.1 快速上手

```python
from langgraph.prebuilt import create_react_agent
from langchain_openai import ChatOpenAI
from langchain_core.messages import HumanMessage

llm = ChatOpenAI(model="gpt-4o")
tools = [query_weather, calculator]

# 3 行代码，一个 Agent 就搭好了
agent = create_react_agent(llm, tools)

# 运行
result = agent.invoke({
"messages": [HumanMessage("北京天气怎么样？23的平方根是多少？")]
})
print(result["messages"][-1].content)
```

**输出过程**：
```
用户：北京天气怎么样？23的平方根是多少？
→ LLM 思考：需要查天气 + 算平方根
→ LLM 输出 tool_calls：[query_weather(city="北京"), calculate(expression="sqrt(23)")]
→ ToolNode 执行两个工具
→ ToolMessage 返回结果
→ LLM 再次思考，整合结果
→ LLM 回复用户
```

### 2.2 内部机制白盒化（源码级拆解）

`create_react_agent` 的循环终止**不是 LLM "说" 要结束**——而是程序检测 LLM 返回的 `AIMessage` 中是否包含 `tool_calls` 字段。看源码：

#### 2.2.1 核心路由函数：`should_continue`

```python
# langgraph/prebuilt/chat_agent_executor.py:770-798
def should_continue(state: StateSchema) -> Union[str, list[Send]]:
messages = _get_state_value(state, "messages")
last_message = messages[-1]
    # 如果最后一条消息不是 AIMessage，或者 AIMessage 没有 tool_calls → 结束
if not isinstance(last_message, AIMessage) or not last_message.tool_calls:
if post_model_hook is not None:
return "post_model_hook"
elif response_format is not None:
return "generate_structured_response"
else:
return END # ← 循环终止！
    # 否则，继续去 tools 节点执行
else:
if version == "v1":
return "tools"
elif version == "v2":
            # v2 用 Send 把每个 tool_call 分发到 tools 节点并行执行
return [
Send("tools", ToolCallWithContext(tool_call=tool_call, state=state))
for tool_call in last_message.tool_calls
]
```

**关键结论**：循环终止的唯一判据是 `last_message.tool_calls` 是否为空列表。
- LLM 返回 `AIMessage(tool_calls=[...])` → 去 `tools` 节点执行
- LLM 返回 `AIMessage(tool_calls=[])` 或普通文本 → 结束循环

这不是 LLM "主动告诉" 程序要结束，而是程序检查 LLM 的输出格式——没有 tool call 请求就意味着 LLM 已经给出了最终答案。

#### 2.2.2 步数限制：防止无限循环

```python
# langgraph/prebuilt/chat_agent_executor.py:63-71
class AgentState(TypedDict):
messages: Annotated[Sequence[BaseMessage], add_messages]
is_last_step: IsLastStep # ← 由 LangGraph 运行时自动管理
remaining_steps: RemainingSteps # ← 由 LangGraph 运行时自动管理
```

```python
# langgraph/prebuilt/chat_agent_executor.py:556-573
def _are_more_steps_needed(state: StateSchema, response: BaseMessage) -> bool:
has_tool_calls = isinstance(response, AIMessage) and response.tool_calls
remaining_steps = _get_state_value(state, "remaining_steps", None)
is_last_step = _get_state_value(state, "is_last_step", False)
return (
        # 情况1：is_last_step=True 且还有 tool_calls → 步数用完了但LLM还想调工具
(remaining_steps is None and is_last_step and has_tool_calls)
        # 情况2：remaining_steps < 2 且还有 tool_calls → 再调一次就会超出限制
or (remaining_steps is not None and remaining_steps < 2 and has_tool_calls)
)
```

在 `call_model` 中（第623行），如果 `_are_more_steps_needed` 返回 True，LLM 的输出会被强制替换为：

```python
return {
"messages": [
AIMessage(
id=response.id,
content="Sorry, need more steps to process this request.",
)
]
}
```

这个强制替换后的 `AIMessage` **没有 `tool_calls`**，所以下一轮 `should_continue` 检查到 `tool_calls` 为空 → 循环结束。

**这是 LangGraph 的安全机制**：即使 LLM 无限想调工具，也会被 `remaining_steps` 或 `is_last_step` 强制截断。

#### 2.2.3 图构建源码

```python
# langgraph/prebuilt/chat_agent_executor.py:800-929
workflow = StateGraph(state_schema=state_schema or AgentState)

# 定义两个节点：agent（调用LLM）和 tools（执行工具）
workflow.add_node("agent", RunnableCallable(call_model, acall_model))
workflow.add_node("tools", tool_node)

# 设置入口：从 agent 节点开始
workflow.set_entry_point("agent")

# agent → should_continue → tools / END
workflow.add_conditional_edges("agent", should_continue, path_map=agent_paths)

# tools 执行完 → 回到 agent（除非有 return_direct 的工具）
workflow.add_edge("tools", "agent")

return workflow.compile(...)
```

**它自动帮你搭了这些**：
1. `agent` 节点：调用 LLM
2. `tools` 节点：ToolNode 执行工具
3. 条件边 `should_continue`：检测 `AIMessage.tool_calls` 决定走向
4. 循环边 `tools → agent`：ToolMessage 回到 LLM 继续思考
5. `AgentState`：自动管理 `messages`、`remaining_steps`、`is_last_step`

### 2.3 State 结构

```python
# create_react_agent 的 State 等价于：
class AgentState(TypedDict):
messages: Annotated[list[AnyMessage], add_messages]
```

**就这一个字段**。简单，但也意味着：
- ❌ 你无法自定义其他状态字段（比如 `retry_count`、`user_role`、`session_id`）
- ❌ 你无法在 State 里存业务数据（比如订单号、用户偏好）
- ✅ 但如果你只需要对话历史，这就够了

### 2.4 传 config 和 system prompt

```python
from langchain_core.messages import SystemMessage

# 方式 1：通过 config 传 system prompt
agent = create_react_agent(
llm, tools,
state_modifier=SystemMessage("你是一个专业的天气助手，回答要简洁。")
)

# 方式 2：通过 invoke 传
result = agent.invoke(
{"messages": [HumanMessage("北京天气？")]},
config={"configurable": {"thread_id": "user-123"}}
)
```

### 2.5 局限性（重点！）

| 局限 | 说明 | 影响 |
|------|------|------|
| **State 不可扩展** | 只有 `messages` 一个字段 | 无法存业务状态 |
| **路由不可定制** | 固定 LLM → Tool → LLM 循环 | 无法根据业务状态走不同分支 |
| **中间过程不可见** | `invoke()` 只给最终结果 | 无法监控每步在做什么 |
| **循环控制受限** | 由内部逻辑决定循环次数 | 无法自定义重试策略 |
| **HITL 支持弱** | 没有原生断点能力 | 无法人工审批中间步骤 |

## 3. `create_react_agent` 扩展参数与多 Agent 模式

LangGraph 团队已经明确：**`create_react_agent` 是唯一的官方推荐预构建 Agent**。它的内部实现直接调用 LLM 的原生 `tool_calling` API，覆盖了所有主流模型（OpenAI、Anthropic、Google、本地模型等）。

### 3.1 `create_react_agent` 的扩展参数（进阶）

除了 model + tools，还有一些实用参数：

```python
from langgraph.prebuilt import create_react_agent

agent = create_react_agent(
model=llm,
tools=tools,
    # 系统提示词
prompt="你是一个专业助手，回答要简洁。",
    # 或传 SystemMessage / Callable / Runnable
    # prompt=SystemMessage("..."),
    # prompt=lambda state: [...],

    # 在调用 LLM 前执行自定义逻辑（如消息裁剪）
    # pre_model_hook=trimming_hook,
)
```

**`pre_model_hook` 实战：消息裁剪**

```python
def trimming_hook(state):
"""在 LLM 调用前自动裁剪历史消息。"""
from langchain_core.messages import RemoveMessage

messages = state["messages"]
if len(messages) > 20:
        # 只保留最近 10 条，移除前面的
to_remove = messages[:-10]
return {
"messages": [RemoveMessage(id=m.id) for m in to_remove]
}
return {}

agent = create_react_agent(llm, tools, pre_model_hook=trimming_hook)
```

### 3.2 多 Agent 预构建模式（LangGraph 生态扩展）

除了单 Agent 的 `create_react_agent`，LangGraph 生态还提供了两个专门用于**多 Agent 协作**的预构建库：

#### 3.2.1 `create_supervisor`（监督者模式）—— 源码级拆解

来自 `langgraph-supervisor`，适合**层级化管理**多个专业化 Agent。

##### 使用示例

```python
from langgraph.prebuilt import create_react_agent
from langgraph_supervisor import create_supervisor

weather_agent = create_react_agent(llm, tools=[query_weather], name="weather_expert")
order_agent = create_react_agent(llm, tools=[query_order, cancel_order], name="order_expert")
refund_agent = create_react_agent(llm, tools=[refund_order], name="refund_expert")

workflow = create_supervisor(
[weather_agent, order_agent, refund_agent],
model=llm,
prompt="你是调度主管，根据用户问题分发给对应专员。",
)
app = workflow.compile()
```

##### 核心源码分析

**（1）Supervisor 的本质：就是一个 `create_react_agent`**

```python
# langgraph_supervisor/supervisor.py:425-434
supervisor_agent = create_react_agent(
name=supervisor_name,
model=model,
tools=tool_node, # ← 关键：tools 里包含了所有 sub-agent 的 handoff_tools
prompt=prompt,
state_schema=supervisor_schema,
...
)
```

**Supervisor 本身不特殊**——它就是一个普通的 `create_react_agent`，只是它的 `tools` 参数里除了常规工具，还自动加入了一组 **handoff tools**。

**（2）Handoff Tool 的自动生成**

```python
# langgraph_supervisor/supervisor.py:175-186
# _prepare_tool_node() 内部自动为每个 sub-agent 创建 handoff tool
handoff_tools = [
create_handoff_tool(
agent_name=agent_name,
name=f"transfer_to_{agent_name}", # 工具名：transfer_to_weather_expert
add_handoff_messages=True,
)
for agent_name in agent_names # ["weather_expert", "order_expert", "refund_expert"]
]
```

**（3）Handoff Tool 的内部实现**

```python
# langgraph_supervisor/handoff.py:84-122
@tool(name, description=description)
def handoff_to_agent(
state: Annotated[dict, InjectedState],
tool_call_id: Annotated[str, InjectedToolCallId],
) -> Command:
tool_message = ToolMessage(
content=f"Successfully transferred to {agent_name}",
name=name,
tool_call_id=tool_call_id,
response_metadata={"__handoff_destination": agent_name},
)
last_ai_message = cast(AIMessage, state["messages"][-1])

    # 并行 handoff 处理（supervisor 同时调用多个 agent）
if len(last_ai_message.tool_calls) > 1:
handoff_messages = state["messages"][:-1]
handoff_messages.extend((
_remove_non_handoff_tool_calls(last_ai_message, tool_call_id),
tool_message,
))
return Command(
graph=Command.PARENT,
goto=[Send(agent_name, {"messages": handoff_messages})],
)
    # 单个 handoff
else:
handoff_messages = state["messages"] + [tool_message]
return Command(
goto=agent_name,
graph=Command.PARENT,
update={**state, "messages": handoff_messages},
)
```

**关键点**：`handoff_to_agent` 不是一个"做业务"的工具——它不查天气、不查订单。它只返回一个 `Command(goto=agent_name)`，告诉 LangGraph 运行时要跳转到哪个 agent 节点。

**（4）图的构建**

```python
# langgraph_supervisor/supervisor.py:436-451
builder = StateGraph(workflow_schema, context_schema=context_schema)

# 1. 添加 supervisor 节点（它是一个完整的 CompiledStateGraph）
builder.add_node(supervisor_agent, destinations=tuple(agent_names) + (END,))
builder.add_edge(START, supervisor_agent.name)

# 2. 为每个 sub-agent 添加节点
for agent in agents:
builder.add_node(
agent.name,
_make_call_agent(agent, output_mode, ...), # 包装 sub-agent 调用
)
    # 3. sub-agent 执行完 → 回到 supervisor
builder.add_edge(agent.name, supervisor_agent.name)
```

**完整的数据流**：

```
用户："查北京天气"
→ START → supervisor (LLM 思考)
→ supervisor 输出 tool_calls: [transfer_to_weather_expert]
→ ToolNode 执行 handoff_to_agent("weather_expert")
→ handoff_to_agent 返回 Command(goto="weather_expert")
→ weather_expert 节点执行（它是一个完整的 create_react_agent）
→ weather_expert 返回结果 → 自动回到 supervisor（add_edge 保证）
→ supervisor 再次 LLM 思考："天气已查到，直接回复用户"
→ supervisor 输出 AIMessage（无 tool_calls）→ END
```

**（5）Sub-agent 输出的处理**

```python
# langgraph_supervisor/supervisor.py:77-99
def _process_output(output: dict) -> dict:
messages = output["messages"]
if output_mode == "last_message":
        # 只保留最后一条消息（sub-agent 的最终回复）
messages = messages[-1:]
if add_handoff_back_messages:
        # 添加 "转回 supervisor" 的标记消息
messages.extend(create_handoff_back_messages(agent.name, supervisor_name))
return {**output, "messages": messages}
```

`output_mode` 控制 sub-agent 的完整对话历史是否带回 supervisor：
- `"full_history"`：sub-agent 内部每一步思考+工具调用都带回
- `"last_message"`（默认）：只带回最终答案，节省 Token

#### 3.2.2 `create_swarm`（群体协作模式）—— 源码级拆解

来自 `langgraph-swarm`，适合**平级 Agent 动态交接**的场景。**与 Supervisor 最大的区别：没有中心调度器。**

##### 使用示例

```python
from langgraph_swarm import create_handoff_tool, create_swarm

alice = create_react_agent(
llm,
tools=[
query_order,
create_handoff_tool(agent_name="Bob", description="技术问题转接给 Bob"),
],
name="Alice",
)

bob = create_react_agent(
llm,
tools=[
query_logs,
create_handoff_tool(agent_name="Alice", description="转回 Alice"),
],
name="Bob",
)

workflow = create_swarm([alice, bob], default_active_agent="Alice")
app = workflow.compile(checkpointer=checkpointer)
```

##### 核心源码分析

**（1）Swarm State：多了一个 `active_agent` 字段**

```python
# langgraph_swarm/swarm.py:13-19
class SwarmState(MessagesState):
active_agent: str | None # ← 关键：记录当前活跃的 agent
```

**（2）图的构建：没有 supervisor，直接从 START 路由**

```python
# langgraph_swarm/swarm.py:270-287
state_schema = _update_state_schema_agent_names(state_schema, agent_names)
builder = StateGraph(state_schema, context_schema)

# 1. 添加 START → active_agent 的条件路由
add_active_agent_router(
builder,
route_to=agent_names, # ["Alice", "Bob"]
default_active_agent="Alice",
)

# 2. 添加每个 agent 作为节点
for agent in agents:
builder.add_node(
agent.name,
agent, # agent 本身就是 CompiledStateGraph
destinations=tuple(get_handoff_destinations(agent)),
)
```

**（3）核心路由器：`route_to_active_agent`**

```python
# langgraph_swarm/swarm.py:163-166
def route_to_active_agent(state: dict) -> str:
return state.get("active_agent", default_active_agent)

builder.add_conditional_edges(START, route_to_active_agent, path_map=route_to)
```

**这是 Swarm 的核心机制**：每次用户输入或 handoff 完成后，图从 `START` 重新开始，根据 `state["active_agent"]` 的值决定进入哪个 agent。

**（4）Handoff Tool 的实现：更新 `active_agent` 状态**

```python
# langgraph_swarm/handoff.py:70-90
@tool(name, description=description)
def handoff_to_agent(
state: Annotated[Any, InjectedState],
tool_call_id: Annotated[str, InjectedToolCallId],
) -> Command:
tool_message = ToolMessage(
content=f"Successfully transferred to {agent_name}",
name=name,
tool_call_id=tool_call_id,
)
return Command(
goto=agent_name,
graph=Command.PARENT,
update={
"messages": [*state["messages"], tool_message],
"active_agent": agent_name, # ← 关键：更新 active_agent 状态！
},
)
```

**与 Supervisor Handoff 的关键区别**：
- Supervisor：handoff tool 返回 `Command(goto=agent_name)`，直接跳转到 sub-agent，执行完通过 `add_edge` 回到 supervisor
- Swarm：handoff tool 返回 `Command(goto=agent_name, update={"active_agent": agent_name})`，跳转到目标 agent 的同时更新状态。目标 agent 执行完后图结束，下次用户输入从 START 重新进入，由 `route_to_active_agent` 根据更新后的 `active_agent` 路由

**完整的数据流**：

```
用户："我订单有问题"
→ START → route_to_active_agent("Alice") → Alice 节点
→ Alice 内部：LLM 思考 → 调用 query_order → 得到结果
→ Alice 内部：LLM 判断需要技术支持 → tool_calls: [transfer_to_Bob]
→ Alice 的 ToolNode 执行 handoff_to_Bob
→ handoff_to_Bob 返回 Command(
goto="Bob",
update={"active_agent": "Bob", "messages": [...]}
)
→ Bob 节点执行 → 解决技术问题 → 最终回复用户
→ Bob 内部 LLM 输出无 tool_calls → 图结束

[等待用户下一次输入]
用户："谢谢"
→ START → route_to_active_agent("Bob") → Bob 节点（因为 active_agent 还是 "Bob"）
→ Bob 回复："不客气"
```

**（5）每个 Agent 的 `destinations` 是怎么确定的**

```python
# langgraph_swarm/handoff.py:96-114
def get_handoff_destinations(agent: CompiledStateGraph, tool_node_name="tools") -> list[str]:
"""从 agent 的 tools 中提取 handoff 目标。"""
nodes = agent.get_graph().nodes
tool_node = nodes[tool_node_name].data
tools = tool_node.tools_by_name.values()
return [
tool.metadata["__handoff_destination"]
for tool in tools
if tool.metadata and "__handoff_destination" in tool.metadata
]
```

LangGraph 编译时会检查 `destinations`——如果 Alice 的 tools 里有 `transfer_to_Bob`，那么 Alice 节点就只能跳转到 Bob（或者 END），不能跳转到其他地方。

**三种模式选型**：

| 模式 | 核心函数 | 来源包 | 适用场景 |
|------|---------|--------|---------|
| 单 Agent | `create_react_agent` | `langgraph` | 一个 LLM + 多个工具 |
| 监督者 | `create_supervisor` | `langgraph-supervisor` | 一个主管调度多个专员 |
| 群体协作 | `create_swarm` | `langgraph-swarm` | 多个专员平级交接 |

> **注意**：`create_supervisor` 和 `create_swarm` 不在核心 `langgraph` 包中，需要额外安装 `pip install langgraph-supervisor langgraph-swarm`。

### 3.3 更复杂的多 Agent 模式：DeepResearch Agent（预告）

除了上面三种预构建模式，LangGraph 生态中还有一类更复杂的多 Agent 架构——**DeepResearch Agent**（深度研究 Agent）。它不是一个简单的 `create_xxx` 函数，而是一套完整的分层多 Agent 系统设计：

```
┌─────────────────────────────────────────────────────────────┐
│ DeepResearch Agent 架构概览 │
├─────────────────────────────────────────────────────────────┤
│ │
│ [用户输入] → [模式路由器] → 规划模式？执行模式？反思模式？ │
│ │ │
│ ┌──────────┼──────────┐ │
│ 规划 执行 反思 │
│ │ │ │ │
│ [计划生成] [子Agent组] [质量检查] │
│ │ │ │ │
│ [任务分解] [并行检索] [循环修正] │
│ │ │ │ │
│ └──────────┼──────────┘ │
│ [结果汇总] → [输出报告] │
│ │
│ 涉及能力：模式路由 / 计划路由循环 / 节点级缓存 / │
│ 任务跟踪系统 / 执行次数控制 / 子图嵌套 │
└─────────────────────────────────────────────────────────────┘
```

**为什么现在不讲？**

DeepResearch Agent 涉及的能力还远不止多 Agent 协作：

| 涉及能力 | 在哪节课学 | 为什么必须先学 |

## 4. 预构建 vs 自定义：决策框架

### 4.1 对比矩阵

```
create_react_agent（黑盒） vs 自定义 StateGraph（白盒）
┌─────────────────────┐ ┌─────────────────────────────────────┐
│ plan → tool → observe│ │ [思考] → [决策] → [工具] → [观察] │
│ ↑___________↓ │ │ ↑___________________________↓ │
│ （内部自动循环） │ │ （每个节点你都可以自定义、监控、干预） │
└─────────────────────┘ └─────────────────────────────────────┘
优点：3行代码搞定 优点：完全可控，中间状态可见
缺点：中间状态不可见，循环控制受限 缺点：代码量多，需要理解图机制
适用：快速原型、标准 ReAct 场景 适用：复杂路由、HITL、自定义状态管理
```

| 维度 | 预构建 Agent | 自定义 StateGraph |
|------|-------------|-------------------|
| 开发速度 | ⭐⭐⭐ 快（3 行代码） | ⭐⭐ 中等 |
| 可控性 | ⭐⭐ 黑盒 | ⭐⭐⭐ 白盒，每步可控 |
| 可观测性 | ⭐⭐ 只能看到输入输出 | ⭐⭐⭐ 每节点状态可见 |
| 复杂路由 | ⭐ 不支持 | ⭐⭐⭐ 任意图结构 |
| HITL 支持 | ⭐ 弱 | ⭐⭐⭐ 强 |
| 自定义 State | ⭐ 只有 messages | ⭐⭐⭐ 任意字段 |

### 4.2 决策树

```
你的需求是什么？
│
├─ 标准 ReAct 循环（思考→工具→观察→再思考）？
│ ├─ 不需要看中间过程？
│ │ └─ 不需要自定义 State？
│ │ └─ 不需要复杂路由？
│ │ → create_react_agent ✅
│ │
│ └─ 需要监控每一步的 State？→ 自定义 StateGraph ✅
│
├─ 需要根据业务状态走不同分支？
│ └─ 自定义 StateGraph ✅（预构建不支持复杂路由）
│
├─ 需要人在中间审批/干预？
│ └─ 自定义 StateGraph ✅（预构建 HITL 能力弱）
│
└─ 需要自定义 State 字段（retry_count、user_role 等）？
└─ 自定义 StateGraph ✅（预构建只有 messages 字段）
```

**关键认知**：预构建 Agent 的"快"是有代价的——你放弃了对中间过程的可见性和控制力。只要出现"需要看内部状态"或"需要干预执行"的需求，就必须切换到自定义 StateGraph。

## 5. 上下文管理：防止 Agent 撑爆窗口

### 5.1 问题根源

预构建 Agent 的 State 只有 `messages` 一个字段，每次 tool 调用都会把结果追加到 messages 里：

```
[HumanMessage] → [AIMessage + tool_calls] → [ToolMessage(结果)]
→ [AIMessage + tool_calls] → [ToolMessage(结果)]
→ ...（循环多次后 messages 越来越长）
```

一个 session 里如果循环 10 次，messages 可能有 30+ 条，Token 轻松过万。

### 5.2 策略 1：Message Trimming（裁剪历史）

```python
from langchain_core.messages import trim_messages

def agent_with_trimming(state):
"""在调用 LLM 前自动裁剪消息历史。"""
    # 只保留最近的 4000 tokens
trimmed = trim_messages(
state["messages"],
max_tokens=4000,
strategy="last",
token_counter=llm
)
return {"messages": trimmed}
```

### 5.3 策略 2：Summarization（总结历史）

```python
def summarize_if_needed(state):
"""当消息太长时，总结早期内容。"""
messages = state["messages"]
if count_tokens(messages) < 6000:
return {}

    # 保留最近 4 条，总结前面的
to_summarize = messages[:-4]
recent = messages[-4:]

summary = llm.invoke([
SystemMessage("请总结以下对话的关键事实和结论，保留所有重要信息。"),
*to_summarize
])

return {"messages": [summary] + recent}
```

### 5.4 策略 3：Tool 结果精简（从源头控制）

在 tool 定义阶段就控制输出长度（参考 LG-02 的上下文管理）：

```python
@tool
def query_weather(city: str) -> str:
"""查询天气，只返回关键信息。"""
raw = query_weather_api(city)
    # 只取关键字段，原始数据可能几十 KB
return f"{city}：{raw['weather'][0]['description']}，{raw['main']['temp']}°C"
```

**三种策略对比**：

| 策略 | 实现难度 | 信息损失 | 适用场景 |
|------|---------|---------|---------|
| Trimming | 低 | 中（丢掉早期消息） | 对话轮数多的场景 |
| Summarization | 中 | 低（智能压缩） | 需要保留上下文的复杂对话 |
| Tool 精简 | 低 | 可控（按需提取） | 所有场景（应该在定义 tool 时就做） |

## 6. OpsBot：预构建与自定义的对比实战

### 案例设计

用两种方式实现同一个运维助手，对比体验：

**需求**：
- 能查服务器状态（CPU、内存、磁盘）
- 能执行重启命令（需要审批）
- 能查日志（结果可能很长）
- 复杂查询时自动拆解成多个工具调用

**版本 A：create_react_agent（3 行代码）**

```python
from langgraph.prebuilt import create_react_agent

tools = [check_cpu, check_memory, check_disk, restart_service, query_logs]
agent = create_react_agent(llm, tools)
```

**版本 B：自定义 StateGraph（完整可控）**

```python
# 自己定义 State
class OpsState(TypedDict):
messages: Annotated[list, add_messages]
user_role: str # 区分普通用户和管理员
pending_approval: bool # 是否有待审批操作
execution_log: list # 执行日志

# 自己搭图：LLM → 判断 → 工具/审批/回复
builder = StateGraph(OpsState)
# ... 完整的路由逻辑 ...
```

### 对比演示要点

**Step 1 - 简单查询对比**：
> "先看一个简单查询：『查一下 CPU 使用率』。两个版本都能正确调用 `check_cpu` 工具，返回结果。预构建版 3 行代码搞定，自定义版写了 30 行。这时候预构建版完胜。"

**Step 2 - 审批场景对比（转折点）**：
> "现在看一个敏感操作：『重启数据库服务』。预构建版直接执行了 `restart_service`，没有任何阻拦。自定义版在 `restart_service` 节点前有一个条件边——检查 `user_role`，如果是普通用户，转到 `request_approval` 节点，等待管理员确认。"
>
> "这就是关键差异：预构建版『快但不可控』，自定义版『慢但安全』。"

**Step 3 - 长日志查询对比**：
> "再看一个查询：『查最近 1000 条错误日志』。预构建版直接把 1000 条日志塞进 message history，Token 暴涨。自定义版在 `query_logs` 节点里做了截断——只返回前 50 条 + 统计信息，并告诉用户『共 1000 条，是否查看全部？』"

**Step 4 - 过程可见性对比**：
> "最后看过程监控。预构建版 `invoke()` 只给最终结果，中间调用了什么工具、传了什么参数，你是看不见的。自定义版用 `stream()` 逐个节点输出，每个工具调用的输入输出都实时可见。配合 Langfuse，还能精确看到每次调用的耗时和 Token 用量。"

### 关键转折点

1. **选型判断**：给出 3 个业务场景，让学员判断用预构建还是自定义
- "内部客服机器人，只需要查知识库" → 预构建
- "银行转账助手，涉及金额需要审批" → 自定义
- "快速原型验证，3 天内出 demo" → 预构建，后期再迁移

2. **上下文爆炸**："如果 Agent 循环了 20 次工具调用，messages 会有多长？怎么解决？"
- 答：40+ 条消息，Token 可能过万。解决方案：Trimming / Summarization / Tool 精简。

## 7. 常见误区与注意事项

| 坑 | 现象 | 预警话术 |
|----|------|---------|
| **预构建 Agent 想加自定义 State** | 发现只能传 messages | "`create_react_agent` 的 State 是固定的，只有 messages 字段。需要自定义字段 → 必须用 StateGraph。" |
| **预构建 Agent 想改路由逻辑** | 发现只能走 LLM→Tool→LLM 循环 | "预构建的内部路由是黑盒，改不了。需要自定义分支 → 必须用 StateGraph。" |
| **messages 太长导致 API 报错** | 超出模型上下文限制 | "预构建 Agent 不会自动裁剪消息。需要自己加 Trimming 逻辑，或控制 tool 返回长度。" |
| **把预构建用于生产环境** | 发现无法审计、无法回滚 | "预构建适合原型和内部工具。生产环境需要可观测、可干预、可审计 → 自定义 StateGraph。" |
| **误以为预构建性能更好** | 其实和自定义差不多 | "预构建底层也是 StateGraph，只是帮你搭好了。性能没有本质差异。" |

### 调试技巧

1. **看内部过程**：用 `stream()` 代替 `invoke()`，可以看到每次 tool 调用
2. **Message 长度监控**：打印 `len(messages)` 和 `count_tokens(messages)`，及时发现膨胀
3. **Langfuse 追踪**：预构建 Agent 也支持 CallbackHandler，可以看到完整的 Tool 调用链路
4. **迁移路径**：从预构建迁移到自定义时，先 copy 预构建的 State 结构，再逐步添加自定义字段

## 8. 阶段小结

### 核心要点回顾

1. **`create_react_agent` 是单 Agent 唯一官方推荐**：LangGraph 1.x 统一用这一个，通吃所有模型
2. **多 Agent 生态扩展**：`create_supervisor`（监督者调度）、`create_swarm`（平级交接），需额外安装
3. **更复杂的多 Agent 架构**：DeepResearch Agent 涉及模式路由、计划循环、子图嵌套等，是 LG-01~LG-06 的综合演练，安排在 LG-08
4. **内部机制**：LLM → Tool → LLM 自动循环，消息历史自动管理
5. **State 限制**：只有 `messages` 一个字段，不可扩展
6. **选型原则**：标准 ReAct + 不需要监控 → 预构建；复杂路由/HITL/自定义 State → 自定义
7. **上下文管理**：预构建不会自动裁剪，需要自己加 Trimming/Summarization
8. **迁移策略**：先用预构建验证需求，确认后再迁移到自定义 StateGraph

### 记忆口诀

> **"create_react_agent 是正统，预构建求快三行搞定，自定义求控全程透明。先用预构建验证，再上 StateGraph 生产。"**
