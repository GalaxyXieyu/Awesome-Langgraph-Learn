create_react_agent
3 行代码搭一个 Agent
from langgraph.prebuilt import create_react_agent from langchain_openai import ChatOpenAI llm = ChatOpenAI(model="gpt-4o") tools = [query_weather, calculator] agent = create_react_agent(llm, tools) # 3 行代码搞定 result = agent.invoke({"messages": [HumanMessage("北京天气？23 的平方根？")]})
用户提问 → LLM 思考 → 输出 tool_calls → ToolNode 执行 → LLM 整合回复
LangGraph 渐进式教程 LG-06 / 11
create_react_agent 是 LangGraph 1.x 唯一官方推荐的预构建 Agent。它内部调用 LLM 的原生 tool_calling API，覆盖所有主流模型。3 行代码就能搭一个带工具调用能力的 Agent。输出过程：用户提问 → LLM 思考需要查天气+算平方根 → 输出 tool_calls → ToolNode 并行执行 → ToolMessage 返回 → LLM 再次思考整合 → 回复用户。
Source Code Deep Dive
循环终止的判据：tool_calls 是否为空

should_continue 路由逻辑

def should_continue(state): last = state["messages"][-1] if not isinstance(last, AIMessage) \ or not last.tool_calls: return END # 循环终止 return "tools" # 继续执行工具

不是 LLM "说" 要结束，而是程序检查输出格式

步数限制：防止无限循环

class AgentState(TypedDict): messages: Annotated[Sequence, add_messages] is_last_step: IsLastStep remaining_steps: RemainingSteps

remaining_steps < 2 且还有 tool_calls → 强制替换为 "Sorry, need more steps..."

agent (LLM) → should_continue → tools / END → tools → agent (循环)
LangGraph 渐进式教程 LG-06 / 11
核心路由函数 should_continue 检查最后一条消息是否是 AIMessage 且包含 tool_calls。如果没有 tool_calls 就结束循环。这是 LangGraph 的安全机制：即使 LLM 无限想调工具，也会被 remaining_steps 或 is_last_step 强制截断。预构建自动帮你搭了 agent 节点、tools 节点、条件边、循环边和 AgentState。
Limitations & Extensions
预构建 Agent 的局限
局限说明影响
State 不可扩展只有 messages 一个字段无法存业务状态
路由不可定制固定 LLM → Tool → LLM 循环无法按业务状态分支
中间过程不可见invoke() 只给最终结果无法监控每步在做什么
循环控制受限由内部逻辑决定循环次数无法自定义重试策略
HITL 支持弱没有原生断点能力无法人工审批中间步骤

传 System Prompt

agent = create_react_agent( llm, tools, state_modifier=SystemMessage("你是专业天气助手") )

pre_model_hook：消息裁剪

def trimming_hook(state): msgs = state["messages"] if len(msgs) > 20: to_remove = msgs[:-10] return {"messages": [RemoveMessage(id=m.id) for m in to_remove]} return {}
LangGraph 渐进式教程 LG-06 / 11
create_react_agent 的 State 等价于只有一个 messages 字段的 TypedDict。这意味着你无法自定义其他状态字段如 retry_count、user_role、session_id。但可以通过 state_modifier 传 system prompt，通过 pre_model_hook 在调用 LLM 前执行自定义逻辑如消息裁剪。
Multi-Agent Patterns
三种预构建多 Agent 模式
模式核心函数来源包适用场景
单 Agentcreate_react_agentlanggraph一个 LLM + 多个工具
监督者create_supervisorlanggraph-supervisor一个主管调度多个专员
群体协作create_swarmlanggraph-swarm多个专员平级交接

create_supervisor：层级调度

Supervisor 本身就是一个 create_react_agent，tools 里包含所有 sub-agent 的 handoff_tools。handoff tool 返回 Command(goto=agent_name) 跳转。

workflow = create_supervisor( [weather_agent, order_agent], model=llm, prompt="你是调度主管" )

create_swarm：平级交接

没有中心调度器。State 多了 active_agent 字段。每次从 START 重新进入，根据 active_agent 路由。handoff 时更新 active_agent 状态。

workflow = create_swarm( [alice, bob], default_active_agent="Alice" )
LangGraph 渐进式教程 LG-06 / 11
create_supervisor 和 create_swarm 不在核心 langgraph 包中，需要额外安装。Supervisor 适合层级化管理，Swarm 适合平级 Agent 动态交接。两者的核心区别：Supervisor 有中心调度器，sub-agent 执行完自动回到 supervisor；Swarm 没有中心调度器，handoff 后更新 active_agent，下次用户输入从 START 重新路由。
Decision Framework
什么时候用预构建，什么时候自定义？
维度预构建 Agent自定义 StateGraph
开发速度快（3 行代码）中等
可控性黑盒白盒，每步可控
可观测性只能看输入输出每节点状态可见
复杂路由不支持任意图结构
HITL 支持弱强
自定义 State只有 messages任意字段

决策树

  • 标准 ReAct 循环 + 不需要监控 + 不需要自定义 State → create_react_agent
  • 需要监控每一步 State / 需要复杂路由 / 需要人工审批 / 需要自定义 State → 自定义 StateGraph
LangGraph 渐进式教程 LG-06 / 11
预构建 Agent 的"快"是有代价的——你放弃了对中间过程的可见性和控制力。只要出现"需要看内部状态"或"需要干预执行"的需求，就必须切换到自定义 StateGraph。建议先用预构建验证需求，确认后再迁移到自定义 StateGraph 生产。
Context Management
防止 Agent 撑爆上下文窗口

策略 1：Trimming

裁剪历史，只保留最近 N 条或 N 个 token

trimmed = trim_messages( messages, max_tokens=4000, strategy="last", token_counter=llm )

难度：低 信息损失：中

策略 2：Summarization

当消息太长时，总结早期内容

if count_tokens(msgs) > 6000: to_sum = msgs[:-4] recent = msgs[-4:] summary = llm.invoke([...]) return {"messages": [summary]+recent}

难度：中 信息损失：低

策略 3：Tool 精简

在 tool 定义阶段就控制输出长度

def query_weather(city): raw = query_api(city) return f"{city}：{raw['weather'][0]['description']}，{raw['main']['temp']}°C"

难度：低 信息损失：可控

LangGraph 渐进式教程 LG-06 / 11
预构建 Agent 的 State 只有 messages，每次 tool 调用都会把结果追加到 messages。一个 session 循环 10 次，messages 可能有 30+ 条，Token 轻松过万。三种策略：Trimming 实现简单但会丢掉早期消息；Summarization 智能压缩信息损失小；Tool 精简从源头控制，应该在定义 tool 时就做。
Case Study: OpsBot
预构建 vs 自定义：同一个运维助手

版本 A：create_react_agent（3 行）

tools = [check_cpu, check_memory, check_disk, restart_service, query_logs] agent = create_react_agent(llm, tools)
  • 简单查询：查 CPU → 直接调用，3 行搞定
  • 敏感操作：重启服务 → 直接执行，无阻拦
  • 长日志：1000 条 → 直接塞进 history
  • 过程可见性 → 黑盒

版本 B：自定义 StateGraph（完整可控）

class OpsState(TypedDict): messages: Annotated[list, add_messages] user_role: str pending_approval: bool execution_log: list
  • 简单查询：查 CPU → 同样正确
  • 敏感操作：检查 user_role → 普通用户需审批
  • 长日志：截断返回前 50 条 + 统计
  • 过程可见性 → stream() 逐节点输出
LangGraph 渐进式教程 LG-06 / 11
用两种方式实现同一个运维助手对比体验。预构建版 3 行代码搞定简单查询，但敏感操作直接执行无阻拦，长日志直接塞进 message history 导致 Token 暴涨，过程完全黑盒。自定义版在 restart_service 前有条件边检查 user_role，query_logs 做截断，stream() 逐节点输出可观测。关键差异：预构建版快但不可控，自定义版慢但安全。
Summary
核心要点回顾

create_react_agent 核心认知

  • LangGraph 1.x 唯一官方推荐的预构建 Agent
  • 内部是 LLM → Tool → LLM 自动循环
  • State 只有 messages，不可扩展
  • 循环终止判据：AIMessage.tool_calls 为空
  • 有 remaining_steps / is_last_step 安全机制

选型与生态

  • 标准 ReAct + 不需要监控 → 预构建
  • 复杂路由 / HITL / 自定义 State → 自定义
  • 多 Agent：supervisor（层级）、swarm（平级）
  • 上下文管理：Trimming / Summarization / Tool 精简
  • 先用预构建验证，再上 StateGraph 生产

"create_react_agent 是正统，预构建求快三行搞定，自定义求控全程透明。先用预构建验证，再上 StateGraph 生产。"

核心要点回顾：create_react_agent 是单 Agent 唯一官方推荐；多 Agent 生态有 create_supervisor 和 create_swarm；内部机制是 LLM → Tool → LLM 自动循环；State 只有 messages 不可扩展；选型原则是标准 ReAct 不需要监控用预构建，复杂路由 HITL 自定义 State 用自定义；上下文管理需要自己做 Trimming/Summarization；建议先用预构建验证需求，确认后再迁移到自定义 StateGraph。
