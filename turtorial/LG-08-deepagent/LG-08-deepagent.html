// part 01 · architecture

DeepAgents 是 LangGraph 之上的
更高层抽象

Planning 任务规划

write_todos / read_todos 自动注入，Agent 自主拆解复杂任务为有序执行列表

Filesystem 文件系统

ls / read_file / write_file / edit_file / glob / grep，持久化工作空间突破上下文限制

Sub-agent 子智能体

同步 SubAgent + 异步 AsyncSubAgent，实现串行和并行深度研究

Skills 领域知识

SKILL.md 三层架构，按需加载领域知识，不浪费上下文窗口

Middleware 中间件栈

10 层默认 Middleware 自动组装，可扩展可排除，洋葱模型执行

Backend 后端系统

State / Store / Filesystem / Composite / Sandbox 多种持久化策略

DeepAgents 不是替代 LangGraph，而是基于 LangGraph 的更高层抽象。它帮你把常见模式封装成可复用的组件。

核心洞察是：简单 Agent 等于 LLM 加 Tools 加循环调用，而 Deep Agent 等于 LLM 加 Tools 加 Planning 加 Filesystem 加 Sub-agents 加 Skills。差异不在 LLM，而在架构 surrounding the loop。

这六个模块协同工作：Planning 负责拆解任务，Filesystem 负责持久化，Sub-agent 负责并行协作，Skills 负责知识管理，Middleware 负责可组合扩展，Backend 负责数据生命周期。

// quick start

create_deep_agent 一键创建生产级 Agent

from deepagents import create_deep_agent from langgraph.checkpoint.memory import InMemorySaver agent = create_deep_agent( model="openai:gpt-4o", tools=[search_tool, calc_tool], checkpointer=InMemorySaver(), ) result = agent.invoke({ "messages": [{"role": "user", "content": "研究量子计算最新进展"}] })

推荐使用 "provider:model" 字符串格式

①
模型解析

resolve_model() 自动初始化对应 BaseChatModel

②
子代理处理

自动构建子代理中间件栈，继承父代理工具

③
自动添加通用子代理

未提供时自动创建 general-purpose 子代理

④
组装中间件栈

10 层默认 Middleware 按序组装

⑤
创建最终 Agent

DeltaChannel 优化，O(N) checkpoint

DeepAgents 的入口非常简单，就是 create_deep_agent 这一个函数。但内部有 7 步组装流程。

第一步模型解析，支持 provider:model 字符串格式。第二步处理子代理规格，自动构建子代理的中间件栈。第三步自动添加通用子代理。第四步组装主代理中间件栈。第五步组装系统提示。第六步应用 Harness Profile。第七步创建最终 Agent。

一个关键优化是 DeltaChannel，把 checkpoint 增长从 O(N²) 降到 O(N)。在长对话中，普通列表每次 checkpoint 都要保存完整消息历史，而 DeltaChannel 只保存增量变化。

// part 02 · planning

Planning 让 Agent 不再迷失在复杂任务中

没有 Planning 的 Agent

用户："分析 2024 年新能源汽车行业" 搜索 → 看到结果 → 不知道下一步 → 随机调用工具 → 零散回答

没有大纲，做到一半忘记下一步该做什么

有 Planning 的 Agent

用户："分析 2024 年新能源汽车行业" write_todos([ "1. 收集市场规模数据", "2. 分析主要厂商份额", "3. 梳理技术趋势", "4. 分析政策影响", "5. 撰写综合报告" ]) → 按大纲执行 → 每完成一项勾选 → 全部完成后总结

TodoList 自动注入，Agent 自主规划有序执行

write_todos()

创建或更新待办事项列表

read_todos()

读取当前待办事项及完成状态

Planning 解决的是 Agent 遇到复杂任务时"迷失"的问题。没有 Planning 的 Agent，搜索看到结果后不知道下一步干什么，随机调用工具，给出零散回答。

有 Planning 的 Agent 会先调用 write_todos 创建任务列表，然后循环执行：read_todos 查看当前任务，选择第一个 pending 任务执行，完成后更新状态，继续下一个。直到所有任务 completed，再生成最终回答。

这两个工具是 TodoListMiddleware 自动注入的，开发者无需手动配置。这是 DeepAgents 最基础也最强大的能力之一。

Backend 选型指南

后端生命周期场景
StateBackend会话结束消失临时工作文件
StoreBackend持久化长期记忆、偏好
FilesystemBackend真实磁盘外部访问文件
CompositeBackend混合复杂场景路由
backend = CompositeBackend( default=StateBackend(), routes={ "/memories/": StoreBackend(store=store), "/preferences/": StoreBackend(store=store), }, )

Filesystem 解决的是 LLM 上下文窗口有限的问题。通常 128K 到 200K tokens，复杂研究任务产生的中间数据很容易超出这个限制。

没有 Filesystem 时，搜索收集了 50 页资料全部塞进 messages，Token 暴涨费用飙升。有 Filesystem 时，资料写入 /workspace/research/raw_data.md，messages 只保留摘要，需要时 read_file 读取，上下文始终可控。

DeepAgents 提供多种后端控制数据持久化方式。CompositeBackend 最灵活，不同路径用不同后端，比如 /workspace 用 StateBackend 临时存储，/memories 用 StoreBackend 持久化。

// part 04 · sub-agent

Sub-agent 实现团队协作式并行深度研究

三种子智能体类型

SubAgent 同步

声明式同步子智能体，最常见，阻塞式委派

CompiledSubAgent 复用

复用现有 LangGraph，挂载预构建复杂 Agent

AsyncSubAgent 异步

非阻塞后台执行，支持长时间运行任务

AsyncSubAgent 5 个管理工具

工具功能
start_async_task启动后台任务，返回任务 ID
check_async_task轮询状态并获取结果
update_async_task向运行中任务发送指令
cancel_async_task停止运行中的任务
list_async_tasks列出所有任务及状态

关键设计：Supervisor 用强模型（推理），Sub-agent 用便宜模型（执行），可节省 60-80% 成本。

Sub-agent 解决的是复杂任务包含多个相对独立子任务时的并行处理问题。比如分析新能源汽车行业，可以拆成市场数据、竞品分析、技术趋势三个子任务，三人并行工作。

DeepAgents 支持三种子智能体。SubAgent 是声明式同步的，最常用。CompiledSubAgent 可以复用已有的 LangGraph。AsyncSubAgent 是 v0.5 引入的，支持非阻塞后台执行。

AsyncSubAgent 暴露 5 个管理工具，典型生命周期是：start_async_task 启动任务返回 task_id，Supervisor 继续做其他事，然后 check_async_task 查询结果。一个关键的成本优化策略是Supervisor 用强模型做推理决策，Sub-agent 用便宜模型做执行。

// part 05 · skills

Skills 实现按需加载领域知识，节省 90% 上下文

三层渐进式披露架构

Layer 1: Discovery 始终存在

轻量级元数据：name + description，始终存在于 System Prompt 中

Layer 2: Activation 使用时加载

完整 SKILL.md：instructions + rules，Agent 决定使用时加载

Layer 3: Deep Context 执行时加载

参考资料、脚本、模板、示例，执行时按需加载

SKILL.md 格式示例

--- name: web-research description: Structured approach to conducting thorough web research license: MIT --- # Web Research Skill ## When to Use - User asks you to research a topic - Need to gather information from sources ## Steps 1. Formulate search queries 2. Execute searches 3. Synthesize findings 4. Cite sources

启动时扫描 skills 目录，解析 YAML frontmatter，将 name + description 注入 System Prompt

Skills 解决的是领域知识太多不能全塞进 prompt 的问题。想象 Agent 需要处理财务分析、法律咨询、医疗诊断、代码审查多个领域。如果全部塞进 System Prompt，上下文会爆炸到 3 万 tokens。

Skills 采用三层渐进式披露架构。第一层 Discovery 只保留 name 和 description 在 System Prompt 中，几百 tokens。第二层 Activation 在 Agent 决定使用时加载完整 SKILL.md。第三层 Deep Context 在执行时按需加载参考资料和模板。

SKILL.md 使用 YAML frontmatter 格式，包含 name、description、license 等字段。启动时自动扫描目录，解析 frontmatter，完成 Skill 自动发现流程。

// part 06 · middleware

10 层 Middleware 栈按洋葱模型顺序执行

#Middleware功能条件
1TodoListMiddleware任务规划（write_todos / read_todos）默认
2SkillsMiddlewareSkills 加载提供 skills 时
3FilesystemMiddleware文件操作（必需，不可排除）默认
4SubAgentMiddleware子智能体委派（必需，不可排除）默认
5SummarizationMiddleware自动摘要默认
6PatchToolCallsMiddleware修复不完整工具调用默认
7用户自定义 Middleware自定义扩展用户提供
8AnthropicPromptCachingMiddleware提示词缓存默认
9MemoryMiddleware长期记忆注入提供 memory 时
10HumanInTheLoopMiddleware人工审批提供 interrupt_on 时

顺序设计精妙：提示缓存放在 Memory 之前（避免缓存失效），HITL 放在最外层（确保所有工具调用经人工审批）

create_deep_agent 自动按特定顺序组装 10 层 Middleware。这个顺序不是随意的，而是经过精心设计的。

FilesystemMiddleware 和 SubAgentMiddleware 是不可排除的脚手架中间件，如果用户尝试排除它们会抛出 ValueError。提示缓存放在 Memory 之前，是因为 MemoryMiddleware 会修改系统提示，如果在缓存之后修改会导致缓存失效。HITL 放在最外层，确保所有工具调用都经过人工审批后再真正执行。

SummarizationMiddleware 放在 PatchToolCallsMiddleware 之前，先压缩历史再修复工具调用格式。用户自定义中间件插入在第 6 层之后，Harness Profile 额外中间件紧随其后。

// production

生产级部署的 3 条核心建议

01
模型异构策略：规划用强模型，执行用便宜模型

Supervisor 推荐 Claude Sonnet / GPT-4o（强推理），Sub-agent 推荐 GPT-4o-mini / Claude Haiku（低成本并行），Summarization 用 GPT-4o-mini。可节省 60-80% 成本。

02
Backend 分层：临时数据 State，长期记忆 Store

开发环境用 StateBackend，生产环境用 CompositeBackend 路由：/workspace/ 用 StateBackend（临时工作文件），/memories/ 用 StoreBackend（跨会话持久化）。

03
避免 8 个常见坑：从 Planning 过细到忘记 checkpointer

Planning 过于详细 → 限制 todo 数量；Filesystem 文件混乱 → 约定目录结构；Sub-agent 过多 → 限制 max_concurrent；Skills 冲突 → 明确适用范围；上下文膨胀 → 启用 SummarizationMiddleware；忘记 checkpointer → Skills 不工作。

三条生产级核心建议。第一条是模型异构策略，这是成本优化的关键。Supervisor 做决策需要强推理能力，用 Claude Sonnet 或 GPT-4o。Sub-agent 做执行不需要强推理，用 GPT-4o-mini 或 Claude Haiku 并行处理，成本大幅降低。

第二条是 Backend 分层。不要把所有数据都持久化，也不要都不持久化。工作文件用 StateBackend 会话结束自动清理，长期记忆和用户偏好用 StoreBackend 持久化存储。

第三条是避免常见坑。Planning 不要太细，5 到 10 个 todo 项足够。文件要有命名规范。Sub-agent 数量要限制。最重要的是checkpointer 是使用 Skills 的必需条件，很多人忘了配导致 Skills 不工作。
