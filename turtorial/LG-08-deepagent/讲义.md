## 2. DeepAgents 架构概述

### 2.1 什么是 DeepAgents？

**DeepAgents** 是 LangChain 官方推出的开源 Agent Harness，基于 LangGraph 构建，专门用于构建能够处理**长周期、多步骤、复杂任务**的智能体。

```
┌─────────────────────────────────────────────────────────────────────┐
│                        DeepAgents 架构                               │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   ┌─────────────┐  ┌─────────────┐  ┌─────────────┐  ┌───────────┐│
│   │  Planning   │  │ Filesystem  │  │  Sub-agent  │  │  Skills   ││
│   │   任务规划   │  │  文件系统   │  │   子智能体   │  │ 领域知识  ││
│   └──────┬──────┘  └──────┬──────┘  └──────┬──────┘  └─────┬─────┘│
│          │                │                │               │      │
│          └────────────────┴────────────────┴───────────────┘      │
│                                   │                               │
│                    ┌──────────────▼──────────────┐               │
│                    │       Middleware 层         │               │
│                    │  TodoList | Filesystem      │               │
│                    │  SubAgent | Summarization   │               │
│                    └─────────────┬───────────────┘               │
│                                  │                                │
│                    ┌─────────────▼────────┐                      │
│                    │    LangGraph 运行时   │                      │
│                    │  StateGraph | Check   │                      │
│                    │  Stream | Human-loop  │                      │
│                    └──────────────────────┘                      │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**核心洞察**：
> 简单 Agent = LLM + Tools + 循环调用
> Deep Agent = LLM + Tools + **Planning** + **Filesystem** + **Sub-agents** + **Skills**
>
> 差异不在 LLM，而在**架构 surrounding the loop**。

### 2.2 四大架构支柱

| 支柱 | 解决的问题 | 类比 |
|------|-----------|------|
| **Planning** | 复杂任务不知道怎么拆 | 写论文先列大纲 |
| **Filesystem** | 上下文窗口装不下全部信息 | 用文件柜存资料 |
| **Sub-agent** | 一个人干不完，需要团队协作 | 找实习生分工 |
| **Skills** | 领域知识太多，不能全塞进 prompt | 按需查手册 |

### 2.3 DeepAgents 与 LangGraph 的关系

```
┌─────────────────────────────────────────┐
│         DeepAgents (高层抽象)            │
│   - 10 层默认 Middleware 栈             │
│   - 预置 Tools (filesystem, todo)       │
│   - Skills 系统                         │
│   - 最佳实践封装                        │
├─────────────────────────────────────────┤
│         LangGraph (编排框架)             │
│   - StateGraph                          │
│   - Checkpointing                       │
│   - Human-in-the-loop                   │
│   - Streaming                           │
├─────────────────────────────────────────┤
│         LangChain (基础组件)             │
│   - LLM 接口                            │
│   - Tools                               │
│   - Prompt Templates                    │
└─────────────────────────────────────────┘
```

DeepAgents 不是替代 LangGraph，而是**基于 LangGraph 的更高层抽象**——它帮你把常见模式封装成可复用的组件。

## 3. create_deep_agent：一键创建 Deep Agent

### 3.1 最简单的使用方式

```python
from deepagents import create_deep_agent
from langgraph.checkpoint.memory import InMemorySaver

agent = create_deep_agent(
    model="openai:gpt-4o",  # 支持 "provider:model" 格式
    tools=[search_tool, calculator_tool],
    checkpointer=InMemorySaver(),  # 会话持久化
)

result = agent.invoke({
    "messages": [{"role": "user", "content": "研究量子计算最新进展"}]
})
```

### 3.2 完整参数签名

```python
create_deep_agent(
    model: str,                          # "provider:model" 格式
    tools: list[BaseTool] = [],          # 自定义工具
    subagents: list[dict | AsyncSubAgent] = [],  # 子智能体定义
    system_prompt: str = "",             # 系统提示
    backend: BaseBackend = None,         # 文件系统后端
    store: BaseStore = None,             # 长期记忆存储
    memory: list[str] = [],              # 记忆文件路径
    checkpointer: Checkpointer = None,   # 检查点（必需）
    skills: list[str] = [],              # Skills 目录路径
    interrupt_on: dict = {},            # 人工审批配置
    middleware: list[AgentMiddleware] = [],  # 自定义中间件
    excluded_middleware: list[str] = [],    # 排除默认中间件
    debug: bool = False,                # 调试模式
)
```

### 3.3 模型字符串格式

**推荐使用 `"provider:model-name"` 格式**，而非预构建模型实例：

```python
# ✅ 推荐：使用字符串
agent = create_deep_agent(
    model="openai:gpt-4o-mini",
)

# ❌ 避免：使用模型实例（可能导致流式崩溃）
model = ChatOpenAI(model="gpt-4o-mini")
agent = create_deep_agent(model=model)
```

**支持的 provider**：`openai`、`anthropic`、`google`、`ollama`、`groq` 等。

## 4. Planning：任务规划系统

### 4.1 为什么需要 Planning？

简单 Agent 遇到复杂任务时会"迷失"——不知道从何下手、做到一半忘记下一步该做什么。

```
用户："分析 2024 年新能源汽车行业格局及未来 5 年趋势"

❌ 没有 Planning 的 Agent：
   搜索 → 看到结果 → 不知道下一步干什么 → 随机调用工具 → 零散回答

✅ 有 Planning 的 Agent：
   写大纲 → 1.收集市场规模 2.分析主要玩家 3.技术趋势 4.政策影响 5.撰写报告
   → 按大纲执行 → 每完成一项勾选 → 全部完成后总结
```

### 4.2 TodoListMiddleware：自动注入 Todo 工具

DeepAgents 的 `create_deep_agent` **自动包含** `TodoListMiddleware`，无需手动配置。

**自动注入的工具**：

| 工具 | 功能 |
|------|------|
| `write_todos(todos: list[str])` | 创建或更新待办事项列表 |
| `read_todos()` | 读取当前待办事项及完成状态 |

```python
# 这些是 Agent 自主调用的（不是开发者直接调用）

# Agent 内部调用示例：
write_todos(todos=[
    "1. 收集 2024 年新能源汽车市场规模数据",
    "2. 分析主要厂商市场份额（BYD、Tesla、理想）",
    "3. 梳理电池技术发展趋势",
    "4. 分析各国政策影响",
    "5. 撰写综合分析报告",
])

read_todos()  # 返回当前 todo 列表和状态
```

### 4.3 Planning 的工作流程

```
用户输入复杂问题
       ↓
[Agent] → 调用 write_todos 创建任务列表
       ↓
循环：
  [Agent] → 调用 read_todos 查看当前任务
       ↓
  [Agent] → 选择第一个 pending 任务执行
       ↓
  [Agent] → 完成任务后，write_todos 更新状态
       ↓
  [Agent] → 继续下一个 pending 任务
       ↓
直到所有任务 completed → 生成最终回答
```

## 5. Filesystem：文件系统工作空间

### 5.1 为什么需要 Filesystem？

LLM 的上下文窗口有限（通常 128K-200K tokens）。复杂研究任务产生的中间数据很容易超出这个限制。

```
❌ 没有 Filesystem：
   搜索收集了 50 页资料 → 全部塞进 messages → Token 暴涨 → 费用飙升

✅ 有 Filesystem：
   搜索收集了 50 页资料 → 写入 /workspace/research/raw_data.md
   → messages 只保留摘要 → 需要时 read_file 读取 → 上下文始终可控
```

### 5.2 FilesystemMiddleware 自动注入的工具

`create_deep_agent` **自动包含** `FilesystemMiddleware`，注入以下工具：

| 工具 | 功能 |
|------|------|
| `ls(path: str)` | 列出目录内容 |
| `read_file(path: str, offset?, limit?)` | 读取文件（支持行范围） |
| `write_file(path: str, content: str)` | 写入新文件 |
| `edit_file(path: str, old_string: str, new_string: str)` | 原地编辑文件 |
| `glob(pattern: str)` | 按模式查找文件 |
| `grep(pattern: str, path?: str)` | 在文件中搜索 |

```python
# Agent 自主调用示例：
write_file(
    path="/workspace/research/market_size.md",
    content="# 2024 新能源汽车市场规模\n\n- 全球销量：1,400 万辆\n..."
)

read_file(path="/workspace/research/market_size.md")

edit_file(
    path="/workspace/research/market_size.md",
    old_string="- 全球销量：1,400 万辆",
    new_string="- 全球销量：1,450 万辆（修订版）"
)
```

### 5.3 Backend 系统：数据持久化

DeepAgents 提供多种后端来控制数据的持久化方式：

```python
from deepagents.backends import (
    StateBackend,           # 默认：每线程临时存储
    FilesystemBackend,      # 真实磁盘访问
    StoreBackend,           # 跨线程持久化
    CompositeBackend,       # 混合路由
)

# CompositeBackend：不同路径用不同后端
backend = CompositeBackend(
    default=StateBackend(),  # /workspace/* → 临时（会话结束消失）
    routes={
        "/memories/": StoreBackend(store=store),      # 持久化
        "/preferences/": StoreBackend(store=store),   # 持久化
    },
)

agent = create_deep_agent(
    model="openai:gpt-4o",
    backend=backend,
)
```

**Backend 选型指南**：

| 后端 | 数据生命周期 | 适用场景 |
|------|-------------|---------|
| `StateBackend` | 会话结束消失 | 临时工作文件 |
| `StoreBackend` | 持久化 | 长期记忆、用户偏好 |
| `FilesystemBackend` | 写入真实磁盘 | 需要外部访问的文件 |
| `CompositeBackend` | 混合 | 复杂场景，不同数据不同策略 |

### 5.4 工作空间目录结构

```
/workspace/                          # StateBackend：会话临时存储
├── plan.md                         # 任务计划
├── research/                       # 研究资料
│   ├── market_size.md
│   ├── competitors.md
│   └── tech_trends.md
├── analysis/                       # 分析结果
│   ├── swot.md
│   └── forecast.md
└── report.md                       # 最终报告

/memories/                          # StoreBackend：持久化
├── agent_notes.md                  # Agent 的长期笔记
└── user_preferences.md             # 用户偏好
```

## 6. Sub-agent Delegation：子智能体委派

### 6.1 为什么需要 Sub-agent？

复杂任务往往包含多个相对独立的子任务，适合并行处理：

```
任务："分析新能源汽车行业"

├─→ 子任务 A：市场数据收集（不需要 B、C 的结果）
├─→ 子任务 B：竞品分析（不需要 A、C 的结果）
└─→ 子任务 C：技术趋势分析（不需要 A、B 的结果）

→ 三个子任务可以并行执行！
```

### 6.2 子智能体类型

DeepAgents 支持三种子智能体：

| 类型 | 描述 | 使用场景 |
|------|------|---------|
| `SubAgent` | 声明式同步子智能体 | 最常见；阻塞式委派 |
| `CompiledSubAgent` | 复用现有 LangGraph | 挂载预构建的复杂 Agent |
| `AsyncSubAgent` | 非阻塞后台执行 | 长时间运行任务 |

### 6.3 定义子智能体

```python
from deepagents import create_deep_agent

# 定义子智能体（声明式）
research_subagent = {
    "name": "research-agent",
    "description": "Used to research in-depth questions with web search",
    "system_prompt": """You are an expert researcher.
    Use internet_search to find information and write findings to files.
    Always summarize your findings concisely.""",
    "tools": [internet_search],           # 子 Agent 可用工具
    "model": "openai:gpt-4o-mini",        # 子 Agent 可使用不同模型
}

analysis_subagent = {
    "name": "data-analyzer",
    "description": "Specialized agent for data analysis",
    "system_prompt": "You are a data analysis expert.",
    "tools": [],
}

# 创建主 Agent，注册子智能体
agent = create_deep_agent(
    model="anthropic:claude-sonnet-4-20250514",  # Supervisor 用强模型
    tools=[get_weather, internet_search],
    subagents=[research_subagent, analysis_subagent],
)
```

**关键设计**：Supervisor 用强模型（推理），Sub-agent 用便宜模型（执行），大幅降低成本。

### 6.4 task 工具：委派子任务

子智能体通过 `task` 工具被调用（自动注入，无需配置）：

```python
# Agent 内部调用示例：
# Supervisor LLM 决定何时调用 task 工具

task(
    description="Search for recent AI papers on arXiv",
    subagent_type="research-agent",  # 指定使用哪个子智能体
)

# 子智能体隔离运行，返回最终结果给 Supervisor
```

### 6.5 AsyncSubAgent：异步非阻塞子智能体

v0.5 引入的 AsyncSubAgent 让子任务可以**并行执行、长时间运行**：

```python
from deepagents import AsyncSubAgent, create_deep_agent

# 定义异步子智能体
async_researcher = AsyncSubAgent(
    name="deep-researcher",
    description="Performs deep research on a topic (may take several minutes).",
    url="https://my-agent-server.dev",  # 远程部署（可选）
    graph_id="research_agent",
)

agent = create_deep_agent(
    model="anthropic:claude-sonnet-4-6",
    subagents=[async_researcher],
)
```

**AsyncSubAgent 暴露的 5 个工具**：

| 工具 | 功能 | 返回 |
|------|------|------|
| `start_async_task` | 启动后台任务 | 任务 ID（立即） |
| `check_async_task` | 轮询状态并获取结果 | 状态 + 结果 |
| `update_async_task` | 向运行中的任务发送后续指令 | 确认 + 更新状态 |
| `cancel_async_task` | 停止运行中的任务 | 确认 |
| `list_async_tasks` | 列出所有任务及状态 | 所有任务摘要 |

**典型生命周期**：

```
Supervisor: start_async_task("deep-researcher", "Research topic X")
<- 返回: task_id: "abc123"

# Supervisor 继续做其他事情...

Supervisor: check_async_task("abc123")
<- 返回: status: "success", result: "findings..."
```

### 6.6 AsyncSubAgent 部署选项

| 传输方式 | 使用场景 | 配置 |
|----------|----------|------|
| **ASGI** | 同进程共存 | 省略 `url` 字段 |
| **HTTP** | 远程独立部署 | 设置 `url` 为远程服务器 |
| **LangSmith Deployments** | 托管部署 | 使用 LangGraph Platform |

## 7. Skills：渐进式领域知识披露

### 7.1 为什么需要 Skills？

想象 Agent 需要处理多个领域的任务：财务分析、法律咨询、医疗诊断、代码审查。如果把所有领域知识都塞进 System Prompt，上下文会爆炸。

```
❌ 没有 Skills：
   System Prompt = "你是专家，懂财务、法律、医疗、代码、设计、营销、运营..."
   → 30,000 tokens → 每次调用都花钱 → Agent 不知道用哪个知识

✅ 有 Skills：
   System Prompt = "你是助手，可以加载专业技能。可用技能：财务分析、法律咨询..."
   → 500 tokens → 需要时调用 load_skill("财务分析") → 只加载相关知识
```

### 7.2 Skills 三层架构

```
┌─────────────────────────────────────────┐
│  Layer 1: Discovery（技能发现）          │
│  轻量级元数据：name + description         │
│  → 始终存在于 System Prompt 中           │
├─────────────────────────────────────────┤
│  Layer 2: Activation（技能激活）         │
│  完整 SKILL.md：instructions + rules     │
│  → Agent 决定使用时加载                  │
├─────────────────────────────────────────┤
│  Layer 3: Deep Context（深度上下文）     │
│  参考资料、脚本、模板、示例               │
│  → 执行时按需加载                        │
└─────────────────────────────────────────┘
```

### 7.3 Skill 目录结构

```
./skills/
├── web-research/
│   ├── SKILL.md          # 必需：YAML frontmatter + markdown 指令
│   └── helper.py         # 可选：辅助文件
├── data-processing/
│   ├── SKILL.md
│   └── templates/
│       └── report_template.md
└── financial-analysis/
    └── SKILL.md
```

### 7.4 SKILL.md 格式

```markdown
---
name: web-research
description: Structured approach to conducting thorough web research
license: MIT
---

# Web Research Skill

## When to Use
- User asks you to research a topic
- Need to gather information from multiple sources

## Steps
1. Formulate search queries
2. Execute searches
3. Synthesize findings
4. Cite sources

## Example
User: "What are the latest AI breakthroughs?"
→ Search: "AI breakthroughs 2024"
→ Search: "latest LLM research"
→ Summarize findings with citations
```

**YAML Frontmatter 字段**：

| 字段 | 描述 | 约束 |
|------|------|------|
| `name` | Skill 标识符 | 最多 64 字符，小写字母、数字、连字符 |
| `description` | Skill 功能描述 | 最多 1024 字符 |
| `license` | 可选许可证 | — |
| `module` | 可选 JS/TS 模块路径 | 用于解释器 skills |

### 7.5 注册和使用 Skills

```python
from deepagents import create_deep_agent

# 方式 1：通过 skills 参数（最简单）
agent = create_deep_agent(
    model="openai:gpt-4",
    skills=["./skills/"],       # 自动启用 SkillsMiddleware
    checkpointer=checkpointer,   # 必需！
)

# 方式 2：手动配置 SkillsMiddleware
from deepagents.middleware.skills import SkillsMiddleware
from deepagents.backends.filesystem import FilesystemBackend

backend = FilesystemBackend(root_dir="/path/to/skills")
middleware = SkillsMiddleware(
    backend=backend,
    sources=[
        "/skills/base/",      # 基础 skills
        "/skills/user/",      # 用户覆盖
        "/skills/project/",   # 项目特定（后者覆盖前者）
    ],
)
```

**Skill 自动发现流程**：
1. 启动时扫描 `./skills/` 目录
2. 解析每个 `SKILL.md` 的 YAML frontmatter
3. 将 `name` + `description` 注入 System Prompt（Layer 1）
4. 当 Agent 决定使用某 skill 时，加载完整内容（Layer 2）

## 8. Middleware 架构：10 层默认栈

### 8.1 默认 Middleware 栈

`create_deep_agent()` 自动按以下顺序组装 middleware：

| 位置 | Middleware | 功能 |
|------|-----------|------|
| 1 | `TodoListMiddleware` | 任务规划（write_todos / read_todos） |
| 2 | `SkillsMiddleware` | Skills 加载（如果提供 skills） |
| 3 | `FilesystemMiddleware` | 文件操作 |
| 4 | `SubAgentMiddleware` | 子智能体委派 |
| 5 | `SummarizationMiddleware` | 自动摘要（默认启用） |
| 6 | `PatchToolCallsMiddleware` | 修复不完整工具调用 |
| — | **用户 Middleware** | 自定义扩展（插入此处） |
| — | `AnthropicPromptCachingMiddleware` | 提示词缓存 |
| — | `MemoryMiddleware` | 长期记忆注入（如果提供 memory） |
| — | `HumanInTheLoopMiddleware` | 人工审批（如果提供 interrupt_on） |

### 8.2 自定义 Middleware

```python
from deepagents.middleware.base import AgentMiddleware

class LoggingMiddleware(AgentMiddleware):
    """自定义 Middleware：记录所有工具调用"""

    def wrap_tool_calls(self, tool_calls):
        for tc in tool_calls:
            print(f"[LOG] 调用工具: {tc.name}")
        return tool_calls

    def wrap_tool_result(self, tool_name, result):
        print(f"[LOG] 工具返回: {tool_name}, 长度: {len(str(result))}")
        return result

# 使用
agent = create_deep_agent(
    model="openai:gpt-4o",
    middleware=[LoggingMiddleware()],
)
```

## 9. 实战：构建深度研究 Agent

### 9.1 完整代码示例

```python
import asyncio
from deepagents import create_deep_agent, AsyncSubAgent
from deepagents.backends import CompositeBackend, StateBackend, StoreBackend
from langchain_core.tools import tool
from langgraph.checkpoint.memory import InMemorySaver
from langgraph.store.memory import InMemoryStore

# ===== 1. 定义自定义工具 =====
@tool
def internet_search(query: str) -> str:
    """Run a web search and return results."""
    return f"Search results for: {query}"

@tool
def get_weather(city: str) -> str:
    """Get the weather in a city."""
    return f"The weather in {city} is sunny."

# ===== 2. 定义子智能体 =====
research_subagent = {
    "name": "research-agent",
    "description": "Used to research in-depth questions with web search",
    "system_prompt": "You are an expert researcher...",
    "tools": [internet_search],
    "model": "openai:gpt-4o-mini",  # 子 Agent 用便宜模型
}

async_researcher = AsyncSubAgent(
    name="deep-researcher",
    description="Performs deep research on a topic.",
    # url="https://my-agent-server.dev",  # 远程部署（可选）
)

# ===== 3. 配置后端 =====
store = InMemoryStore()
backend = CompositeBackend(
    default=StateBackend(),
    routes={
        "/memories/": StoreBackend(store=store),
        "/workspace/": StateBackend(),
    }
)

# ===== 4. 创建 Deep Agent =====
agent = create_deep_agent(
    model="anthropic:claude-sonnet-4-20250514",
    tools=[get_weather, internet_search],
    subagents=[research_subagent, async_researcher],
    system_prompt="""You are an expert research assistant.
    When working on complex tasks:
    - Break them down using todo lists
    - Save intermediate results to files
    - Delegate to subagents when appropriate
    """,
    backend=backend,
    store=store,
    memory=["/memories/agent_notes.md"],
    checkpointer=InMemorySaver(),
    skills=["./skills/"],
    interrupt_on={"execute": True},
    debug=True,
)

# ===== 5. 运行 =====
async def main():
    result = await agent.ainvoke({
        "messages": [{
            "role": "user",
            "content": "Research quantum computing and save a report"
        }]
    })
    print(result["messages"][-1].content)

asyncio.run(main())
```

       ↓

## 11. 常见误区与注意事项

| 坑 | 现象 | 解决方案 |
|----|------|---------|
| **Planning 过于详细** | Todo 列表 50 项，执行缓慢 | 限制 todo 数量，合并相关任务 |
| **Filesystem 文件混乱** | 文件命名随意，找不到资料 | 约定目录结构和命名规范 |
| **Sub-agent 过多** | 启动 20 个子任务，系统崩溃 | 限制 max_concurrent |
| **Skills 冲突** | 多个 Skills 给出矛盾指令 | 明确 Skills 的适用范围 |
| **上下文膨胀** | 即使用了 Filesystem，messages 还是太长 | 启用 SummarizationMiddleware |
| **Sub-agent 结果不一致** | 并行任务返回矛盾数据 | Supervisor 层做数据校验 |
| **模型实例格式** | 流式输出崩溃 | 使用 `"provider:model"` 字符串格式 |
| **忘记 checkpointer** | Skills 不工作 | checkpointer 是使用 Skills 的必需条件 |

## 14. 源码架构深度解析

前面我们从使用者的角度介绍了 DeepAgents 的 API。这一节我们深入源码，理解 `create_deep_agent()` 是如何把各个组件组装起来的，以及中间件栈的内部工作机制。

### 14.1 `create_deep_agent()` 的组装流程

`create_deep_agent()` 位于 `deepagents/graph.py`，是整个框架的核心入口。它的内部逻辑可以分解为以下步骤：

#### Step 1: 模型解析

```python
# graph.py 第477-498行
_model_spec: str | None = model if isinstance(model, str) else None

if model is None:
    model = _build_default_model()  # ChatAnthropic("claude-sonnet-4-6")
else:
    model = resolve_model(model)    # "provider:model" 字符串解析

_profile = _harness_profile_for_model(model, _model_spec)
```

**关键设计**：
- 支持 `provider:model` 字符串格式（如 `"openai:gpt-4o-mini"`），通过 `resolve_model()` 自动初始化为对应的 `BaseChatModel`
- 每个模型都有对应的 `HarnessProfile`，定义了该模型特有的系统提示后缀、工具描述重写、额外中间件等
- 默认模型 `claude-sonnet-4-6` 将在 1.0.0 版本中移除，必须显式指定

#### Step 2: 验证不可排除的中间件

```python
# graph.py 第187-214行
_REQUIRED_MIDDLEWARE = (
    (FilesystemMiddleware, ()),
    (SubAgentMiddleware, ()),
)
```

`FilesystemMiddleware` 和 `SubAgentMiddleware` 是**不可排除的脚手架中间件**：
- `FilesystemMiddleware` 提供所有内置文件工具（ls、read_file、write_file 等），并强制执行 `permissions` 安全规则
- `SubAgentMiddleware` 提供 `task` 工具，是子代理委派的基础
- 如果用户尝试通过 `HarnessProfile.excluded_middleware` 排除它们，会抛出 `ValueError`

#### Step 3: 处理子代理规格

```python
# graph.py 第526-612行
for spec in subagents or []:
    if "graph_id" in spec:
        # AsyncSubAgent — 路由到 AsyncSubAgentMiddleware
        async_subagents.append(spec)
    elif "runnable" in spec:
        # CompiledSubAgent — 预编译的可运行对象
        inline_subagents.append(spec)
    else:
        # SubAgent — 声明式规格，需要构建完整中间件栈
        subagent_model = resolve_model(spec.get("model", model))
        subagent_profile = _harness_profile_for_model(subagent_model, ...)
        # 构建子代理的中间件栈：Todo + Filesystem + Summarization + Patch + Skills + ...
        # 继承父代理的 tools、permissions、interrupt_on
```

**子代理继承规则**：
| 属性 | 继承行为 |
|------|---------|
| `tools` | 若子代理未声明，继承父代理工具 |
| `permissions` | 若子代理未声明，继承父代理权限规则 |
| `interrupt_on` | 若子代理未声明，继承父代理 HITL 配置 |
| `model` | 默认继承父代理模型，可独立覆盖 |
| `skills` | 独立配置，不继承 |

#### Step 4: 自动添加通用子代理

```python
# graph.py 第618-669行
gp_profile = _profile.general_purpose_subagent or GeneralPurposeSubagentProfile()
if gp_profile.enabled is not False and \
   not any(spec["name"] == "general-purpose" for spec in inline_subagents):
    # 自动创建默认的 general-purpose 子代理
    general_purpose_spec = {
        **GENERAL_PURPOSE_SUBAGENT,
        "model": model,
        "tools": _tools or [],
        "middleware": gp_middleware,
    }
```

如果用户没有提供名为 `general-purpose` 的子代理，且 harness profile 未禁用，DeepAgents **自动创建一个通用的子代理**。这个默认子代理：
- 拥有完整的文件系统访问权限
- 可以访问父代理的所有工具
- 拥有与父代理相同的模型和系统提示
- 可以通过 `GeneralPurposeSubagentProfile(enabled=False)` 禁用

#### Step 5: 组装主代理中间件栈

```python
# graph.py 第672-730行
deepagent_middleware = [
    TodoListMiddleware(),                               # 1. 任务规划
]
if skills is not None:
    deepagent_middleware.append(SkillsMiddleware(...))  # 2. 技能加载（条件）
deepagent_middleware.extend([
    FilesystemMiddleware(backend=backend, ...),         # 3. 文件系统（必需）
    SubAgentMiddleware(backend=backend, ...),           # 4. 子代理（条件）
    create_summarization_middleware(model, backend),    # 5. 自动摘要
    PatchToolCallsMiddleware(),                         # 6. 工具调用修复
])
if async_subagents:
    deepagent_middleware.append(AsyncSubAgentMiddleware(...))  # 7. 异步子代理
# 用户自定义中间件插入点
deepagent_middleware.extend(middleware)                 # 8. 用户中间件
# Harness Profile 额外中间件
deepagent_middleware.extend(_profile.materialize_extra_middleware())
# 工具排除中间件（如果有 excluded_tools）
deepagent_middleware.append(_ToolExclusionMiddleware(...))
# Anthropic 提示缓存（无条件）
deepagent_middleware.append(AnthropicPromptCachingMiddleware(...))
# 记忆中间件（条件）
if memory is not None:
    deepagent_middleware.append(MemoryMiddleware(...))
# 人工介入（条件）
if interrupt_on is not None:
    deepagent_middleware.append(HumanInTheLoopMiddleware(...))
```

**中间件栈顺序的精妙设计**：

```
请求流向 →
┌─────────────────────────────────────────────────────────────┐
│ 1. TodoListMiddleware        (任务规划，最早介入)            │
│ 2. SkillsMiddleware          (技能注入系统提示)              │
│ 3. FilesystemMiddleware      (文件工具注入，安全权限检查)     │
│ 4. SubAgentMiddleware        (task 工具注入)                 │
│ 5. SummarizationMiddleware   (上下文压缩)                    │
│ 6. PatchToolCallsMiddleware  (修复工具调用格式)              │
│ 7. [用户自定义中间件]          (在此插入)                     │
│ 8. [Harness Profile 额外中间件]                             │
│ 9. _ToolExclusionMiddleware  (过滤 excluded_tools)          │
│ 10. AnthropicPromptCachingMiddleware (提示缓存断点)          │
│ 11. MemoryMiddleware          (记忆注入系统提示)             │
│ 12. HumanInTheLoopMiddleware  (人工审批，最后执行)            │
└─────────────────────────────────────────────────────────────┘
← 响应流向
```

**顺序的关键原因**：
- **提示缓存放在 Memory 之前**：因为 MemoryMiddleware 会修改系统提示（注入记忆内容），如果在缓存之后修改，会导致缓存失效
- **HITL 放在最外层**：确保所有工具调用都经过人工审批后再真正执行
- **Summarization 放在 Patch 之前**：先压缩历史，再修复工具调用格式

#### Step 6: 组装系统提示

```python
# graph.py 第749-755行
base_prompt = _apply_profile_prompt(_profile, BASE_AGENT_PROMPT)
if system_prompt is None:
    final_system_prompt = base_prompt
elif isinstance(system_prompt, SystemMessage):
    final_system_prompt = SystemMessage(
        content_blocks=[...system_prompt.content_blocks, 
                       {"type": "text", "text": f"\n\n{base_prompt}"}]
    )
else:
    final_system_prompt = system_prompt + "\n\n" + base_prompt
```

**提示组装顺序**：`USER` → (`BASE` 或 `CUSTOM`) → `SUFFIX`

| 部分 | 来源 | 作用 |
|------|------|------|
| `USER` | `system_prompt=` 参数 | 用户自定义指令，优先级最高 |
| `BASE` | `BASE_AGENT_PROMPT` 常量 | SDK 默认的 Deep Agent 行为指南 |
| `CUSTOM` | `HarnessProfile.base_system_prompt` | 覆盖 BASE，用于模型调优 |
| `SUFFIX` | `HarnessProfile.system_prompt_suffix` | 模型特定调优，放在最靠近历史的位置 |

**为什么 USER 在前、SUFFIX 在后？**
- `USER` 在前：用户的自定义指令优先于 SDK 默认行为
- `SUFFIX` 在后：模型对系统提示末尾的内容关注度最高（attention 机制），所以模型调优指令放最后

#### Step 7: 创建最终 Agent

```python
# graph.py 第765-788行
return create_agent(
    model,
    system_prompt=final_system_prompt,
    tools=_tools,
    middleware=deepagent_middleware,
    ...
    state_schema=_DeepAgentState,  # 自定义状态模式
    transformers=[_subagent_factory],
).with_config({
    "recursion_limit": 9_999,  # 超高递归限制
    "metadata": {
        "ls_integration": "deepagents",
        "versions": {"deepagents": __version__},
    },
})
```

**`_DeepAgentState`** 是关键的优化设计：

```python
class _DeepAgentState(AgentState):
    messages: Required[Annotated[
        list[AnyMessage], 
        DeltaChannel(_messages_delta_reducer, snapshot_frequency=50)
    ]]
```

使用 `DeltaChannel` 替代普通列表，将 checkpoint 增长从 **O(N²) 降到 O(N)**。在长对话中，普通列表每次 checkpoint 都要保存完整消息历史，而 DeltaChannel 只保存增量变化。

### 14.2 中间件为什么不是普通工具？

```
普通工具（tools=[]）                          中间件（AgentMiddleware）
┌──────────────────────┐                    ┌────────────────────────────┐
│ 只在被 LLM 调用时执行  │                    │ 拦截每次 LLM 请求前后        │
│ 不能修改系统提示      │                    │ 可以注入/修改系统提示        │
│ 不能过滤工具列表      │                    │ 可以动态增删工具             │
│ 无跨轮次状态         │                    │ 可读写持久化状态             │
│ 简单函数即可          │                    │ 需要继承 AgentMiddleware     │
└──────────────────────┘                    └────────────────────────────┘
```

中间件通过重写 `wrap_model_call()` 钩子来拦截每个 LLM 请求：

```python
class AgentMiddleware(ABC):
    def wrap_model_call(self, model_call):
        """在 model_call 前后执行自定义逻辑"""
        def wrapped(state, tools, config):
            # 1. 前置处理：修改 tools、注入系统提示
            modified_tools = self.before_call(state, tools)
            # 2. 调用底层模型
            response = model_call(state, modified_tools, config)
            # 3. 后置处理：修改响应、更新状态
            return self.after_call(response, state)
        return wrapped
```

def _check_fs_permission(rules, operation, path) -> "allow" | "deny":
    for rule in rules:
        if operation not in rule.operations:
            continue
        if any(wcglob.globmatch(path, pattern) for pattern in rule.paths):

    return "allow"  # 默认允许

### 14.7 自定义状态模式：DeltaChannel 优化

DeepAgents 使用自定义的 `_DeepAgentState` 来优化 checkpoint 性能：

```python
from langgraph.channels.delta import DeltaChannel

class _DeepAgentState(AgentState):
    messages: Required[Annotated[
        list[AnyMessage],
        DeltaChannel(_messages_delta_reducer, snapshot_frequency=50)
    ]]
```

**普通 Channel 的问题**：
- 普通 `list` Channel 每次 checkpoint 保存完整消息列表
- N 轮对话 = N 次 checkpoint = 1 + 2 + 3 + ... + N = O(N²) 的总存储量

**DeltaChannel 的优化**：
- 只保存自上次 checkpoint 以来的**增量变化**
- N 轮对话 = O(N) 的总存储量
- `snapshot_frequency=50`：每 50 步强制做一次完整快照，防止增量链过长

```python
def _messages_delta_reducer(existing, update):
    """合并消息增量更新"""
    if existing is None:
        return update
    return existing + update
```

## 15. 沙箱系统与代码执行

DeepAgents 的核心能力之一是**在隔离环境中执行代码**。本节深入分析沙箱系统的架构和实现。

### 15.1 Backend 协议层次

DeepAgents 的后端系统采用**协议分层**设计：

```
┌──────────────────────────────────────────────────────┐
│        SandboxBackendProtocol                        │
│  (BackendProtocol + execute() + id 属性)             │
├──────────────────────────────────────────────────────┤
│        BackendProtocol                               │
│  ls() / read() / write() / edit()                    │
│  grep() / glob() / upload_files() / download_files() │
└──────────────────────────────────────────────────────┘
```

**`SandboxBackendProtocol`** 在 `BackendProtocol` 基础上增加了：
- `execute(command, *, timeout) -> ExecuteResponse`：执行 shell 命令
- `id` 属性：沙箱实例唯一标识
- `aexecute()`：异步版本

### 15.2 BaseSandbox：沙箱的通用实现

`BaseSandbox`（`deepagents/backends/sandbox.py`）是所有沙箱后端的**抽象基类**。它的设计哲学是：**子类只需要实现 `execute()` 和文件传输，其他操作都通过服务器端脚本完成**。

#### 核心抽象方法

```python
class BaseSandbox(SandboxBackendProtocol, ABC):
    @abstractmethod
    def execute(self, command: str, *, timeout: int | None = None) -> ExecuteResponse:
        """在沙箱中执行 shell 命令"""

    @abstractmethod
    def upload_files(self, files: list[tuple[str, bytes]]) -> list[FileUploadResponse]:
        """上传文件到沙箱"""

    @abstractmethod
    def download_files(self, paths: list[str]) -> list[FileDownloadResponse]:
        """从沙箱下载文件"""

    @property
    @abstractmethod
    def id(self) -> str:
        """沙箱唯一标识"""
```

#### 派生操作的实现策略

BaseSandbox 的所有文件操作都是通过**服务器端 Python 脚本**完成的：

| 操作 | 实现方式 | 关键设计 |
|------|---------|---------|
| `ls()` | 执行 Python 脚本调用 `os.scandir()` | 输出 JSON Lines 格式 |
| `read()` | 执行 Python 脚本做分页读取 | 仅返回请求的页面，避免全文件传输 |
| `write()` | `upload_files()` + 预检查脚本 | 先检查文件不存在，再上传 |
| `edit()` | 执行 Python 脚本做字符串替换 | 小payload内联，大payload临时文件 |
| `grep()` | 执行 `grep -rHnF` | `-F` 固定字符串模式（非正则） |
| `glob()` | 执行 Python 脚本调用 `glob.glob()` | base64 编码参数避免 shell 注入 |

#### read() 的服务器端分页

```python
_READ_COMMAND_TEMPLATE = """python3 -c "
import codecs, os, stat as _stat, sys, base64, json

MAX_OUTPUT_BYTES = 500 * 1024  # 500KB 输出上限
MAX_BINARY_BYTES = 500 * 1024  # 500KB 二进制上限

path = base64.b64decode('{path_b64}').decode('utf-8')

# 1. 检查文件类型和大小
# 2. 检测二进制文件（非 UTF-8）→ base64 编码
# 3. 文本文件 → 按 offset/limit 分页读取
# 4. 输出超过 MAX_OUTPUT_BYTES 时截断并提示
" 2>&1"""
```

**为什么分页在服务器端做？**
- 避免传输整个大文件，只传输请求的片段
- 减少网络带宽消耗
- 让 LLM 可以通过多次 `read_file` 调用来读取大文件的不同部分

#### edit() 的双路径实现

```python
def edit(self, file_path, old_string, new_string, replace_all=False):
    payload_size = len(old_string.encode()) + len(new_string.encode())

    if payload_size <= 50_000:  # 50KB 以内
        return self._edit_inline(file_path, old_string, new_string, replace_all)
    else:
        return self._edit_via_upload(file_path, old_string, new_string, replace_all)
```

**内联编辑**（小 payload）：
- 将 path/old/new/replace_all 打包为 base64 JSON
- 通过 heredoc 传递给服务器端 Python 脚本
- 单次 `execute()` 调用完成

**上传编辑**（大 payload）：
- 将 old_string 和 new_string 上传为临时文件
- 服务器端脚本读取临时文件，在源文件上做替换
- **源文件始终不离开沙箱**，只传输新旧字符串
- 完成后自动清理临时文件

**CRLF 智能处理**：
- LLM 收到的内容是 LF 格式（read() 做了规范化）
- 但磁盘文件可能是 CRLF 格式
- 编辑脚本尝试三种匹配：原样、CRLF 变体、LF 变体
- 保留文件原有的换行风格

#### ModalSandbox 实现细节

```python
class ModalSandbox(BaseSandbox):
    def __init__(self, *, sandbox: modal.Sandbox):
        self._sandbox = sandbox
        self._default_timeout = 30 * 60  # 30分钟默认超时

    def execute(self, command, *, timeout=None):
        effective_timeout = timeout or self._default_timeout
        process = self._sandbox.exec("bash", "-c", command, timeout=effective_timeout)
        process.wait()
        return ExecuteResponse(
            output=process.stdout.read() + "\n" + process.stderr.read(),
            exit_code=process.returncode,
        )

    def upload_files(self, files):
        return [self._write_file(path, content) for path, content in files]

    def _write_file(self, path, content):
        f = self._sandbox.open(path, "wb")
        try:
            f.write(content)
        finally:
            f.close()
        return FileUploadResponse(path=path, error=None)
```

class DaytonaSandbox(BaseSandbox):
    def execute(self, command, *, timeout=None):
        effective_timeout = timeout or self._default_timeout
        session_id = str(uuid4())
        self._sandbox.process.create_session(session_id)
        try:
            result = self._sandbox.process.execute_session_command(
                session_id,
                SessionExecuteRequest(command=command, run_async=True),
            )
            # 轮询等待完成
            while True:
                command_result = self._sandbox.process.get_session_command(
                    session_id, result.cmd_id
                )
                if command_result.exit_code is not None:
                    break
                time.sleep(self._sync_polling_interval)
            logs = self._sandbox.process.get_session_command_logs(
                session_id, result.cmd_id
            )
        finally:
            self._sandbox.process.delete_session(session_id)

        return ExecuteResponse(
            output=logs.stdout + "\n<stderr>" + logs.stderr + "</stderr>",
            exit_code=command_result.exit_code,
        )

### 15.8 自定义沙箱后端

你可以通过继承 `BaseSandbox` 创建自定义沙箱：

```python
from deepagents.backends.sandbox import BaseSandbox
from deepagents.backends.protocol import ExecuteResponse, FileUploadResponse, FileDownloadResponse

class DockerSandbox(BaseSandbox):
    """基于 Docker 容器的沙箱后端"""

    def __init__(self, container_name: str):
        self._container = container_name
        self._default_timeout = 300

    @property
    def id(self) -> str:
        return self._container

    def execute(self, command: str, *, timeout: int | None = None) -> ExecuteResponse:
        import subprocess
        effective = timeout or self._default_timeout
        result = subprocess.run(
            ["docker", "exec", self._container, "sh", "-c", command],
            capture_output=True, text=True, timeout=effective
        )
        return ExecuteResponse(
            output=result.stdout + ("\n" + result.stderr if result.stderr else ""),
            exit_code=result.returncode,
        )

    def upload_files(self, files: list[tuple[str, bytes]]) -> list[FileUploadResponse]:
        import tempfile, os
        responses = []
        for path, content in files:
            with tempfile.NamedTemporaryFile(delete=False) as tmp:
                tmp.write(content)
                tmp.flush()
                subprocess.run(
                    ["docker", "cp", tmp.name, f"{self._container}:{path}"],
                    check=True
                )
                os.unlink(tmp.name)
            responses.append(FileUploadResponse(path=path, error=None))
        return responses

    def download_files(self, paths: list[str]) -> list[FileDownloadResponse]:
        import tempfile, subprocess
        responses = []
        for path in paths:
            with tempfile.NamedTemporaryFile(delete=False) as tmp:
                result = subprocess.run(
                    ["docker", "cp", f"{self._container}:{path}", tmp.name],
                    capture_output=True
                )
                if result.returncode == 0:
                    with open(tmp.name, "rb") as f:
                        content = f.read()
                    responses.append(FileDownloadResponse(path=path, content=content, error=None))
                else:
                    responses.append(FileDownloadResponse(path=path, content=None, error="file_not_found"))
                os.unlink(tmp.name)
        return responses

# 使用
backend = DockerSandbox(container_name="my-sandbox")
agent = create_deep_agent(
    model="anthropic:claude-sonnet-4-20250514",
    backend=backend,
)
```

**自定义沙箱的关键点**：
1. 必须实现 `execute()`、`upload_files()`、`download_files()` 和 `id` 属性
2. 继承 `BaseSandbox` 后自动获得 `ls()`、`read()`、`edit()`、`grep()`、`glob()` 的实现
3. 文件操作通过服务器端脚本完成，无需自己实现
4. 注意处理异常，返回 `FileUploadResponse`/`FileDownloadResponse` 中的 `error` 字段

uvx deepagents-cli --sandbox runloop --sandbox-setup ./setup.sh

# Daytona

### 16.1 核心知识点回顾

1. **Planning**：`write_todos` / `read_todos`，Agent 自主任务分解

6. **Middleware**：10 层默认栈，可扩展可排除，洋葱模型执行
7. **源码**：`create_deep_agent()` 7 步组装，DeltaChannel O(N) checkpoint，服务器端脚本沙箱
8. **沙箱**：Modal（GPU）、Daytona（TTL 清理）、Runloop（快速启动）、LocalShell（本地开发）
