02 编译时配置
interrupt_before vs interrupt_after

interrupt_before —— 执行前拦截

节点代码完全不执行，resume 后第一次执行

graph = builder.compile(
    checkpointer=InMemorySaver(),
    interrupt_before=["publish"]
)
# 审批员先看输入，决定是否让 AI 处理

interrupt_after —— 执行后审查

节点已执行完，writes 已保存，再中断

graph = builder.compile(
    checkpointer=InMemorySaver(),
    interrupt_after=["generate"]
)
# AI 生成内容后让人类确认
维度interrupt_beforeinterrupt_after
节点执行了吗否是
resume 后节点执行吗是（第一次）否（已执行过）
state 有输出吗否是
适用场景审批输入/前置确认审查输出/后置确认

03 interrupt() 陷阱
重执行陷阱与拆分节点模式
interrupt() 恢复时，节点从头重新执行！interrupt() 前的代码会重跑。

错误写法（计算 + 中断混在一起）

def bad_node(state):
    result = expensive_compute(state)  # resume 时重跑！
    user_input = interrupt({"data": result})
    return {"output": process(user_input)}

正确写法（拆分节点）

def compute_node(state):
    result = expensive_compute(state)
    return {"analysis_result": result}  # 写入 state

def human_confirm_node(state):
    result = state["analysis_result"]  # 从 state 读取
    decision = interrupt({"content": result})
    return {"confirmed": decision == "approve"}

拆分原则：计算节点不抛 interrupt；中断节点只读 state，不执行副作用。

04 预构建 Agent
create_react_agent 的 HITL 配置
agent = create_react_agent(
    "openai:gpt-4",
    tools=[check_weather, send_email],
    interrupt_before=["tools"],   # 工具执行前确认（防止误操作）
    interrupt_after=["agent"],    # LLM 输出后审查（质量控制）
    checkpointer=InMemorySaver(),
)

工具调用前审批

LLM 决定调用工具后，先暂停等用户确认。用户看到 "AI 要执行 send_email，确认吗？"

LLM 输出后审查

LLM 每次输出后暂停，用户可审查/修改。如果没有 tool_call，直接审查答案。

注意：create_react_agent 的 interrupt_before/after 直接透传给 workflow.compile()，节点名固定为 "agent" 和 "tools"。

05 生产级封装
ToolInteractionEnvelope 结构化通信
工具不直接中断，返回信封让上层决定。信封包含状态、渲染契约、动作请求。
class ToolInteractionEnvelope(TypedDict):
    version: str              # 协议版本
    status: str               # ok / needs_input / needs_confirmation / error
    domain: str               # 业务域
    summary: str              # 给 LLM 看的摘要
    data: dict                # 结构化数据
    render_schema: dict       # 前端渲染契约
    action_request: dict      # 动作请求（risk / requires_confirmation）

RenderSchema 前端契约

标准字段类型：static_text、single_select_card、checkbox、text_input、date_input、time_input、file_upload

Resume 协议

Operation: submit / continue / revise / regenerate / retry
Decision: confirm -> accept, cancel -> reject, edit -> edit + args

05 生产级封装
双模式工具包装器

Copilot 模式（自动执行）

适合只读查询。auto=True 时工具直接执行，无需确认。

# web_search_tool、calculator_tool
# 搜索、计算——自动执行

Interactive 模式（需要确认）

适合写操作。auto=False 时工具返回信封，触发中断。

# library_booking_tool、send_email_tool
# 订位、发邮件——需要确认

运行时上下文注入

writer 通过 ContextVar 注入，不暴露在工具的 args_schema 中。LLM 看不到 writer 参数，但工具内部可安全使用。

06 常见误区
HITL 最容易踩的坑
坑现象正确做法
忘记 checkpointerinterrupt 后无法恢复HITL 必须配 checkpointer！
interrupt 和异常混用Exception 终止执行HITL 必须用 interrupt() 原语
在工具里用 interrupt()工具异常，图不暂停用 interrupt_before=["tools"]
resume 值格式不对被中断节点收不到值Command(resume=...) 的值就是 interrupt() 返回值
Hooks 里修改 StatePre-hook 改了影响后续用 llm_input_messages 临时注入
审批流设计太复杂中断 5 次，体验差HITL 是安全阀，不是每一步都问
多层中断冲突before 和 in-node 同时触发设计时确保逻辑不重叠

调试技巧：graph.get_state(config) 看 .next 字段；snapshot.tasks 看 interrupts 字段；节点内加 print 检测重执行。

07 阶段小结
核心要点回顾

三层架构

编译时 -> 预构建 Agent -> 节点内动态，粒度递增，风险递增

interrupt_before

节点不执行，resume 后第一次执行

interrupt_after

节点已执行完，resume 直接流转

interrupt() 陷阱

恢复时从头重新执行，必须拆分节点

生产级封装

信封标准化 + RenderSchema 驱动 UI + Resume 统一协议

双模式包装

Copilot 自动执行 / Interactive 人工确认

记忆口诀

compile 配 before/after，不动代码就生效；
节点内写 interrupt，千万要拆分防重跑；
Agent 工具要审批，tools 节点暂停好；
resume 是 playback，state 已在 checkpoint 存好。
