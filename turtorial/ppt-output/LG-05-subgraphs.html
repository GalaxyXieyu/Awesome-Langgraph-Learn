核心内容
Fan-out / Fan-in
一个节点同时触发多个下游节点（Fan-out），多个节点完成后汇聚到一个节点（Fan-in）。

Fan-out：并行分发

parse_intent 节点有 4 条出边，分别指向 4 个数据源节点。LangGraph 自动并行执行。

parent_builder.add_edge("parse", "finance")
parent_builder.add_edge("parse", "news")
parent_builder.add_edge("parse", "competitor")
parent_builder.add_edge("parse", "expert")

Fan-in：自动汇聚

4 个节点都完成后，compile_report 才会执行。LangGraph 自动等待所有前置节点。

# 4 条边都指向 collect
parent_builder.add_edge("finance", "collect")
parent_builder.add_edge("news", "collect")
parent_builder.add_edge("competitor", "collect")
parent_builder.add_edge("expert", "collect")
Fan-out 像项目经理同时发 4 封邮件；Fan-in 像等 4 个人都回复后才变绿

核心内容
Reducer：并行的和平协议
多个并行节点同时修改同一个 State 字段时，没有 Reducer 就会写覆盖。

内置 Reducer

from operator import add

class AgentState(TypedDict):
    # 多个节点往 findings 加内容，自动合并
    findings: Annotated[list, add]

operator.add：列表/数值累加

自定义 Reducer

class AgentState(TypedDict):
    # 保留最近的 10 条日志
    logs: Annotated[list,
        lambda x, y: (x + y)[-10:]]

    # 去重合并
    tags: Annotated[set,
        lambda x, y: x | y]
关键：并行节点共享的字段必须用 Annotated + Reducer，否则是竞态条件

核心内容
Send 动态 Map-Reduce
运行时决定分发多少子任务，并行执行后自动 Fan-in 汇聚。
from langgraph.types import Send

def dispatch(state: ParentState):
    """Map 阶段：动态分发每个公司到子图"""
    return [
        Send("analyze_company", {"company": c})
        for c in state["companies"]
    ]

# 条件边：dispatch 返回 Send 列表，动态分发
parent_builder.add_conditional_edges(
    START, dispatch, ["analyze_company"]
)

Send 参数

Send("target_node", {"key": value})
第二个参数是子图的初始 State

状态隔离

子图从 Send 参数开始，独立于父图执行

自动汇聚

所有子图实例完成后，LangGraph 自动触发 Fan-in

核心内容
Subgraph：模块化复用
子图是独立的 StateGraph，有自己的 State Schema，可单独测试、可嵌套复用。

定义子图

child_builder = StateGraph(ChildState)
child_builder.add_node("fetch", fetch_data)
child_builder.add_node("summarize", summarize)
child_builder.add_edge(START, "fetch")
child_builder.add_edge("fetch", "summarize")
child_graph = child_builder.compile()

嵌入父图

# 关键：编译后的子图作为节点
parent_builder.add_node("research", child_graph)

# input/output 映射（字段名不同时）
parent_builder.add_node(
    "research", child_graph,
    input=lambda s: {"query": s["topic"]},
    output=lambda s: {"findings": s["results"]},
)
核心：子图必须先 .compile()，然后作为 action 传给父图 add_node。子图默认继承父图 checkpointer。

核心内容
状态隔离与通信
子图 State 与父图默认隔离，有三种跨图通信方式。
方式适用场景代码示例
公共字段父子图有相同字段名messages 在父子图中同名，自动透传
input/output 映射字段名不同，需要格式转换input=lambda s: {"query": s["topic"]}
Store（长期记忆）多个子图需要共享数据store.aput(("shared", uid), "key", data)

隔离的好处

  • 封装性：中间状态不污染父图
  • 可复用性：同一子图嵌入多个父图
  • 可测试性：独立运行和调试

最佳实践

  • 先独立测试子图
  • 再嵌入父图，逐步集成
  • 优先用公共字段，减少映射
  • 子图 State 最小化

常见误区
避坑指南
坑现象预警
并行节点修改非 Reducer 字段数据互相覆盖并行节点共享字段必须用 Annotated + Reducer
Fan-in 前置有循环Fan-in 永远等不到确保循环有明确终止条件
子图 State 与父图不兼容字段映射错误，数据传不过去input/output 映射必须显式配置
Send 分发太多子任务内存/连接池耗尽做上限控制，max_concurrency 限制并发
递归限制不当并行 + 循环导致超级步过多生产环境合理设置 recursion_limit
调试技巧：先不并行，串行跑通每个节点，再改成并行；Reducer 验证打印合并前后 State；子图单独测试再嵌入。

阶段小结
核心要点回顾

并行机制

  • Fan-out：一个节点触发多个下游同时执行
  • Fan-in：多个节点完成后才触发下一个
  • Reducer：定义并行字段合并策略
  • Send：运行时动态分发子任务

子图机制

  • Subgraph：独立 State、独立逻辑、可复用
  • input/output 映射：显式配置字段传递
  • 公共字段：同名自动透传
  • Store：跨子图共享数据
"Fan-out 是群发，Fan-in 是等齐人；Reducer 是合并器，Send 是动态派活，Subgraph 是模块包。"
