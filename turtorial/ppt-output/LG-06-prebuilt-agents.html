LangGraph 渐进式教程3 / 10
核心内容
循环终止的判据：tool_calls 是否为空

should_continue 路由逻辑

def should_continue(state):
    last = state["messages"][-1]
    if not isinstance(last, AIMessage) \
       or not last.tool_calls:
        return END      # 循环终止
    return "tools"    # 继续执行工具

不是 LLM "说" 要结束，而是程序检查输出格式

步数限制：防止无限循环

class AgentState(TypedDict):
    messages: Annotated[Sequence, add_messages]
    is_last_step: IsLastStep
    remaining_steps: RemainingSteps

remaining_steps < 2 且还有 tool_calls → 强制替换为 "Sorry, need more steps..."

agent (LLM)
→
should_continue
→
tools / END
→
tools → agent (循环)
LangGraph 渐进式教程4 / 10
核心内容
预构建 Agent 的局限
局限说明影响
State 不可扩展只有 messages 一个字段无法存业务状态
路由不可定制固定 LLM → Tool → LLM 循环无法按业务状态分支
中间过程不可见invoke() 只给最终结果无法监控每步在做什么
循环控制受限由内部逻辑决定循环次数无法自定义重试策略
HITL 支持弱没有原生断点能力无法人工审批中间步骤

传 System Prompt

agent = create_react_agent(
    llm, tools,
    state_modifier=SystemMessage("你是专业天气助手")
)

pre_model_hook：消息裁剪

def trimming_hook(state):
    msgs = state["messages"]
    if len(msgs) > 20:
        to_remove = msgs[:-10]
        return {"messages": [RemoveMessage(id=m.id) for m in to_remove]}
    return {}

三种预构建多 Agent 模式
模式核心函数来源包适用场景
单 Agentcreate_react_agentlanggraph一个 LLM + 多个工具
监督者create_supervisorlanggraph-supervisor一个主管调度多个专员
群体协作create_swarmlanggraph-swarm多个专员平级交接

create_supervisor：层级调度

Supervisor 本身就是 create_react_agent，tools 里包含所有 sub-agent 的 handoff_tools。handoff tool 返回 Command(goto=agent_name) 跳转。

workflow = create_supervisor(
    [weather_agent, order_agent],
    model=llm,
    prompt="你是调度主管"
)

create_swarm：平级交接

没有中心调度器。State 多了 active_agent 字段。每次从 START 重新进入，根据 active_agent 路由。handoff 时更新 active_agent 状态。

workflow = create_swarm(
    [alice, bob],
    default_active_agent="Alice"
)

核心内容
什么时候用预构建，什么时候自定义？
维度预构建 Agent自定义 StateGraph
开发速度快（3 行代码）中等
可控性黑盒白盒，每步可控
可观测性只能看输入输出每节点状态可见
复杂路由不支持任意图结构
HITL 支持弱强
自定义 State只有 messages任意字段

决策树

  • 标准 ReAct 循环 + 不需要监控 + 不需要自定义 State → create_react_agent
  • 需要监控每一步 State / 需要复杂路由 / 需要人工审批 / 需要自定义 State → 自定义 StateGraph
LangGraph 渐进式教程7 / 10
核心内容
防止 Agent 撑爆上下文窗口

策略 1：Trimming

裁剪历史，只保留最近 N 条或 N 个 token

trimmed = trim_messages(
    messages,
    max_tokens=4000,
    strategy="last",
    token_counter=llm
)

难度：低 信息损失：中

策略 2：Summarization

当消息太长时，总结早期内容

if count_tokens(msgs) > 6000:
    to_sum = msgs[:-4]
    recent = msgs[-4:]
    summary = llm.invoke([...])
    return {"messages": [summary]+recent}

难度：中 信息损失：低

策略 3：Tool 精简

在 tool 定义阶段就控制输出长度

def query_weather(city):
    raw = query_api(city)
    return f"{city}：{raw['weather'][0]['description']}，{raw['main']['temp']}°C"

难度：低 信息损失：可控

LangGraph 渐进式教程8 / 10
实战对比
预构建 vs 自定义：同一个运维助手

版本 A：create_react_agent（3 行）

tools = [check_cpu, check_memory,
         check_disk, restart_service,
         query_logs]
agent = create_react_agent(llm, tools)
  • 简单查询：查 CPU → 直接调用，3 行搞定
  • 敏感操作：重启服务 → 直接执行，无阻拦
  • 长日志：1000 条 → 直接塞进 history
  • 过程可见性 → 黑盒

版本 B：自定义 StateGraph（完整可控）

class OpsState(TypedDict):
    messages: Annotated[list, add_messages]
    user_role: str
    pending_approval: bool
    execution_log: list
  • 简单查询：查 CPU → 同样正确
  • 敏感操作：检查 user_role → 普通用户需审批
  • 长日志：截断返回前 50 条 + 统计
  • 过程可见性 → stream() 逐节点输出

阶段小结
核心要点回顾

create_react_agent 核心认知

  • LangGraph 1.x 唯一官方推荐的预构建 Agent
  • 内部是 LLM → Tool → LLM 自动循环
  • State 只有 messages，不可扩展
  • 循环终止判据：AIMessage.tool_calls 为空
  • 有 remaining_steps / is_last_step 安全机制

选型与生态

  • 标准 ReAct + 不需要监控 → 预构建
  • 复杂路由 / HITL / 自定义 State → 自定义
  • 多 Agent：supervisor（层级）、swarm（平级）
  • 上下文管理：Trimming / Summarization / Tool 精简
  • 先用预构建验证，再上 StateGraph 生产
