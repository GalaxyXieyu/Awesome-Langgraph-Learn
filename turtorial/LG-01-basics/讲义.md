## 1. 开场引入

LangChain 的 Chain 写法 `prompt | llm | parser` 很优雅，但业务逻辑一复杂就开始失控。需要根据 LLM 输出决定下一步走哪条分支、某个步骤失败后重试、用户中途打断修改需求 —— Chain 就变成了意大利面条，到处都是 `RunnableLambda` 和条件判断。

LangGraph 的思路很简单：把工作流从"一条直线"变成"一张图"。

先展示 WeatherBot 的最终效果：输入"北京今天天气怎么样"，图先判断意图，再走天气查询分支，节点逐个高亮执行，输出结果。如果用户问的是时间，图会走另一条分支。今天这节课结束，你们就能写出这个带判断的 WeatherBot。

---

## 2. 总览：学 LangGraph 到底要掌握什么？

一张图讲清楚五个核心概念的关系：

```
State（状态）= 贯穿整个图的共享数据
│
├── Node（节点）= 处理函数，接收 State，返回更新
│ │
│ └── 可以是任何 Python 函数、LangChain Runnable、甚至另一个图
│
├── Edge（边）= 节点之间的普通连接，A 做完必定到 B
│
└── Conditional Edge（条件边）= 根据 State 决定下一步走哪条分支
│
└── 由 Router 函数控制：返回字符串，对应目标节点的名字

compile() = 把上面这些东西编译成可执行的图对象

invoke() / stream() = 运行图，前者直接给结果，后者逐个节点输出中间状态
```

用一句话概括：

> **State 是数据，Node 是处理，Edge 是路由，compile 是生成，invoke/stream 是运行。**

今天的目标：搞懂这五个东西怎么配合，然后写出带路由判断的 WeatherBot。

## 3. 逐一拆解

### 3.1 State —— 图里的共享数据

State 是一个 TypedDict，定义了图中所有节点共享的数据结构。

```python
from typing import TypedDict, Annotated
from operator import add
from langchain_core.messages import HumanMessage, AIMessage

class WeatherState(TypedDict):
messages: Annotated[list, add] # 对话历史，用 add 累加
intent: str # 用户意图：weather / time / other
city: str # 用户提到的城市
weather_data: dict # API 返回的原始数据
response: str # 最终回复内容
```

关键点：

- 每个节点接收**完整 State**。Node A 可以看到 Node B 产生的数据，反之亦然。
- 节点返回**增量更新**。`return {"city": "北京"}` 只会更新 `city` 字段，其他字段保留原值。
- `Annotated[field, reducer]` 决定字段怎么合并。`add` 表示追加，`lambda x,y: y` 表示覆盖。不写的字段默认覆盖。

**常见坑**：节点内直接 `state["city"] = "北京"` 不会生效！必须通过返回值让框架合并。

---

#### 深入：Annotated 与 Reducer 到底是怎么回事？

`Annotated[type, reducer]` 是 Python 的类型注解语法，LangGraph 利用它给每个字段附加一条**合并规则**。节点返回增量时，框架会问这个字段："新值来了，怎么跟旧值合？"

**默认行为（不写 Annotated）= last-write-wins**。节点返回 `{"intent": "weather"}`，直接覆盖旧值。这适用于大多数单一值字段。

**列表/消息字段必须加 Reducer**，否则每次都被覆盖。LangGraph 内置了两个最常用 reducer：

| Reducer | 来源 | 作用 | 适用场景 |
|---------|------|------|---------|
| `operator.add` | Python 标准库 | 直接拼接两个列表 | 普通列表累加，如日志、标签 |
| `add_messages` | `langgraph.graph.message` | 拼接消息 + 按 ID 去重 + 支持 RemoveMessage | 对话历史，AI 回复、工具调用循环 |

**`operator.add` 示例** —— 普通列表追加：

```python
from typing import Annotated, TypedDict
import operator

class AgentState(TypedDict):
tags: Annotated[list[str], operator.add] # 节点返回新标签会自动追加
query: str # 无 reducer，直接覆盖

# Node A 返回: {"tags": ["weather"]}
# Node B 返回: {"tags": ["beijing"]}
# 最终 tags: ["weather", "beijing"]
```

**`add_messages` 示例** —— 对话消息专用：

```python
from typing import Annotated, TypedDict
from langgraph.graph.message import add_messages
from langchain_core.messages import AnyMessage

class ChatState(TypedDict):
messages: Annotated[list[AnyMessage], add_messages]

# 核心优势：
# 1. 自动追加新消息
# 2. 如果消息 ID 已存在，会原地更新而不是重复追加（对 tool-calling 循环至关重要）
# 3. 支持 RemoveMessage(id="xxx") 删除指定消息
```

> **为什么 `messages` 推荐用 `add_messages` 而不是 `operator.add`？**
> 
> 在 tool-calling 场景里，LLM 可能多次生成响应并更新同一条消息。`add_messages` 通过 message ID 去重，避免对话历史里出现重复消息；`operator.add` 只会无脑追加，导致上下文膨胀。

**自定义 Reducer** —— 自己写合并逻辑：

Reducer 本质上是一个函数：`(old_value, new_value) -> merged_value`。必须满足三条规则：
1. **纯函数** —— 不能有副作用，不要原地修改输入
2. **处理初始空值** —— 首次调用时 old_value 可能是空列表/空字符串
3. **返回类型匹配** —— 返回值类型要跟字段声明一致

常见自定义模式：

```python
# 模式 1：滑动窗口，只保留最近 N 条
from typing import Annotated, TypedDict

def keep_last_n(n: int):
def _reducer(old: list, new: list) -> list:
return (old + new)[-n:]
return _reducer

class BoundedState(TypedDict):
history: Annotated[list, keep_last_n(10)] # 只保留最近 10 条

# 模式 2：字典浅合并
from typing import Annotated, TypedDict

def merge_dicts(old: dict, new: dict) -> dict:
return {**old, **new}

class MetaState(TypedDict):
metadata: Annotated[dict, merge_dicts]

# 模式 3：去重追加
from typing import Annotated, TypedDict

def unique_append(old: list, new: list) -> list:
seen = set(old)
return old + [x for x in new if x not in seen]

class TagState(TypedDict):
tags: Annotated[list[str], unique_append]
```

**什么时候必须用 Reducer？**

- 列表/消息字段要累加 → 必须用
- 多个节点**并行执行**且可能返回同个 key → 必须用，否则 LangGraph 会抛 `InvalidUpdateError`
- 单一值字段只需要覆盖 → 不用写 Annotated，默认行为就是对的

**Reducer 现场验证**：

问学员：如果 `messages` 不用 `Annotated[list, add]`，第二次调用会发生什么？
答：消息历史被覆盖，只保留一条。

### 3.2 Node —— 图里的处理单元

Node 就是一个函数：接收 `state`，返回一个 `dict`。

```python
def parse_intent(state: WeatherState):
"""解析用户意图，判断是问天气、问时间还是其他"""
text = state["messages"][-1].content
if "天气" in text or "温度" in text:
intent = "weather"
elif "时间" in text or "几点" in text:
intent = "time"
else:
intent = "other"
return {"intent": intent}

def extract_city(state: WeatherState):
"""从用户输入中提取城市"""
text = state["messages"][-1].content
city = "北京" if "北京" in text else "上海" # 简化版
return {"city": city}

def query_weather(state: WeatherState):
"""调用天气 API"""
    # 实际调用 weather API，这里 mock
return {"weather_data": {"temp": 25, "condition": "晴"}}

def get_time(state: WeatherState):
"""获取当前时间"""
from datetime import datetime
return {"response": f"现在是 {datetime.now().strftime('%H:%M')}"}

def format_weather_reply(state: WeatherState):
"""生成天气回复"""
data = state["weather_data"]
reply = f"{state['city']}今天{data['condition']}，{data['temp']}度"
return {"response": reply}

def fallback_reply(state: WeatherState):
"""兜底回复"""
return {"response": "抱歉，我暂时只能回答天气和时间相关的问题。"}
```

关键点：

- Node 可以是任何 Python 函数，不限于 AI 调用。数据库查询、API 调用、文件处理，直接放进去就是节点。
- 也可以是 LangChain Runnable、甚至另一个完整的图（后面讲 Subgraph）。
- 返回值必须是 dict。如果想返回字符串，包装成 `{"result": string}`。

```python
# LangGraph 版：30 行 → 40 行，只加了 10 行
def query_weather(state):
try:
data = query_weather_api(state["city"])
return {"weather_data": data, "retry_count": 0}
except APIError:
return {"retry_count": state.get("retry_count", 0) + 1}

def should_retry(state) -> str:
if state.get("retry_count", 0) >= 3:
return "fallback_reply"
return "query_weather" # 回到自己，重试！

builder.add_conditional_edges("query_weather", should_retry)
```

> 重试逻辑被抽成了一个独立的条件边，和业务节点完全解耦。而且所有节点都可以复用同一个重试模式——不需要复制粘贴 `for` 循环。

**核心差异**：纯函数的重试是"函数内部写循环"，重试代码和业务代码**缠在一起**。LangGraph 的重试是"图上画个圈"，重试逻辑是**独立的路由规则**。

**到底什么时候用纯函数，什么时候用 LangGraph？**

| 你的场景 | 用哪个 | 原因 |
|---------|--------|------|
| A→B→C 直线流程，不循环、不重试、不分支 | **纯函数** | 简单直接，不需要框架 overhead |
| 需要循环/重试（API 失败重试、迭代优化） | **LangGraph** | 图上画个圈，不用写嵌套循环 |
| 需要条件分支（根据 LLM 输出决定下一步） | **LangGraph** | 图画出来，分支逻辑一目了然 |
| 需要并行执行（同时调多个 API） | **LangGraph** | 声明式并行，不用造线程轮子 |
| 需要暂停等用户/外部系统响应 | **LangGraph** | 原生支持 HITL，不用自己写会话管理 |
| 需要调试中间态（看每个节点收到了什么） | **LangGraph** | stream + Studio，全程透明 |

> **一句话**：纯函数做"一次性的事"，LangGraph 做"有状态、可编排、可恢复的事"。今天这节课的 WeatherBot 用纯函数也能写——但需求每升级一次，纯函数就要重写一次；LangGraph 只需要加一条边。

#### 幕后机制：LangGraph 到底在背后做了什么？

四个需求对比完了，你心里可能还有一丝不安全感——"LangGraph 看起来是魔法，它到底是怎么做到的？" 不是魔法，是四个关键机制：

**机制 1：State 是全局共享池（不是参数管道）**

纯函数的数据传递是**参数管道**——`parse_intent(text)` 返回 `intent`，你要显式把 `intent` 传给下一个函数。五个节点要传五个参数，改一个字段要改五个函数签名。

LangGraph 的 State 是**全局共享池**——你在图的最外层定义一个 `TypedDict`（比如 `WeatherState`），所有节点接收的是同一个 State 对象。节点 A 返回 `{"city": "北京"}`，节点 B 直接读 `state["city"]`，不需要传参。框架在背后做了两件事：
1. 把节点返回的增量 dict 合并到 State（按字段的 Reducer 规则）
2. 把更新后的 State 传给下一个节点

这个"共享池"机制，是后面所有高级能力的基础。

**机制 2：循环是图拓扑（不是代码控制流）**

纯函数的循环是**代码控制流**——`for` 或递归，每循环一次栈帧加深一层，递归深度有限制（默认 1000），异常处理必须自己写 try/except。

LangGraph 的循环是**图拓扑**——框架内部维护一个"当前走到哪个节点"的指针。节点返回 `"query_weather"`，框架就把指针移回这个节点，重新执行。没有函数调用栈，没有深度限制，异常由框架统一捕获。你只是在"描述图的结构"，真正的执行调度是框架在管。

**机制 3：Checkpoint 自动存档（不是函数栈帧）**

纯函数执行时，数据存在**函数调用栈**的局部变量里。函数 return 之后，栈帧销毁，所有中间变量（比如 `city="北京"`、`data={...}`）全部释放。除非你自己写代码把状态存到数据库或 Redis，否则没有任何办法"回到中间某个步骤"。

LangGraph 的每个节点跑完后，框架**自动把完整 State 保存到 checkpoint**（后面 LG-05 会深入讲）。checkpoint 就是一个"存档点"——记录了"当前在哪个节点、State 里每个字段的值是什么"。这类似于游戏的存档功能，你可以随时读取任意存档，从那个位置继续玩，不用从头打。

Studio 的"改 State 重跑"，本质上就是：读取 `extract_city` 执行后的 checkpoint → 把里面的 `city` 改成"上海" → 从这个存档点继续执行。前面的 `parse_intent` 已经跑过了，不需要重做。

**机制 4：并行是框架级调度（不是操作系统级线程）**

纯函数的并行是**操作系统级并行**——你自己创建 `Thread` 或 `Task`，手动 `await` 或 `.result()`，一个超时了另一个还要不要等？异常在哪个线程抛的？结果怎么合并成统一的返回值？这些边界情况全要自己处理。

LangGraph 的并行是**框架级调度**——你写 `add_edge(START, ["query_weather", "query_aqi"])`，框架内部的调度器（基于 asyncio 事件循环）会自动为每个节点创建独立的执行上下文，并发跑完。两个节点都返回后，框架调用每个字段的 Reducer（比如 `operator.add` 或 `add_messages`）自动合并它们的 State 增量。超时策略、异常捕获、结果合并，框架内置了默认行为，你也可以通过配置覆盖。

---

**四个机制的关系**：

```
State 共享池 ──→ 节点之间天然共享数据
│
├──→ 图拓扑调度 ──→ 支持循环、条件分支
│
├──→ Checkpoint 存档 ──→ 支持断点续跑、人机协作
│
└──→ 框架级调度器 ──→ 支持并行执行、自动合并
```

> **关键认知转变**：写纯函数时，你在写"怎么执行"；写 LangGraph 时，你在写"图长什么样"。执行是框架的事，你只管描述结构。

### 3.3 Edge —— 普通边，确定性的连接

普通边：A 做完必定到 B。

```python
builder.add_edge("extract_city", "query_weather")
builder.add_edge("query_weather", "format_weather_reply")
builder.add_edge("get_time", END)
builder.add_edge("format_weather_reply", END)
builder.add_edge("fallback_reply", END)
```

START 和 END：图的入口和出口。

```python
from langgraph.graph import START, END

builder.add_edge(START, "parse_intent")
builder.add_edge("parse_intent", "extract_city") # 解析完意图，先到提取城市
```

**常见坑**：`START` 和 `END` 需要从 `langgraph.graph` 导入，不是字符串 `'START'`。

### 3.4 Conditional Edge —— 条件边，图的灵魂

条件边是 LangGraph 最强大的特性之一。它让图能根据 State 的内容，动态决定下一步走哪个节点。

**Router 函数**：接收 `state`，返回一个**字符串**，这个字符串必须是目标节点的名字。

```python
def route_by_intent(state: WeatherState) -> str:
"""根据意图决定下一步走哪个分支"""
if state["intent"] == "weather":
return "extract_city" # 走天气查询分支
elif state["intent"] == "time":
return "get_time" # 走时间查询分支
else:
return "fallback_reply" # 走兜底回复分支

# 条件边：parse_intent 做完后，由 route_by_intent 决定下一步
def route_after_city(state: WeatherState) -> str:
"""提取城市后，如果城市不为空就查天气，否则兜底"""
if state.get("city"):
return "query_weather"
return "fallback_reply"
```

注册条件边：

```python
# 第一个参数：源节点（做完后触发路由判断）
# 第二个参数：router 函数
# 第三个参数：可选，把所有可能的目标节点名字写清楚，便于可视化
builder.add_conditional_edges("parse_intent", route_by_intent, {
"extract_city": "extract_city",
"get_time": "get_time",
"fallback_reply": "fallback_reply"
})

builder.add_conditional_edges("extract_city", route_after_city, {
"query_weather": "query_weather",
"fallback_reply": "fallback_reply"
})
```

**关键点**：

- Router 函数返回的字符串必须对应一个已注册的节点名，否则会报错。
- `add_conditional_edges` 的第三个参数是可选的，但建议写上，这样 Mermaid 图能画出所有可能的分支。
- 一个节点可以同时有普通边和条件边吗？**不能**。一个节点的出边要么是确定的（普通边），要么是动态的（条件边），二选一。

**常见坑**：Router 函数返回了节点名字符串，但忘了在 `add_node` 中注册这个节点，运行时会说找不到目标节点。

### 3.5 compile —— 生成可执行的图

五步走：定义 State → 添加 Node → 连接 Edge（普通 + 条件）→ Compile → 运行。

```python
from langgraph.graph import StateGraph

builder = StateGraph(WeatherState)

# 1. 添加所有节点
builder.add_node("parse_intent", parse_intent)
builder.add_node("extract_city", extract_city)
builder.add_node("query_weather", query_weather)
builder.add_node("format_weather_reply", format_weather_reply)
builder.add_node("get_time", get_time)
builder.add_node("fallback_reply", fallback_reply)

# 2. 连接普通边
builder.add_edge(START, "parse_intent")
builder.add_edge("query_weather", "format_weather_reply")
builder.add_edge("format_weather_reply", END)
builder.add_edge("get_time", END)
builder.add_edge("fallback_reply", END)

# 3. 连接条件边
builder.add_conditional_edges("parse_intent", route_by_intent)
builder.add_conditional_edges("extract_city", route_after_city)

# 4. 编译
graph = builder.compile()
```

`compile()` 做了什么？

- 类型检查：State 是不是 TypedDict，Node 签名对不对
- 边验证：有没有孤立的节点，Router 返回的目标节点是否存在
- 生成可执行对象：编译后的 `graph` 可以 `invoke()` 或 `stream()`

**常见坑**：不 compile 直接 invoke 会报错。compile 是必需的。

### 3.6 invoke vs stream —— 两种运行方式

**invoke**：直接给最终结果。

```python
result = graph.invoke({
"messages": [HumanMessage("北京今天天气怎么样？")]
})
print(result["response"]) # 北京今天晴，25度
```

**stream**：逐个节点输出中间状态，适合调试和观察流程。

```python
for chunk in graph.stream({
"messages": [HumanMessage("北京今天天气怎么样？")]
}):
print(chunk) # 可以看到每个节点执行后的状态变化
```

**什么时候用 stream**？

- 调试时：看每个节点接收了什么 State，返回了什么更新
- 生产环境：需要实时反馈进度给用户
- 排查问题时：定位哪个节点出错了

## 4. 完整案例：带路由的 WeatherBot

### 需求

用户输入一句话，WeatherBot 先判断意图：
- 问天气 → 提取城市 → 查天气 API → 格式化回复
- 问时间 → 直接返回当前时间
- 其他 → 兜底回复

### 完整代码

```python
from typing import TypedDict, Annotated
from operator import add
from datetime import datetime
from langchain_core.messages import HumanMessage, AIMessage
from langgraph.graph import StateGraph, START, END

# === State ===
class WeatherState(TypedDict):
messages: Annotated[list, add]
intent: str
city: str
weather_data: dict
response: str

# === Nodes ===
def parse_intent(state: WeatherState):
text = state["messages"][-1].content
if "天气" in text or "温度" in text:
return {"intent": "weather"}
elif "时间" in text or "几点" in text:
return {"intent": "time"}
return {"intent": "other"}

def extract_city(state: WeatherState):
text = state["messages"][-1].content
city = "北京" if "北京" in text else "上海" if "上海" in text else ""
return {"city": city}

def query_weather(state: WeatherState):
return {"weather_data": {"temp": 25, "condition": "晴"}}

def format_weather_reply(state: WeatherState):
data = state["weather_data"]
reply = f"{state['city']}今天{data['condition']}，{data['temp']}度"
return {"response": reply}

def get_time(state: WeatherState):
return {"response": f"现在是 {datetime.now().strftime('%H:%M')}"}

def fallback_reply(state: WeatherState):
return {"response": "抱歉，我暂时只能回答天气和时间相关的问题。"}

# === Routers ===
def route_by_intent(state: WeatherState) -> str:
intent = state["intent"]
if intent == "weather":
return "extract_city"
elif intent == "time":
return "get_time"
return "fallback_reply"

def route_after_city(state: WeatherState) -> str:
if state.get("city"):
return "query_weather"
return "fallback_reply"

# === Build Graph ===
builder = StateGraph(WeatherState)
builder.add_node("parse_intent", parse_intent)
builder.add_node("extract_city", extract_city)
builder.add_node("query_weather", query_weather)
builder.add_node("format_weather_reply", format_weather_reply)
builder.add_node("get_time", get_time)
builder.add_node("fallback_reply", fallback_reply)

builder.add_edge(START, "parse_intent")
builder.add_edge("query_weather", "format_weather_reply")
builder.add_edge("format_weather_reply", END)
builder.add_edge("get_time", END)
builder.add_edge("fallback_reply", END)

builder.add_conditional_edges("parse_intent", route_by_intent)
builder.add_conditional_edges("extract_city", route_after_city)

graph = builder.compile()

# === Run ===
for query in ["北京今天天气怎么样？", "现在几点了？", "讲个笑话"]:
result = graph.invoke({"messages": [HumanMessage(query)]})
print(f"用户: {query}")
print(f"Bot: {result['response']}\n")
```

## 5. MessagesState —— 聊天场景的预制模板

如果只是为了做聊天机器人，LangGraph 已经预制了一个最常用 State 模板：

```python
from langgraph.graph import MessagesState

# 等价于：
# class MessagesState(TypedDict):
#     messages: Annotated[list, add_messages]

builder = StateGraph(MessagesState)
# 直接开始写 Node，不用自己定义 State
```

`MessagesState` 就是一个带 `messages` 字段的 TypedDict，`messages` 自动用 `add_messages` Reducer 管理。适合快速上手，但建议学完基础后还是用自己的 TypedDict，更灵活。

## 6. 常见误区

| 坑 | 现象 | 怎么避免 |
|----|------|---------|
| State 写成普通 dict | 编译通过但运行时报错 | 必须用 TypedDict 或 Pydantic BaseModel |
| Reducer 遗漏 | 列表字段每次被覆盖 | 会累积的字段用 `Annotated` + Reducer |
| 节点返回字符串 | 报 TypeError | 节点必须返回 dict |
| 忘记 import START/END | 找不到 START 符号 | 从 `langgraph.graph` 导入 |
| 节点内直接改 state | 修改不生效 | 必须通过返回值让框架合并 |
| 忘记 compile | 报错没有 invoke 方法 | compile 是必需的 |
| Router 返回未注册的节点名 | 运行时找不到目标节点 | 确保 Router 返回的字符串和 add_node 的名字一致 |
| 一个节点同时有普通边和条件边 | 行为不可预期 | 一个节点的出边只能是普通边或条件边，二选一 |

## 7. 阶段小结

今天学了五个东西：

1. **State** —— TypedDict，定义共享数据结构，Annotated + Reducer 控制合并策略
2. **Node** —— 接收 state 返回 dict 的函数，可以是任何 Python 代码
3. **Edge** —— 普通边是确定性连接，条件边让图能根据 State 动态路由
4. **compile** —— 类型检查、边验证、生成可执行图对象
5. **invoke / stream** —— 运行图的两种方式，后者适合调试和实时反馈

构建流程：定义 StateSchema → 添加 Node → 连接 Edge（普通 + 条件）→ Compile → invoke/stream。

口诀：

> 状态是数据，节点是工厂，边是传送带，条件边是分岔口，编译出图纸，stream 看过程。

## 8. 与 LG-02 的衔接

今天 WeatherBot 已经能根据意图走不同分支了。但现实世界不会这么简单：

- 查询 API 失败了需要重试怎么办？—— 这就需要**循环**
- 用户说"不对，我问的是上海不是北京"，需要回到前面的节点重新走怎么办？—— 这也需要**循环**
- 想让 AI 调用外部工具（搜索、计算器、数据库）怎么办？—— 这就需要**Tools**

下一节课 LG-02，我们要让这张图更"聪明"：
- **循环（Cycles）** —— 让图能够重试、迭代、回到之前的节点
- **工具（Tools）** —— 让 AI 能够调用外部 API
- **Human-in-the-loop** —— 让人在流程中间介入审核

这些都是让 WeatherBot 变成真正可用的生产级系统的关键能力。
