Core Concepts
五个核心概念一句话说清关系

State（状态）= 贯穿整个图的共享数据

TypedDict 定义，所有节点接收完整 State，返回增量更新

↓

Node（节点）= 处理函数

接收 State，返回 dict。可以是任何 Python 函数、LangChain Runnable、甚至另一个图

Edge（边）= 节点之间的连接

普通边：A 做完必定到 B
条件边：根据 State 动态决定下一步

↓

compile() = 把上面这些东西编译成可执行的图对象

↓

invoke() / stream() = 运行图

invoke 直接给结果，stream 逐个节点输出中间状态

LangGraph 教程 3 / 10
这张图讲清楚五个核心概念的关系。用一句话概括：State 是数据，Node 是处理，Edge 是路由，compile 是生成，invoke/stream 是运行。今天的目标就是搞懂这五个东西怎么配合，然后写出带路由判断的 WeatherBot。
State
State —— 图里的共享数据
class WeatherState(TypedDict):
    messages: Annotated[list, add]   # 对话历史，用 add 累加
    intent: str                       # 用户意图
    city: str                         # 城市
    weather_data: dict               # API 返回数据
    response: str                   # 最终回复

三个关键点

  • 每个节点接收完整 State
  • 节点返回增量更新，只改需要的字段
  • Annotated[field, reducer] 决定字段怎么合并

两个核心 Reducer

  • operator.add：列表直接拼接
  • add_messages：拼接 + 按 ID 去重 + 支持删除

常见坑

节点内直接 state["city"] = "北京" 不会生效！必须通过返回值让框架合并。

LangGraph 教程 4 / 10
State 是一个 TypedDict，定义了图中所有节点共享的数据结构。每个节点接收完整 State，节点 A 可以看到 Node B 产生的数据。节点返回增量更新，return {"city": "北京"} 只会更新 city 字段。Annotated 加 reducer 决定字段怎么合并。最常见的坑：节点内直接修改 state 不会生效，必须通过返回值让框架合并。
Node
Node —— 图里的处理单元
def parse_intent(state: WeatherState):
    text = state["messages"][-1].content
    if "天气" in text: return {"intent": "weather"}
    elif "时间" in text: return {"intent": "time"}
    return {"intent": "other"}

def query_weather(state: WeatherState):
    return {"weather_data": {"temp": 25, "condition": "晴"}}

def format_reply(state: WeatherState):
    data = state["weather_data"]
    reply = f"{state['city']}今天{data['condition']}，{data['temp']}度"
    return {"response": reply}

Node 的本质

  • 就是一个函数：接收 state，返回一个 dict
  • 可以是任何 Python 函数，不限于 AI 调用（数据库查询、API 调用、文件处理）
  • 也可以是 LangChain Runnable、甚至另一个完整的图（Subgraph）
  • 返回值必须是 dict，字符串要包装成 {"result": string}
LangGraph 教程 5 / 10
Node 就是一个函数，接收 state，返回一个 dict。看这三个例子：parse_intent 解析用户意图，query_weather 调用天气 API（mock），format_reply 生成回复文本。Node 的本质很简单：可以是任何 Python 函数，不限于 AI 调用。唯一约束是返回值必须是 dict。
Edge
Edge —— 普通边与条件边

普通边（确定性连接）

builder.add_edge(START, "parse_intent")
builder.add_edge("extract_city", "query_weather")
builder.add_edge("format_reply", END)

A 做完必定到 B。START/END 需从 langgraph.graph 导入。

条件边（图的灵魂）

def route_by_intent(state) -> str:
    if state["intent"] == "weather":
        return "extract_city"
    elif state["intent"] == "time":
        return "get_time"
    return "fallback"

builder.add_conditional_edges(
    "parse_intent", route_by_intent)

关键约束

一个节点的出边只能是普通边或条件边，二选一，不能同时有。Router 返回的字符串必须对应已注册的节点名。

LangGraph 教程 6 / 10
边分两种。普通边是确定性连接。条件边让图能根据 State 内容动态决定下一步走哪个节点。Router 函数接收 state，返回目标节点的名字字符串。add_conditional_edges 的第三个参数建议写上目标节点映射，便于 Mermaid 可视化。关键约束：一个节点的出边只能是普通边或条件边，二选一。
Build & Run
五步构建 + 两种运行方式
# 1. 定义 StateSchema
class WeatherState(TypedDict): ...

# 2. 创建 Builder，添加所有节点
builder = StateGraph(WeatherState)
builder.add_node("parse_intent", parse_intent)
builder.add_node("query_weather", query_weather)

# 3. 连接普通边
builder.add_edge(START, "parse_intent")

# 4. 连接条件边
builder.add_conditional_edges("parse_intent", route_by_intent)

# 5. 编译
graph = builder.compile()

invoke —— 直接给最终结果

result = graph.invoke({"messages": [HumanMessage("北京天气")]})
# 输出: 北京今天晴，25度

stream —— 逐个节点输出中间状态

for chunk in graph.stream({...}):
    print(chunk)  # 调试 + 实时进度反馈
LangGraph 教程 7 / 10
构建图的五步走：定义 StateSchema、添加 Node、连接普通边、连接条件边、compile 编译。compile 会做类型检查、边验证、生成可执行对象。不 compile 直接 invoke 会报错。invoke 直接给最终结果，适合生产；stream 逐个节点输出中间状态，适合调试和实时反馈。

Comparison
需求升级时，代码量怎么变？
需求升级 纯函数代码量 LangGraph 代码量

纯函数的困境

  • 重试代码和业务代码缠在一起
  • 并行需要自己造线程轮子
  • 没有"暂停等用户"的概念
  • 每加一个需求就要重构已有代码

LangGraph 的优势

  • 重试 = 图上画个圈，独立路由规则
  • 并行 = 声明式，框架自动调度
  • HITL = 加一个节点 + 一条边
  • 加需求就是加节点/加边，不影响已有结构
"纯函数做'一次性的事'，LangGraph 做'有状态、可编排、可恢复的事'。"
LangGraph 教程 9 / 10
这是今天最重要的一张对比图。纯函数的复杂度是指数级爆炸，每加一个需求就要重构已有代码。LangGraph 的复杂度是线性增长，加需求就是加节点加边，不影响已有结构。纯函数做一次性的事，LangGraph 做有状态、可编排、可恢复的事。
