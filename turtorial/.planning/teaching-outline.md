## LG-01：LangGraph 基础与图构建
**预计时长**: 3-4 小时（含 1 小时实战）

---

### 1. 开场引入（5-10 分钟）

#### 引入策略：从「熟悉的痛苦」切入

**开场白建议**：
> "大家用 LangChain 写过 Chain 吧？`prompt | llm | parser` 这种管道式写法很优雅，但你们有没有遇到过这种情况：当业务逻辑稍微复杂一点——比如需要根据 LLM 的输出决定下一步做什么、或者某个步骤失败后要重试——Chain 就开始变得像意大利面条一样，到处都是 `RunnableLambda` 和条件判断？"

#### 生活化类比：餐厅点餐流水线

用「餐厅后厨」类比 LangChain → LangGraph 的演进：

- **LangChain（线性流水线）**: 像快餐店的固定流水线：接单→做汉堡→打包→出餐。每个步骤固定，无法灵活应对。
- **LangGraph（有向图）**: 像正餐餐厅的智能调度系统：接单→（判断：简单菜品？直接做 / 复杂菜品？分解步骤）→各站并行烹饪→汇总摆盘→出餐。每个订单可以根据复杂程度走不同的路径。

> "LangGraph 的核心理念就是：把工作流从『一条直线』变成『一张图』。状态（State）是这张图的血脉，节点（Node）是处理站，边（Edge）是传送带。"

#### 与上一阶段的衔接（首阶段无前置，直接点题）

> "今天这堂课是整个课程的基石。后面所有的复杂工作流——路由、循环、并行、多智能体——都是在『图』这个概念上长出来的。理解好 LG-01，后面会事半功倍。"

**激发兴趣的点**：
- 展示一个 WeatherBot 的最终效果动画（节点高亮 + 状态实时变化）
- 预告："今天 3 小时后，你们就能独立写出这个 WeatherBot"

### 2. 概念讲解顺序（核心部分，约 90 分钟）

#### 2.1 先讲「State」—— 图的血脉（20 分钟）

**为什么先讲 State？**  
State 是贯穿整个 LangGraph 的核心抽象。学员如果不理解 State 如何在节点间流动，后面的 Node 和 Edge 都会变成空中楼阁。

**讲解要点**：
1. `State` 是 `TypedDict`：强类型的共享字典（5 分钟）
2. State 的流转机制：每个节点接收完整 State，返回部分更新（10 分钟）
3. `Annotated` + `Reducer`：状态字段的合并策略（5 分钟）

**必须画图的点**：
```
┌─────────────────────────────────────┐
│  State = {                          │
│    "messages": [...],     ← Annotated[list, add]  │
│    "city": "北京",        ← 普通字段，直接覆盖    │
│    "step_count": 3        ← Annotated[int, lambda x,y: y] │
│  }                                  │
└─────────────────────────────────────┘
         ↓ 传入 Node A
   Node A 返回 {"city": "上海"}
         ↓ State 更新
  "city" 被覆盖，messages 保留，step_count 保留
```

**类比**：State 像是一个「公共白板」，每个节点（工人）可以看到白板上的全部信息，然后添加或修改自己负责的那一栏。

---

#### 2.2 再讲「Node」—— 图的处理站（15 分钟）

**为什么第二个讲 Node？**  Node 是 State 的操作者，理解了 State 再讲 Node 是自然的递进。

**讲解要点**：
1. Node 的本质：接收 `state` 字典，返回 `dict` 或 `Command` 的函数（5 分钟）
2. Node 的注册：`builder.add_node("node_name", node_function)`（5 分钟）
3. Node 的灵活性：可以是任意 Python 函数、LangChain Runnable、甚至是另一个图（伏笔 LG-06）（5 分钟）

**代码演示节奏**：
```python
# 先写一个最简单的 Node
def parse_intent(state: AgentState):
    # 接收完整 state
    user_input = state["messages"][-1].content
    # 返回部分更新
    return {"intent": "weather_query", "city": extract_city(user_input)}
```

**关键话术**：
> "注意这个返回值——它只是一个字典。LangGraph 会自动把这个字典合并到 State 中。不需要你手动去更新整个 State，框架帮你做合并。"

---

#### 2.3 然后讲「Edge」—— 图的传送带（15 分钟）

**讲解要点**：
1. 普通边：`add_edge("node_a", "node_b")` —— 确定性的顺序执行（5 分钟）
2. `START` 和 `END`：图的入口和出口（3 分钟）
3. 条件边预告（详细内容留到 LG-02，这里只提一句）："边可以是动态的，根据 State 决定下一步去哪"（2 分钟）
4. 边的可视化：用 ASCII 图或 Mermaid 语法展示（5 分钟）

**板书建议**：
```
START → [parse_intent] → [query_weather] → [format_reply] → END
         (Node)           (Node)            (Node)
```

---

#### 2.4 最后讲「StateGraph 构建与编译」（20 分钟）

**讲解要点**：
1. 构建流程：`StateGraph(StateSchema) → add_node → add_edge → compile`（10 分钟）
2. `compile()` 的作用：类型检查、边验证、生成可执行对象（5 分钟）
3. 图的执行：`graph.invoke({"messages": [user_msg]})`（5 分钟）

**演示节奏**：
- 先写代码构建图（10 分钟）
- 再执行一次，展示结果（5 分钟）
- 再画一次图的结构，让学员「看到」代码与图的对应关系（5 分钟）

---

#### 2.5 进阶：`MessagesState` 与 Channel（20 分钟）

**讲解策略**：
- `MessagesState`：「如果你只是做聊天机器人，LangGraph 已经为你预制了一个最常用 State 模板」（5 分钟）
- Channel 机制：选讲，用「管道系统」类比 —— LastValue（单槽位覆盖）、Topic（广播）、BinaryOperatorAggregate（累加器）。建议初学者先跳过，标注为「进阶阅读」（15 分钟）

### 3. 案例驱动逻辑：WeatherBot

#### 案例与概念的一一对应

| 概念 | WeatherBot 中的体现 |
|------|-------------------|
| State | `AgentState = {"messages": [...], "city": str, "weather_data": dict}` |
| Node | `parse_intent` → `query_weather` → `format_reply` |
| Edge | `START→parse_intent→query_weather→format_reply→END` |
| Reducer | `messages` 字段用 `Annotated[list, add]` 累积对话历史 |
| compile() | 编译后生成 `graph`，调用 `graph.invoke()` |

#### 代码演示的讲解话术

**Step 1 - 定义 State 时**：
> "大家看，我们定义了一个 `AgentState`，它是一个 TypedDict。这里有三个字段：`messages` 用来存对话历史，`city` 存用户提到的城市，`weather_data` 存 API 返回的原始数据。注意 `messages` 前面有个 `Annotated[list, add]` —— 这告诉 LangGraph：当节点返回 messages 时，不要覆盖，而是追加到列表末尾。"

**Step 2 - 写 Node 时**：
> "现在写第一个节点 `parse_intent`。它接收整个 state，但只用到了 `messages[-1]`——也就是用户的最新输入。它返回一个字典 `{"city": city}`。LangGraph 收到这个返回值后，会自动把 `state["city"]` 更新为这个新值。"

**Step 3 - 构建图时**：
> "现在到了最关键的步骤——把节点连起来。大家看这段代码：`builder.add_edge("parse_intent", "query_weather")`。这就像在说：parse_intent 做完后，下一站永远是 query_weather。"

**Step 4 - 执行时**：
> "执行！`result = graph.invoke({"messages": [HumanMessage("北京今天天气怎么样？")]})`。大家注意看这个返回值——它包含了完整的 state，包括所有中间过程产生数据。"

#### 关键转折点（需要停下来确认学员理解）

1. **Reducer 时刻**：讲完 `Annotated[list, add]` 后，停下来问："如果不用 add，用普通 list，第二次调用会发生什么？"（期望回答：消息历史被覆盖，只保留最后一条）
2. **State 合并时刻**：讲完节点返回字典后，停下来问："如果一个节点返回了 state 中没有的新字段，会发生什么？"（期望回答：LangGraph 会自动添加这个字段）
3. **编译时刻**：问 "不 compile 直接 invoke 会怎样？"（期望回答：编译时检查，比如有节点没连边、有死代码等）

### 4. 常见误区与注意事项

#### 学员容易踩的坑

| 坑 | 现象 | 预警话术 |
|----|------|---------|
| **忘记 TypedDict** | 把 State 写成普通 dict，没有类型提示，后续编译通过但运行时报错 | "State 必须是 TypedDict 或 Pydantic BaseModel，不能是普通 dict。框架靠这个类型信息做校验。" |
| **Reducer 遗漏** | 列表字段每次被覆盖而不是追加 | "凡是会累积的字段（messages、history、logs），一定要用 `Annotated` + Reducer。" |
| **节点返回格式错误** | 节点返回了字符串而不是 dict，报错 `TypeError` | "节点必须返回 dict。如果想返回字符串，包装成 `{"result": string}`。" |
| **忘记 import START/END** | 找不到 START 符号 | "START 和 END 需要从 `langgraph.graph` 导入，不是字符串 'START'。" |
| **在节点里修改 state 直接** | 节点内 `state["x"] = y`，以为能持久化 | "节点内直接修改 state 不会生效！必须通过返回值让框架合并。" |

#### 调试技巧

1. **打印 State**：在每个节点开头加 `print(f"[Node: {name}] State: {state}")`
2. **用 LangGraph Studio**：如果有条件，展示 Studio 的可视化调试
3. **逐步执行**：`graph.stream()` 替代 `graph.invoke()`，逐个节点观察输出

### 5. 阶段小结（5 分钟）

#### 核心要点回顾

1. **LangGraph 的核心思想**：用「图」替代「链」，状态驱动执行
2. **三大基石**：State（共享数据）+ Node（处理单元）+ Edge（控制流）
3. **State 更新机制**：节点返回增量更新，框架自动合并（Reducer 决定合并策略）
4. **构建四步走**：定义 StateSchema → 添加 Node → 连接 Edge → Compile
5. **MessagesState**：聊天场景的预制模板，开箱即用

#### 记忆口诀

> **"状态是血脉，节点是工厂，边是传送带，编译出图纸。"**

#### 思维导图建议

```
LangGraph 基础
├── State（TypedDict）
│   ├── 普通字段 → 覆盖更新
│   └── Annotated + Reducer → 累积/合并更新
├── Node
│   ├── 接收 state → 返回 dict
│   └── 可以是函数 / Runnable / 子图
├── Edge
│   ├── 普通边 → 确定性流转
│   └── 条件边 → 动态路由（LG-02）
└── compile() → 类型检查 + 生成可执行图
```

---

### 6. 与下一阶段的衔接

> "今天我们做了一个最简单的直线型图：接单→解析→查询→回复。但现实世界的流程从来不会这么简单——如果用户问的不是天气而是时间呢？如果查询 API 失败了需要重试呢？如果用户要求退款需要走审批流程呢？
>
> 下一节课 LG-02，我们要让这张图『聪明』起来——让它学会自己做决定。我们要学**条件边（Conditional Edge）**，让图根据状态选择不同的分支；要学**循环（Cycles）**，让图能够重试和迭代；还要学**工具（Tools）**，让图能够调用外部 API。这些都是让 WeatherBot 变成 SmartRouter 的关键能力。"

## LG-02：路由、工具与条件控制
**预计时长**: 4-5 小时（含 1.5 小时实战）

---

### 1. 开场引入（5-10 分钟）

#### 引入策略：从 LG-01 的「直线图」引入

**开场白建议**：
> "上节课我们画了第一张图——WeatherBot，一条直线走到底。但我们没有回答一个问题：如果用户输入『北京天气』，走天气查询分支；如果输入『我要退款』，走退款流程分支；如果输入『转人工』，直接通知客服——这张图该怎么画？
>
> 换句话说，我们要让图学会『看情况办事』。这就是今天主题：路由（Routing）与条件控制。"

#### 生活化类比：机场安检分流

> "想象你在机场安检：走到入口处，系统扫描你的证件 → 判断你是『国内航班』还是『国际航班』→ 国内走左边通道，国际走右边通道。每个通道内部又有自己的流程（国内：安检→登机口；国际：安检→边检→登机口）。
>
> 条件边就是这个『判断 + 分流』的机制。"

#### 与 LG-01 的衔接

> "上节课我们学了普通边 `add_edge(A, B)`——A 做完永远是 B。今天要学 `add_conditional_edges(A, router_func)`——A 做完，让 `router_func` 根据当前状态决定下一步去哪。"

---

### 2. 概念讲解顺序（核心部分，约 150 分钟）

#### 2.1 条件边 Conditional Edge（30 分钟）

**讲解顺序**：
1. **概念引入**：用机场分流类比（5 分钟）
2. **API 讲解**：`add_conditional_edges(source, router, path_map)`（10 分钟）
3. **router 函数的签名**：接收 state，返回字符串（下一个节点的名字）（10 分钟）
4. **path_map 的作用**：限制 router 的返回值范围（安全性）（5 分钟）

**必须画图的点**：
```
                    ┌─→ [天气查询] ──┐
[意图识别] → router ─┼─→ [退款流程] ──┼─→ [统一回复] → END
                    └─→ [人工客服] ──┘
```

**关键话术**：
> "大家注意，router 函数返回的是一个字符串——这个字符串必须是 path_map 中列出的某个节点名称。如果返回了不存在的节点，图会报错。这是 LangGraph 帮你做的一道安全检查。"

---

#### 2.2 循环 Cycles（25 分钟）

**讲解顺序**：
1. **为什么需要循环**：重试机制、迭代优化、ReAct 模式（5 分钟）
2. **循环的实现**：某个节点的边指向它自己，或指向前面已经执行过的节点（10 分钟）
3. **防止无限循环**：`recursion_limit` 参数（5 分钟）
4. **ReAct Pattern**：推理→行动→观察→再推理（5 分钟）

**必须画图的点**：
```
[思考] → [行动] → [观察] ─┐
   ↑─────────────────────┘
   （循环：如果还没得到答案，回去再思考）
```

**类比**：医生诊断 —— 问诊 → 开检查单 → 看报告 → （如果还不确定）→ 再问再查。

---

#### 2.3 工具集成 ToolNode（25 分钟）

**讲解顺序**：
1. **Tool 的定义**：`@tool` 装饰器 + 函数描述（10 分钟）
2. **ToolNode**：把工具集合包装成图节点（5 分钟）
3. **`tools_condition`**：自动判断 LLM 是否需要调用工具（10 分钟）

**代码演示节奏**：
```python
@tool
def query_weather(city: str) -> str:
    """查询指定城市的天气"""
    ...

tools = [query_weather]
tool_node = ToolNode(tools)

# 条件边：LLM 如果需要工具，去 tool_node；如果不需要，去 END
builder.add_conditional_edges("agent", tools_condition)
```

**关键话术**：
> "`tools_condition` 是一个 LangGraph 内置的条件函数。它会检查 LLM 的最后一条消息——如果消息里包含了 `tool_calls`，就说明 LLM 想要调用工具，图就走到 `tool_node`；如果没有，就直接结束。"

---

#### 2.4 预构建 Agent 深度对比（35 分钟，v1.1 新增重点）

**讲解策略**：这是 v1.1 新增的重点模块，要讲清楚「什么时候用预构建，什么时候自定义」。

**讲解顺序**：
1. **`create_react_agent` 的内部机制**：plan→tool_call→observe→(loop)（15 分钟）
2. **`create_tool_calling_agent`**：轻量级，无显式 plan 步骤（10 分钟）
3. **预构建 vs 自定义 StateGraph 的决策框架**（10 分钟）

**必须画对比图的点**：

```
create_react_agent（黑盒）           vs          自定义 StateGraph（白盒）
┌─────────────────────┐              ┌─────────────────────────────────────┐
│ plan → tool → observe│              │ [思考] → [决策] → [工具] → [观察]     │
│    ↑___________↓    │              │    ↑___________________________↓    │
│  （内部自动循环）      │              │  （每个节点你都可以自定义、监控、干预）  │
└─────────────────────┘              └─────────────────────────────────────┘
优点：3行代码搞定                          优点：完全可控，中间状态可见
缺点：中间状态不可见，循环控制受限            缺点：代码量多，需要理解图机制
适用：快速原型、标准 ReAct 场景              适用：复杂路由、HITL、自定义状态管理
```

**决策框架（板书重点）**：

| 维度 | 预构建 Agent | 自定义 StateGraph |
|------|-------------|-------------------|
| 开发速度 | ⭐⭐⭐ 快（3 行代码） | ⭐⭐ 中等 |
| 可控性 | ⭐⭐ 黑盒 | ⭐⭐⭐ 白盒，每步可控 |
| 可观测性 | ⭐⭐ 只能看到输入输出 | ⭐⭐⭐ 每节点状态可见 |
| 复杂路由 | ⭐ 不支持 | ⭐⭐⭐ 任意图结构 |
| HITL 支持 | ⭐ 弱 | ⭐⭐⭐ 强 |

**关键话术**：
> "选型决策很简单：如果需求是一个标准 ReAct 循环（思考→工具→观察→再思考），且你不需要监控中间过程，用 `create_react_agent`；只要出现以下任何一种情况——需要人在中间审批、需要根据业务状态走不同分支、需要自定义状态字段——就必须用自定义 StateGraph。"

### 3. 案例驱动逻辑：SmartRouter

#### 案例与概念的一一对应

| 概念 | SmartRouter 中的体现 |
|------|-------------------|
| 条件边 | 意图识别后路由到不同分支 |
| 循环 | 工具调用失败后的重试循环 |
| ToolNode | 各分支的工具集合（天气 API、订单查询、退款接口） |
| `tools_condition` | 预构建 Agent 分支的自动工具判断 |
| 预构建 vs 自定义 | 同一分支用两种方式实现，对比执行路径 |

#### 代码演示的讲解话术

**Step 1 - 意图识别节点**：
> "首先看 `intent_classifier` 节点。它接收用户输入，用 LLM 判断意图——是天气查询、订单查询、退款申请还是转人工？它返回 `{"intent": "weather"}`。"

**Step 2 - 条件路由**：

**Step 4 - 循环演示**：
> "再看这个重试场景。查询天气 API 时超时了——在自定义版本中，我们设计了一个循环：如果 API 失败，最多重试 3 次。大家看这张图，`query_api` 节点的条件边指向自己，这就是循环的实现。"

### 4. 常见误区与注意事项

| 坑 | 现象 | 预警话术 |
|----|------|---------|
| **条件边返回值不在 path_map 中** | 运行时抛异常 | "router 返回值必须是 path_map 中的字符串，建议用 Enum 或常量定义。" |

## LG-03：流式输出与实时监控
**预计时长**: 3-4 小时（含 1.5 小时实战）

---

### 1. 开场引入（5-10 分钟）

#### 引入策略：从「黑盒焦虑」引入

**开场白建议**：
> "大家用过 ChatGPT 吧？它的打字机效果——一个字一个字往外蹦——不只是为了好看，更重要的是消除等待焦虑。你知道它在工作，你看到了进度。
>
> 但我们上两节课写的图都是『黑盒执行』：调用 `invoke()`，等几秒或几十秒，然后蹦出一个最终结果。用户在这几十秒里完全不知道发生了什么。
>
> 今天的主题是：**把黑盒变成透明盒**——让每一个中间状态、每一条生成的 token、每一次节点执行，都实时推送到用户眼前。"

#### 生活化类比：外卖订单追踪

> "点外卖时，你最焦虑的不是等 30 分钟，而是『外卖到哪了』的未知。所以美团给你显示了完整的流程：商家已接单→骑手已到店→骑手已取餐→骑手距您 500 米。
>
> 流式输出就是在给我们的 LangGraph 应用做『外卖追踪』。"

#### 与 LG-02 的衔接

> "上节课我们实现了 SmartRouter，能分流、能循环、能调工具。但用户看到的是：输入→（黑盒等待）→输出。今天我们要让用户看到盒子里发生的一切。"

---

### 2. 概念讲解顺序（核心部分，约 120 分钟）

#### 2.1 `stream_mode="values"`（20 分钟）

**讲解顺序**：
1. **概念**：每次 State 完整快照的流式推送（5 分钟）
2. **适用场景**：需要实时看到 State 变化的仪表盘（5 分钟）
3. **代码演示**：`for event in graph.stream(input, stream_mode="values")`（10 分钟）

**类比**：「每次节点执行完，给你拍一张照片（完整 State），然后快速播放这些照片，就是一部电影。」

---

#### 2.2 `stream_mode="updates"`（20 分钟）

**讲解顺序**：
1. **概念**：只推送 State 的增量更新（5 分钟）
2. **与 values 的区别**：values 给完整照片，updates 只给变化的部分（10 分钟）
3. **适用场景**：网络带宽敏感、只需要增量更新的前端（5 分钟）

**类比**："values 像是你每次把整张纸给我看；updates 像是你只把修改的那一行指给我看。"

---

#### 2.3 `stream_mode="messages"`（25 分钟，重点）

**讲解顺序**：
1. **概念**：token 级别的流式推送——每个生成的字都会推过来（5 分钟）
2. **适用场景**：打字机效果、实时对话（5 分钟）
3. **代码演示**：前端如何拼接 token（10 分钟）
4. **messages 与其他 mode 的组合使用**（5 分钟）

**必须演示的效果**：
> 现场运行代码，展示 token 一个一个蹦出来的效果。这是最有冲击力的演示。

---

#### 2.4 `stream_mode="custom"`（20 分钟）

**讲解顺序**：
1. **概念**：节点内通过 `yield` 推送自定义事件（5 分钟）
2. **适用场景**：进度百分比、自定义状态通知、业务事件（5 分钟）
3. **代码演示**：在 Node 中 `yield {"progress": 50, "status": "searching"}`（10 分钟）

**类比**："自定义事件就像你在做一道菜时，不只在最后说『做好了』，而是中途说『水烧开了』、『调料放完了』、『还有 2 分钟出锅』。"

---

#### 2.5 `stream_mode="tasks"` 与 `astream_events`（20 分钟）

**讲解顺序**：
1. **tasks**：任务级生命周期事件（pending→running→completed）（10 分钟）
2. **astream_events**：最细粒度的事件流（v2 API）（10 分钟）

---

#### 2.6 多 Mode 组合与事件过滤（15 分钟）

**讲解顺序**：
1. 如何组合多种 stream_mode：`stream_mode=["values", "messages"]`
2. Node Metadata / Tags 用于事件过滤
3. 前端如何区分不同来源的事件

### 3. 案例驱动逻辑：LiveWriter

#### 案例与概念的一一对应

| 概念 | LiveWriter 中的体现 |
|------|-------------------|
| values | 实时状态面板显示当前执行到哪个节点 |
| updates | 段落进度条只接收新增的内容 |
| messages | 打字机效果，逐字生成文章 |

#### 代码演示的讲解话术

**Step 1 - 大纲规划节点 + values**：

**Step 2 - 并行资料检索 + tasks**：

### 4. 常见误区与注意事项

| 坑 | 现象 | 预警话术 |
|----|------|---------|
| **用 values 做打字机效果** | 每次推送完整 State，前端需要 diff 才能知道新增了什么 | "打字机效果必须用 messages mode，values 不适合 token 级流式。" |
| **忘记前端拼接 token** | 每个 token 覆盖了上一个，只显示最后一个字 | "messages mode 的 token 是增量，前端需要 `text += new_token`，不是 `text = new_token`。" |
| **custom yield 不在 Node 中** | 自定义事件没有推送 | "custom mode 的 yield 必须在 Node 函数内部，且图要用 astream。" |
| **混淆 sync/async** | `stream` 和 `astream` 混用导致不流式 | "异步图必须用 `astream`，同步图用 `stream`。LangGraph 的流式依赖 async generator。" |
| **事件类型过滤不当** | 前端收到大量不需要的事件，UI 卡顿 | "给 Node 加 tags，前端按 tag 过滤事件，不要全量接收。" |

1. **打印每个 event**：先不用前端，直接在终端 `print(event)` 观察事件结构
2. **区分 event 类型**：`event["event"]` 字段标识事件类型（on_chain_start、on_chat_model_stream 等）
3. **用 Gradio 快速验证**：Gradio 的 `chatbot` 组件原生支持流式消息

## LG-04：持久化与记忆系统
**预计时长**: 5-6 小时（含 2.5 小时实战）

---

### 1. 开场引入（5-10 分钟）

#### 引入策略：从「失忆」引入

**开场白建议**：
> "想象你和一个朋友聊天，每次你刷新页面，他都忘了你是谁、你们之前聊过什么、你的名字叫什么。你会不会很崩溃？
>
> 但我们前两节课写的图就是这样的『失忆症患者』。每次 `invoke()` 都是全新的开始，state 在内存里，程序一关就清零。
>
> 今天的主题是：**给图装上记忆**——短期记忆（一次对话内记住上下文）和长期记忆（跨对话记住你是谁）。"

#### 生活化类比：人类记忆系统

> "人的记忆分三层：
> - **工作记忆**：你现在正在想的事情，容量有限，断电（睡觉）就清空 → 对应 LangGraph 的内存 State
> - **短期记忆**：今天发生的事情，能记住几天 → 对应 LangGraph 的 Checkpoint（线程级持久化）
> - **长期记忆**：你的名字、你的喜好、你学过的知识，能记很久 → 对应 LangGraph 的 Store（用户级持久化）
>
> 我们要让图同时拥有这三种记忆。"

#### 与 LG-03 的衔接

> "上节课我们学会了实时流式输出，用户能『看到』图在运行。但用户关闭浏览器后，图就『忘记』了一切。今天我们要让图『记住』。"

### 2. 概念讲解顺序（核心部分，约 180 分钟）

#### 2.1 Checkpoint 机制（30 分钟）

**讲解顺序**：
1. **什么是 Checkpoint**：图执行过程中的状态快照（5 分钟）
2. **超级步（Super-step）**：LangGraph 的执行单元，每个超级步结束生成一个 Checkpoint（10 分钟）
3. **恢复机制**：从任意 Checkpoint 恢复执行（5 分钟）
4. **可视化**：用时间线展示 Checkpoints 序列（10 分钟）

**必须画图的点**：
```

```

**类比**："Checkpoint 就像游戏里的存档点。你打到一个存档点，保存进度。如果后面挂了，可以从存档点继续，不用从头打。"

#### 2.2 Checkpointer / Saver（35 分钟）

**讲解顺序**：
1. **MemorySaver**：内存中的 Checkpoint，开发测试用（10 分钟）
2. **PostgresSaver**：PostgreSQL 持久化，生产环境用（15 分钟）
3. **RedisSaver**：Redis 高速持久化，适合高频会话（10 分钟）

**代码对比演示**：
```python
# 开发环境

graph = builder.compile(checkpointer=MemorySaver())

# 生产环境

graph = builder.compile(checkpointer=PostgresSaver(conn_string))
```

**关键话术**：
> "切换 Saver 只需要改一行代码——从 MemorySaver 换成 PostgresSaver。图的定义完全不需要变。这就是 LangGraph 的设计哲学：持久化是『插件』，不是『侵入』。"

**类比**："thread_id 就像聊天窗口的 ID。同一个窗口里的消息是连续的，换了一个窗口就是新的对话。"

| PostgresSaver | 生产环境持久化对话历史 |
| RedisSaver | 高频访问的用户偏好缓存 |
| thread_id | 每个聊天窗口的唯一 ID |
| Store (aput/aget) | 记住用户偏好：语言、风格、兴趣标签 |
| Store (asearch) | 语义检索用户的历史记忆 |
| 混合架构 | Redis 缓存偏好，Postgres 持久化对话 |
| 上下文裁剪 | 超长对话自动裁剪，显示 token 用量 |

#### 代码演示的讲解话术

**Step 1 - 基础持久化**：
> "看这段代码——我们给图加了一个 `PostgresSaver`。现在用户说『你好，我叫小明』。然后用户刷新页面，再次发送『我喜欢 Python』。注意看这个 thread_id——两次调用用了同一个 thread_id，所以第二次调用时，LangGraph 自动从 Postgres 加载了上一次的 Checkpoint，LLM 知道『你叫小明』。"

**Step 2 - 长期记忆 Store**：
> "Checkpoint 记住了『这次对话里发生了什么』，但它不会主动提炼出『用户的偏好』。所以我们要用 Store。看这里——当用户说『我喜欢简洁的回答』时，我们在 `preference_store` 节点里执行 `store.aput(user_id, "preferences", {"style": "concise"})`。这个信息就永久保存了。"

**Step 3 - 混合架构**：
> "现在看生产环境的混合架构。用户偏好数据——每次对话都要查——存在 Redis 里，1 毫秒就能拿到；完整的对话历史——只有用户翻记录时才查——存在 Postgres 里。如果 Redis 挂了，系统会自动降级到直接从 Postgres 读。"

**Step 4 - 上下文裁剪**：
> "这个用户聊了 100 轮，消息历史已经 8000 tokens 了。看 `trim_messages` 的效果——它自动保留了最近的 4000 tokens，把早期的消息裁剪掉了。前端这里显示了一个 token 用量仪表盘，绿色表示充足，黄色表示接近限制，红色表示已经裁剪。"

#### 关键转折点（需要停下来确认学员理解）

1. **Checkpoint vs Store**：问 "用户的对话历史存在哪里？用户的偏好设置存在哪里？"（期望：历史在 Checkpoint，偏好在 Store）
2. **thread_id 的作用**：问 "两个用户使用同一个 thread_id 会怎样？"（期望：他们会看到对方的对话历史）
3. **混合架构一致性**：问 "Redis 和 Postgres 数据不一致怎么办？"（期望：以 Postgres 为准，Redis 只是缓存）

---

### 4. 常见误区与注意事项

| 坑 | 现象 | 预警话术 |
|----|------|---------|
| **忘记传 config** | `thread_id` 没传，每次调用都是新对话 | "持久化必须传 `config={\"configurable\": {\"thread_id\": \"...\"}}`，否则每次调用都是全新 State。" |
| **Checkpoint 和 Store 混用** | 把应该长期保存的数据放到 Checkpoint | "Checkpoint 是执行快照，有清理策略；Store 是长期存储。用户偏好、知识库用 Store，对话历史用 Checkpoint。" |
| **Store 不用 user_id 隔离** | 用户 A 看到了用户 B 的数据 | "Store 的 aput/aget 必须传入 user_id 作为命名空间，这是数据隔离的底线。" |
| **Redis 当主数据库** | Redis 重启后数据丢失 | "Redis 是缓存层，不是持久层。关键数据必须同步到 Postgres。" |
| **上下文裁剪太激进** | LLM 丢失了关键上下文 | "裁剪策略要平衡——保留系统消息、最近的 N 轮、以及摘要化的早期内容。" |

#### 调试技巧

1. **查看 Checkpoint**：直接查 Postgres `checkpoints` 表，看保存的 State 结构
2. **Store 数据检查**：`store.aget(user_id, key)` 验证数据是否写入
3. **Token 计数器**：在 State 中维护 `token_count` 字段，实时显示用量

#### 核心要点回顾

1. **Checkpoint = 执行快照**：自动保存图执行状态，支持崩溃恢复
2. **Saver = 持久化后端**：MemorySaver（开发）、PostgresSaver（生产持久化）、RedisSaver（高速缓存）
3. **thread_id = 会话标识**：同一个 thread_id 自动续接对话
4. **Store = 长期记忆**：用户偏好、知识、历史记录，支持语义检索
5. **混合架构 = 速度 + 安全**：Redis 热数据 + Postgres 冷数据
6. **上下文工程 = 成本控制**：trim、摘要、过滤，保持上下文在合理大小

#### 记忆口诀

> **"Checkpoint 是存档，Store 是笔记本；thread_id 是窗口，Redis+Postgres 是双保险。"**

## LG-05：人机协作与 Hooks
**预计时长**: 4-5 小时（含 2 小时实战）

---

### 1. 开场引入（5-10 分钟）

#### 引入策略：从「AI 失控」引入

**开场白建议**：
> "假设你是一家电商公司的运营。你用 LangGraph 做了一个自动发布营销文案的系统。AI 生成了文案，直接自动发到官方微博、微信公众号、抖音...
>
> 然后 AI 把『五一促销』写成了『五一大促，全场免费送』——因为训练数据里的某个梗。等你看手机时，已经 10 万转发，公司准备赔钱了。
>
> 这就是『完全自动化』的风险。在很多场景下，AI 可以生成，但人类必须审批。今天的主题是：**让图学会『停下来问人』**。"

#### 生活化类比：电梯的急停按钮

> "自动扶梯是自动运行的，但每部扶梯都有一个红色的急停按钮。紧急情况下，任何人都可以按下它，让扶梯停下来。
>
> HITL（Human-in-the-Loop）就是这个急停按钮——图在正常执行，但在关键节点，它可以停下来等人类输入，然后再决定是继续、修改还是终止。"

#### 与 LG-04 的衔接

> "上节课我们让图有了记忆。今天我们要让图学会『请示』——在关键时刻暂停执行，等人类批准后继续前行。"

### 2. 概念讲解顺序（核心部分，约 150 分钟）

#### 2.1 `interrupt()` 原语（30 分钟）

**讲解顺序**：
1. **概念**：在节点中抛出中断，等待人类输入（5 分钟）
2. **代码演示**：`def approval_node(state): raise interrupt({"type": "approval", "content": ...})`（15 分钟）
3. **中断后的状态**：图暂停，State 已持久化到 Checkpoint（10 分钟）

**类比**："`interrupt()` 就像你在做菜时突然发现酱油没了——你停下来，喊一嗓子『谁帮我买瓶酱油』，然后站在原地等。菜还在锅里（State 已保存），但你不再执行下一步。"

---

#### 2.2 `interrupt_before` / `interrupt_after`（25 分钟）

**讲解顺序**：
1. **`interrupt_before`**：节点执行前中断（比如审批员先看输入再决定让不让 AI 处理）（10 分钟）
2. **`interrupt_after`**：节点执行后中断（比如 AI 生成完后让人类确认）（10 分钟）
3. **配置方式**：在 `compile()` 时传入（5 分钟）

**对比演示**：
```python
# 方式 1：在 compile 时配置
graph = builder.compile(
    checkpointer=checkpointer,
    interrupt_before=["publish"],  # 发布前必须审批
    interrupt_after=["generate"]   # 生成后让人确认
)

# 方式 2：在节点内手动 raise interrupt
```

---

#### 2.3 `Command(resume=...)` 恢复（25 分钟）

**讲解顺序**：
1. **恢复机制**：人类输入后，用 `Command(resume=value)` 继续执行（10 分钟）
2. **resume 值的传递**：被中断的节点收到 resume 值作为输入（10 分钟）
3. **可视化演示**：中断→等待→恢复→继续执行的时间线（5 分钟）

**代码演示**：
```python
# 用户审批通过
graph.invoke(Command(resume={"decision": "approve"}), config)

# 用户要求修改
graph.invoke(Command(resume={"decision": "revise", "feedback": "语气太正式了"}), config)
```

---

#### 2.4 Pre/Post Model Hooks（35 分钟）

**讲解顺序**：
1. **Pre-model Hook**：LLM 调用前的预处理（注入上下文、安全检查）（15 分钟）
2. **Post-model Hook**：LLM 调用后的处理（输出审查、格式化）（15 分钟）
3. **`llm_input_messages`**：临时注入，不修改 State（5 分钟）

**类比**："Pre-hook 就像给 LLM 戴耳机——在它『听』问题之前，你给加点提示；Post-hook 就像给 LLM 装个过滤器——它『说』完之后，你先检查一下再放出去。"

#### 2.5 Guardrails 与安全（15 分钟）

**讲解顺序**：
1. 输出安全审查的基本策略
2. `RemoveMessage` 用于删除/替换消息
3. HITL 在安全流程中的位置

### 3. 案例驱动逻辑：ContentGuard

#### 案例与概念的一一对应

| 概念 | ContentGuard 中的体现 |
|------|----------------------|
| interrupt_before | 高风险操作（发布到官方账号）前要求审批 |
| interrupt_after | AI 生成文案后让人类确认 |
| Command(resume) | 审批员选择「通过/拒绝/修改」后恢复执行 |
| Pre-model Hook | 调用 LLM 前注入品牌规范和安全提示 |
| Post-model Hook | LLM 输出后进行敏感词检查和格式化 |
| llm_input_messages | 临时注入当前活动的促销规则（不保存到 State） |

#### 代码演示的讲解话术

**Step 1 - 生成文案**：
> "首先 AI 生成营销文案。大家看 `generate` 节点——它调用 LLM 写文案。但因为我们在 compile 时配置了 `interrupt_after=["generate"]`，所以这个节点执行完后，图会自动暂停。"

**Step 2 - 人工审批**：
> "现在前端弹出了审批面板。审批员看到 AI 生成的文案：『五一大促，全场免费送』。审批员觉得『免费送』有风险，点击『修改』，输入反馈：『把免费送改成满 100 减 50』。"

**Step 3 - 恢复执行**：
> "前端发送 `Command(resume={"decision": "revise", "feedback": "..."})`。图从中断点恢复，这个 resume 值被传给下一个节点。`revise` 节点收到反馈后，调用 LLM 重新生成文案。"

**Step 4 - 发布前再次审批**：
> "修改后的文案再次经过 `interrupt_after=["generate"]` 的审批。这次审批员通过了。然后图走到 `publish` 节点——但我们在 compile 时配置了 `interrupt_before=["publish"]`，所以在正式发布前，还有一次最终确认。"

**Step 5 - Hooks 演示**：
> "再看 Pre/Post Hooks。每次调用 LLM 前，Pre-hook 自动注入『品牌规范』：『我们是高端家电品牌，语气要优雅，不要用网络梗』。LLM 输出后，Post-hook 自动检查敏感词和合规性。"

#### 关键转折点（需要停下来确认学员理解）

1. **中断后的 State**：问 "图中断后，之前节点的执行结果还在吗？"（期望回答：在，已保存到 Checkpoint）
2. **resume 的传递**：问 "Command(resume) 的值被谁接收？"（期望回答：被中断的节点或下一个节点）
3. **interrupt_before vs after**：给出 3 个业务场景，让学员判断用 before 还是 after

### 4. 常见误区与注意事项

| 坑 | 现象 | 预警话术 |
|----|------|---------|
| **忘记 checkpointer** | interrupt 后无法恢复，State 丢失 | "HITL 必须配 checkpointer！没有持久化，中断后就丢失了。" |
| **interrupt 和异常混用** | 用 raise Exception 代替 interrupt | "Exception 会终止执行，interrupt 是暂停。HITL 必须用 `interrupt()` 原语。" |
| **resume 值格式不对** | 被中断节点收不到 resume 值 | "resume 可以是任意类型，但要和被中断节点的期望输入匹配。" |
| **Hooks 里修改 State** | Pre-hook 改了 State，影响后续节点 | "Pre-hook 应该用 `llm_input_messages` 临时注入，不要直接改 State。" |
| **审批流设计太复杂** | 一个流程里中断 5 次，用户体验差 | "HITL 是『安全阀』不是『每一步都问』。只在高风险/高成本操作前中断。" |

#### 调试技巧

1. **模拟审批**：开发时用 `Command(resume=...)` 模拟不同审批结果
2. **查看中断状态**：`graph.get_state(config)` 查看当前是否处于中断状态
3. **审计日志**：所有 HITL 操作（谁、什么时候、做了什么决定）记录到 Store

### 5. 阶段小结（5 分钟）

#### 核心要点回顾

1. **interrupt() = 暂停**：图在节点内或节点前后暂停，等待人类输入
2. **Command(resume) = 继续**：人类输入后恢复执行，resume 值传给后续节点
3. **interrupt_before/after = 声明式中断**：在 compile 时配置，无需改节点代码
4. **Pre-hook = 预处理**：LLM 调用前注入上下文、安全检查
5. **Post-hook = 后处理**：LLM 输出后审查、格式化
6. **HITL 设计原则**：只在关键时刻中断，审批流要简洁

#### 记忆口诀

> **"interrupt 是暂停键，resume 是播放键；Hooks 是耳机和滤镜，HITL 是安全网。"**

## LG-06：并行执行与子图
**预计时长**: 5-6 小时（含 2.5 小时实战）

---

### 1. 开场引入（5-10 分钟）

#### 引入策略：从「等待焦虑」引入

**开场白建议**：
> "假设你要写一份行业研究报告。需要做四件事：查财报数据、查新闻舆情、查竞品动态、查专家观点。如果你一个人做，要一件一件来——做完第一件再做第二件，总时间是 4 个任务之和。
>
> 但如果你有一个团队：你负责统筹，4 个研究员同时分头查，你在汇总点等他们都完成后整合报告。总时间就是最慢的那个研究员的时间。
>
> 这就是并行执行的意义。今天的主题是：**让图学会『分工合作』**。"

#### 生活化类比：餐厅后厨的并行出餐

> "你去 restaurant 点了一份套餐：主菜、配菜、汤、饮料。后厨不会等主菜做完了才做配菜——四个厨师同时开工，各自负责一道，然后在『出餐台』汇聚成一份完整的套餐端给你。
>
> 并行节点就是这四个厨师，Fan-in 汇聚点就是出餐台。"

#### 与 LG-05 的衔接

> "上节课我们学会了在关键节点让人类审批。今天我们要加速图的执行——让独立的任务并行跑，让复杂的系统模块化拆分。"

---

### 2. 概念讲解顺序（核心部分，约 180 分钟）

#### 2.1 Fan-out / Fan-in（30 分钟）

**讲解顺序**：
1. **Fan-out**：一个节点同时触发多个下游节点（10 分钟）
2. **Fan-in**：多个节点都完成后，汇聚到一个节点（10 分钟）
3. **LangGraph 的自动 Fan-in**：多个上游边指向同一个节点即自动汇聚（10 分钟）

**必须画图的点**：
```
           ┌─→ [查财报] ──┐
[主题输入] ─┼─→ [查新闻] ──┼─→ [汇总报告] → END
           ├─→ [查竞品] ──┤
           └─→ [查专家] ──┘
            ↑ Fan-out      ↑ Fan-in
```

**类比**："Fan-out 像是一个项目经理同时给 4 个人发邮件分配任务；Fan-in 像是项目经理在 Excel 里设置了条件格式——等 4 个人都回复了，这行才变绿。"

---

#### 2.2 Reducer 机制（35 分钟）

**讲解顺序**：
1. **为什么需要 Reducer**：多个节点同时修改同一个 State 字段（10 分钟）
2. **`operator.add`**：列表/数值累加（10 分钟）
3. **自定义 Reducer**：去重、保留最近 N 个、排序合并（15 分钟）

**代码演示**：
```python
from operator import add

class AgentState(TypedDict):
    # 多个并行节点都往 findings 里加内容，自动合并
    findings: Annotated[list, add]
    # 自定义 reducer：保留最近的 10 条
    logs: Annotated[list, lambda x, y: (x + y)[-10:]]
```

**关键话术**：
> "并行节点都往同一个字段写数据时，如果没有 Reducer，就会『写覆盖』——最后一个完成的节点把前面节点的数据全盖掉了。Reducer 是并行的『和平协议』——规定大家怎么合并结果。"

---

#### 2.3 `Send` 动态 Map-Reduce（25 分钟）

**讲解顺序**：
1. **什么是 Map-Reduce**：动态分发子任务 → 并行执行 → 汇聚结果（5 分钟）
2. **`Send` API**：`Send("target_node", {"key": value})`（10 分钟）
3. **动态分发**：运行时决定分发几个子任务（10 分钟）

**类比**："Map-Reduce 像是一个老师批改作文——有 50 篇作文，分给 5 个助教，每人 10 篇，最后汇总成绩。`Send` 就是『分发』这个动作。"

---

#### 2.4 Subgraph 构建（25 分钟）

**讲解顺序**：
1. **为什么需要子图**：模块化、复用、隔离复杂度（5 分钟）
2. **子图的定义**：独立的 `StateGraph`，有自己的 State Schema（10 分钟）
3. **子图作为节点**：`builder.add_node("subgraph", subgraph.compile())`（10 分钟）

---

#### 2.5 子图状态隔离与映射（35 分钟，v1.1 新增重点）

**讲解顺序**：
1. **状态隔离**：子图拥有独立的 State Schema（10 分钟）
2. **`input` / `output` 映射**：父图与子图的字段映射配置（15 分钟）
3. **共享 state 字段**：通过公共字段或 Store 实现子图间通信（10 分钟）

**必须画图的点**：

### 3. 案例驱动逻辑：ResearchForge

#### 案例与概念的一一对应

| 概念 | ResearchForge 中的体现 |
|------|----------------------|
| Fan-out | 主题输入后并行分发给 4 个数据源 |
| Fan-in | 4 个数据源都完成后才进入汇总节点 |
| Reducer | 各分支的 findings 用 `add` 合并 |
| Send | 动态分发多个子查询到搜索节点 |
| Subgraph | 每个数据源是一个独立子图（财经子图、新闻子图等） |
| input/output 映射 | 父图的 "topic" → 子图的 "query" |
| 状态隔离 | 各子图有自己的中间状态，互不干扰 |

#### 代码演示的讲解话术

**Step 1 - Fan-out 并行检索**：
> "看这张图——输入主题『新能源汽车』后，4 个箭头同时亮起来。这就是 Fan-out。`parse_intent` 节点有 4 条出边，分别指向 4 个数据源节点。LangGraph 会自动并行执行这 4 个节点。"

**Step 2 - Reducer 合并结果**：
> "4 个节点并行执行，每个节点都会往 `findings` 里加数据。注意 State 定义——`findings: Annotated[list, add]`。`add` Reducer 保证了 4 个节点的结果不会互相覆盖，而是自动合并成一个列表。"

**Step 3 - Fan-in 汇聚**：
> "4 个节点都完成后，`compile_report` 节点才会执行。这就是 Fan-in——LangGraph 自动等待所有前置节点完成。注意看这张图，`compile_report` 的入度是 4，4 条边都『变绿』后它才开始。"

**Step 4 - 子图结构**：
> "点击这个『财经数据』节点，展开它的内部结构。大家看——它不是一个简单的函数，而是一个完整的子图！有自己的 State（`FinanceState`）、自己的节点（`fetch_stock` → `analyze_trend`）、自己的边。子图执行完后，通过 output 映射把 `results` 传回父图的 `findings`。"

**Step 5 - 动态 Map-Reduce**：
> "再看这个动态分发场景。系统分析了主题后，发现需要查 7 个公司的数据。用 `Send` 动态分发 7 个子任务：`[Send("fetch_company", {"company": c}) for c in companies]`。这 7 个任务并行执行，结果通过 Reducer 合并。"

#### 关键转折点（需要停下来确认学员理解）

1. **Reducer 必要性**：问 "如果 4 个并行节点都返回 `{"result": "..."}`，不用 Reducer 会怎样？"（期望回答：互相覆盖，只有最后一个结果保留）
2. **Fan-in 触发条件**：问 "如果 4 个并行节点中有一个失败了，Fan-in 会触发吗？"（期望回答：LangGraph 默认等待所有节点，失败可以用 try/except 或特殊 Reducer 处理）
3. **子图 vs 普通节点**：问 "子图和普通函数节点相比，优势是什么？"（期望回答：模块化、复用、状态隔离、内部复杂度隐藏）

### 4. 常见误区与注意事项

| 坑 | 现象 | 预警话术 |
|----|------|---------|
| **并行节点修改同一个非 Reducer 字段** | 数据互相覆盖 | "并行节点共享的字段必须用 Annotated + Reducer，否则是竞态条件。" |
| **Fan-in 节点前置有循环** | Fan-in 触发条件不明确 | "如果并行分支里有循环，确保循环有明确终止条件，否则 Fan-in 永远等不到。" |
| **子图 State 与父图不兼容** | 字段映射错误，数据传不过去 | "子图的 input/output 映射必须显式配置，字段名不一致时不会自动匹配。" |
| **Send 分发太多子任务** | 内存/连接池耗尽 | "Send 是动态分发，要做上限控制。`max_concurrency` 限制同时执行的子任务数。" |
| **递归限制不当** | 并行 + 循环导致超级步过多 | "并行图更容易触达 recursion_limit，生产环境要合理设置。" |

#### 调试技巧

1. **逐个节点验证**：先不并行，串行跑通每个节点，再改成并行
2. **Reducer 验证**：打印合并前后的 State，确认没有数据丢失
3. **子图单独测试**：每个子图作为独立图先跑通，再嵌入父图

### 5. 阶段小结（5 分钟）

#### 核心要点回顾

1. **Fan-out = 并行分发**：一个节点触发多个下游节点同时执行
2. **Fan-in = 汇聚等待**：多个节点都完成后才触发下一个节点
3. **Reducer = 并行和平协议**：定义多个节点同时修改同一字段时的合并策略
4. **Send = 动态 Map-Reduce**：运行时决定分发多少子任务
5. **Subgraph = 模块化**：独立 State、独立逻辑、可复用、可嵌套
6. **input/output 映射 = 父子通信**：显式配置字段传递规则

#### 记忆口诀

> **"Fan-out 是群发，Fan-in 是等齐人；Reducer 是合并器，Send 是动态派活，Subgraph 是模块包。"**

#### 完整架构映射

```
用户输入
   ↓
[记忆助手] ──→ 加载用户历史偏好 + 长期记忆 (Store)
   ↓
[模式路由] ──→ 判断复杂度
```

| 概念 | DeepResearch 中的体现 |
|------|----------------------|
| 模式路由 | 根据问题复杂度选择 react / deep |

| 节点级缓存 | 搜索/分析/报告不同 TTL |
| 任务跟踪 | `@track_execute` 追踪所有节点 |
| max_executions | 防止循环无限执行的安全阀 |
| 子图 | 搜索子图、分析子图、报告子图 |
| 并行 | 信息搜索阶段的多源并行检索 |
| Stream | 任务进度实时推送到前端 |
| Store | 记忆助手加载用户长期偏好 |

#### 代码演示的讲解话术

**Step 1 - 模式路由决策**：
> "看用户输入：『分析新能源汽车行业』。模式路由节点运行——它分析这个查询：涉及多个维度、需要数据支撑、不是简单事实查询 → 判断为『复杂』→ 走 deep 分支。如果是『北京今天天气』，就会走 react 快答分支。"

**Step 2 - 深度研究循环**：
> "进入 deep 分支后，首先任务规划节点生成研究计划。然后信息搜索节点并行检索多个数据源。搜索完成后，计划路由节点评估：『信息足够了吗？』→ 不够 → 返回继续搜索。这就是循环。"

**Step 3 - 计划路由动态调整**：
> "第三轮搜索后，计划路由发现已经有了足够的市场数据，但缺少竞品对比。于是它决定下一步不是『继续搜索』，而是『数据分析』。注意——计划是动态调整的，不是一开始定死的。"

**Step 4 - 缓存效果演示**：
> "第一次搜索某数据源花了 3 秒，API 费用 $0.02。5 分钟后另一个用户问类似问题，再次走到搜索节点——缓存命中！耗时 0.01 秒，费用 $0。看面板上的缓存统计：节省 97% 的 API 调用。"

**Step 5 - 任务跟踪看板**：
> "所有节点都有 `@track_execute` 装饰。看这个时间线——每个节点的状态变化一目了然。这个节点 pending 了 0.1s，running 了 2.3s，completed。整体进度 75%。"

#### 关键转折点（需要停下来确认学员理解）

1. **模式路由 vs 计划路由**：问 "模式路由和计划路由的区别是什么？各在哪个层级工作？"（期望：模式路由在顶层选分支，计划路由在 deep 分支内做循环决策）
2. **循环终止条件**：问 "计划路由循环怎么保证不无限执行？"（期望：max_executions + 状态判断 + 内容饱和度评估）
3. **缓存 key 设计**：问 "不同用户的相同查询，缓存应该共享吗？"（期望：看业务需求，通用数据可共享，个性化数据要隔离）

### 4. 常见误区与注意事项

| 坑 | 现象 | 预警话术 |
|----|------|---------|
| **模式路由判断不准** | 简单问题走了 deep，浪费资源；复杂问题走了 react，回答质量差 | "模式路由的决策逻辑要持续优化，可以用历史数据训练分类器，不要只靠关键词匹配。" |
| **计划路由循环不收敛** | 永远觉得『信息不够』，无限循环 | "必须设 max_executions + 信息饱和度评估函数，强制循环收敛。" |
| **缓存 key 过于宽泛** | 不同查询命中同一缓存，返回错误结果 | "缓存 key 要包含查询语义、用户标识（如果需要隔离）、时间窗口。" |
| **任务跟踪性能开销** | `@track_execute` 本身带来延迟 | "跟踪数据异步写入，不要阻塞主流程；生产环境可以采样跟踪。" |
| **子图状态共享不当** | 各子图互相影响状态 | "子图优先用状态隔离，通过显式 input/output 映射传递数据。" |

#### 调试技巧

1. **模式路由日志**：记录每个查询的路由决策和原因，定期分析误判
2. **计划路由轨迹**：保存每次循环的计划状态和决策，可视化循环路径
3. **缓存命中率监控**：低于 30% 要检查 key 设计；高于 90% 要检查是否过度缓存

### 5. 阶段小结（5 分钟）

#### 核心要点回顾

1. **模式路由 = 顶层分诊**：根据复杂度选择执行策略（快答 vs 深度）
2. **计划路由 = 动态调度**：循环执行，根据中间结果调整下一步
3. **节点级缓存 = 成本优化**：不同节点不同 TTL，精准控制
4. **任务跟踪 = 可观测性**：每个节点的状态实时可见
5. **max_executions = 安全阀**：防止无限循环的底线
6. **DeepResearch = 综合应用**：前面 6 个阶段知识的终极组合

#### 记忆口诀

> **"模式路由选赛道，计划路由跑接力；缓存省钱跟踪看进度，max_executions 保安全。"**

### 1. 开场引入（5-10 分钟）

#### 引入策略：从「上线恐惧」引入

**开场白建议**：
> "假设你花了两个月做了一个 DeepResearch Agent，本地跑得很好。老板让你明天上线。你突然想到：
> - 系统崩溃了怎么办？有监控吗？
> - API 费用暴涨怎么办？能预警吗？
> - Prompt 里的『系统提示词』改了 20 版，现在用的是哪一版？
> - 用户投诉回答质量差，你能复现当时的情况吗？
> - 开发环境、测试环境、生产环境的配置怎么隔离？
>
> 这就是生产部署的现实。今天的主题是：**从『能跑』到『能上线』**。"

#### 生活化类比：餐厅后厨监控

> "想象你是一家连锁餐厅的老板。后厨里有 20 个厨师同时做菜，你需要知道：
> - **Trace = 订单追踪**：3 号桌的宫保鸡丁现在做到哪一步了？厨师 A 在切菜，厨师 B 在炒菜，预计还有 5 分钟出锅。
> - **Score = 菜品评分**：今天宫保鸡丁的平均评分 4.5 分，但有 3 个客人说太咸。是哪个厨师做的？
> - **Dataset = 顾客反馈收集**：把每天的『差评订单』自动收集到一个本子里，每周复盘改进。
> - **Prompt = 菜谱管理**：宫保鸡丁的菜谱改了 3 版——1 版太辣、2 版太甜、3 版刚好。所有分店都要用第 3 版。
>
> Langfuse 就是后厨的『监控大屏』+『评分系统』+『反馈收集器』+『菜谱管理中心』。"

#### 与 LG-07 的衔接

> "上节课我们解析了 DeepResearch Agent 的架构——它能跑、功能完整。今天我们要让它『能上线』——可监控、可追踪、可回滚、可优化。"

4. **自动拉取**：本地不存在时从 Hub 自动拉取（5 分钟）
5. **推送发布**：本地推送到 Hub，自动打 `labels: ["production"]`（5 分钟）

**必须画 Prompt 工作流图的点**：
```
开发环境                    Langfuse Hub                 生产环境
┌─────────┐                ┌─────────────┐             ┌─────────┐
│ YAML v1 │ ──push──→      │ Prompt Hub  │ ←──pull──   │ YAML v1 │
│ (本地编辑)│                │ (版本管理)   │             │ (运行时) │
└────┬────┘                └──────┬──────┘             └─────────┘
     │                           │

┌─────────┐                ┌─────────────┐
```

> 要回滚？把上一版打上 production 标签，生产环境自动切回来。要 A/B 测试？v1 打标签给 A 组，v2 给 B 组。"

### 3. 案例驱动逻辑：OpsMate

#### 案例与概念的一一对应

| 概念 | OpsMate 中的体现 |
|------|-----------------|
| 预构建 vs 自定义对比 | 同一查询两种实现，实时展示路径差异 |
| 缓存监控 | 节点级缓存命中/未命中的实时色块矩阵 |
| Redis + Postgres | 双后端健康看板，模拟故障降级 |
| RAG | 运维知识库向量检索，相似度热力图 |
| Langfuse Trace | 每次图执行的完整链路瀑布图 |
| Langfuse Score | 缓存命中率、输出质量、响应时间时序图 |
| Langfuse Dataset | QA Capture 采集的数据集管理 |
| Langfuse Tags | 按环境/服务/用户筛选 Trace |
| Prompt YAML | 本地 Prompt 版本管理 |
| Prompt Hub 同步 | 本地 vs 云端版本差异对比、同步状态灯 |

#### 代码演示的讲解话术

**Step 1 - Langfuse Trace 演示**：
> "看 Langfuse 的 Trace 大屏。这条 Trace 是用户『查服务器 CPU 告警』的请求。点击展开——大家看这个时间瀑布图：意图识别 23ms、模式路由 15ms、LLM 调用 312ms... 总耗时 1.2s。哪个节点最慢？一目了然。"

**Step 2 - Score 看板**：
> "看 Score 看板。昨天的缓存命中率平均 87%，但凌晨 3 点掉到了 45%。点击那个时间点——发现是一次批处理任务触发了大量冷查询。这就是 Score 的业务价值：不只是技术数字，是『成本预警器』。"

**Step 3 - Dataset QA Capture**：
> "这个 Dataset 是自动采集的。每次用户查询 + 系统回答 + 用户反馈（👍/👎）自动保存到这里。我们可以选中差评数据，一键重新运行评估——看改了 Prompt 后，同样的输入输出质量有没有提升。"

**Step 4 - Prompt 同步演示**：
> "看 Prompt 同步看板。左边是本地 YAML 文件的内容，右边是 Langfuse Hub 上的版本。红色高亮表示『本地删除了这行』，绿色表示『本地新增了这行』。同步状态灯是黄色——表示『本地有更新，待推送』。点击一键推送，灯变绿色。"

**Step 5 - 故障模拟**：
> "现在模拟 Redis 故障。看双后端监控面板——Redis 状态从绿色变成红色，系统自动降级到 Postgres。虽然查询延迟从 2ms 变成了 50ms，但服务没有中断。"

#### 关键转折点（需要停下来确认学员理解）

1. **Trace vs Score vs Dataset**：问 "Trace、Score、Dataset 三者的关系是什么？各自解决什么问题？"（期望：Trace 是单次执行的详细记录，Score 是聚合指标，Dataset 是结构化数据集用于评估）
2. **Prompt 版本管理**：问 "生产环境出问题了，怎么快速回滚 Prompt？"（期望：在 Langfuse Hub 把上一版打上 production 标签，应用自动拉取）
3. **缓存策略权衡**：问 "TTL 设太长和太短各有什么问题？"（期望：太长→数据过期；太短→缓存命中率低、API 费用高）

### 4. 常见误区与注意事项

| 坑 | 现象 | 预警话术 |
|----|------|---------|
| **Langfuse 没配环境变量** | 追踪数据没上报 | "Langfuse 的 `secret_key` 和 `public_key` 必须配好，且不同环境用不同 project。" |
| **Prompt 本地和 Hub 不一致** | 生产环境用了旧版 Prompt | "上线前必须确认 Prompt 同步状态灯是绿色。黄色=待推送，红色=Hub 有新版本待拉取。" |
| **缓存雪崩** | 大量缓存同时过期，瞬间打爆 API | "TTL 加随机偏移（±10%），避免同时过期；热点数据用互斥锁防穿透。" |
| **RAG 检索质量差** | 检索结果不相关，影响生成质量 | "RAG 效果要持续评估：命中率、相关性评分、Bad Case 收集。检索不行，生成肯定不行。" |
| **状态太大导致性能下降** | Checkpoint 保存/恢复越来越慢 | "定期裁剪 State，大文件（如图片）存对象存储，State 里只存引用。" |
| **忘记区分环境** | 开发环境的 Trace 混到生产环境 | "Tags 里必须打 env:dev/test/prod，Langfuse 项目也要分开。" |

#### 调试技巧

1. **Langfuse Trace 排查**：找到慢请求 → 看瀑布图找瓶颈节点 → 优化该节点
2. **Prompt 版本对比**：本地 vs Hub 并排对比，快速定位差异
3. **缓存效果验证**：对比「有缓存」和「无缓存」的响应时间和成本
4. **RAG 效果评估**：定期跑评测集，看命中率和相关性趋势

#### 核心要点回顾

1. **Langfuse Trace = X 光片**：每次请求的完整执行链路，精准定位问题

6. **生产部署 = 全流程**：Docker 容器化、健康检查、性能调优、安全加固

> **"Trace 看链路，Score 看指标，Dataset 做评估；YAML 管本地，Hub 管云端，同步保一致。"**
