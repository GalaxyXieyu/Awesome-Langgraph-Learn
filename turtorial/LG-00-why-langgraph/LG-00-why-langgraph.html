本节课定位是导论，不跑 Demo，先建立选型认知。共 6 个主题，从企业需求出发，对比各框架，最终给出务实建议。
The Gap
Demo 很美好，生产很残酷
"Demo 要证明 AI 能做，企业级要确保 AI 能一直可靠地做。"

第一层：业务流程不是一次性的

  • 用户填到一半的表单明天再打开 —— 状态必须能中断和恢复
  • 审批不通过要打回重走 —— 流程不能写死成线性步骤
  • 涉及金额、合规、安全的决策 —— 必须有人工审核

第二层：系统不会永远稳定

  • 进程重启、容器漂移 —— 状态必须持久化到数据库
  • 一个 Agent 不可能精通所有领域 —— 必须多 Agent 协作

第三层：出了问题不能靠猜

  • AI 黑盒推理无从复现 —— 必须全链路追踪、可视化调试
  • 任务跑 30 秒没反馈 —— 必须流式输出实时进度
LangGraph 教程 3 / 9
把 AI 塞进真实业务流程，三层问题会逐个找上门。企业级买的不是 AI 的能力，而是 AI 的可靠性、可控性和可维护性。
Framework Landscape
主流 Agent 框架一句话定位

LangGraph 本课主角

基于有向图的 Agent 工作流编排。状态驱动，传统业务与 AI 代码平等编排。需要精确控制流程、状态透明时选它。

CrewAI

角色扮演驱动的多 Agent 协作。角色抽象自然，但流程控制以角色为核心，持久化和图级状态管理薄弱。

Agno（原 Phidata）

轻量级 Agent 构建工具包，带 AgentOS 运行时。极致速度和低内存，但 checkpoint 基于 session 而非流程级。

AutoGen（Microsoft）

对话驱动的多 Agent 框架。Agent 之间通过对话协作，多 Agent 聊天场景是强项，但对话模型强耦合。

Pydantic AI

类型安全的 Agent 框架。Pydantic 重度用户适用，但生态较新，企业级能力还在发展。

LlamaIndex Workflows

LlamaIndex 生态的工作流编排。与 LlamaIndex 强绑定，通用性不如 LangGraph。

LangGraph 教程 5 / 9
注意"传统业务接入"这一行，这是 LangGraph 和其他框架的本质区别。LangGraph 是编排层，不是封闭的 Agent 黑盒。
Reason 1
图级别的原生深度
其他框架在单点上并不弱，但企业级需要的是一整套能力的系统性、原生深度集成。

状态在节点间显式传递

每个节点知道完整上下文，不是隐式传递或黑盒内部状态。

Checkpoint 精确到任意节点

服务重启后可以从任意位置恢复，不是 session 级别粗粒度存档。

循环和条件是图结构本身

可视化、可维护、防死循环，不是嵌套在函数里的控制流。

7 项能力彼此连贯

全部内建在图的核心层，不是插件拼凑的松散组合。

LangGraph 教程 6 / 9
状态、持久化、人机协作、循环控制、可视化、多 Agent、流式输出——7 项能力在 LangGraph 里是一体化的，彼此连贯。
Reason 2
LangGraph 是编排层，不是封闭的 Agent 黑盒
企业级项目从来不是纯 AI 的。你的系统里已有几百万行跑了多年的业务代码。

LangGraph：以图为核心

  • 节点可以是任何 Python 函数
  • 传统业务代码一行不改，直接放进去就是节点
  • 传统代码和 AI 代码是平等的一等公民

其他框架：以 Agent 为中心

  • 传统业务代码只能作为工具或回调硬塞进去
  • AI 步骤要定义成 Agent
  • 流程可见性和可控性大打折扣
电商订单示例：用户下单 → 查询库存（传统 SQL）→ AI 评估风险 → 调用支付（传统 API）→ 高风险时人工审核 → 发送通知（传统短信接口）。在 LangGraph 里，这 6 个步骤在一张图里平铺直叙。
LangGraph 教程 7 / 9
这意味着三件事：代码复用率极高，开发效率高（传统开发和 AI 开发可以并行），维护成本低（业务逻辑和 AI 逻辑在一张图里清晰可见）。
When Not to Use
LangGraph 不是银弹
场景 推荐框架 理由
5 分钟搭个 Demo Agno 最快上手，代码最少
纯角色扮演协作 CrewAI 角色抽象最自然
Java 技术栈 Spring AI 生态一致
多 Agent 聊天实验 AutoGen 对话模型成熟
极致性能/低内存 Agno 官方宣称 5000x 更快启动
"对于 99% 的公司来说，框架性能不是瓶颈。能不能快速上手、能不能找到解决方案、团队能不能协同开发，这些才是瓶颈。"
LangGraph 教程 8 / 9
给新手的务实建议：先用 LangGraph 把系统搭起来跑通业务，等真有性能瓶颈时再考虑优化。迁移成本远小于"因为选了个冷门框架，团队卡住半年上不了线"的成本。
Summary
阶段小结

核心认知

  • 框架选型看场景：没有最好的，只有最合适的
  • LangGraph 核心优势：图级原生深度 + 传统业务零改动 + 最成熟生态
  • 适用场景：复杂工作流、精确控制、生产级部署、已有系统升级

务实策略

  • 先选生态最成熟的框架把系统跑起来
  • 等真有性能瓶颈时再考虑优化
  • 纠结"哪个更快"不如纠结"哪个能让我这周上线"
