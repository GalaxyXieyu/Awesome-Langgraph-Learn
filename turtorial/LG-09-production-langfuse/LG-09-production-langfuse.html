// part 01 · cache & architecture

缓存策略与Redis + Postgres混合架构平衡速度与安全

缓存策略三要素

CachePolicy 深化配置

TTL 过期、主动失效、版本号控制

缓存失效策略

TTL 加随机偏移（±10%），避免同时过期引发雪崩

穿透/雪崩防护

热点数据用互斥锁防穿透，批量预热避免冷启动

双后端数据流

# 开发环境 backend = StateBackend() # 生产环境：Redis 热数据 + Postgres 冷数据 backend = CompositeBackend( default=StateBackend(), routes={ "/memories/": StoreBackend( store=postgres_store ), "/workspace/": StateBackend(), } )
策略一致性复杂度
Cache-Aside最终一致低
Write-Through强一致中

缓存是生产环境的第一道防线。没有缓存，每次请求都调 LLM，API 费用会暴涨。但缓存不是简单的「存一下」，需要考虑 TTL、失效策略、穿透和雪崩防护。

一个常见的坑是缓存雪崩：大量缓存同时过期，瞬间打爆 API。解决方案是 TTL 加随机偏移，避免同时过期。热点数据用互斥锁防穿透。

混合架构方面，Redis 存热数据，Postgres 存冷数据。开发环境用 StateBackend 就够了，生产环境用 CompositeBackend 做路由，不同路径用不同后端策略。

// part 02 · trace

Langfuse Trace 是每次请求的「X 光片」

Trace 链路示例

Trace: 用户查询 "新能源汽车分析"
Node: intent_classifier 23ms
Node: mode_router 15ms → LLM Call 312ms
Node: task_planner 45ms → LLM Call 456ms
Node: search_parallel 1203ms
└─ Subgraph: finance 800ms
└─ Subgraph: news 600ms
└─ Subgraph: competitor 750ms
Node: report_generator 2345ms → LLM Call 2300ms

Langfuse 核心 API

# 1. 从环境变量初始化 init_langfuse_observability() # 2. CallbackHandler 自动追踪图执行 handler = CallbackHandler() # 3. 为每次执行创建追踪实例 create_callback_handler() # 4. 一键获取 config 字典 get_langfuse_config() # 5. Tags 标签系统 tags=["env:prod", "service:agent"] # 6. debug=True 开启追踪 agent = create_deep_agent( model="openai:gpt-4o", debug=True, )

出了 bug？找到那条 Trace，一步一步看是哪个节点出了问题。

Langfuse Trace 就是一个「执行家谱」。用户的每次请求是一个 Trace，Trace 里每个节点、每次 LLM 调用、每次工具调用，都是一条记录。你点击任何一条，都能看到当时的输入、输出、耗时、token 用量。

看这个示例 Trace，总耗时约 4 秒。哪个节点最慢？一目了然——report_generator 花了 2345 毫秒，其中 LLM Call 占了 2300 毫秒。search_parallel 三个子图并行执行，总耗时由最慢的 finance 子图决定。

Langfuse 的核心 API 很简单：init_langfuse_observability 从环境变量初始化，CallbackHandler 自动追踪图执行，create_callback_handler 为每次执行创建追踪实例。Tags 标签系统用于按环境、服务、用户维度筛选 Trace。

// part 03 · score

Score 不只是技术监控，更是业务指标

三类核心 Score

缓存命中率 成本指标

report_cache_hit_rate() 上报。85% 命中率意味着省了 85% 的 API 费用

响应时间 体验指标

P99 < 2s 意味着用户体验达标。凌晨 3 点掉到 45% 要排查

输出质量 质量指标

评分下降意味着 Prompt 需要优化。Score 时序图追踪趋势变化

Score 的业务价值

# 上报缓存命中率 report_cache_hit_rate( trace_id=trace_id, hit_rate=0.87, # 87% ) # 自定义评分：输出质量 score_output_quality( trace_id=trace_id, score=4.5, # 1-5 分 comment="回答完整，引用准确", ) # 自定义评分：用户满意度 score_user_satisfaction( trace_id=trace_id, score="thumbs_up", )

昨天的缓存命中率平均 87%，但凌晨 3 点掉到了 45%。点击那个时间点——发现是一次批处理任务触发了大量冷查询。

Score 不只是「技术监控」，更是「业务指标」。缓存命中率 85% 意味着你省了 85% 的 API 费用；响应时间 P99 小于 2 秒意味着用户体验达标；输出质量评分下降意味着 Prompt 需要优化。

看一个真实案例：昨天的缓存命中率平均 87%，但凌晨 3 点掉到了 45%。点击那个时间点，发现是一次批处理任务触发了大量冷查询。这就是 Score 的业务价值——不只是技术数字，是「成本预警器」。

Langfuse 支持自定义评分，你可以上报任何业务指标。关键是把 Score 当作持续优化的数据基础，而不是一次性的监控数字。

// part 04 · dataset

Dataset 是自动采集的「病历本」，用于回归测试

QA Capture 自动采集

# 每次用户查询自动保存 Dataset Item: - input: 用户查询 - expected_output: 系统回答 - metadata: - user_feedback: 👍 / 👎 - trace_id: xxx - timestamp: 2025-01-15T10:30:00Z # 选中差评数据，一键重新运行评估 run_evaluation( dataset="bad-cases-jan", prompt_version="v2.1", )

选中差评数据，一键重新运行评估——看改了 Prompt 后，同样的输入输出质量有没有提升。

Dataset 管理流程

①
自动采集

每次用户查询 + 系统回答 + 用户反馈自动保存

②
分类筛选

按评分、标签、时间筛选，聚焦问题数据

③
回归测试

选中数据集，用新 Prompt 版本重新运行

④
效果对比

对比新旧版本的评分差异，量化改进效果

⑤
持续迭代

把改进后的数据加入基准集，形成闭环

Dataset 是 Langfuse 的第三个核心功能。如果说 Trace 是「X 光片」，Score 是「体检报告」，那 Dataset 就是「病历本」。

每次用户查询、系统回答、用户反馈（点赞或点踩）自动保存到 Dataset。你可以选中差评数据，一键重新运行评估——看改了 Prompt 后，同样的输入输出质量有没有提升。

Dataset 的管理流程是五步走：自动采集、分类筛选、回归测试、效果对比、持续迭代。这个闭环让 Prompt 优化从「凭感觉」变成「用数据说话」。

// part 05 · prompt management

Prompt YAML + Hub 实现「菜谱管理」式版本控制

Prompt 工作流

# 开发环境 Langfuse Hub 生产环境 # ┌─────────┐ ┌─────────────┐ ┌─────────┐ # │ YAML v1 │ ──push──→│ Prompt Hub │←──pull──│ YAML v1 │ # │(本地编辑)│ │ (版本管理) │ │ (运行时)│ # └────┬────┘ └──────┬──────┘ └─────────┘ # │ │ # ↓ ↓ # ┌─────────┐ ┌─────────────┐ # │ YAML v2 │ ──push──→│ labels: │ # │(迭代中) │ │ ["latest"] │ # └─────────┘ │ ["production"]│ # └─────────────┘ # PromptManager API get_prompt(name, label="production") build_langchain_prompt(name) save_prompt(name, content) push_to_hub(name, labels=["production"])

Prompt 管理 6 步法

①
YAML 本地存储

Prompt 以 YAML 文件管理，支持 Jinja2 和 f-string

②
元数据配置

版本、标签、适用场景写入 YAML frontmatter

③
Langfuse Prompt Hub

云端托管与版本管理，支持标签系统

④
自动拉取

本地不存在时从 Hub 自动拉取 production 标签版本

⑤
推送发布

本地推送到 Hub，自动打 labels: ["production"]

⑥
快速回滚

把上一版打上 production 标签，生产环境自动切回

Prompt 管理就像「菜谱管理」。你先在本地用 YAML 写菜谱，改了几版后觉得 OK 了，推送到云端菜谱库 Langfuse Prompt Hub，打上 production 标签。生产环境的应用自动拉取带 production 标签的最新版。

要回滚？把上一版打上 production 标签，生产环境自动切回来。要 A/B 测试？v1 打标签给 A 组，v2 给 B 组。

PromptManager API 提供四个核心方法：get_prompt 按标签获取，build_langchain_prompt 构建 LangChain 格式，save_prompt 本地保存，push_to_hub 推送到云端。上线前必须确认同步状态灯是绿色，黄色表示待推送，红色表示 Hub 有新版本待拉取。

// part 06 · deployment

Docker 容器化是生产部署的标准起点

Dockerfile 多阶段构建

# Stage 1: Builder FROM python:3.11-slim AS builder WORKDIR /app COPY requirements.txt . RUN pip install --user -r requirements.txt # Stage 2: Runtime FROM python:3.11-slim WORKDIR /app COPY --from=builder /root/.local /root/.local COPY . . ENV PATH=/root/.local/bin:$PATH ENV LANGFUSE_SECRET_KEY=${LANGFUSE_SECRET_KEY} ENV LANGFUSE_PUBLIC_KEY=${LANGFUSE_PUBLIC_KEY} HEALTHCHECK --interval=30s --timeout=5s \ CMD curl -f http://localhost:8000/health CMD ["python", "-m", "app.main"]

生产部署 checklist

①
多阶段构建

减小镜像体积，分离构建依赖和运行时

②
健康检查

HEALTHCHECK 指令，30s 间隔检测服务状态

③
环境变量隔离

dev/test/prod 用不同 Langfuse project 和 key

④
日志结构化

JSON 格式输出，便于集中收集和分析

⑤
性能调优

并发控制、递归限制、状态大小优化、连接池

⑥
安全加固

API 认证、速率限制、输入校验、敏感信息脱敏

Docker 容器化是生产部署的标准起点。多阶段构建可以大幅减小镜像体积，第一阶段安装依赖，第二阶段只复制必要的运行时文件。

健康检查很重要，HEALTHCHECK 指令让容器编排系统知道服务是否正常运行。环境变量隔离是关键——不同环境必须用不同的 Langfuse project 和 API key，否则开发环境的 Trace 会混到生产环境。

性能调优方面要注意并发控制、递归限制、状态大小优化。安全方面要做 API 认证、速率限制、输入校验。LangGraph Cloud 是另一个部署选项，适合不想自己管理基础设施的团队。

// pitfalls

生产环境最常见的6 个坑及解决方案

坑现象解决方案
Langfuse 没配环境变量追踪数据没上报secret_key 和 public_key 必须配好，不同环境用不同 project
Prompt 本地和 Hub 不一致生产环境用了旧版 Prompt上线前确认同步状态灯是绿色
缓存雪崩大量缓存同时过期，瞬间打爆 APITTL 加随机偏移（±10%），热点数据用互斥锁
RAG 检索质量差检索结果不相关，影响生成质量持续评估：命中率、相关性评分、Bad Case 收集
状态太大导致性能下降Checkpoint 保存/恢复越来越慢定期裁剪 State，大文件存对象存储
忘记区分环境开发环境的 Trace 混到生产环境Tags 里必须打 env:dev/test/prod

这 6 个坑是我从实际项目中总结出来的，每一个都踩过。

第一个坑是 Langfuse 环境变量没配好，追踪数据没上报，出了问题找不到原因。第二个坑是 Prompt 本地和 Hub 不一致，生产环境用了旧版 Prompt 导致输出质量下降。

第三个坑缓存雪崩最危险，可能瞬间打爆 API 配额。第四个坑 RAG 检索质量差，很多人只关注生成质量，其实检索不行生成肯定不行。第五个坑状态太大，checkpoint 保存越来越慢。第六个坑环境没隔离，开发环境的 Trace 混到生产环境。

// debugging

4 个调试技巧让问题定位效率提升 10 倍

① Langfuse Trace 排查

找到慢请求 → 看瀑布图找瓶颈节点 → 优化该节点。Trace 是「时间显微镜」，精确到毫秒级。

② Prompt 版本对比

本地 vs Hub 并排对比，红色高亮删除、绿色高亮新增。快速定位差异，避免「我以为推送了」的坑。

③ 缓存效果验证

对比「有缓存」和「无缓存」的响应时间和成本。用 Score 时序图验证缓存策略是否生效。

④ RAG 效果评估

定期跑评测集，看命中率和相关性趋势。检索质量是生成质量的上限，必须持续监控。

四个调试技巧，每一个都能大幅提升问题定位效率。

第一个是 Langfuse Trace 排查，找到慢请求后看瀑布图，瓶颈节点一目了然。第二个是 Prompt 版本对比，本地和 Hub 并排对比，颜色标记差异，避免「我以为推送了」的坑。

第三个是缓存效果验证，用 Score 时序图对比有缓存和无缓存的情况。第四个是 RAG 效果评估，定期跑评测集，记住这句话：检索质量是生成质量的上限。
