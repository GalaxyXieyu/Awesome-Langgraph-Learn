#### 生活化类比：「做饭」全流程

> "想象你要做一顿饭：
> - **模式路由** = 判断今天是『煮泡面』还是『做大餐』。如果就你一个人饿急了，煮泡面；如果家里来客人了，启动大餐流程。
> - **计划路由** = 做大餐时的动态调度。你计划了『买菜→切菜→炒菜→装盘』，但做到一半发现缺酱油——计划路由告诉你下一步是『暂停炒菜，去买酱油』，而不是死板地继续。
> - **任务跟踪** = 厨房里的计时器和任务板。每道菜谁在做、做了多久、做好了没，一目了然。
>
> DeepResearch Agent 就是这样做研究的——简单问题快答，复杂问题深度研究，每个步骤动态调整，全程可监控。"

### 2. 核心内容

#### 2.1 Multi-Agent 架构模式

**讲解顺序**：
1. **Hierarchical Teams**：主管分配任务，Worker 执行
2. **Network / Peer-to-Peer**：智能体平等协作
3. **Supervisor Pattern**：监督者协调多个 Worker
4. **Agent Handoff**：任务交接机制

**必须画架构对比图的点**：
```
Hierarchical: Supervisor: Network:
[Boss] [Supervisor] [A] ←→ [B]
/ | \ / | \ ↕ ↕
[W1][W2][W3] [W1] [W2] [W3] [C] ←→ [D]
（命令式） （协调式） （协商式）
```

---

#### 2.2 模式路由 Mode Router

**讲解顺序**：
1. **概念**：根据查询复杂度自动选择执行模式
2. **「煮泡面 vs 做大餐」类比详解**：
- 简单问题 → `create_react_agent` 快速问答（react 模式）
- 复杂问题 → 深度研究流程（deep 模式）
3. **路由决策逻辑**：基于关键词、意图分类、历史模式的综合判断

**必须画图的点**：
```
[用户输入] → [记忆助手加载上下文] → [模式路由]
├─→ 简单？→ [快速问答(react)] → [追问建议] → END
└─→ 复杂？→ [深度研究分支]
```

**讲解话术**：
> "模式路由就像急诊分诊台。病人来了，护士先判断：轻微感冒？去普通门诊。严重外伤？直接进手术室。DeepResearch 的分诊标准是什么？——问题的复杂度。如果用户问『北京天气』，走 react 快答；如果问『分析 2024 年新能源汽车行业格局及未来 5 年趋势』，走 deep 深度研究。"

---

#### 2.3 计划路由 Plan Router

**讲解顺序**：
1. **概念**：循环结构，根据任务完成状态决定下一步
2. **状态机设计**：`pending → running → completed/failed`
3. **动态计划调整**：根据中间结果修改后续计划
4. **「做饭」类比详解**：
- 计划：买菜→切菜→炒菜→装盘
- 执行到切菜时发现没酱油 → 计划路由插入「买酱油」→ 继续

**必须画循环状态图的点**：
```
┌──────────────────────────────────┐
↓ │
[任务规划] → [信息搜索] → [计划路由] ─┬─→ [继续搜索] ──┘
（判断） ├─→ [数据分析]
├─→ [代码执行]
├─→ [报告撰写]
└─→ [结果总结] → END
```

**讲解话术**：
> "计划路由是 DeepResearch 的心脏。它不是『一次计划，死板执行』——而是『走一步，看一步，再决定下一步』。
>
> 就像你做饭：原计划是切完菜炒菜，但切到一半发现酱油没了。计划路由不会让你『按原计划继续炒菜（没酱油的炒菜）』，而是动态调整：下一步去买酱油，买完回来继续。
>
> 在研究场景中，搜索了一轮后发现信息不够——计划路由决定『再搜索一轮』；分析后发现需要跑代码验证——计划路由决定『进入代码执行节点』。"

---

#### 2.4 节点级缓存 CachePolicy

**讲解顺序**：
1. **为什么需要节点级缓存**：不同节点有不同的缓存需求
2. **`CachePolicy` 配置**：`ttl=3600, key_func=...`
3. **差异化 TTL 策略**：搜索结果 1h、分析结果 30min、报告 24h

**讲解话术**：
> "缓存不是『一刀切』。搜索结果 1 小时内有效——因为新闻在变；分析报告 30 分钟有效——因为基于特定数据；生成的报告 24 小时有效——因为报告的『保质期』更长。
>
> 节点级缓存让你可以为每个节点单独配置策略，而不是整个图用一个策略。"

---

#### 2.5 任务跟踪系统

**讲解顺序**：
1. **`@track_execute` 装饰器**：自动追踪节点执行状态
2. **状态机**：`pending → running → completed/failed`
3. **进度推送**：与 LG-07 stream 结合，实时推送前端

**讲解话术**：
> "`@track_execute` 就像给每个节点装了一个『智能手环』。节点开始执行时，状态变成 running；完成时变成 completed；报错时变成 failed。这些信息自动汇聚成一个『任务跟踪看板』，用户可以实时看到每个步骤的进度。"

---

#### 2.6 执行次数控制

**讲解顺序**：
1. **为什么需要**：防止搜索循环无限执行
2. **`max_executions` 配置**：安全阀机制
3. **与计划路由的配合**：达到上限时优雅退出

---

#### 2.7 追问建议

**讲解顺序**：
1. 研究完成后的 `follow-up` 节点
2. 基于研究内容生成 3 条深入问题
3. 引导用户继续探索

### 3. DeepResearch Agent 架构全解析

#### 完整架构映射

```
用户输入
↓
[记忆助手] ──→ 加载用户历史偏好 + 长期记忆 (Store)
↓
[模式路由] ──→ 判断复杂度
├─→ react 模式 ──→ [create_react_agent 快答] ──→ [追问建议] ──→ END
└─→ deep 模式 ──→ [任务规划]
↓
[信息搜索] ←─────┐
↓ │
[计划路由] ──────┘ 循环
（判断下一步）
├─→ [继续搜索]
├─→ [数据分析]
├─→ [代码执行]
├─→ [报告撰写]
└─→ [结果总结]
↓
[追问建议] ──→ END
```

#### 案例与概念的一一对应

| 概念 | DeepResearch 中的体现 |
|------|----------------------|
| 模式路由 | 根据问题复杂度选择 react / deep |
| 计划路由 | 深度研究分支内的循环决策 |
| 节点级缓存 | 搜索/分析/报告不同 TTL |
| 任务跟踪 | `@track_execute` 追踪所有节点 |
| max_executions | 防止循环无限执行的安全阀 |
| 子图 | 搜索子图、分析子图、报告子图 |
| 并行 | 信息搜索阶段的多源并行检索 |
| Stream | 任务进度实时推送到前端 |
| Store | 记忆助手加载用户长期偏好 |

#### 代码演示的讲解话术

**Step 1 - 模式路由决策**：
> "看用户输入：『分析新能源汽车行业』。模式路由节点运行——它分析这个查询：涉及多个维度、需要数据支撑、不是简单事实查询 → 判断为『复杂』→ 走 deep 分支。如果是『北京今天天气』，就会走 react 快答分支。"

**Step 2 - 深度研究循环**：
> "进入 deep 分支后，首先任务规划节点生成研究计划。然后信息搜索节点并行检索多个数据源。搜索完成后，计划路由节点评估：『信息足够了吗？』→ 不够 → 返回继续搜索。这就是循环。"

**Step 3 - 计划路由动态调整**：
> "第三轮搜索后，计划路由发现已经有了足够的市场数据，但缺少竞品对比。于是它决定下一步不是『继续搜索』，而是『数据分析』。注意——计划是动态调整的，不是一开始定死的。"

**Step 4 - 缓存效果演示**：
> "第一次搜索某数据源花了 3 秒，API 费用 $0.02。5 分钟后另一个用户问类似问题，再次走到搜索节点——缓存命中！耗时 0.01 秒，费用 $0。看面板上的缓存统计：节省 97% 的 API 调用。"

**Step 5 - 任务跟踪看板**：
> "所有节点都有 `@track_execute` 装饰。看这个时间线——每个节点的状态变化一目了然。这个节点 pending 了 0.1s，running 了 2.3s，completed。整体进度 75%。"

#### 关键转折点（需要停下来确认学员理解）

1. **模式路由 vs 计划路由**：问 "模式路由和计划路由的区别是什么？各在哪个层级工作？"（期望：模式路由在顶层选分支，计划路由在 deep 分支内做循环决策）
2. **循环终止条件**：问 "计划路由循环怎么保证不无限执行？"（期望：max_executions + 状态判断 + 内容饱和度评估）
3. **缓存 key 设计**：问 "不同用户的相同查询，缓存应该共享吗？"（期望：看业务需求，通用数据可共享，个性化数据要隔离）

### 4. 常见误区与注意事项

| 坑 | 现象 | 预警话术 |
|----|------|---------|
| **模式路由判断不准** | 简单问题走了 deep，浪费资源；复杂问题走了 react，回答质量差 | "模式路由的决策逻辑要持续优化，可以用历史数据训练分类器，不要只靠关键词匹配。" |
| **计划路由循环不收敛** | 永远觉得『信息不够』，无限循环 | "必须设 max_executions + 信息饱和度评估函数，强制循环收敛。" |
| **缓存 key 过于宽泛** | 不同查询命中同一缓存，返回错误结果 | "缓存 key 要包含查询语义、用户标识（如果需要隔离）、时间窗口。" |
| **任务跟踪性能开销** | `@track_execute` 本身带来延迟 | "跟踪数据异步写入，不要阻塞主流程；生产环境可以采样跟踪。" |
| **子图状态共享不当** | 各子图互相影响状态 | "子图优先用状态隔离，通过显式 input/output 映射传递数据。" |

#### 调试技巧

1. **模式路由日志**：记录每个查询的路由决策和原因，定期分析误判
2. **计划路由轨迹**：保存每次循环的计划状态和决策，可视化循环路径
3. **缓存命中率监控**：低于 30% 要检查 key 设计；高于 90% 要检查是否过度缓存

### 5. 阶段小结

#### 核心要点回顾

1. **模式路由 = 顶层分诊**：根据复杂度选择执行策略（快答 vs 深度）
2. **计划路由 = 动态调度**：循环执行，根据中间结果调整下一步
3. **节点级缓存 = 成本优化**：不同节点不同 TTL，精准控制
4. **任务跟踪 = 可观测性**：每个节点的状态实时可见
5. **max_executions = 安全阀**：防止无限循环的底线
6. **DeepResearch = 综合应用**：前面 6 个阶段知识的终极组合

#### 记忆口诀

> **"模式路由选赛道，计划路由跑接力；缓存省钱跟踪看进度，max_executions 保安全。"**
