# html-ppt-skill 架构问题分析与重构方案

## 一、当前架构的核心问题

### 1.1 问题：数据分析被当作「独立输出格式」，而非「底层能力」

当前的决策逻辑：

```
用户提需求
├── 分析/报告/可视化 → 交互式HTML报告（独立路线）
├── 演讲/技术分享 → 静态HTML PPT（独立路线）
└── 需要.pptx → HTML→PPTX（独立路线）
```

**问题**：三条路线平行并列，数据分析路线和PPT路线没有打通。用户如果有数据+演讲需求，会被迫二选一，而不是「先分析数据，再把洞察放进PPT」。

### 1.2 问题：数据洞察没有「自动注入」到内容中

当前的数据分析结果是独立交付物（HTML报告或终端输出），没有机制把：
- 核心结论
- KPI数字
- 趋势图表
- 异常预警

自动转化为PPT的slide内容。用户需要手动复制粘贴。

### 1.3 问题：需求解析层缺失

没有统一的「需求解析」步骤来判断：
- 用户给了数据吗？（Excel/CSV/表格）
- 用户的真实意图是「看数字」还是「用数字讲故事」？
- 交付场景是什么？（演讲/邮件/自读/打印）

导致AI可能：
- 把演讲需求做成了静态数据报告
- 把深度分析需求做成了没有数据支撑的空洞PPT

## 二、重构后的目标架构

### 2.1 统一工作流：三层架构

```
┌─────────────────────────────────────────────────────────────┐
│  第一层：需求解析（Demand Parser）                            │
│  —— 理解用户真正需要什么                                       │
├─────────────────────────────────────────────────────────────┤
│  输入：用户query + 附件（Excel/CSV/文字/链接）                  │
│                                                             │
│  输出决策：                                                   │
│  ├─ 是否有数据输入？     → 触发/跳过 数据层                    │
│  ├─ 核心意图是什么？     → 演讲叙事 / 分析报告 / 混合           │
│  ├─ 受众是谁？           → 工程师/高管/客户/投资人/学生        │
│  └─ 交付场景？           → 投影/屏幕共享/自读/手机/打印        │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  第二层：数据层（Data Engine）【条件触发】                     │
│  —— 有数据则分析，无数据则跳过                                  │
├─────────────────────────────────────────────────────────────┤
│  输入：Excel / CSV / 表格数据 / 用户描述的数字                  │
│                                                             │
│  处理：                                                      │
│  ├─ 读取 → 清洗 → 结构化                                      │
│  ├─ 计算（ROI/同比环比/占比/趋势）                             │
│  ├─ 异常检测（波动>阈值/负向指标/遗漏数据）                    │
│  └─ 洞察生成（结论卡片：标题+数字+趋势+建议）                  │
│                                                             │
│  输出：洞察卡片数组（可直接映射到slide）                       │
│  [                                                            │
│    { title: "Q3转化率提升23%", number: "23%", trend: "up",    │
│      detail: "归因：落地页优化+投放定向收紧",                   │
│      recommendation: "Q4继续收紧定向，预算向高转化渠道倾斜" }, │
│    ...                                                        │
│  ]                                                            │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  第三层：内容编排层（Content Orchestrator）                   │
│  —— 把素材编排成有叙事弧线的大纲                                │
├─────────────────────────────────────────────────────────────┤
│  输入：用户原始素材 + 数据层洞察卡片（如有）                    │
│                                                             │
│  编排逻辑：                                                   │
│  ├─ Hook（封面）：核心结论的一句话版本                         │
│  ├─ Context（背景）：问题/机会/数据概况                        │
│  ├─ Build（论证）：分论点 + 数据支撑                           │
│  ├─ Proof（证据）：KPI/图表/对比                               │
│  ├─ Plan（行动）：建议 + 时间线                                │
│  └─ Close（收尾）：总结/CTA                                    │
│                                                             │
│  输出：标准化大纲（每页 = 断言标题 + 叙事角色 + 布局 + 备注）   │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  第四层：输出层（Output Adapter）                             │
│  —— 根据交付场景选择「容器格式」                                │
├─────────────────────────────────────────────────────────────┤
│  场景 → 容器：                                                │
│  ├─ 演讲/技术分享/有逐字稿需求   → 静态HTML PPT                │
│  ├─ 分析/报告/需要交互图表       → 交互式HTML报告              │
│  ├─ 客户需要.pptx文件           → HTML→PPTX                   │
│  └─ 快速看数字/探索性分析        → Terminal + Markdown        │
│                                                             │
│  关键：同一个大纲可以渲染成多种容器格式！                        │
│  内容（故事弧线+数据洞察）是稳定的，容器是灵活的。               │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 关键变化

| 维度 | 当前架构 | 目标架构 |
|------|---------|---------|
| 数据分析 | 独立输出格式 | 条件触发的底层能力 |
| 决策时机 | 先选格式 | 先解析需求，再选容器 |
| 数据→PPT | 手动搬运 | 洞察卡片自动注入slide |
| 内容复用 | 低（每条路线独立） | 高（同一大纲→多容器） |
| 用户负担 | 需要明确说「我要数据分析+PPT」 | 只需给数据和需求，自动判断 |

## 三、具体改进点（落地到 SKILL.md）

### 3.1 重写「输出格式决策」为「需求解析协议」

**当前（要改的）**：
```
| 用户意图 | 输出格式 | 何时用 |
| 分析/报告/可视化 | 交互式HTML报告 | ... |
| 演讲/技术分享 | 静态HTML PPT | ... |
```

**目标**：
```
收到用户请求后，先执行需求解析：

1. 【数据检测】用户是否提供了Excel/CSV/表格/数字？
   → 是 → 自动触发数据层（read_excel.py → 分析 → 洞察卡片）
   → 否 → 跳过数据层，进入纯内容编排

2. 【意图识别】用户的真实目的是什么？
   → "给团队讲" / "技术分享" / "演讲" → 演讲叙事权重高 → 静态HTML PPT
   → "分析" / "报告" / "复盘" / "看看数据" → 分析权重高 → 交互式HTML报告
   → "发给客户" / "需要编辑" → 可编辑权重高 → HTML→PPTX
   → 混合需求 → 先给交互HTML报告（详细数据）+ 再给静态PPT（演讲精简版）

3. 【受众识别】谁在消费这个内容？
   → 工程师 → 技术主题 + 代码/架构布局
   → 高管 → 结论先行 + KPI网格 + 极简
   → 投资人 → 数据密度高 + 增长曲线 + 对比
   → 客户 → 温暖叙事 + 前后对比 + 利益点

4. 【交付识别】在哪里看？
   → 投影仪 → 高对比度暗色主题 + 大字号
   → 屏幕共享 → 亮色主题 + 清晰
   → 自读 → 信息密度可以高
   → 手机 → 竖版/少列/大触摸目标
```

### 3.2 新增「洞察卡片」中间格式

在数据层和内容层之间，定义标准数据结构：

```markdown
## 洞察卡片格式（数据层 → 编排层）

每张洞察卡片必须包含：
- `headline`: 断言式标题（不超过10个字）
- `number`: 核心数字（带单位，如 "23%", "¥1.2M"）
- `trend`: up/down/neutral（决定颜色：绿/红/灰）
- `context`: 一句话解释（30字内）
- `evidence`: 支撑数据（同比/环比/对标）
- `action`: 可执行建议（如果有）
- `chart_type`: 推荐图表类型（bar/line/pie/kpi/table）

示例：
```yaml
- headline: "Q3转化率提升23%"
  number: "23%"
  trend: up
  context: "落地页优化+投放定向收紧"
  evidence: "Q2: 3.1% → Q3: 3.8%（行业均值2.5%）"
  action: "Q4继续收紧定向，预算向高转化渠道倾斜"
  chart_type: line
```

编排层直接将洞察卡片映射到布局：
- 1张卡片 + 大数字 → `stat-highlight.html`
- 4张卡片并排 → `kpi-grid.html`
- 带趋势图 → `chart-line.html`
- 带详细表格 → `table.html`
```

### 3.3 新增「混合需求」处理策略

很多真实场景是混合的：
- "帮我分析这个月投放数据，然后做个给老板汇报的PPT"
- "做个技术分享，要包含性能测试数据"

**当前问题**：AI只能选一个格式。

**目标策略**：
```
混合需求 → 双轨交付
├── 轨道A（详细版）：交互式HTML报告
│   └── 包含：完整数据表、所有图表、详细分析
│   └── 用途：会前发给老板预习、留档、后续查阅
│
└── 轨道B（演讲版）：静态HTML PPT（5-10页精简版）
    └── 包含：核心结论、3个关键数字、建议行动
    └── 用途：现场演讲，配合口述展开

或者用户明确要一个格式时，根据「信息密度」自动调整：
- 演讲场景 → 从洞察卡片中精选3-5张，配大字体+极简视觉
- 报告场景 → 全部洞察卡片展开，配详细图表+表格
```

### 3.4 模板推荐应该是「智能路由」，不是让用户选

当前：问用户3个问题，推荐2-3个主题。

目标：根据需求解析结果，直接推荐1个最佳主题+1个备选，并说明为什么。

```
示例：
"基于你的需求（技术分享+投影仪+工程师受众），
我推荐【tokyo-night】——深蓝夜空+霓虹高光，代码和架构图对比度充足，
是技术演讲的默认安全牌。

备选【catppuccin-mocha】——如果你希望氛围更柔和、长时间观看不累眼。"
```

## 五、一句话总结

**当前**：数据分析 和 PPT制作 是两台独立的机器，用户需要手动把数据从A搬到B。

**目标**：建立一个「中央厨房」——用户给需求和数据，系统自动完成「分析→洞察→编排→渲染」，数据分析是内置的底层能力，不是可选配件。
